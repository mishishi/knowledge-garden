# 05. 内容矩阵设计: 一鱼多吃, 1 篇扩 10 篇

去年 11 月我写过一篇 6000 字的英文长文,讲怎么用 Claude Code 给一个真实的 Next.js 项目做重构,跑了 28 个 PR 的 diff 分析。写完第二天我瘫在椅子上,觉得"行了,这周产出了"。三天后我把同一份内容拆成了 8 条推、1 个 9 分钟的 YouTube 脚本、3 篇公众号、2 条小红书图文、1 份邮件 newsletter,总共触达比原文多 40 倍,但我额外花的时间只有 6 个小时。

这才是内容矩阵真正该做的事:不是"多平台铺开",是"一份母题,按平台和用户路径改写"。

但我必须先承认一个反直觉的事:矩阵做过头,反而比不做更糟。我去年 8 月试过那种"一篇长文 → 自动改写 20 条短帖"的工具流,产出了一堆我自己都看不下去的 AI 味短文,粉丝没涨,取关反而多了。后来我花了两个月想明白一件事:**复用是省力气,不是省脑子**。改写也是创作,不是机器干的活。

这一章我想把我后来摸出来的边界讲清楚,什么该复用、什么必须重写、怎么用工具提效但不掉质量。

## 我的矩阵长什么样,先给你看个真东西

我现在稳定在跑的"母题-子产物"结构大概是这样:

一篇 4000-8000 字的长文(发在自己 newsletter 或者博客),写完后 48 小时内会变成:

- 6-10 条 X(原 Twitter)短帖,每条 1-4 句,带 hook
- 1 个 6-12 分钟的 YouTube 长视频(口播 + 屏幕录制)
- 2-3 条短视频(竖屏,90-180 秒,剪 YouTube 的片段)
- 1 篇公众号(略改排版,加更长的引子)
- 1 份 newsletter 摘要版(给已经订阅的人,精简到 1500 字)
- 1 条即刻动态 + 1 条小红书图文(国内双开)
- 1 份免费资源(代码 repo / 模板 / prompt 集)挂在 newsletter 落地页

一条母题,大概能稳出 10-13 个子产物,跨 5-6 个平台。但这是**我现在的状态**,不是第一周就这样。我从 2024 年 3 月开始做副业,前 3 个月根本没矩阵,就是死磕英文推,踩了一堆坑才慢慢扩到现在的结构。

讲几个我犯过的具体错误,你别再走一遍。

第一个错误是"长文压缩成短帖就直接发"。我 2024 年 4 月干过这事,把一篇 5000 字的 GitHub Copilot 评测压缩成 10 条推,每条 280 字,带链接回原文。结果:阅读量是平时的 1/3,完读率只有原文的 1/4。我后来才搞明白,短帖不是"浓缩",是"换骨架"。一篇 5000 字的长文可能核心论证只有 3 个点,但每个点都需要 1500 字铺垫,这 1500 字铺垫在推上全是噪音。短帖要的是**单点穿透**,不是全景压缩。

第二个错误是"视频就是念稿"。我 5 月拍过第一条 YouTube,对着摄像头念我自己写的长文,11 分钟,播放量 800。我当时觉得是"我没露脸所以没人看",后来跟一些做 YouTube 的朋友聊才发现不是露脸的问题,是**视觉节奏**——11 分钟我人就坐在那里,屏幕切了 3 次,完播率 22%,算法直接不给推。

第三个错误最致命,我以为"多平台 = 多账号同时铺",所以一开始就开了 X、即刻、小红书、知乎、公众号 5 个号,每个号都同步发。结果我很快发现,这不是"矩阵",这是"复制粘贴",每个平台的用户期待完全不同——X 上我的英文技术读者喜欢"短平快 + 截图",小红书用户喜欢"教程感 + 颜值封面",知乎要"长论证 + 数据来源",公众号要"情绪 + 故事线"。同一份内容硬塞五个平台,两边都不讨好。

这三个错误加起来,让我前 3 个月的有效内容产出其实非常低。我看着自己每天花 3-4 小时在内容上,粉丝数从 0 涨到 800,但转化是 0。

转折点是我开始把"矩阵"这个词彻底忘掉,改成想"内容路径"。

## 母题怎么选,为什么不是所有长文都值得拆

工程师很容易掉进一个坑:觉得自己写的每一篇技术文都很有价值,都应该扩成矩阵。错。

我现在的判断标准很简单,**一个母题值不值得拆,看它有没有"可被拆解的论证单元"**。

拿我那个 Claude Code 重构的真实案例举例,原文 6000 字,核心论证其实是这 4 个:

1. 重构前先建 eval set,不要边改边测(具体: 我建了 28 个 PR 的 test set)
2. 让 AI 先读 git history,比直接给指令高效(具体: 它从 3 个 commit 里找到了原作者的设计意图)
3. prompt 里要带"不要改 X"的负面约束(具体: 它擅自重构了我故意保留的兼容层)
4. 长上下文比多轮对话更省 token(具体: 一次塞 12 个文件 vs 拆 4 轮)

这 4 个点,每个都独立成立,每个都带具体数字,每个都能被引用、被截图、被做视频。这就是**好的母题**——它的论证单元可以脱开原文生存。

反过来,我写过一篇讲"我为什么从 VSCode 转到 Cursor"的日常吐槽文,3000 字,我自己读着挺爽,但拆不动。因为它没有"可被拆解的论证单元",全是个人感受和场景描写,拆成短帖就是"我转了 + 截图",没有任何信息增量。这种文章我就让它安安静静发在 newsletter,不去硬拆。

具体怎么判断?我现在的经验是写完初稿后,**回头看一遍,数一下"如果只能保留 3 段,是哪 3 段"**。如果这 3 段每一段都能脱离上下文被独立理解,那母题合格。如果这 3 段互相引用、互相铺垫,那就是叙事型长文,不适合拆,适合整篇发。

还有一个我后来加的判断标准:这篇母题能不能产生"可被复用的资源"。比如代码 repo、prompt 模板、配置脚本、cheat sheet。这些东西是矩阵的**杠杆点**——一篇长文可能没人转,但一个 GitHub 模板可能被人 star 几百次,然后顺着 repo 找到你。我那个 Claude Code 重构的案例,挂了一个 800 行的 prompt 模板,repo 现在 1400 star,比我那个 newsletter 的订阅数还多。

## 复用 vs 重写,边界到底在哪

这块是我花最多时间才想明白的。说几个我现在的硬规则,你可以直接拿去用。

**先讲"哪些东西可以直接复用"。**

第一,核心数据和事实可以原样搬。比如"我跑了 28 个 PR""token 消耗从 1.2M 降到 680K""完播率 22%"这些数字,在 X 上是这些数字,在公众号也是这些数字,在 YouTube 视频里我念出来还是这些数字。这种硬数据是矩阵的骨架,改它反而会引入不一致风险。

第二,代码块和配置可以原样搬。Markdown 的 ```code fence``` 直接复制到 newsletter,截图到小红书,贴到 YouTube 视频描述里,完全 OK。代码本身是中性的,不挑平台。

第三,核心结论的"金句"可以原样搬。我有一句话反复用在不同平台:"AI 写代码不是替代你思考,是逼你把思考写下来。"这句话在 X 上我当 hook 用,在 YouTube 视频里我当开篇第一句,在公众号里我当小标题,在小红书里我当封面文案。这种"母句"是矩阵里的稳定锚点,读者在不同平台看到会认出来"啊是你"。

**接下来讲"哪些东西必须重写"。**

第一个必须重写的是**开头的 hook**。每个平台的用户滑到内容的第一秒,期待完全不同。X 上是"反常识 + 数字"(我直接写"我用 Claude Code 重构了 28 个 PR,有一个发现让我重新思考 prompt 设计")。小红书上必须"教程感 + 收益前置"(我会写"程序员必看|用 AI 重构老项目,我整理了这 4 个关键 prompt,代码开源")。公众号要"故事感 + 共情"(开头我会写"上个月接了一个烂尾的 Next.js 项目,前任留下的代码我自己看了一周,直到我让 AI 先去翻 git log")。同一个论证,开篇三种写法,这是矩阵的核心成本,省不掉。

第二个必须重写的是**论证顺序**。X 是瀑布流,你必须在第一条就把最有冲击力的结论扔出来,然后用后续推文补充。公众号是长卷,你要先铺场景,再给方法,最后给数据。YouTube 是线性,前 30 秒留人,中间给方法,结尾给行动号召。三种叙事结构,不能复用。

第三个必须重写的是**视觉资产**。我吃过大亏——同一张截图,X 上能看,小红书上糊成一片。同一段代码,X 上用 ```code fence```,小红书必须截图,而且字体要放大到 14 号以上。YouTube 视频里我得录屏讲解,不能贴静态图。这个工作流是矩阵里**最容易被低估的工时**,我后面会专门讲怎么用工具压。

讲一个具体的改写例子,同一段内容的三种开头。

X 版本(140 字以内,英文):
"Spent 2 weeks refactoring a Next.js codebase with Claude Code. 28 PRs. 1 finding that completely changed how I write prompts: let the AI read git history FIRST, before giving any instructions. The diff it produced was 3x cleaner than my 'expert' prompts. Why? Thread 🧵"

小红书版本(中文,带 emoji 教程感):
"📌 程序员的 AI 重构笔记 | 这 4 个 prompt 让我 2 周清完 28 个 PR
·
最近接了一个烂尾的 Next.js 项目,前任留下的代码我看了 1 周才理清。用了 Claude Code 之后,2 周清完 28 个 PR。
·
今天把 4 个最关键的 prompt 整理出来,代码也开源了👇
[配 3 张截图 + 1 个 repo 链接]
·
#AI编程 #ClaudeCode #程序员副业"

YouTube 视频开头(口播,前 30 秒):
"我上个月接了一个烂尾项目,前任的 Next.js 代码我自己看了一周。然后我让 Claude Code 接手,它第一件事不是改代码,是去翻 git log。我当时觉得这 AI 是不是有问题,结果它从 3 个 commit 里找到了原作者当初的设计意图,这个项目我才知道'为什么要这么写'。今天我想跟你讲 4 个我用 AI 重构老项目时发现的关键 prompt,这 4 个 prompt 让我的 diff 质量比我自己手写的还干净。"

三个开头,核心信息一样,但骨架完全不同。这就是矩阵的本质:**复用信息,重写骨架**。

## 工具链:怎么把改写成本压到 20%

我必须老实讲,我现在矩阵的改写工作流里,**AI 工具的参与度大概只有 20%**,剩下 80% 还是我自己干。这 20% 主要用在这几个地方。

第一个用途是**生成多版本 hook**。我写完一个母题后,会让 Claude 给我生成 5-8 个不同风格的开头(反常识型 / 数字型 / 故事型 / 争议型 / 教程型),然后我挑 2-3 个改一下直接用。这个工作 AI 做得很稳,因为 hook 本身套路性强,改写风险低。我用 Claude Code 的时候会给它一个简单的 prompt:

```
这是我的长文核心论证:[粘贴 4 个核心点]
请基于这个母题生成 8 个不同风格的开头 hook,覆盖以下场景:
- X 短帖(英文,140 字以内,反常识型)
- 小红书图文开头(中文,带 emoji,教程型)
- 公众号引子(中文,故事型,300 字)
- YouTube 视频前 30 秒口播脚本(中文,口语化)
每个 hook 单独成段,用 --- 分隔。
```

这个 prompt 我存成了 snippet,每次写完长文跑一次,5 分钟出结果,我自己改 15 分钟,比从零写 1 小时省 3 倍时间。但我必须强调:**AI 给的 hook 我从来不直接用,每次都要改 30%-50%**。直接用 AI 输出的 hook,读者一眼能看出来,那种"AI 味的完美"反而是最不人味的。

第二个用途是**改写时的语言转换**。我的母题主要用英文写(X 和 YouTube 的主战场),但公众号、小红书、即刻需要中文。Claude 在中英互译 + 调整平台语调这件事上非常强。我会给它一个示例:

```
把下面这段英文技术内容改写成适合小红书的图文开头,要求:
- 中文人称,口语化,带 2-3 个 emoji
- 前 50 字必须出现"收益"或"数字"
- 段落短,每段不超过 30 字
- 不要"赋能/沉淀/抓手"这种大词
- 不要"宝子们/家人们"这种小红书烂梗
原文:[粘贴]
```

第三个用途是**自动生成社交媒体卡片的文案**。我用 V0(不是 Vercel v0,是另一个工具)+ Canva 的组合,但这个流程我自己还在调,下次再讲。短视频剪辑我用 Descript,自动去 silence + 自动加字幕,这个工具我自己用了 1 年多,矩阵里所有 YouTube 视频都用它剪,效率提升大概 3 倍。Descript 不算 AI 工具,但它的 transcript-based 编辑范式对工程师非常友好,你可以像编辑 markdown 一样剪视频。

讲一个反例,我必须警告你:**别用 AI 工具"全自动改写"长文成短帖**。我 2024 年 8 月试过一个叫某 AI Repurpose 的工具,号称"一篇博客自动生成 30 条推",我跑了一篇 5000 字的英文文,出来 28 条推,每条都带 emoji 标签,看起来很专业。我挑了 10 条发,结果:

- 第 1 天阅读量是平时手工发推的 1/5
- 第 3 天开始收到 3 条取关消息,说"你最近发的内容感觉很 AI"
- 第 7 天我手工写了 2 条推,阅读量立刻回到正常水平

那个月我的粉丝曲线是平的,甚至微跌。这是我副业早期最贵的一课:**自动化产出会污染你的内容指纹**。读者关注你是因为你的"调性",而调性来自你的判断、你的取舍、你的"哪些话我要说、哪些话我不要"。AI 工具替你做了这些决定,调性就崩了。

我现在用 AI 工具的边界很清晰:**它只做"扩写"和"换语种",不做"定调"**。一篇文章"该怎么说、该强调哪个点、用什么故事开头",必须我自己定。AI 只在我定好调之后,帮我把语言风格快速切换到不同平台。

## 我踩过的最贵的一个坑:矩阵做了,但漏了"收口"

我必须单独讲这个坑,因为它最隐蔽,也最致命。

2024 年下半年,我的矩阵跑得很顺,一个母题稳定出 10+ 个子产物,触达数据也好看。但我发现一个诡异的事:**触达涨了,订阅没涨;阅读涨了,付费没涨**。我当时算了一笔账,从 8 月到 11 月,触达人数涨了 4 倍,但付费转化人数只涨了 30%。

我花了两周拆解这个问题,后来跟一个做 SaaS 的朋友吃饭,他一句话点醒我:"你的矩阵全是'曝光型',没有'收口型'。"

他说得对。我当时的所有子产物都指向"更多曝光"——X 推文引流到长文,长文引流到 newsletter,newsletter 引流到下一个母题。但**所有的引流都是"看更多",不是"做下一步"**。读者看完就走了,没有理由留下来。

后来我做的调整是,每个矩阵必须有一个**明确的"收口动作"**。比如:

- X 上的爆款短帖 → 末尾带"完整 prompt 模板在置顶" → 引导到 GitHub repo
- YouTube 视频 → 中段 5 分钟处明确说"完整流程我写成了 newsletter,链接在简介" → 引导到 newsletter
- 小红书图文 → 文末"完整版代码在评论区置顶" → 引导到小红书私信 / 微信
- newsletter → 末尾的 PS 永远带"如果你想每周收到这种内容,订阅在这里" → 引导到付费订阅

我加了这层"收口"之后,转化率从 0.8% 涨到 3.2%。不是文案写得多好,是我终于给了读者一个"做完这次阅读之后的下一个动作"。

工程师特别容易犯这个错,因为我们本能地觉得"好东西自然会传播",但**内容产品的传播不是自然现象,是你设计出来的**。收口是设计的一部分,不是营销的附加。

## 给你今晚能动手的两件事

如果你现在有 1 篇已经写好的长文,或者正准备写一篇,今晚就能做这两件事。

第一件事:**回到那篇长文,数一下"可被拆解的论证单元"**。如果只有 1-2 个,那就别硬拆,让它整篇发,把时间花在下一篇上。如果有 3-5 个,那你已经有一个合格的母题,可以往下走。判断标准就是"这 3-5 个点能不能脱开原文被独立理解",能就拆,不能就攒着。

第二件事:**打开你的手机,选 3 个你常用的内容平台,写下每个平台的"用户滑到内容的前 3 秒期待什么"**。X / 推是"反常识 + 数字",小红书是"教程感 + 收益",即刻是"圈层认同 + 个人观察",公众号是"故事 + 共情",知乎是"论证 + 数据来源",B 站是"开箱 + 演示"。把这个写下来,贴在你写内容的工作区旁边。下次你写完一个母题要改写时,看一眼这 3 行,改写就快了。

矩阵这件事,工程师最容易陷入"系统化强迫症"——想做一套完美的内容生产管线,每个平台自动同步,每个母题自动扩 10 篇。我走过这条路,浪费了 3 个月。真正能跑起来的矩阵,核心永远是"你自己的判断",工具和自动化只是把你的判断放大,不是替代。
