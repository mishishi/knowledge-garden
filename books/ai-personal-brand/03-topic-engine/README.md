# 03. AI 选题引擎：用 LLM 找选题，验证真需求

我做内容的第一年，选题全靠"我觉得"。觉得 Cursor 写代码很爽，就写一篇 Cursor 教程；觉得 MCP 协议很新，就写一篇 MCP 入门。结果发出去，阅读量稳定在 200-400，跟写日记差不多。

转折点是我开始用 LLM 跑选题批量验证。三个月后，同样的发文频率（每周 1-2 篇），平均阅读从 350 涨到 5800，有 4 篇过了 1.5 万。最离谱的一篇是 47 万阅读，直接给我账号加了 6000 粉。

不是因为我文笔变好了，是选题变了。

这一章讲我怎么用 AI 把"我觉得"变成"数据告诉我"。这套流程跑下来，每周的选题时间从 4 小时压到 40 分钟，命中率（点赞 / 阅读 > 3%）从 12% 涨到 38%。

## 我踩过的选题坑，先讲清楚

刚开始做副业那会儿，我对"选题"的理解特别朴素：今天写了点代码，学了个新东西，那就分享出来。工程师嘛，输出就是这样的，有什么好纠结的。

结果发了几十篇，账号数据像心电图一样平。我那会儿在 V2EX 上看到一个哥们儿做 AI 工具测评，三个月做到 2 万粉，写的内容跟我差不多深度，差距就是选题——他专挑"Claude vs GPT-4 谁更好用"这种有争议的题，我写"今天我用 Cursor 写了个爬虫"。

差距不是写作能力，是**他知道我想要什么**。

我后来反思，那段时间我犯了三个错，现在看来特别典型：

**第一个错是只写自己感兴趣的。** 这是工程师最常犯的。我对编译器、对 V8 内部实现、对 TypeScript 类型体操很感兴趣，但这些内容对副业目标读者（想用 AI 提效的工程师）来说，付费意愿极低。你写一篇深入剖析 V8 优化原理的文章，点赞可能 50，转发可能 5 个，没有一个能转化。

**第二个错是凭直觉判断需求真假。** 我会看到 X 上有人说"想要 XXX"，就觉得这是真需求。但单条反馈什么都不证明。一个人说想要，可能有一万人在用替代方案或者根本不 care。

**第三个错是不做内容差异化分析。** 看到别人写什么火就跟风写，结果发现同主题的文章早就有人写过，而且写得比我好。我有一篇写"AI 写代码工具对比"，发出去阅读 300，因为同类内容有 200 篇。

这些坑的本质都是同一件事：**我用"产品直觉"代替了"市场验证"**。做工程产品我知道要做用户调研、做 A/B 测试，但做内容的时候我就忘了，凭感觉干。

现在我跑选题的第一步，永远是先问：这个需求有没有被验证过？有没有数据支持？市场上同类内容竞争烈度怎么样？

这套思路跟做 SaaS 选赛道是一样的逻辑：TAM、竞争、差异化、可触达。内容选题也是产品，工程师应该用工程师的思维方式去做。

## 四象限筛选题：兴趣 / 能力 / 付费 / 流量

2025 年中我开始认真做内容矩阵的时候，画了一张表。横轴是"我能写"，纵轴是"有人想看"，切成四个象限。但光这两个维度不够，我又加了"付费意愿"和"搜索流量"，变成四维评估。

四象限是这样的：

**左下角（兴趣 + 能力，但没人看没人付钱）：** 这是我以前的舒适区。TypeScript 高阶类型、SQL 优化技巧、VSCode 配置。写起来很爽，但阅读量稳定 500 以下，也没人愿意为此付费。我有 60% 的早期内容都在这个象限，全是垃圾时间。

**右下角（能力 + 流量，但我不感兴趣）：** 这种题有搜索量，比如"如何安装 Python"、"VSCode 快捷键大全"。但工程师做这类内容就是自降身价，而且跟大厂文档、官方教程竞争没任何优势。我偶尔会写一两篇当流量入口，导流到主账号，但不主攻。

**左上角（兴趣 + 付费，但没流量）：** 这种题有付费意愿但搜索量小。比如"独立开发者如何定价"、"B2B SaaS 冷启动"。目标用户少但精准，适合做邮件列表、做付费社群。LTV 高，但起量慢。我现在大概 20% 的内容在这个象限，主要放在 newsletter 上。

**右上角（兴趣 + 能力 + 流量 + 付费都有）：** 这是金矿，但也是最稀缺的。我 2025 年下半年找到 3 个这样的主题：AI 编程工具实战对比（Cursor / Claude Code / Codex）、副业产品从 0 到 1 的工程实践、LLM 评测方法论。每个主题我都能写 30-50 篇不重样。

**这四象限怎么用 AI 跑？** 我现在的做法是这样的：

第一步是 LLM 生成候选。我用 Claude Code 写了一个 prompt，每次开会话让它生成 20 个候选主题，约束条件是：必须在我兴趣范围内、必须有真实付费意愿、必须有搜索流量支撑、必须跟前面 3 篇内容不重复。这个 prompt 我迭代了 11 版，下面会贴一版能用的。

第二步是数据验证。每个候选我会丢给 Perplexity 和 5118 工具查数据。Perplexity 查"这个主题最近 6 个月被讨论的热度"、"最常被问的 5 个相关问题"。5118（中文关键词工具）查"百度指数 / 微信指数 / 小红书话题量"。两个数据源结合起来判断是真需求还是伪需求。

第三步是竞争分析。我会让 LLM 列出"这个主题下 top 20 文章"，评估平均质量、覆盖角度、还缺什么角度。竞争越激烈，我越要找差异化切口。

我每周跑 1 次，一次产生 30-50 个候选，经过筛选最终留下 3-5 个作为本周内容。命中率（点赞 / 阅读 > 3%）从 12% 涨到 38%，就是这个流程跑出来的。

讲一个真实案例。2025 年 9 月，我想写"Cursor 怎么用更爽"这类选题。LLM 一开始给的候选是"Cursor 入门教程"、"Cursor 快捷键大全"、"Cursor vs VSCode"。我把这些丢到 Perplexity 一查，发现 Cursor 入门类内容在小红书、知乎已经有 500+ 篇，竞争饱和。

我换了个角度，prompt 里加了约束："这个主题必须跟'副业变现'或'效率提升的量化数据'挂钩"。LLM 重新生成的候选里有一个击中了我："我用 Cursor 做了一个 SaaS，从代码到上线只花 7 天"。

这个题目我从来没见人写过。竞争烈度低（5118 显示长尾词，每月搜索 800），但付费意愿强（副业主题普遍转化好）。我写出来发到 X，第一周 1.2 万阅读，公众号 3800 阅读，导流到我的产品页面转化 4 个付费用户，¥497 一个，¥1988 当天到账。

这就是 AI 跑选题的威力：**它能 5 分钟内给你 50 个候选，你只需要挑出有数据支撑的那个**。传统方法你要泡在 V2EX、刷小红书刷一下午，AI 直接压缩到一次会议时间。

## Perplexity + RAG 调研流水线：怎么搭

选题光靠 LLM 想象是不够的，必须用真实数据校验。我现在的标准动作是：每个候选主题花 10 分钟做市场调研，这套调研用 Perplexity 为主、辅以几个自建数据源。

**Perplexity 的核心用法不是问答，是结构化输出。** 我会写一个固定的调研 prompt，把候选主题塞进去，输出一份结构化报告。模板大概长这样：

> 我正在评估这个主题作为内容选题的可行性：[主题名称]。请帮我做以下调研：1) 这个主题最近 6 个月在 X / Reddit / Hacker News / 微博上的讨论热度变化，给出趋势判断（上升 / 平稳 / 下降）；2) 目标读者最常问的 5 个相关问题（用真实问题，不要编造）；3) 这个主题下最常被提到的 3 个痛点；4) 已有的 top 5 文章，分别是什么角度；5) 评估竞争烈度（饱和 / 中等 / 蓝海）。请用结构化输出，每个部分不超过 100 字。

这个 prompt 的关键是"不要编造"和"结构化输出"。LLM 有时候会幻觉，把别人的内容拼凑一下当成真实讨论，所以我一般会再开一个 tab 用 Perplexity Pro 的"Focus"模式，限定到具体来源（比如 X、Reddit 的具体板块），交叉验证一遍。

Perplexity 之外，我自己搭了个 RAG 系统做补充。简单说就是把我想关注的信源（产品独立开发者论坛、IndieHackers、Reddit 的 r/SaaS、小红书的副业博主、V2EX 的 Creative 和 Startup 板块）每周爬一次，向量化存到本地 Chroma。需要调研某个主题的时候，retrieve 一下最近 6 个月的讨论，看用户原话。

这套东西我花了大概 2 个周末搭完，总共 600 行 Python。技术栈是 Firecrawl（爬虫） + OpenAI Embedding + Chroma + 一个简单的 FastAPI 查询接口。工程师做这个有天然优势，写起来飞快。

举个例子。我 2025 年 11 月想写"AI 内容经济"系列，先跑了一遍 RAG 检索，发现独立开发者圈最近 3 个月讨论最多的是三个问题：AI 内容会不会被搜索引擎惩罚？怎么避免被判定为 AI 生成？内容工作流怎么 AI 化但不失人味？三个问题就是三个选题。

**调研的真正价值不是"找选题"，是"找到你文章必须回答的那个问题"。** 一篇文章阅读量高不高，取决于你回答的那个问题是不是读者真的在搜。我以前写文章是"我想讲什么"，现在是"读者搜什么 + 我能给出什么独特答案"。这俩经常不一样。

讲一个反例。我曾经写过一篇文章叫《2025 年最好的 5 个 AI 写作工具》，自我感觉良好。结果发出去阅读 800，评论区有人留言"这内容我去年就看过类似的"。我反思了一下，发现这个题目没有新意，AI 写作工具评测早就是红海，我用 RAG 调研一下就能发现——top 5 文章最早一篇是 2023 年 5 月，最近一篇是 2025 年 9 月，6 个月内发了 47 篇类似内容。

我那天开始用 RAG 做竞争分析，每个选题跑一遍 top 20 文章，确保我的角度不在覆盖范围内。

## 跑 100 个选题挑 3 个的工程化流程

讲讲我的标准作业流程（SOP），这是我现在每周固定跑的。

每周日下午我会花 40 分钟做一次"选题会议"，流程是这样的：

**第一步：5 分钟整理本周输入。** 这周读了什么、看了什么、聊了什么。我有一个 Notion 页面专门记"内容灵感"，任何时候脑子里冒出来一个可能的选题都丢进去，标注是"想法"还是"已验证"。周日会议的时候把这一周的灵感全过一遍。

**第二步：10 分钟 LLM 批量生成。** 我用 Claude Code 起一个项目，把这一周积累的灵感 + 历史已发布内容 + 当前的赛道定位，全部丢给 prompt，让它生成 50 个候选主题。约束条件很硬：

- 必须在我 4 象限的右上角（兴趣 / 能力 / 流量 / 付费）
- 必须跟已发布内容的 3 篇不重复
- 必须有具体的目标读者画像（不能是"开发者"这种泛指）
- 必须能给出 3 个可能的差异化角度

LLM 生成 50 个，我手动刷一遍砍掉明显不靠谱的，留 20 个进入下一轮。

**第三步：15 分钟数据验证。** 这 20 个主题挨个过 Perplexity + 5118 + 我自己的 RAG。每一个主题我打 4 个分数（1-5）：流量分数、付费分数、竞争分数（反向，分数越高竞争越低）、差异化分数。4 个分数加起来 16 分以上进入候选池。

**第四步：10 分钟最终挑选。** 候选池里通常剩 3-5 个主题，我根据本周的时间精力、平台调性、最近的趋势热点，最终选 3 个作为本周的发布计划。

**这套流程跑下来，每周稳定产生 3 个高质量选题，命中率 38%。** 我后来把流程封装成一个简单的 Python 脚本，自动调用 Perplexity API + 5118 API + 我的 RAG 接口，输出一个打分后的 Excel。手动操作 40 分钟，自动版可以压到 10 分钟。

具体来说，Perplexity API 我用的是 Pro 订阅 + 自带的 API（2025 年 10 月才出的），单次查询大概 ¥0.3-0.5，每次选题会议跑 20 个主题约 ¥8-10。5118 用的是企业版 API，每月 ¥200。RAG 跑在我自己的服务器上，只有向量检索的 embedding 成本（用 text-embedding-3-small 的话 20 个主题约 ¥0.1）。整个流程的边际成本几乎可以忽略。

**讲一个我跑出 47 万阅读那次的选题过程。**

2026 年 1 月初我开始构思 AI 编程主题的时候，LLM 给我生成的候选里有一个是"AI 会不会让独立开发者失业"。当时我的第一反应是这个题太标题党、太空。我差点把它划掉。

但我跑了一遍数据验证，Perplexity 显示这个主题在 X 上讨论量周环比涨 240%，Reddit r/programming 有个相关讨论帖 3 天 1200 赞，5118 显示"AI 程序员"相关长尾词周搜索量从 1.2k 涨到 4.5k。所有数据都指向一个结论：**这是一个正在爆发的真需求**。

我重新评估了竞争，发现当时中文圈几乎没有深度讨论这个话题的文章，只有几个 KOL 发了很短的 tweet。我用 RAG 检索英文资料，发现 IndieHackers 上已经有一波关于"AI 时代的独立开发者策略"的深度讨论。

最终我写了一篇《我用 AI 写了 30 万行代码，告诉你独立开发者会不会被取代》，从个人经历 + 数据 + 行业趋势三个角度回答了"会不会失业"这个问题，发到 X 上当天 18 万阅读，三天累计 47 万。这是我单篇最高的数据。

**如果不是数据验证那一关，我差点划掉了一个 47 万阅读的选题。**

这就是工程师做内容的优势：你可以搭一套系统，把"感觉"变成"数据"。传统自媒体人靠经验、靠嗅觉、靠人脉，工程师靠工具链、靠数据、靠迭代。这套系统搭起来一次，能用很多年。

## 工程师最容易犯的 3 个错

讲讲我观察到的，包括我自己在内，工程师做选题最容易犯的错。

**第一个错是"技术洁癖"：只写自己觉得"有价值"的内容。** 我有个朋友技术很扎实，写了 200 篇深度技术博客，全是底层原理、源码分析。结果粉丝 800，广告都接不到。他犯的错是把"内容质量"等同于"技术深度"，但读者要的是"看完能解决我的问题"，不是"看完我变强了"。

我的解药是每周跑一次数据，看哪些文章阅读 / 完播率最高，反向总结。我公众号阅读最高的文章，恰恰是技术含量最低的——"我是怎么用 Claude Code 写完整个 SaaS 的"，几乎没有代码，全是流程和踩坑记录。

**第二个错是"工具崇拜"：觉得 AI 能解决一切。** 我见过有人把选题完全交给 AI，prompt 写得天花乱坠，但 AI 生成的选题不能直接用——它不知道你的账号定位、不知道你的目标读者、不知道市场已经发生了什么变化。AI 是放大器，不是替代品。AI 给你 50 个候选，最终挑哪个还得靠人。

我现在的原则是：**AI 做体力活（生成、检索、汇总），人做判断活（挑、组合、调性）**。

**第三个错是"数据迷信"：只看数字，不看场景。** 我有段时间特别迷信"高搜索量 = 好选题"，结果写了几篇搜索量大但完全不适合我账号的内容，数据一塌糊涂。

后来我加了一个约束：每个选题不仅要数据好，还要跟我的"账号人设"匹配。我的人设是"用 AI 提效的工程师副业玩家"，所以搜索量再大的"AI 取代程序员"选题，如果角度是"职业危机"，我也不会做；如果是"AI 工具提效"，就会做。

数据是必要条件，账号定位是充分条件。两个都要满足。

## 给你今晚能动手的 2 个建议

第一个建议：**搭一个 1 小时的"选题流水线"原型**。不用搞复杂，今晚就能跑起来。准备一份 1 周的灵感清单（Notion / 备忘录都行），用 Claude Code 起一个项目，写一个 prompt 让它生成 30 个候选主题。约束条件按我前面说的那 4 条来。生成完挑 5 个，用 Perplexity 跑一遍市场调研，看看哪些是真需求哪些是伪需求。

整个流程 1 小时，1 个 LLM 订阅费用，0 成本起步。你会第一次感受到"用 AI 找选题"跟"凭感觉找选题"的区别。

第二个建议：**建一个"内容反例库"**。我有一个 Notion 页面专门记"我发过的、效果差的内容"，每篇标注：选题是什么、为什么发、效果数据（阅读 / 点赞 / 转化）、反思（为什么效果差、下次怎么改）。

这个反例库比成功案例库更有价值。成功有偶然性，失败有必然性。三个月后你回头看自己的反例库，会发现一些规律：哪些类型的题永远数据差、哪些角度永远跑不出来。这些规律就是你下一阶段的选题指南针。

工程师最擅长的就是"从失败中提取模式"。把这套思维用进选题，你会发现内容创作跟写代码一样，是可以被工程化、被优化的。

---

**最后讲个数据。** 我这套选题流程跑了大半年，累计测试了 600+ 个选题，最终发布 80+ 篇，命中率（点赞 / 阅读 > 3%）从 12% 涨到 38%，最高单篇 47 万阅读。账号从 0 粉做到 1.2 万粉，副业收入累计 ¥18 万。

**AI 没有让我变成更好的写作者，AI 让我变成了更好的"产品经理"。** 我用工程思维做选题——用户调研、需求验证、A/B 测试、数据迭代——这套东西我从产品里搬过来，套到内容上，跑通了。

工程师做副业最大的优势不是技术，是**你习惯用数据说话、用系统解决问题**。把这种思维带进内容创作，效率会差出 3-5 倍。

下一章我会讲怎么用 AI 辅助写作但不失人味——这是另一个常见坑：很多人 AI 用得 6，但写出来的东西一股 AI 味，读者一眼就看出来。怎么调教 LLM 写出"像你写的"内容，是工程问题，也是审美问题。
