# 03. 三段式口述：vibe coding 的标准方法

vibe coding 听起来很玄，其实有个标准三段式：**初始需求 → AI 生成（带瑕疵）→ 精准修正口令**。这一章把这个方法论讲透。

## 第一段：初始需求口述

好需求的三要素：**目标、约束、验收标准**。缺一项 AI 就会乱填。

差的写法：

> "写个登录接口。"

AI 会自己拍脑袋：用什么 session？JWT 还是 cookie？密码加密用 bcrypt 还是 argon2？错误返回什么格式？字段命名驼峰还是下划线？每个问题都可能跟你的项目规范不一致。

好的写法：

> "基于现有 NestJS 项目加一个 /api/auth/login 接口。请求体 { username: string, password: string }，用 bcrypt 校验（项目里 user 表 password_hash 字段已经存 bcrypt 哈希），成功返回 JWT（用项目已有的 JwtService，exp 7 天），失败返回 401 + { code: 401, message: '用户名或密码错误' }。路由前缀跟 /api/users 一致。错误处理走全局 ExceptionFilter（项目已有）。"

字数差不多，但好版本把所有"AI 容易猜错"的地方都钉死了。

**口诀**：**先告诉 AI 你项目的现状，再说你想要什么，最后说验收标准**。

## 第二段：AI 生成（带瑕疵）

这一段有个反直觉的点：**你应该预期 AI 第一版会有 bug**。vibe coding 不是"一次到位"，是"AI 出 70 分稿，你补到 100 分"。

70 分稿的常见瑕疵：

- 字段名跟项目不一致
- 缺边界异常（null、空字符串、超长）
- 性能陷阱（N+1 查询、循环里 new 对象）
- 安全漏洞（明文密钥、SQL 注入、XSS）
- 缺日志 / 监控
- 跟现有代码风格不一致（缩进、import 顺序）

**不要在 AI 第一版出来时急着 commit**。先看 30 秒结构，确认大方向对，然后就进第三段。

## 第三段：精准修正口令

修正口令的关键：**具体位置 + 具体问题 + 具体改法**。

差的修正："这个不对，再改改。" — AI 不知道改什么，会瞎试。

好的修正：

> "路由 /api/auth/login 没问题，但三个地方要修：第一 userService.findByUsername 没处理 undefined 返回，会 NPE，加空判断；第二 bcrypt.compare 没 try/catch 包，DB 异常会 500 改成 401；第三返回的 JWT 里没带 userId，前端要拿 userId 拼请求头，加到 payload。"

这种修正口令 AI 能一次到位，不用再迭代。

**口诀**：**修正口令里说"哪个位置 + 什么问题 + 改成什么"**。三件套缺一不可。

## 反例对照

差的迭代 6 轮：用户不断说"还是不行"，AI 不断瞎试，每次换一处，最后代码逻辑全乱。

好的迭代 1-2 轮：第一轮说清需求，AI 出 70 分稿；第二轮一次列 3-5 个具体瑕疵，AI 一次修完。

我自己的真实数据：把需求口述写扎实 + 修正口令具体化，平均迭代 1.8 轮能出可交付代码。差的写法平均 5.4 轮，还经常改坏其他文件。

## 模板

我现在每个 vibe coding 项目开头都会写一份"口述模板"贴给 AI：

```
项目背景：[一两句话说清技术栈和现有架构]
本任务目标：[具体要做什么]
约束：[性能、安全、命名规范、错误处理等已有约定]
验收标准：[怎么算"做完了"]
```

写好模板，复制粘贴到每个新需求的开头，vibe coding 效率立刻翻倍。下一章讲第一个真坑——N+1 性能陷阱。
