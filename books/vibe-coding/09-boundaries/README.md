# 09. vibe coding 的边界：什么时候该停

vibe coding 不是万能的。2025 年 Karpathy 提出这个概念时附了一句**免责声明**：

> "It's not too bad for throwaway weekend projects or code that you rarely use, but it's still not something I would use for anything serious."

2026 年 vibe coding 已经从"周末项目"扩展到严肃项目，但**它依然有自己的边界**。这一章讲清楚哪些场景 vibe coding 能用，哪些场景必须停。

## vibe coding 能用的场景

**原型阶段** — 半天出 MVP，向老板 / 投资人 / 用户演示。这是 vibe coding 最适合的场景，速度优势最大化、长期成本最小化。代码用完就扔。

**独立项目** — owner 就是自己，没有团队协作，没有合规要求。我自己的个人工具全 vibe。

**遗留项目批量重构** — 接手别人代码不想逐行读。vibe coding 帮你理解 70%，剩下 30% 关键决策你来做。

**一次性脚本** — 数据迁移、批量处理、定时任务、报告生成。代码用完就扔，错了大不了重跑。

**教学 / 内部 demo** — 不需要长期维护，只用来教学或演示。

## vibe coding 不能用的场景

**金融 / 医疗 / 政企** — 合规要求逐行审计（SOX、HIPAA、GDPR、等保 2.0）。AI 写的代码审计不过。

**开源核心库** — 社区要 review、要测试覆盖、要语义化版本控制。AI 写的代码社区不会接受。

**高频性能路径** — HFT、游戏引擎、底层算法。AI 默认走最容易想到的实现，性能不够。

**安全敏感模块** — 鉴权 / 加密 / 支付 / 密钥管理。AI 写出来的代码 2000+ 漏洞案例（Escape 公司扫的 5600 个应用），安全审计成本远高于省下的开发时间。

**生产数据库迁移** — DDL 操作、数据迁移脚本。一旦写错就是数据丢失。

## 三个判断"该不该停"的信号

**信号 1：AI 开始"幻觉"**

AI 引用了一个不存在的库、不存在的函数、不存在的 API。比如你说"用 project 里的 UserService"，AI 编出一个 `UserService.getActiveUsersPremium()`，但项目里没这个方法。

**出现幻觉就停**，回去补上下文或者自己写。

**信号 2：迭代 3 轮还在改同一处 bug**

第 1 轮 AI 改 A 坏了 B，第 2 轮 AI 修 B 坏了 C，第 3 轮 AI 改 C 坏了 A——经典死循环。

**3 轮没修好就停**，要么自己读代码手动修，要么把这段代码标记为"AI 禁区"。

**信号 3：AI 拒绝继续**

直接说"This code is too complex for me to safely modify" 或者反复给同一份代码做无关紧要的格式调整。

**AI 拒绝就尊重**，别硬逼。AI 的拒绝通常是基于它真没把握。

## vibe coding 的长期成本

很多人忽略的一点：**vibe coding 出来的代码维护成本是手写代码的 2-3 倍**。

原因：vibe coding 出来的代码**结构不统一**（每个 AI 生成的函数风格略不同）、**注释少**（AI 默认不写注释）、**测试少**（AI 默认不写测试）、**文档缺**（AI 默认不写文档）。

半年后你自己回头看，或者换个人接手，要花 2-3 倍时间理解它、维护它、debug 它。**省下的开发时间全在维护阶段还回去**。

所以 vibe coding 适合**短生命周期项目**，不适合**长生命周期项目**。

## 我自己的边界

**纯 vibe**：个人工具、demo、一次性脚本、内部自动化。

**vibe + 人工 review + 补测试**：业余项目、客户 prototype、内部工具。

**vibe + 人工重写关键模块**：正式客户项目、生产环境项目、需要维护半年以上的项目。

**不用 vibe**：金融、医疗、政企、高性能、安全敏感、生产数据库迁移。

## 一个反直觉的发现

我跟踪自己一年 vibe coding 数据：**纯 vibe 的代码半年后我还能流畅维护的比例只有 38%**——剩下 62% 我都要回去读 + 重构。

但 **vibe + 人工 review + 补测试** 的代码半年后能维护的比例升到 **84%**。

关键不在 vibe 不 vibe，**关键在 review 和测试**。AI 写的代码也要走和手写代码一样的质量门：单元测试 + 集成测试 + code review + 性能 benchmark + 安全审计。
