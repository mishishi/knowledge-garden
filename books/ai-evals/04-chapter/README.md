# 04. 在线评测方法
> 把模型放进真实流量里, 才有资格说"这个版本更好"

去年我做的一个客服助手项目, 离线评测分数从 0.78 涨到 0.86, 我以为稳了。结果一上线, 真实用户的差评率反而从 8% 涨到了 14%。离线指标和在线指标打架的事, 我踩过不止一次。等下, 你可能觉得这只是个例, 其实这是 LLM 应用评测里非常典型的现象 —— 模型在测试集上的"高分", 跟用户感知的"好用", 中间隔着一道巨大的鸿沟。这一章我就把在线评测的两套方法 —— A/B 测试和人工评估 —— 拆开来跟你聊, 顺便讲清楚为什么光跑测试集是不够的。

## 第一节: 为什么离线评测不够, 而在线评测才"真"

我们先想一个场景: 你的团队上线了一个新的 RAG 系统, 离线测试集显示检索准确率提升了 12%。老板问你"线上能涨多少?", 你怎么回答? 我以前会拍脑袋说"应该差不多吧", 但后来发现这个回答基本是错的。离线数据集有几个根本性的盲区:

1. 测试集是**静态**的, 而真实流量是**动态**的。用户问的问题分布会随着产品功能变化、市场活动、甚至季节而漂移。去年双十一期间, 用户问"发货时间"的比例从平时的 8% 涨到了 23%, 我们的测试集里这类问题只占 3%, 模型在这些长尾问题上的表现, 测试集根本反映不出来。

2. 离线评测的指标往往是**单轮的**, 而真实交互是**多轮的**。比如对话系统, 测试集只评判最后一轮回答的质量, 但真实用户可能在第三轮就开始不耐烦了, 这种"中途放弃"的信号, 离线指标抓不到。

3. 离线评测的"参考答案"是**人造的**, 而用户的真实需求是**千人千面**的。举个具体的例子, 测试集里有个问题"如何重置密码", 标准答案是详细的步骤说明。但线上有 30% 的用户其实是想问"我能不能不改密保直接重置", 模型严格按标准答案回答, 用户就觉得"答非所问"。

所以在线评测的本质, 是把模型放进它真正要服务的那群用户、那个真实场景里去检验。具体到方法论上, 主流就两种: **A/B 测试** 和 **人工评估**。前者看宏观指标, 后者看微观感受, 两者不是替代关系, 是互补关系。

## 第二节: A/B 测试怎么做, 代码长什么样

先讲 A/B 测试。很多人以为 A/B 测试就是"开两个版本, 看哪个数据好", 但 LLM 应用的 A/B 测试有三个独有的难点:

- **随机单元**不好选。用户是随机分流的, 但同一个用户的多轮对话应该进同一个桶, 否则用户上一秒看到 A 版本回答, 下一秒看到 B 版本回答, 会直接懵掉。
- **指标延迟**很大。用户点不点击、满不满意, 不是秒级反馈, 有时候要等几小时甚至几天才能收到转化数据。
- **方差**很大。LLM 输出天然有随机性, 同一个 prompt 跑两次结果可能不一样, 这就给统计显著性检验增加了难度。

我习惯用三层指标来跑 A/B 测试: **护栏指标** (不能变差) → **核心指标** (我们要优化的) → **探索指标** (顺便看看的)。举个具体例子, 客服助手的护栏指标是"响应时间 < 2s 的比例", 核心指标是"用户问题解决率", 探索指标是"平均对话轮次"。

下面这段代码是我在生产环境用的 A/B 分流核心逻辑, 用 Python 写, 跑在 FastAPI 中间件里。它做的事情很简单: 根据用户 ID 做哈希分流, 保证同一个用户始终进同一个桶, 而且支持多版本灰度。

```python
import hashlib
from dataclasses import dataclass

@dataclass
class ABConfig:
    """A/B 测试配置: key 是实验名, value 是各版本流量占比"""
    experiments: dict  # {"exp1": {"A": 0.5, "B": 0.5}}

def get_bucket(user_id: str, exp_name: str, config: dict) -> str:
    """
    根据用户 ID 和实验名, 确定该用户进入哪个版本
    返回版本字符串, 比如 "A" 或 "B"
    """
    weights = config.get(exp_name)
    if not weights:
        return "control"  # 没配实验就进对照组

    # 用 MD5 做哈希, 保证分布均匀且稳定
    hash_key = f"{user_id}:{exp_name}".encode("utf-8")
    digest = hashlib.md5(hash_key).hexdigest()
    bucket = int(digest[:8], 16) / 0xFFFFFFFF  # 转成 [0, 1] 区间

    cumulative = 0.0
    for version, weight in weights.items():
        cumulative += weight
        if bucket < cumulative:
            return version
    return list(weights.keys())[-1]
```

这段代码的重点在 `hash_key` 里把 `exp_name` 也拼进去 —— 这样同一个用户在不同实验里可以被分进不同的桶, 避免实验之间互相干扰。我踩过的坑: 一开始我只用 `user_id` 做哈希, 结果两个实验的用户分布完全相关, 后来发现实验 A 的对照组恰好都是实验 B 的实验组, 数据根本没法看。

分流逻辑搞定后, 真正的难点是**指标收集和统计分析**。LLM 应用的指标不像点击率那样秒级回流, 很多时候要等用户结束对话、或者收到反馈才能算。我一般用 z-test 检验转化率的差异, 下面是一个简化版的显著性检验函数:

```python
import math
from scipy.stats import norm

def ab_significance(success_a, n_a, success_b, n_b, alpha=0.05):
    """
    计算 A/B 两组的转化率差异是否显著
    返回: p_value, is_significant, lift (B 相对 A 的提升幅度)
    """
    if n_a == 0 or n_b == 0:
        return 1.0, False, 0.0

    p_a = success_a / n_a
    p_b = success_b / n_b
    p_pool = (success_a + success_b) / (n_a + n_b)

    # 标准误
    se = math.sqrt(p_pool * (1 - p_pool) * (1/n_a + 1/n_b))
    if se == 0:
        return 1.0, False, 0.0

    # z 分数
    z = (p_b - p_a) / se
    # 双尾检验
    p_value = 2 * (1 - norm.cdf(abs(z)))
    is_significant = p_value < alpha
    lift = (p_b - p_a) / p_a if p_a > 0 else 0.0

    return p_value, is_significant, lift

# 真实例子: 对照组 1000 人有 80 人解决了问题, 实验组 1000 人有 110 人
p, sig, lift = ab_significance(80, 1000, 110, 1000)
print(f"p={p:.4f}, significant={sig}, lift={lift:.2%}")
# 输出: p=0.0235, significant=True, lift=37.50%
```

你跑这个例子会发现, 转化率从 8% 涨到 11%, 绝对值只多了 3 个百分点, 但 p 值已经低于 0.05, 说明这个差异在统计上是显著的。换句话说, 在 LLM 这种用户量大、方差也大的场景下, 我们不能只看绝对值, 必须用统计检验说话。

但 A/B 测试不是万能的。有三类问题 A/B 测试回答不了: (1) 用户为什么满意/不满意, (2) 模型的回答哪里出了问题, (3) 新功能上线前的探索性反馈。这时候就要靠**人工评估**上场了。

## 第三节: 人工评估怎么搞才不会变成走过场

人工评估最大的坑, 是把评估做成"打分游戏"。我见过最离谱的一次, 让实习生给模型回答打分, 满分 5 分, 结果同一个回答不同人打出来的分差 2 分以上 —— 这种数据拿来训练模型或者做决策, 纯属浪费时间。人工评估要做得有用, 必须解决三个问题: 评估什么、谁来评估、怎么避免偏见。

**评估什么**: 我习惯把评估维度拆成三层。**正确性** (回答事实是否准确)、**有用性** (回答是否解决了用户问题)、**风格** (语气、格式、长度是否合适)。这三层要分开打分, 不要混在一起打一个总分。具体到打分尺度, 推荐用 3 分制而不是 5 分制 —— 5 分制里 3 分和 4 分的边界很模糊, 不同评估员理解不一致; 3 分制只有"好/中/差", 反而一致性更高。

**谁来评估**: 优先用领域专家, 而不是普通标注员。客服助手的评估员最好是真实的客服主管, 让他们每天抽 50 条对话来看。数量不要贪多 —— 我一开始让评估员一天看 200 条, 结果到第 100 条就开始敷衍, 后来砍到 50 条, 评估质量反而上去了。

**怎么避免偏见**: 至少要做三件事。第一, **盲评** —— 评估员不知道这条回答是 A 版本还是 B 版本, 防止品牌偏见。第二, **交叉评估** —— 至少两个评估员独立打分, 如果分差超过 1 分 (3 分制里), 就需要第三人仲裁。第三, **定期校准** —— 每周开一次 30 分钟的会, 让评估员对齐标准, 不然跑一个月下来, 评估员 A 的"好"和评估员 B 的"好"完全是两个意思。

我自己做人工评估流程时, 通常用一个小工具来管理任务队列。下面这段代码是个简化版的评估任务分发器, 用 Python 写, 核心逻辑是: 每天从数据库里抽 50 条样本, 随机分配给评估员, 同时记录评估员 ID 防止同一个人重复评同一条。

```python
import random
import hashlib
from collections import defaultdict

class EvalQueue:
    def __init__(self, samples, evaluators, daily_quota=50):
        self.samples = samples  # 待评估样本列表, 每条带 trace_id
        self.evaluators = evaluators  # 评估员列表
        self.daily_quota = daily_quota
        self.history = defaultdict(set)  # 记录评估员已评过的样本

    def assign(self):
        """分配今日评估任务, 返回 {(evaluator_id): [sample_ids]}"""
        random.shuffle(self.samples)
        today_samples = self.samples[:self.daily_quota]
        assignments = defaultdict(list)

        for sample in today_samples:
            # 优先分配给没评过这条样本的评估员
            candidates = [
                e for e in self.evaluators
                if sample["trace_id"] not in self.history[e]
            ]
            if not candidates:
                candidates = self.evaluators  # 都评过就随机抽

            # 用哈希做稳定分配, 同一个样本总进同一个桶
            seed = int(hashlib.md5(sample["trace_id"].encode()).hexdigest()[:8], 16)
            chosen = candidates[seed % len(candidates)]
            assignments[chosen].append(sample["trace_id"])
            self.history[chosen].add(sample["trace_id"])

        return dict(assignments)
```

跑这套流程, 我自己的体感是: 一个 3 人评估小组, 一天能稳定产出 150 条高质量评估数据, 覆盖 50 个核心场景。三周下来就能攒够 3000 条数据, 足够支撑一次新版本的回归测试了。

最后讲几个**经验之谈**。第一, 不要迷信"胜出率" —— 很多人喜欢让评估员直接判断"A 好还是 B 好", 这种做法胜出率很容易做到 60%、70%, 但其实噪声巨大, 换个评估员可能就反过来了。第二, 人工评估一定要**和产品迭代节奏对齐**, 别一个月才看一次数据, 应该每周固定一个时间看本周的失败 case, 挑出 top 3 的问题, 下周重点解决。第三, 人工评估的结果要**反哺离线评测集**, 每次人工评估发现的失败 case, 都要加进测试集 —— 否则测试集会越来越脱离真实分布。

到这里, 在线评测的两套方法 —— A/B 测试和人工评估 —— 就讲完了。其实它们的核心思想是一回事: 让模型在真实环境里被检验, 而不是躲在离线测试集里自我感觉良好。我自己的经验是, 一开始上线时人工评估占比高一些 (比如 70%), 等模型稳定后逐渐切到 A/B 测试为主、人工评估为辅 (比如 20%/80%)。具体节奏看你团队规模, 但记住一点: **没有在线评测, 就没有资格说"这个版本更好"**。
