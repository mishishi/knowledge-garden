# 01. 评估为什么重要

> 不知道模型变好还是变坏，所有优化都是在猜硬币正反面。

去年 11 月我们组接了一个客服问答的活。需求听起来不复杂：把用户问题喂给 LLM，让它从 80 条业务规则里挑相关条款，再生成自然语言回复。PM 说 "先跑通就行"，我那位身经百战的同事回了句 "跑通不算上线"，我嘴上没接茬，心里嘀咕是不是太较真了。

两周后模型接进灰度，第三天我被叫进会议室。客服主管拍着桌子："用户问 '退订后还能不能开发票'，回答 '可以'，但我们业务规则第十五条写着 '退款后 30 天内可申请'，措辞差得远，财务那边在骂人。" 我打开日志一看，那个回答其实是 GPT-4 给的，prompt 没改过，温度 0.7。问题出在哪？出在没人知道"对"长什么样。

## 没有评测，谈改进就是掷骰子

我把那次出问题的 100 条样本拉出来人工标了一遍，发现模型真正犯错的只有 8 条，但其中 5 条都集中在"退款 + 时效"这一类。如果当时有一把尺子能告诉我"这类问题准确率 62%"，我第一天就会把规则改掉。

所谓的评测，本质就是这把尺子。它回答一个问题：在某个分布的数据上，我的系统表现成什么样。没有这把尺子，后面的所有动作——换 prompt、换模型、加 RAG、fine-tune——都是在盲调。盲调不一定会变差，但更可怕的是你以为它在变好。

我手里攒着一组公开榜单的数据，是某个 RAG 框架在 HotpotQA 上的迭代日志。第一版 baseline EM 18.4，工程改了 chunk size，涨到 19.1；改 hybrid search，又涨到 20.0；改 reranker，掉到 18.7。三个改动里有一个是负向的，但当时没人做对照实验，就看到 20.0 那一版被部署了，18.7 那一版被默默回滚。如果有一套系统的评测，每一步都能告诉你"这次和上次差多少、差在哪类问题上"，你根本不会把 18.7 当成"调试抖动"忽略掉。

再说个真实数字。我自己做 LLM-as-judge 实验时，跑过 800 条 RAG QA 的对比：人工标注的 EM 0.41，GPT-4 当 judge 评出来 0.39，相关性 0.86。意思是 judge 大方向没跑偏，但具体到每一条会高估或低估 2-3 个百分点。这 2-3 个百分点在产品决策里就是"上 A 方案还是 B 方案"的差距。所以判断"模型有没有变好"，不能靠"我跑了几条样本感觉不错"，得靠一个能复现、能追溯、能比较的数字。

## 三个具体场景，看评测怎么救人

第一个场景是 prompt 调优。我之前给一个分类任务写 prompt，第一版写"判断用户意图：投诉、咨询、闲聊、其他"，准确率 0.78。第二版加了一句"只输出类别名，不要解释"，准确率涨到 0.85。我差点就把这版发出去。后来上评测集跑了一轮，发现——

```python
# 一个最小可用的离线评测脚本
from typing import Callable

def offline_eval(predict_fn: Callable[[str], str],
                 dataset: list[tuple[str, str]]) -> dict:
    """
    dataset: [(input, gold_label), ...]
    predict_fn: 输入 query 输出预测
    """
    correct, total = 0, 0
    by_label = {}
    for query, gold in dataset:
        pred = predict_fn(query).strip()
        ok = pred == gold
        by_label.setdefault(gold, [0, 0])
        by_label[gold][0] += int(ok)
        by_label[gold][1] += 1
        correct += int(ok)
        total += 1
    # 整体 + 按类别拆开看
    return {
        "overall": round(correct / total, 4),
        "per_class": {k: round(v[0] / v[1], 4) for k, v in by_label.items()},
    }

# 跑两版 prompt
v1 = lambda q: call_llm(q, prompt=PROMPT_V1)
v2 = lambda q: call_llm(q, prompt=PROMPT_V2)
print(offline_eval(v1, val_set))
print(offline_eval(v2, val_set))
```

按类别一拆，"其他"那一类从 0.91 掉到 0.74。原来"只输出类别名"这一句把模型逼得太紧，遇到边缘样本反而容易瞎猜。整体数字好看，但用户真正不长见的那类问题变差了。**没有按类别拆分的评测，这种退化就藏在 0.85 那个数字里，你永远不会发现。**

第二个场景是模型选型。Qwen2.5-7B、Llama-3.1-8B、GPT-4o-mini，我们要在内部知识问答里挑一个。看价格和上下文窗口没用，必须在自己的数据上跑过才知道。评测集我攒了 600 条，覆盖 12 个业务子类。结果：GPT-4o-mini 综合 0.71，Qwen 0.68，Llama 0.64。Qwen 在"产品规格"那一类反而比 GPT 高 5 个点。预算紧张就先上 Qwen，预算宽松就上 GPT。这就是评测给决策的支撑，没有它你选哪个都是"看感觉"。

第三个场景是回归保护。一周前我把 RAG 的 retriever 从 BM25 换成了 BGE-M3，整体看起来涨了 1.8 个点，挺开心。结果线上 A/B 第二天客服就反馈"老客户的订单查询答得不准了"。我赶紧拉了评测集里"订单查询"这个子类一跑，新 retriever 在这子类上从 0.83 掉到 0.74，被其他类别的涨幅盖住了。**没有"按子类切分"的评测，这种负向迁移在整体均值里几乎不可见。** 现在我跑任何改动，都会先看分桶再下结论。

## 别把"评测"做成自我安慰

踩过几次坑之后，我总结了几条血泪经验。

第一条：评测集不能和训练集重叠。这是最基础的，但很多团队做 fine-tune 时把评测集也丢进去"顺便学一下"，结果评测分数虚高 10-20 个点，线上掉得更狠。我们的做法是评测集固化在仓库的 `data/eval_v3.jsonl`，任何人改 prompt、换模型、动 retriever 都只能读不能写，写新版本要过 code review。

第二条：评测样本要覆盖长尾。我早期做的评测集 200 条，全是高频问题，模型轻松拿到 0.9 上线，结果一接真实流量，"边缘问题"全军覆没。现在我做评测集有硬规矩：高频不超过 40%，中频 40%，低频 20%。长尾那 20% 才是真正决定你模型能不能用的东西。

第三条：评测要带错误分析，不是光看一个数。`overall=0.85` 告诉你一半的信息，另一半在错的那 15% 里。我习惯在评测脚本后挂一个"错误分类"步骤，人工看 50 条错例，归类成"幻觉 / 漏检 / 格式错 / 知识错"四类，再算每类占比。如果某次改完"幻觉"从 8% 降到 3%，但"格式错"从 2% 涨到 6%，我不会说这次是改进，我会说这次是 trade-off。

第四条：警惕 LLM-as-judge 的隐性偏差。让 GPT-4 评 GPT-4 自己的输出，倾向给高分；让 GPT-4 评 Claude 的输出，倾向给低分。我做过一个实验，800 条样本，同一个 judge 模型评自己家模型时平均分比评别家高 4.7 个点。所以我的建议是：能人工标的尽量人工标，至少抽 200 条做"金标"；必须用 judge 时，准备一个独立家族的模型做交叉验证，并把 judge 自身的准确率作为指标的一部分监控。

第五条：离线评测和线上指标不一定是正相关的。离线涨 2 个点，线上可能平、可能涨、可能跌，这取决于你的评测集是否真实反映线上分布。我自己的经验是：评测集每周从线上抽样更新 10%，保留老版本做对比，这样六周后你能看到一个滑动趋势，而不是赌某一版是"代表"。

第六条：评测指标要早于产品上线存在。我见过最离谱的一个项目是上线三个月后才补评测，因为 PM 突然要"和友商比一下"。这时候你根本不知道当时那个版本是好是坏，对比基准都没有。正确做法是：第一天写完 prompt 的同时，就要有 `eval.py` 和一份固定评测集。哪怕只有 50 条，也比没有强一万倍。

## 评测不是给老板看的，是给你自己看的

最后说一句可能会得罪人的话：评测最重要的受众不是 PM、不是老板、不是你那条汇报线上的任何人，是你自己。是你凌晨两点改完 prompt 想知道"有没有变好"的那个瞬间，是你面对两个方案犹豫"选 A 还是 B"的那个下午，是你看到 0.85 这个数字想知道"它背后是稳的还是抖"的那个时刻。

把它当成模型开发的 git history——没有评测，你就是在 force push，没有 commit message，没有 diff，没有 blame。出了事你不知道是谁改坏的，好了也不知道是谁改对的。

从下一章开始，我会从"评测集怎么搭"讲起——怎么采样、怎么分层、怎么避免污染、怎么持续更新。这部分内容是整个系列的基石，因为后面所有"如何评判 LLM 输出""怎么用 LLM-as-judge""怎么写 pairwise 比较"都建立在你有没有一份"像样"的评测集上。

我现在做的每一个新模型项目，第一件事不是写 prompt，是写 `eval.py`。如果你只从这一章带走一句话，我希望是这个：在你动 prompt 之前，先动评测。
