# 08. 评测驱动迭代
> 评估结果不是终点，而是模型下一次进化的起点。

去年我们在做一个客服摘要项目，模型上线时 F1 看着还行，结果人工抽查发现 14% 的摘要把用户的核心诉求写反了。我盯着评估报告里那一列 "faithfulness"，数字是 0.86, 挺好看的, 但根本没反映出这种"语义反转"的错误。从那之后我就在想，评估结果到底怎么才能真正反哺到模型优化里去? 这一章我想把我后来摸索出来的一套打法讲清楚。

先说个让我尴尬的事。我们当时的第一反应是 "F1 还行啊，应该没事吧"，直到产品经理拿着标注好的 200 条 bad case 拍到我脸上，我才意识到一个残酷的事实：**离线指标和线上体验之间隔着一堵墙**。评估结果如果不拆解、不归因、不行动，就是一堆漂亮的数字躺在 dashboard 里吃灰。

# 01. Hook：从"看完即弃"到"看完即改"

我后来总结了一个简单的循环：**评估 → 拆解 → 假设 → 改 → 再评估**。听起来像废话，但这五个环节里最容易断的就是从"评估"到"拆解"这一步。很多人跑完评估就 `exit()` 了，看一眼总分，"嗯，比 baseline 高 2 个点"，然后就忙着去训下一个版本了。

我踩过的坑是，当时跑的是 GPT-4o-mini 在 HotpotQA 上的一个子集，指标是 exact match。上线前评估显示 EM 0.72, 看着挺美。结果线上跑了一周，客诉里冒出来一类奇怪的反馈: "回答是错的, 但我没法说清哪里错"。我拉了 50 条线上 case 回来, 让两个标注员盲标, 发现其中 9 条的 reference answer 其实有歧义——两个标注员给的答案都是合理的，但和我们 eval 集里的 ground truth 对不上。

换句话说，**评估集的噪声在悄悄吃掉你的优化信号**。这不是模型的问题, 是 eval 本身的问题。所以我现在的第一原则是：每次评估完，第一件事不是去看模型，而是去看 eval set。这一步被我叫做 "sanity check the eval"。

具体怎么做呢？我会随机抽 50 条 eval 样本, 让两个标注员独立标, 然后算 Cohen's Kappa。如果 κ < 0.7, 那这个 eval 集本身就不稳, 我会先去修 eval, 而不是去修模型。我之前有个项目就是这么干的：κ 从 0.58 修到 0.81 之后, 模型分数反而 "下降" 了 1.5 个点, 但线上的体感反而好了——因为之前那些 "高分" 其实是在拟合 eval 的噪声。

# 02. 拆解：从一个总分到错误画像

好，假设你的 eval 集是干净的（κ > 0.7），下一步是**错误画像**。我不会只看一个总分，我会按错误类型把 bad case 切开。

我常用的错误分类有这几类：
- **幻觉型**：模型编造了不存在的信息
- **遗漏型**：该提的没提
- **顺序型**：逻辑顺序反了
- **格式型**：结构不对（比如 JSON 字段缺失）
- **风格型**：语气/长度不达标

举个例子, 我之前做 RAG 评估的时候, 错误画像大概是: 幻觉型 38%, 遗漏型 27%, 顺序型 15%, 格式型 12%, 风格型 8%。看到这个分布, 优化方向就很清楚了——**优先解决幻觉，而不是去纠结那 8% 的风格问题**。这就是评估反哺优化的第一层价值：它帮你找到 ROI 最高的优化点。

我写过一个简单的错误画像脚本, 跑起来很快:

```python
# error_profiler.py - 按维度拆解 bad case
from collections import Counter
import json

def profile_errors(eval_results, error_taxonomy):
    """eval_results: [{'id': ..., 'is_correct': False, 'raw_output': ..., 'reference': ...}, ...]
       error_taxonomy: {'hallucination': ['编造', '不存在'], 'omission': ['缺少', '遗漏']}
    """
    error_buckets = Counter()
    examples = {k: [] for k in error_taxonomy}

    for item in eval_results:
        if item['is_correct']:
            continue
        # 用关键词做粗分类, 实际项目里会用 LLM-as-judge
        output = item['raw_output']
        matched = False
        for err_type, keywords in error_taxonomy.items():
            if any(kw in output for kw in keywords):
                error_buckets[err_type] += 1
                if len(examples[err_type]) < 3:
                    examples[err_type].append(item)
                matched = True
                break
        if not matched:
            error_buckets['unknown'] += 1

    total_errors = sum(error_buckets.values())
    print(f"总错误数: {total_errors}")
    for err_type, count in error_buckets.most_common():
        pct = 100.0 * count / total_errors if total_errors else 0
        print(f"  {err_type}: {count} ({pct:.1f}%)")
    return error_buckets, examples

# 用法
taxonomy = {
    'hallucination': ['编造', '据我了解', '不存在'],
    'omission': ['缺少', '未提及'],
    'format': ['JSON', '格式错误']
}
results = json.load(open('eval_output.jsonl'))
buckets, samples = profile_errors(results, taxonomy)
```

跑完这个你会得到一张饼图, 但比饼图更有价值的是 `samples` 那个字典——每类错误下保留 3 条典型 bad case。这 3 条就是你和模型搏斗时的"锚点"。

# 03. 从画像到假设：从 12% 到 1.8% 的真实路径

讲个具体的故事。我们当时做一个法律文书摘要任务, baseline 模型 (Llama-3-8B) 的事实性错误率是 12.3%, 这数字看起来还行, 但客户说"出错一条就要赔钱"。

错误画像跑出来, 幻觉型占了 71%, 而且几乎都是 "引用了不存在的法条"。我盯着 30 条 bad case 看了两个小时, 发现一个规律: 模型在生成时, 会自信满满地编出 "根据《XXX法》第 Y 条" 这种东西, 而 ground truth 里根本就没提法条。

**我的假设是：模型在长尾法律知识上过拟合了, 学到了"法条格式"但没学到"具体法条内容"**。于是我做了三件事:

1. **改 prompt**：从 "请总结这段文书" 改成 "请只总结原文中明确出现的事实, 不要补充任何外部法律知识"
2. **加一道后处理**：用正则检测 "第 X 条" 模式, 如果法条号不在原文里就删除
3. **数据增强**：在训练集里加了一类负样本, 即 "原文无此法条 → 输出应不含法条"

改完之后, 事实性错误率从 12.3% 降到了 1.8%, 整整一个数量级。但更重要的是——**这个 1.8% 不是拍脑袋调出来的, 是从错误画像里精确瞄准出来的**。如果我当时去优化那 8% 的风格错误, 估计再调两周也降不到 5% 以下。

这里我想强调一个反直觉的点：**不是所有错误都值得优化**。你会发现, 有些错误是 "标注噪声" 引起的, 有些是 "任务定义不清" 引起的, 有些是 "模型真的不行" 引起的。只有最后那一类才值得花力气。我自己的经验法则是: 如果一类错误人工标注员之间都达不成一致 (κ < 0.5), 那就先去和 PM 吵清楚任务定义, 而不是去调模型。

# 04. 评估驱动的几种典型反哺路径

我把评估反哺优化的路径归纳成四类, 按实施成本从低到高排:

**路径一：Prompt 迭代**。成本最低, 反馈最快。如果你发现错误集中在某一种类型, 比如 "总是用第一人称", 那很可能就是 prompt 没写清。我一般会保留一个 `prompt_history.md`, 每次改 prompt 都记下来: 改了哪句、为什么改、评估分变没变。这文件比任何 paper 都值钱。

**路径二：In-context 增强**。把 bad case 当作 few-shot 塞回 prompt 里。比如我发现模型老是在某个子任务上出错, 我会人工写 3 条 gold example 放进去。这招对 8B-70B 的模型特别管用, 往往能把那类错误率压掉一半。我之前在一个分类任务上试过, 加 3 条针对性 example 之后, 目标类别的 F1 从 0.61 拉到 0.79, 省了大概 47 秒的推理时间 (因为不用调超参)。

**路径三：数据增强与重训**。当错误是系统性的、prompt 救不回来的时候, 就该动训练数据了。我一般会从 bad case 里挑 100-300 条, 人工改写或补充, 混进训练集重训。这里有个坑我踩过：**别全用 bad case 改, 要保留一定比例的原始分布**。我有一次贪心, 拿 500 条 bad case 全塞进去, 模型在新 eval 上反而掉了 3 个点——过拟合到了错误模式上。后来改成 300 条 bad case + 700 条随机样本的 7:3 比例, 才稳下来。

**路径四：架构改造**。比如 RAG 召回错了, 那得改 retriever; 比如推理时漏了步骤, 那得改 agent 的 plan 模块。这是核武器, 别轻易用, 因为改架构意味着前面所有评估结果都可能作废。

# 05. 我常犯的几个错误

最后讲几个我反复犯的错误, 你们能避就避。

**错误一：看总分不看分布**。我之前有个项目, 总分从 0.81 涨到 0.83, 我开心得不行, 结果线上发现 "拒答率" 从 2% 飙到 15%——模型变 "谨慎" 了, 一堆该答的也拒了。**任何总体指标的提升, 都要拆到子指标上看**。我现在评估报告必须包含 5-8 个分维度, 缺一不可。

**错误二：用同一份 eval 集迭代 10 次以上**。你会过拟合到这份 eval 上。我在某个项目上连续迭代了 8 版 prompt, 每次都在涨, 第 8 版比第 1 版高了 4.2 个点。结果拿到线上 A/B, 反而是第 3 版效果最好。**eval 集要用两份: 一份 dev (用来迭代), 一份 holdout (用来最终决策)**。这份 holdout 从你开始第一次评估起就要锁死, 再也不能看。

**错误三：把 LLM-as-judge 当真理**。我后来发现, GPT-4 当 judge 评出来的分, 和人工标注的相关系数大概 0.7-0.85, 取决于任务复杂度。也就是说, **有 15-30% 的概率 judge 是错的**。所以我不会只看 judge 的分, 还会每月抽 100 条让真人复审。这个习惯救过我好几次。

**错误四：没有 baseline 对照**。你改了一版, 涨了 1 个点, 但你不知道这是真的涨还是随机波动。我的经验是, 同一个模型跑 3 次取均值, 波动范围一般在 ±0.5 个点以内。如果涨幅小于 1 个点, 我会当作噪声处理, 不上线上。

说了这么多, 其实核心就一句话: **评估不是交付物, 是对话**。它和模型之间应该是一个持续的、双向的拉扯过程, 而不是一次性的体检报告。当你把评估结果当作 "下一步该改什么" 的信号源, 而不是 "做得好不好" 的判决书时, 你才算真正用上了它。
