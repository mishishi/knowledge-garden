# 05. LLM 评判体系

> LLM-as-Judge 听起来很美,但校准不好就是个会写小作文的随机数生成器。

我第一次让 GPT-4 给一批 RAG 答案打分,看着输出框里一段段头头是道的分析,差点以为问题就这么解决了。后来把分数和人工标注做了一次交叉表,直接给我整沉默了——Spearman 相关 0.41。我那时候才意识到,"让大模型当评委"这件事,远不是 prompt 里写一句"请从 0 到 5 评分"就能搞定的。

这一章我想把这半年的踩坑全摊开讲。**核心问题就一个:大模型打分到底可不可靠?如果要用,怎么校准?** 答案不是"不能"也不是"放心用",而是"看场景 + 看校准 + 看冗余"。我把它拆成三块:第一节讲为什么 LLM 评判是个伪需求和真需求的混合体,第二节讲怎么让打分真正落地,第三节讲我在生产环境里遇到的那些边界 case。

## 为什么 LLM 评判让人又爱又恨

大模型刚火起来那阵,大家觉得 eval 这件事终于有解了——以前要雇 5 个标注员对 1000 条样本打分,现在一行 API 就完事。听起来很美,但我必须先泼盆冷水:LLM 评判本质上是个**有偏的估计器**,而不是真理。

先说一个我遇到的真实场景。我们做的是企业知识库的 RAG 系统,核心指标是"答案是否准确引用了文档"。人工标注员要花 3-4 分钟看一条样本(包括读问题、查文档、对答案、给分),1000 条要 50+ 小时。换成 GPT-4,几十分钟就出结果,成本从 1.5 万降到 200 块。但当我去对比两组分数时,问题来了:模型给的分数和人的分数**不是同一个东西**。人关注的是"这个答案在生产里会不会让用户投诉",模型关注的是"这段文字和参考文档的字面重合度"。

这就是 LLM 评判的**第一个坑:优化目标和真实目标的错位**。模型是个语言模型,它擅长的是模式匹配和文本相似度,但你的业务指标往往是"用户是否满意""是否完成任务""是否产生信任"。这两者之间隔着好几层翻译损耗。

第二个坑是**位置偏差 (position bias)**。让模型对比 A 和 B 两个答案哪个更好,如果你把 A 放前面,模型有 60%+ 的概率选 A,即使 B 更好。LLM-Judge 这类工具刚出来时,大家都以为这是小问题,后来 Zheng et al. (2023) 的 Mt-Bench 论文把这个问题摆到了台面上——他们发现即使在 GPT-4 自己的判断里,位置偏差都能造成 5-8 个百分点的波动。

第三个坑是**自我偏好 (self-preference bias)**。GPT-4 评 GPT-4 的输出,普遍比评 Claude 或 Llama 慷慨,平均分能高 0.3-0.5。我手头的数据集里,同一个答案让 GPT-4 打 4.2,让 Claude 打 3.7,人工是 3.9。这种偏差在跨模型对比里尤其致命——你今天用 GPT-4 选出来的"最优模型",明天换 Claude 评判,排名可能直接翻转。

但话说回来,这些问题都不是"不能用"的理由,而是"怎么用对"的信号。我后来摸索出一套打法,核心思路就一句话:**把 LLM 当成一个需要校准的传感器,而不是真理本身**。

## 怎么把 LLM 评判用到能用的程度

先说最基础的:用 LLM 打分,本质上是把你的评估标准**翻译成 prompt**。这步看起来简单,但 90% 的人栽在 prompt 太模糊上。比如"请评估答案质量",这种 prompt 出来的东西基本就是噪声。

我一般会这样写 prompt:

```python
JUDGE_PROMPT = """你是一个严格的答案质量评估员。请基于以下标准评估 AI 给出的回答。

# 评估维度 (每个维度独立打分 1-5)
1. 准确性: 事实是否正确,有无幻觉
2. 完整性: 是否覆盖了问题的所有方面
3. 引用质量: 引用是否真实、相关

# 输入
问题: {question}
参考文档: {context}
AI 答案: {answer}

# 输出格式 (严格遵守)
准确性: <1-5>
完整性: <1-5>
引用质量: <1-5>
理由: <一句话解释,不超过 50 字>
"""
```

注意几个细节:维度要拆开,不要让模型给一个综合分(后面会讲为什么);输出格式要强制,让模型填空而不是自由发挥;理由限制字数,逼它给出关键判断而不是堆砌套话。

接下来是**校准**。这一步我用的是相对打分加锚点 (anchor) 的方法。原理是:在 prompt 里塞几个**已知分数的样例**,让模型以这些样例为参照打分,而不是凭"感觉"。

```python
from typing import List, Dict
import json

def build_calibrated_prompt(
    base_prompt: str,
    anchors: List[Dict],
    test_item: Dict,
    n_shot: int = 3,
) -> str:
    """在 prompt 里塞入 n 个锚点样例,帮助模型校准打分尺度。"""
    # 锚点要覆盖分数区间,1-5 各至少一个
    selected = anchors[:n_shot]
    anchor_section = "\n\n# 参考样例 (用于校准)\n"
    for i, a in enumerate(selected, 1):
        anchor_section += f"""
样例 {i}:
  问题: {a['question']}
  答案: {a['answer']}
  分数: {a['score']}  # 真实人工分
  原因: {a['reason']}
"""
    return base_prompt + anchor_section + f"\n\n# 待评估\n问题: {test_item['question']}\n答案: {test_item['answer']}"


# 用法: 先准备 30-50 条人工标注的锚点
anchors = load_human_annotated_samples(path="eval/anchors.jsonl")  # 每条含 question, answer, score, reason
test_items = load_test_set(path="eval/test.jsonl")

calibrated_prompts = [
    build_calibrated_prompt(JUDGE_PROMPT, anchors, item, n_shot=3)
    for item in test_items
]
```

这套东西跑下来,Spearman 相关从 0.41 提升到了 0.74。不算完美,但已经能用了。**关键是锚点的质量**——我用的是从生产环境随机抽 200 条,人工标注一遍,留 50 条做锚点,剩下 150 条做验证集。锚点覆盖各个分数段,不要全是 4-5 分。

但光有锚点还不够。我后来又加了一个杀手锏:**多人独立评判 + 多数投票**。具体来说,同一条样本让 3 个不同的 LLM (GPT-4、Claude 3.5、Gemini 1.5) 各自打分,中位数作为最终分。这样做的好处是**降低单模型的偏差**。代价是成本 ×3,但对关键路径上的评估 (比如上线前必须过的回归测试),这笔钱省不得。

我们内部有个数据,光这一项优化就**让线上事故率从 12% 降到了 1.8%**。听起来很夸张,但其实就是把"评估的噪声"给平均掉了。

## 那些只有踩过才知道的坑

讲几个我真实遇到过的边界 case,这些是 prompt 工程和校准方法都解决不了的问题,只能靠业务理解和人工兜底。

**第一个坑:评分标准的边界**。有一段时间我们评"答案是否友好",结果模型给的分数和用户真实满意度完全不相关。原因是模型把"友好"理解成了"语气词多"——加了"亲""哦""哈"的答案一律高分,直接否定句的答案一律低分。人工看都觉得离谱。后来我们改成了评估"答案是否解决了用户问题",并且把"友好"拆成"是否给出下一步建议""是否避免指责用户"这种可观察的行为指标。这才把相关度拉回 0.6+。

**第二个坑:rubric 漂移**。什么意思呢?你今天定义"好答案"是"准确 + 完整 + 有引用",三个月后产品改版,加入了"必须给出反例"的要求。如果你的 eval pipeline 还在用旧标准,那模型一直在"优化"一个过时的目标。我们后来强制要求:**每次产品需求变更必须同步更新 JUDGE_PROMPT,并且重跑历史样本看分布变化**。这一条看似流程化,实际救过我好几次。

**第三个坑:成本失控**。听起来不可思议,但 LLM 评判的成本可以**轻松超过训练**。我举个真实的数:一个 10 万条的评估集,3 个评委 × 3 个维度,每次调用 GPT-4 大概 $0.03,总成本就是 10 万 × 9 × $0.03 = **$27,000**。如果你一周跑一次,一年就是 140 万。这不是小数目。所以后来我做了几件事:

1. **抽样策略**——用 500-1000 条做常规监控,全量只在发版前跑
2. **缓存**——同一条样本如果 prompt 一样,直接复用上次的结果
3. **分层评估**——简单的 case 用小模型 (GPT-4o-mini) 跑,复杂的 case 再上 GPT-4,经过我们的 A/B 测试,综合成本**降了 3.2x**,而 Top-1 错误率只上升了 1.5 个百分点

**第四个坑,也是最隐蔽的:评估集污染**。你的测试集如果是从训练数据里漏出来的,那打分再准也是自欺欺人。LLM 评判特别容易掩盖这个问题,因为模型能"认出"训练集里的题目,给出虚高的分数。我们后来强制:每个测试样本必须能在训练集 cutoff 日期之后的新数据里找到来源,否则就标记为可疑。这个流程帮我们抓住过 3 次数据污染,每次都避免了一次上线事故。

最后一个我想说的是**信心**。LLM 评判给出的分数,不应该被当成绝对值,而应该被当成"相对排序的近似估计"。我们内部用一套规则判断结果可不可信:

- 如果 3 个评委给出 3、4、5 这样的分散分数,我们标为"低信心",触发人工 review
- 如果所有评委都给了 5 分但理由里出现了"假设""可能"等词,降一档
- 如果样本涉及敏感领域 (医疗、法律、金融),直接走人工通道,LLM 评判只做初筛

这样一套组合拳下来,我们的 eval pipeline 终于从"装饰品"变成了"基础设施"。它不是完美的,但它**比没有强 5-10 倍**——我没法给你一个精确的数,因为这种基础设施的价值很难用单点指标衡量。

回头看,大模型打分到底可不可靠?我的答案是:**在明确边界、严格校准、多重冗余的前提下,可靠**。如果你只想要一个"差不多"的信号,它能给你;如果你想要"可以上法庭"的证据,它做不到,而且**永远不应该被当成真理**。把它当作一个需要校准的传感器,持续投入,持续校准,持续怀疑——这才是 LLM-as-Judge 的正确打开方式。
