# 06. 指标体系搭建
> 把业务目标拆成可测指标的那条线，比想象中要细得多

去年我第一次给一个 RAG 系统搭指标体系的时候，信心满满。业务方说"我们要提高答案质量"，我就建了一个 dashboard，上面挂了一堆向量召回率、生成多样性、BLEU 分数。第二天 PM 走过来说："这些数我看了，但不知道该把哪个调上去。"我愣了三秒。

那一秒是我开始认真对待"指标体系"这件事的起点。后来的半年里我推翻了三次表格，把一个看上去很完整的指标矩阵砍到只剩 7 个字段，效果反而好了。这一章我想把这条路讲清楚——怎么从一句模糊的业务诉求，走到一排你真的能拿来决策的数字。

## 从业务目标到指标的三个断层

"提高答案质量"和"答案质量 +12%"之间不是同义反复，而是隔着三层翻译。每漏掉一层，做出来的指标都会偏。

第一层是**定义口径**。什么叫质量？是用户满意？是事实正确？是覆盖全面？我第一次做的时候就栽在这里：我在 Notion 里写"答案质量 = LLM-as-judge 打分 ≥ 4"，但 PM 想要的是"用户能直接拿去用的比例"。这两个定义差了至少三个数量级——judge 觉得好的答案，用户复制粘贴后还是要改半天。

后来我学会了一个笨办法：去找 5 个真实的"好样本"和 5 个真实的"坏样本"，让业务方逐个说为什么好、为什么坏。我那个项目里，业务方眼里"好"的样本，60% 是因为"格式刚好契合他们下游的表格模板"，跟 LLM judge 看的内容质量一点关系都没有。这件事不坐下来聊，根本发现不了。

第二层是**指标分类**。一个 RAG 系统至少有四个层级可以打指标：

- 离线层：召回率、MRR、NDCG、生成多样性
- 在线层：点击率、改写率、停留时长
- 任务层：用户任务完成率、单次会话成本
- 业务层：续费率、客单价、人工介入率

我把这些抄在一张表里之后发现一个尴尬的事：我做了一年这个产品，所有人盯的都是离线层那两列，而真正的业务结果其实是任务层和业务层在驱动。离线指标的提升和业务结果的相关性，我没测过；后来粗算了一下，大概是 0.3 左右。这意味着我把 100% 的精力投在了只能解释 30% 业务变动的指标上。

第三层是**量级与可比性**。一个常见错误是把不同量纲的指标堆在同一个图里。"准确率从 0.72 升到 0.78"和"响应时间从 1.4s 降到 1.1s"——这两个数字看起来都是"涨了 / 降了"，但它们对一个具体决策的指向完全不同。我现在强迫自己做的第一件事是给每个指标定一个"基线值"和"目标值"，中间划三档：恶化 / 持平 / 改善，再附上一个"这个档位意味着业务要采取什么动作"的备注。

## 一段代码：业务目标 → 指标拆解

拆解这件事如果只靠脑子想，很容易做成"我以为是"。我后来写了一个小工具，强迫自己把每个目标都走一遍拆解流程，落到字段上。核心想法是：业务目标本身不能直接当指标，必须先拆成可观测的行为，再把行为翻译成数据字段。

```python
# 把业务目标拆成可测指标的小工具
from dataclasses import dataclass, field
from typing import List

@dataclass
class Metric:
    name: str                 # 指标名
    layer: str                # 离线 / 在线 / 任务 / 业务
    formula: str              # 计算口径
    baseline: float           # 当前基线
    target: float             # 目标值
    decision_rule: str        # 该指标触发什么动作

@dataclass
class Goal:
    statement: str            # 业务方原话
    decomposed_to: List[Metric] = field(default_factory=list)

def decompose(goal: Goal, metrics: List[Metric]) -> Goal:
    # 要求：每个业务目标必须被拆到至少 2 个不同 layer
    layers = {m.layer for m in metrics}
    assert len(layers) >= 2, f"目标 '{goal.statement}' 只拆到了单一 layer，拆解失败"
    goal.decomposed_to = metrics
    return goal

# 实际使用
goal = Goal(statement="提高答案质量，让用户能直接复制使用")
goal = decompose(goal, [
    Metric("首条采纳率", "在线", "首次回复即被复制 / 首次回复数",
           0.41, 0.60, "采纳率 < 0.35 时回滚到上个 prompt 版本"),
    Metric("改写率", "在线", "用户二次输入且修改了首条 / 总会话数",
           0.38, 0.20, "改写率 > 0.30 时启动答案模板专项 review"),
    Metric("任务完成率", "任务", "标记为完成且无人工接管 / 总会话数",
           0.62, 0.80, "完成率 < 0.70 时召集业务方做案例复盘"),
])
print(f"{goal.statement} -> {len(goal.decomposed_to)} 个指标")
```

这段代码看起来挺土，但它强制我回答两个问题：第一，这个目标能不能拆到至少两个不同的层级；第二，每个指标触发什么动作。我后来做指标评审的时候，每次 PM 提"我想加一个指标"，我都拿这个工具跑一遍；如果拆不到两个 layer，就打回重想。

顺便说一句，rule 里写"采纳率 < 0.35 时回滚"这种话，看起来像废话，但实际跑系统的时候非常救命。我曾经在一次 prompt 调整后看到采纳率从 0.41 升到 0.49，正在高兴，结果改写率从 0.38 涨到 0.55——意味着用户拿到答案后大量二次改写，根本没用。这个 case 让我后来在指标体系里永远要同时看"采纳"和"改写"两个面，单看任何一个都会骗人。

## 指标之间的张力与边界

指标搭起来之后，真正的麻烦不是缺指标，而是指标打架。

我印象最深的一次，是做一个企业内部知识助手。PM 希望"召回更多相关内容"，算法同事希望"提升 Top-1 准确率"。这两件事在多数场景下是反的：召回多了，Top-1 容易掺水；Top-1 准了，召回就会收紧。我们三个人为这个吵了三轮，最后定下的折中方案是"Top-3 召回率必须 ≥ 0.92，Top-1 准确率必须 ≥ 0.68"，两个都看，谁掉到阈值外就阻塞发版。

这件事让我后来养成了一个习惯：每个核心指标都要有一个"对手指标"。比如：

- 召回率 vs Top-1 准确率
- 采纳率 vs 改写率
- 响应时间 vs 答案完整度
- 多样性 vs 一致性

指标体系如果只有"我想达成的"，没有"我可能要付出的代价"，做出来的产品往往是单点最优、全局次优。我们那个项目上线三个月后统计，发现单纯追"Top-1 准确率"的版本虽然单项高了 3.2 个点，但用户的整体任务完成时间反而从平均 47 秒涨到了 71 秒——因为他们要反复换 query 才能拿到想要的答案。

我踩过另一个坑是**指标的"采样偏差"**。我做用户反馈数据的时候，前三个月只看打过分的用户，发现好评率 87%。后来一查，发现只有 12% 的用户会主动打分，剩下的 88% 沉默用户里抽样看，真实满意率大概在 54%。这个差距让我后来做指标体系多了一条铁律：任何用户自报数据，都必须问"分母是什么、采样是否偏置"。如果是 NPS 这种主动反馈，得分高不代表真的好，只是打分的人本来就更愿意打高分。

还有一个边界问题：**指标体系的保质期**。我曾经维护过一张指标表，用了 11 个月没动过，结果在第 12 个月发现它的核心指标已经完全失真。原因很简单：业务场景变了。最初我们做的是"通用问答"，指标围着"覆盖广度"和"准确率"打；后来业务转到"合同审查"，核心指标变成"漏检条款数"和"误报条款数"，原来的指标一个都用不上。我后来给自己定了一个规矩：每季度至少做一次"指标体系健康度检查"，问三个问题——这个指标还在被谁看？这个指标还能区分好坏版本吗？这个指标的提升还和业务结果相关吗？任何一个答"否"，就标记为待清理。

## 怎么判断指标体系本身好不好

这件事我想留到最后讲，因为大多数人搭完指标就完了，从不回头看指标体系本身的质量。

我自己用的一个内部 checklist，六个问题：

第一，**每个指标都有明确的反例吗？** 也就是说，给一个具体的失败场景，指标值会明显变差。如果指标对好坏版本反应迟钝，这个指标没用。

第二，**指标之间有没有重复解释？** 如果两个指标的相关性长期高于 0.85，大概率有一个是冗余的，砍掉一个决策反而更清楚。

第三，**业务方能背出至少三个指标吗？** 如果业务方一个都背不出来，说明指标体系和他们的认知是脱节的。我那个项目做完第二版后，让 PM 闭眼写三个她记得的指标，她写出了"首条采纳率、任务完成率、改写率"——这是我最开心的瞬间之一。

第四，**指标的基线和目标值是怎么定的？** 我见过太多指标表，目标值是"提升 10%"这种空话。基线和目标必须来自历史数据 + 业务访谈，不能拍脑袋。

第五，**指标变化能归因吗？** 一个指标从 0.72 跌到 0.65，你知道是什么引起的吗？如果归因链条拉不到具体动作，这个指标的诊断价值就有限。

第六，**这个指标体系本身能不能被评估？** 比如把它放在两个相似项目上，输出的指标集合是否稳定？我之前做过一次实验，把同一个产品换到两个不同的用户群，指标体系跑出来的 Top-5 关键指标居然有 3 个不一样——这意味着指标体系本身就和人群耦合，不是普适的。后来我就把指标分成了"通用层"和"场景层"，通用层稳定，场景层可调。

写到这里我想说一句可能会被吐槽的话：**好的指标体系不是搭出来的，是砍出来的。** 我最开始给那个 RAG 系统做了 23 个指标，跑了一个月，最后留下来的只有 7 个。被砍掉的那些不是没价值，而是它们之间的相关性太高、解释力重叠、或者业务方根本不会看。

我有个不太成熟的体会：指标体系的成熟度，跟指标数量成反比。一个团队如果 dashboard 上挂了 30 个图表没人看，那这个团队其实没有指标体系——只有一堆数。一张能被三个人同时讲清楚、并且能驱动具体决策的表，比一百个炫酷的图都值钱。

下一章我想聊聊另一个更隐蔽的问题：当你有了指标，怎么知道指标的"提升"是真的提升，而不是指标在骗你。LMS 随机化、A/B 桶设计、显著性检验，这些东西听起来工程化，但在 AI 系统里特别容易被绕过。
