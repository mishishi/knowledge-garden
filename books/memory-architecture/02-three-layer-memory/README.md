# 02. 三层记忆架构

长期记忆不是"一个超大数据库"，而是**3 层不同性质的存储 + 互相协调**。这一章讲清 3 层架构的设计原理、各自的存储选型、互相怎么协作。

## 整体架构图

```
┌─────────────────────────────────────┐
│ Working Memory（工作记忆）            │
│ - 存储：当前会话 context window      │
│ - 生命周期：单次会话                 │
│ - 容量：100K-1M token                │
│ - 速度：< 10ms 访问                  │
└─────────────────────────────────────┘
                ↑↓
┌─────────────────────────────────────┐
│ Episodic Memory（情景记忆）           │
│ - 存储：原始事件流（问答、行为、结果）│
│ - 生命周期：永久                     │
│ - 容量：TB 级                        │
│ - 速度：< 100ms 检索                 │
└─────────────────────────────────────┘
                ↑↓
┌─────────────────────────────────────┐
│ Semantic Memory（语义记忆）           │
│ - 存储：抽象知识、规则、偏好          │
│ - 生命周期：永久                     │
│ - 容量：GB 级（提炼后的精华）         │
│ - 速度：< 50ms 检索                  │
└─────────────────────────────────────┘
```

数据流：

```
[用户输入] → Working Memory（处理）
                ↓ 重要事件
            Episodic Memory（存储原始事件）
                ↓ 周期提炼
            Semantic Memory（提炼抽象知识）
                ↓ 检索时注入
            Working Memory（增强上下文）
```

## Working Memory（工作记忆）

**是什么**：当前对话的 context window。

**存储**：LLM 自身的 KV cache，不需要额外存储。

**生命周期**：单次会话。session 结束清空。

**容量**：100K-1M token（Claude Opus 4.7 是 200K，GPT 是 128K）。

**速度**：< 10ms 访问。

**典型数据**：

- 当前用户问题
- 当前 agent 回复
- 当前会话的工具调用历史
- 短期任务状态

**优化手段**：

- **Compaction** — 长会话定期压缩：把 100K token 的旧对话压缩成 5K token 的摘要
- **Pruning** — 删除无关紧要的中间步骤
- **Recitation** — 把关键信息"背诵"到 context 前面，防止遗忘

## Episodic Memory（情景记忆）

**是什么**：具体的"事件"，原始的、不可变的、可追溯的。

**存储**：通常用向量数据库（pgvector / Pinecone / Weaviate / Milvus）+ 关系数据库（PostgreSQL / SQLite）。

**生命周期**：永久。事件一旦写入就不改。

**容量**：TB 级。1 个用户 1 年可能产生 10 万条事件。

**速度**：< 100ms 检索（向量检索 + 元数据过滤）。

**典型数据**：

```
{
  "event_id": "evt_20260601_001",
  "user_id": "u_123",
  "timestamp": "2026-06-01T10:23:45Z",
  "type": "question",  // question / action / observation
  "content": "上海地区 25-35 岁违约率是多少",
  "embedding": [0.23, -0.41, ...],  // 1536 维向量
  "metadata": {
    "agent": "risk-analyst",
    "tools_used": ["postgres_query"],
    "result_summary": "违约率 2.3%",
    "user_satisfaction": 4.5,
    "tags": ["risk", "shanghai", "demographics"]
  }
}
```

**关键特征**：

- **不可变** — 写入后不改（append-only），保证可追溯
- **带时间戳** — 所有事件都有时间，方便按时间窗口检索
- **带 embedding** — 支持语义检索（"类似的问题"）
- **带 metadata** — 支持结构化过滤（"上海地区""2026 Q1"）

## Semantic Memory（语义记忆）

**是什么**：从情景记忆提炼出的抽象知识、规则、偏好。

**存储**：通常用知识图谱（Neo4j / Memgraph）+ 键值存储（Redis）+ 关系数据库。

**生命周期**：永久，但会随时间更新（知识会过时）。

**容量**：GB 级（提炼后的精华）。

**速度**：< 50ms 检索。

**典型数据**：

```
// 用户偏好
{
  "user_id": "u_123",
  "type": "preference",
  "key": "coffee",
  "value": "不喝",
  "confidence": 0.95,
  "source_events": ["evt_20260115_003", "evt_20260320_007"],
  "last_updated": "2026-03-20"
}

// 项目规范
{
  "project_id": "p_456",
  "type": "convention",
  "key": "naming.api",
  "value": "camelCase",
  "confidence": 1.0,
  "source_events": ["evt_20260210_001"]
}

// 通用知识
{
  "type": "world_knowledge",
  "key": "上海地区人口",
  "value": "约 2500 万",
  "confidence": 0.9,
  "last_updated": "2026-01-01"
}
```

**关键特征**：

- **抽象性** — 不是具体事件，是提炼出的规律
- **可更新** — 知识会过时（用户改口、项目规范改了）
- **可推理** — 知识图谱支持关系推理（"用户不喝咖啡 → 用户喜欢清淡 → 推荐龙井茶"）
- **可追溯** — 每条知识都标注来源（哪些事件提炼出来的）

## 3 层的协作

**写入路径**：

```
[用户输入]
   ↓ 进 working memory
[LLM 处理 + Agent 执行]
   ↓ 关键事件（如用户表达偏好、完成任务、出错）
[写 episodic memory]    ← 原始事件
   ↓ 周期任务（每日 / 每周）
[提炼 semantic memory]  ← 抽象知识
```

**读取路径**：

```
[新用户输入]
   ↓
[检索 semantic memory]  ← 抽象偏好 / 知识（少量 token）
   ↓
[检索 episodic memory]  ← 相关历史案例（按需）
   ↓
[组装 working memory]   ← 注入 context
   ↓
[LLM 处理]
```

## 关键设计决策

**决策 1：什么写 episodic？什么写 semantic？**

我的经验法则：

- **原始事件全写 episodic**（成本低，10 万事件才几 GB）
- **明确表达的偏好 / 规则才写 semantic**（成本高，每次更新都要更新图谱）

**决策 2：semantic 怎么更新？**

3 种策略：

- **覆盖**——新值直接覆盖旧值（适合明确修改）
- **版本化**——保留所有历史版本（适合可追溯场景）
- **置信度加权**——新旧值同时存，按置信度使用（适合不确定场景）

**推荐版本化**。理由：AI 提炼的知识经常错，保留版本方便回溯。

**决策 3：episodic 的保留期**

3 种策略：

- **永久保留**（GDPR 合规复杂）
- **按时间保留**（通常 1-3 年）
- **按重要性保留**（自动判断哪些事件重要、保留哪些）

**推荐按时间保留 2 年 + 重要事件永久**。GDPR 要求用户可删除，2 年后自动清理普通事件。
