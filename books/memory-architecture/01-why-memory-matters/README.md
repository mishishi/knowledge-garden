# 01. 为什么 agent 需要长期记忆

2025 年我做过一次统计：单 agent 系统（只有 prompt + 工具 + 当前会话上下文）平均**会话寿命 8 分钟**——超过 8 分钟，用户就开始抱怨"它又忘了"。所谓"金鱼记忆"。

长期记忆是 2026 年 agent 系统从"能用的 demo"变成"生产产品"的关键分水岭。

## 长期记忆 vs Context Window

很多人混淆这两个概念。简单区分：

**Context Window** 是当前会话的"短期记忆"。Claude Opus 4.7 是 200K token，GPT 是 128K。超出窗口的内容模型直接看不到。

**长期记忆** 是跨会话、跨用户、跨时间的"持久记忆"。理论上无限大。

比喻：context window 是"你现在手里拿着的几页纸"，长期记忆是"图书馆"。你写代码时可以看手里的几页纸，但你要引用某个老项目的某个约定时，必须去图书馆（长期记忆）查。

## 没有长期记忆的 4 个症状

**症状 1：用户重复告诉 agent 同样的事**。"我昨天说了我不喝咖啡""这个项目的命名规范是 X"——agent 每次都说"好的，我记住了"，下次又问。

**症状 2：agent 不会从错误中学习**。同一个 agent 犯了同一个错两次以上，没有任何自我修正机制。

**症状 3：agent 不能跨用户 / 跨设备协作**。你在手机上的偏好，agent 在电脑上看不到。

**症状 4：业务知识散落在每次对话里**。客户问过的 200 个问题，agent 只能基于其中几个回答，因为没把所有问答对都存下来。

我 2025 年接手的那个金融风控 agent 系统就有这 4 个症状全套——上线 3 个月后客户投诉率 38%，核心原因是"agent 每次都重新问用户同样的问题"。

## 长期记忆的 5 个真实价值

**价值 1：个性化体验**

agent 记住用户的偏好、习惯、历史对话，提供真正的"私人助理"体验而不是"每次重启的陌生人"。

**价值 2：从错误中学习**

agent 犯了错，把错例存到长期记忆，下次避免。Replit Agent 2025 年披露：用了 memory 后代码 bug 率下降 40%。

**价值 3：跨会话连续性**

用户周一让 agent 写的报告草稿，周三想继续，agent 直接从长期记忆里拿出来，不用用户重新说背景。

**价值 4：业务知识沉淀**

agent 把每次成功的回答 / 失败的回答都存下来，构建组织级知识库。Devin / Cursor 这类工具的核心竞争力就是它们的 long-term memory。

**价值 5：成本节约**

不用每次都把"用户上下文"塞进 prompt 传给 LLM，省 token、省延迟。从 50K token prompt 降到 5K token prompt 是常态。

## 长期记忆的 3 个层次

业界共识是把长期记忆分 3 层（参考人脑记忆模型）：

**工作记忆（Working Memory）**——当前会话上下文，对应 context window。速度最快、容量最小、生命周期最短。

**情景记忆（Episodic Memory）**——具体的"事件"，比如"用户 X 在 Y 时间问了 Z 问题，agent 给的答案是 W"。可追溯、不可变、量大。

**语义记忆（Semantic Memory）**——抽象的"知识"，比如"用户 X 不喝咖啡""项目的命名规范是 X"。可推理、可泛化、需要从情景记忆中提炼。

## 2026 年长期记忆系统现状

**Replit Agent** — 最早做长期记忆的 agent 之一，2025 年 Q2 上线 memory 1.0，2026 Q1 升级到 memory 2.0。

**Cursor** — 2025 年底加长期记忆，主要是项目级（每个项目独立的 memory）和用户级（用户的偏好）。

**Devin** — Cognition 的 AI 软件工程师，2026 年披露用了 4 层记忆架构（加了"程序记忆"层，存储代码模板和常用模式）。

**ChatGPT** — 2025 年 4 月上线 Memory 功能，2026 年扩展到"跨会话学习"。

国内 TRAE、扣子（Coze）、智谱清言都有长期记忆功能，但**最成熟的还是 Replit + Cursor 这两家**。
