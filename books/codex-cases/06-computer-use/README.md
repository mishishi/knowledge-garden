# 06. Computer Use 实战：Codex 接管你电脑的能力图谱

2026 年 5 月 29 日，OpenAI 发布 Codex App v26.527，**Windows 平台正式获得 Computer Use 能力**。这一章讲清楚这个能力到底是什么、它实际能做什么、不能做什么、独立开发者怎么用。

## Computer Use 不只是"AI 看屏幕"

我之前以为 Computer Use 就是"AI 截图 + 点按钮"，错了。Codex Windows Computer Use 实际是**三层能力的组合**：

**第一层：感知（看见）**

实时截图，理解当前界面状态。**这跟 RPA 完全不同**——RPA 依赖 UI 元素选择器，界面一变脚本就死；Codex 是基于视觉理解，每次重新判断当前状态。

实测：Codex 在某个企业内部 ERP 系统的"反审批"流程里，**界面升级 3 次它都能照样工作**，因为它不是按坐标点，是按"语义"理解。

**第二层：操控（动手）**

- 移动鼠标、点击按钮、菜单、对话框
- 模拟键盘输入
- 跨应用工作（浏览器 / IDE / 企业软件 / 遗留系统）

**只要屏幕能显示，就能操控**。这打破了一个关键限制：**"必须有 API 才能自动化"**。

**第三层：安全沙箱**

- 不能在终端或 Codex 自身内运行
- 不能以管理员身份执行
- 操作账户、支付、凭证相关设置时**主动暂停**并询问

官方明确表示 Codex 涉及安全操作会**强制人工确认**。这个边界是写死在 app 里的。

## Windows 版 vs Mac 版的关键差异

很多国内开发者关心 Windows 版。Codex Windows Computer Use 跟 Mac 版有 3 个关键区别：

**差异 1：运行模式**

- **Windows：前台运行**，必须接管鼠标键盘
- **Mac：支持后台 / 锁屏运行**

Windows 版要等任务跑完才能用电脑。这是最重要的限制。

**差异 2：任务期间能否继续用电脑**

- Windows：**不能**，必须等
- Mac：可以锁屏后继续

Mac 用户的体验"AI 在后台跑、人在前面做事"Windows 暂时做不到。

**差异 3：安全沙箱**

- Windows：PowerShell Sandbox / WSL2
- Mac：macOS 沙箱

两者安全性相当，但 WSL2 模式下文件隔离更彻底。

## 3 个 Windows 前台限制的应对方案

我自己试 Windows 版时，前台限制最烦。三个解决方案：

**方案 1：手机 App 远程监控**

OpenAI 同步上线了手机远程控制 Windows 的功能。**手机当指挥台，PC 专心跑任务**。

流程：

1. PC 启动 Codex 跑 Computer Use 任务
2. 你离开 PC，用手机 App 监控
3. 任务需要确认时，手机弹出通知 → 你点确认
4. 任务继续

5 月 14 日 ChatGPT 手机 App 远程控制 Mac 上线；5 月 29 日同步支持 Windows。

**方案 2：在 Windows VM 里跑 Codex**

如果你有多台电脑，把 Codex 装在**虚拟机**里，Computer Use 接管的是 VM 桌面。**你主桌面完全不受影响**。

实测：Win11 Hyper-V 开 Ubuntu 22.04 VM + WSL2 Codex，跑 Computer Use 任务，主机 0 卡顿。代价是 VM 占内存（建议 8GB+）。

**方案 3：远程桌面 / 云 PC**

用 Azure 云 PC / AWS Workspaces / 阿里云无影。**Codex 跑在云端，本地远程桌面连过去看进度**。

这个方案最稳，但成本最高。**适合企业 / 团队场景**。

## 实战 1：截图转前端代码

Codex Computer Use 最让我惊艳的能力是**截图转代码**。官方在 developers.openai.com/codex/use-cases/frontend-designs 给了一整套工作流。

**第一步：把截图丢给 Codex**

```bash
codex chat "用项目里的设计系统组件，根据我提供的截图实现这套 UI：
- 截图路径：/Users/xxx/Desktop/dashboard.png
- 桌面端 1440px 和移动端 375px 都要响应式
- 用项目里既有的 Tailwind 配置和颜色 token"
```

**第二步：Codex 自动写代码 + 调浏览器**

Codex 调 Playwright 打开真实浏览器，**截图跟原图对比**，发现不对继续改。**这形成自检循环**：

```
Codex 生成代码
   ↓
Playwright 打开浏览器
   ↓
截图
   ↓
跟原图对比（像素 + 视觉）
   ↓
发现差异 → Codex 改代码 → 回到 Playwright
   ↓
（循环直到匹配）
```

**第三步：人 review**

最后 Codex 提交 PR，你 review。**Codex 不是一锤子买卖，是带验收的迭代**。

**关键经验：截图越丰富结果越准**

一张截图起步，但只有一张截图，Codex 能猜的地方就多。建议**多状态覆盖**：

- 桌面端的样子
- 移动端的样子
- 鼠标悬停效果
- 选中状态
- 数据为空状态
- 加载中状态

**这些不需要精确稿，粗略截图都行**——关键是让 Codex 对间距、层级、整体方向有清晰认知。

## 实战 2：Windows 应用 UI 测试

我跑过这个流程 5 次，**最有用**。

场景：新开发用户 dashboard，用 Codex 走一遍"注册 → 支付 → Dashboard"完整流程，自动截图 + 记录 bug。

```bash
codex chat "用 Computer Use 测试 dashboard：
1. 启动本地 dev server (pnpm dev)
2. 打开 Edge 浏览器访问 http://localhost:3000
3. 注册一个新用户
4. 走支付流程（用 Stripe 测试卡 4242...）
5. 进入 Dashboard
6. 每个步骤截图
7. 发现的 UI bug 写到 BUGS.md
8. 严重 bug 自动创建 GitHub Issue
"
```

**实测**：

- 跑了 47 分钟（人干这事 2 小时）
- 找到 14 个 UI bug（3 个严重）
- 浏览器不限于 Chromium，Edge / Brave / Firefox 均可
- **不依赖 DevTools，纯像素级操作**

## 实战 3：操作无 API 的遗留系统

这个场景最现实。

很多企业内部的 ERP / 财务 / 报表工具**没 CLI 没 REST API**，只有 GUI。**传统自动化做不了**。

```
场景：每天早上 9 点，把昨天销售数据从老 ERP 系统导出 CSV
       传到新 BI 系统的指定目录
```

传统 RPA 方案：

- 写脚本（依赖 UI 元素选择器）
- 系统升级 → 脚本失效 → 重写（**平均每月 1 次**）

Codex Computer Use 方案：

- 截图说明步骤（带鼠标位置标注）
- Codex 每天跑，遇到变化自己适应
- 维护成本 = 0

**对中小企业这是杀手级应用**。一堆"用 GUI 操作"的工作，**Codex 都能代理**。

## 实战 4：跨应用多步工作流

我自己每周用 1-2 次的工作流：

```
1. 在 VS Code 跑单元测试
2. 把失败测试的代码块截图
3. 把失败原因复制到 Markdown 报告
4. 在浏览器打开 GitHub 对应 Issue
5. 填写 issue 描述（含截图 + 错误信息 + 修复建议）
6. 提交
```

6 步 6 个 app，**Codex 一次跑完**。我只需要 review 最后的 issue 描述。

## 跟 RPA 工具的对比

很多人会问：这不是 RPA 吗？

| 维度 | UiPath / Automation Anywhere | Codex Computer Use |
|------|----------------------------|-------------------|
| 自动化方式 | 预设 UI 元素选择器 | 视觉理解 + 推理 |
| 界面变化适应 | 脆弱，脚本失效 | 容错，每次重新判断 |
| 开发成本 | 高（专人写脚本）| 低（自然语言）|
| 维护成本 | 高（界面变要重写）| 低（自动适应）|
| 适合场景 | 工业级 RPA、长期稳定 | 开发者场景、探索性 |
| 价格 | $5000+/年/用户 | $20-200/月（ChatGPT 订阅）|

**Codex Computer Use 不是工业级 RPA 替代品，但开发者场景里它是降维打击**。

## 5 个不能做的事

虽然 Computer Use 强大，但有几个边界：

**1. 不能在 Codex 自身内运行**（防止绕过安全策略）

**2. 不能以管理员身份操作**

**3. 不能通过安全权限弹窗**（防误授权）

**4. Windows 不能后台运行**（截止 2026 年 5 月）

**5. EEA / 英国 / 瑞士不可用**（地区限制）

第 1-3 条是安全设计，第 4 条是产品状态，第 5 条是地区法规。

## 独立开发者能用 Computer Use 做什么

**用法 1：UI 自动化测试**

每个新功能写完，让 Codex 跑一遍 E2E 测试。**开发效率 + 30%，bug 漏出 -50%**。

**用法 2：截图转代码**

新页面开发 = 设计师出图 → Codex 转代码 → 你 review。**前端时间从 4 小时降到 1 小时**。

**用法 3：竞品分析**

把竞品网站截图发给 Codex，让它逐页分析交互、布局、信息架构。**从"看 30 分钟"变成"5 分钟拿到结构化报告"**。

**用法 4：跨工具数据搬运**

把数据从老软件导到新软件，**中间不需要 API**。我前面那个销售数据搬运场景就是这个用法。

**用法 5：自己开发 SaaS 时给非技术用户做"AI 操作员"**

我有个项目 AIGrader 早期版本，教师用户不会操作数据库。**用 Codex Computer Use 录一遍操作流程**——他们点"AI 操作"按钮，Codex 自动跑整个流程。

这个思路是**"用 AI 代理代替 UI 重做"**——老旧 ERP / 桌面软件**根本不用重写**，让 AI 学会怎么用就行了。

## 风险与控制

Computer Use 的风险是**它会做错事**。3 个常见翻车：

**翻车 1：点错按钮**

Codex 视觉理解有误差，可能点错位置。**缓解：操作前 Codex 会显示"即将操作的应用列表"，人工点确认**。

**翻车 2：陷入死循环**

某个操作失败，Codex 反复重试。**缓解：设置 max_iterations**（Codex 内部默认有，但你也可以自己加个超时）。

**翻车 3：操作敏感应用误授权**

Codex 涉及"账户、支付、凭证"操作会主动暂停。**但有些边界场景它判断不准确**——比如"修改密码"它可能不认为是敏感操作。

**缓解：永远在沙箱环境跑 Computer Use 任务**。生产账号、生产数据不要让 Codex 接触。

## 我的判断

**短期（3-6 个月）**：Computer Use 成为独立开发者的**日常工具**。截图转代码、UI 自动化测试这两个场景**人手必备**。

**中期（6-12 个月）**：Computer Use 接入企业 SaaS，**成为企业 RPA 的平替**。UiPath 的"传统 RPA 客户"会开始迁移。

**长期（12+ 个月）**：**"用 AI 学会怎么用"代替"重新写 UI"**——大量没有 API 的遗留系统会被 AI 接管，**而不是被重写**。
