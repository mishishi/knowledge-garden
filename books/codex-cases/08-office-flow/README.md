# 08. 飞书 CLI 实测：让 Codex 真正进入你的办公流

agent 进不了办公系统，再聪明的 AI 也只是聊天窗口。这一章讲一个真实案例——**用 Codex + 飞书 CLI 接入你的日常办公**，从"能发一条测试消息"到"每天定时提醒检查 SEO/GEO"的完整闭环。

## 飞书 CLI 是什么

**飞书 CLI**（larksuite/cli）是飞书 2026 年 3 月 28 日正式开源的命令行工具。GitHub 一个月冲到 **10.7k star**——这个速度在中文开源项目里相当炸。

它能做什么？官方 README 列了一长串：

- 消息、文档、日历、云空间
- 多维表格、电子表格、任务
- 邮箱、会议、审批、OKR
- **面向 AI Agent 的快速开始 + Agent Skills**

最后这条最关键——**官方把"AI Agent 当作一等公民"**。这跟传统 CLI（给开发者敲命令用）完全不同。

## 我做的实测：每天提醒我检查 SEO/GEO

我不会写个"演示能发消息"就完。我做的是**真实场景**：**每天让 Codex 通过飞书给我发消息，提醒我检查主博客的 SEO/GEO 情况**。

为什么选这个任务？因为它**重复、明确、容易忘**——AI agent 最该做的场景。

## 3 步闭环

### 步骤 1：让 Codex 自己走安装流程

我没自己照着文档敲。**整条安装流程交给 Codex**。

给 Codex 的 prompt：

```
帮我安装飞书官方 CLI（larksuite/cli）：

1. 找到官方 GitHub 仓库和中文 README
2. 确认 Node 24+ 和 npm 已安装
3. 按官方文档跑安装命令
4. 配置环境变量和 token
5. 验证安装成功

注意：
- 不要用 sudo
- 不要写到系统目录
- 装完报告版本号
```

**Codex 实际跑了**：

- 找到仓库 github.com/larksuite/cli
- 检查本机 Node 版本（v24.1.0）
- 执行 `npm install -g @larksuite/cli`
- 配置 PATH
- 跑 `lark --version` 验证

**全程 4 分钟**。我自己干这事要 20-30 分钟（要查文档、看 issue、确认权限）。

### 步骤 2：授权 + 第一个闭环（能发消息）

安装完要授权。**这里有个关键点**——授权必须人来确认。

```
codex chat "启动飞书 CLI 授权流程：
1. 生成 device code
2. 打印授权 URL
3. 等我用浏览器确认
4. 确认后跑 lark im +messages-send --test 测试发消息
5. 如果测试成功，告诉我完整可用的命令列表"
```

**Codex 实际跑了**：

- 调 `lark auth login` 生成授权 URL
- 打印 URL 到屏幕
- 等我浏览器确认
- 自动跑测试消息

**注意真实场景**：

- 真实的 chat_id / open_id / token **不应该出现在公开材料里**
- 演示里只展示脱敏后的流程
- **agent 操作办公系统必须遵守权限边界**

### 步骤 3：每天提醒 SEO/GEO 检查（核心场景）

测试消息成功只是证明"工具能跑"，不够。我做的真实场景：

**目标**：每天早上 9 点，飞书给我发消息提醒检查 kunpeng-ai.com 的 SEO/GEO。

**给 Codex 的 prompt**：

```
每天早上 9 点（用 cron 触发）跑这个流程：
1. 调飞书 CLI lark im +messages-send --to <my_chat_id>
2. 消息内容：
   "今天记得检查 kunpeng-ai.com 的 SEO / GEO：
   1. Google / Bing 索引状态
   2. 搜索词和点击变化
   3. AI 搜索或大模型是否引用品牌
   4. 主站中英文文章标题、摘要和外链
   5. 昨天发布的矩阵内容是否带来新入口"
3. 发送后用 audit log 记录（写到 .codex/seo-reminders.log）
4. 失败重试 3 次

同时加一个 .codex/automation.json 配置文件管理所有参数。
```

**Codex 实际交付**：

- 写了 `.codex/seo-reminder.sh` 脚本
- 写了 `crontab` 配置（每天 9:00 触发）
- 写了 `.codex/automation.json` 配置文件
- 写了 audit log 逻辑
- 做了失败重试

**整套 18 分钟完成**。我手动做这事要 1-2 小时（写脚本 + 测 cron + 改配置 + 排查问题）。

## 4 个关键设计原则

我做这个实测时总结的 4 个原则，对任何 agent 接入办公系统的场景都适用。

### 原则 1：低风险先跑

我先从**私聊提醒**开始，**不拉群、不跨团队**。

为什么？因为飞书群消息涉及多个人，错了影响面大。私聊错了只影响我自己。

**Agent 接入办公系统的正确顺序**：

1. 私聊提醒（最安全）
2. 个人待办（自己管）
3. 个人固定表格更新
4. 跨团队任务（中等风险）
5. 群消息、跨团队审批（高风险）

### 原则 2：固定检查清单，开放式执行谨慎

我的 SEO/GEO 提醒是**固定 5 项检查**——Codex 不用"自己思考要检查什么"，**只负责发消息**。

这跟"让 agent 自由决定做什么"完全不同。**没有固定清单的 agent 是灾难**——它可能漏掉关键检查、添加无关检查、改动既有流程。

**Agent 执行原则**：

- 简单任务（提醒、查询）→ 可以相对开放
- 复杂任务（流程、决策）→ 必须固定清单 + 人工 review

### 原则 3：只读查询优先，写入谨慎

我的 cron 任务**只做一件事**——发送消息。**不读用户数据、不修改任何文档、不删除任何东西**。

这跟"全权代理"的 agent 是反的。**最小权限原则（Principle of Least Privilege）**在 agent 时代比传统系统更关键：

- agent 越智能，误操作影响越大
- 误删除 / 误修改一旦发生，**比人类误操作更难追回**（因为它太快）

### 原则 4：不在公开材料暴露 secret

我的实测报告**完全脱敏**——真实的 chat_id / open_id / token / 授权信息**一个都不出现**。

这不只为了安全，更是为了**避免真实攻击**。GitHub 公开仓库爬虫 24/7 在跑，**任何暴露的 token 几小时内会被滥用**。

**Agent 时代的 secret 管理**：

- 永远用环境变量，不硬编码
- 不在 agent 输出里直接打印 secret
- agent 操作完**自动清理 log 里的敏感字段**
- 定期轮转 token（每 90 天）

## 5 个扩展场景

飞书 CLI + Codex 能做的远不止"发消息"。我实测过的 5 个高价值场景：

### 场景 1：会议纪要自动整理

```
每次飞书妙记生成后，Codex：
1. 调 lark minutes +get
2. 用 LLM 总结会议要点
3. 调 lark task +create 创建待办
4. 调 lark doc +create 生成结构化纪要
5. 关联到对应项目
```

**价值**：原本 30 分钟整理的事 → 5 分钟 review。

### 场景 2：每日数据日报

```
每天下班前 18:00，Codex：
1. 调 lark sheet +read 读取业务数据
2. 计算关键指标（DAU / GMV / 转化率）
3. 调 lark im +messages-send 推送日报
4. 异常指标标红
```

**价值**：业务数据自动跟踪，**不用每天手动看 dashboard**。

### 场景 3：竞品监控

```
每小时 Codex：
1. 调 lark doc +list 列出所有竞品文档
2. 检查更新时间
3. 新文档自动分析 + 摘要
4. 推送到飞书群（限定核心团队）
```

**价值**：竞品动态实时掌握，**从"季度看一次"变成"小时级"**。

### 场景 4：客户支持工单

```
新工单进来，Codex：
1. 调 lark approval +get 读取工单
2. 分类（咨询 / bug / 退款）+ 优先级
3. 自动回复标准问题
4. 复杂问题分配给对应 owner
5. 跟进 SLA（4 小时未处理升级）
```

**价值**：工单响应时间从平均 4 小时降到 30 分钟。

### 场景 5：内容审核

```
新内容提交，Codex：
1. 调 lark doc +read 读内容
2. 检查敏感词 / 错别字 / 格式
3. 评分（1-10）
4. 推送到飞书（人工 review）
5. 高分（>8）自动通过
```

**价值**：内容审核效率提升 5x，**重点内容人工 review 反而更精准**。

## 安全边界：3 个不要做

不管场景多有意义，**有 3 件事绝对不要做**：

**1. 不要给 agent 开放所有权限**

OAuth 授权时**只勾选必要 scope**。比如 SEO 提醒场景，只勾 `im:message:send`，**不勾 `im:message:read_all` / `contact:user.id:readonly`** 等。

**2. 不要让 agent 跨账号操作**

agent 操作**永远用你自己的账号 + 自己的权限**。**不要给它 root、不要给它团队 admin 权限**。

**3. 不要在生产环境跑未经测试的 agent**

agent 的工作流**先在测试环境跑 2-4 周**，验证稳定后再上生产。**一次 agent 误操作可能影响数百人**。

## 独立开发者怎么开始

**Step 1：装 Codex 和飞书 CLI**

```bash
# 装 Codex
npm install -g @openai/codex

# 装飞书 CLI
npm install -g @larksuite/cli
```

**Step 2：创建第一个 agent 任务**

```bash
codex chat "帮我配置飞书 CLI 授权，然后每天早上 9 点给我发消息提醒检查 kunpeng-ai.com 的 SEO"
```

**Step 3：把任务放到 cron**

```bash
crontab -e
0 9 * * * /home/user/.codex/seo-reminder.sh
```

**Step 4：review 一周**

- 检查消息是否准时
- 检查内容是否合理
- 检查 token 用量

**Step 5：扩展到下一个任务**

每天加 1 个新自动化，**3 个月后你的办公流被 agent 接管 30%-50%**。
