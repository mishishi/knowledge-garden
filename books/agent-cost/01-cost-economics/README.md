# 01. 为什么 Agent 经济学跟 SaaS 不一样

去年 9 月我的 agent side project 周活刚过 100，OpenAI 账单是 $420。之前的 4 个月一直是 $40-60。一个月翻 7 倍。

我盯着那张账单想了 30 分钟。第一个反应是"用户涨了，账单涨不是应该的吗"。但 100 → 130 用户只涨 30%，token 涨 7 倍——这个比例对不上。

这就是我开这个系列的原因。

## SaaS 经济学是算术题

传统 SaaS 的成本模型很简单：你卖 1 个用户收 $X，服务他花 $Y，$X - $Y = 利润。$X 是订阅，$Y 是云资源（EC2 / RDS / S3），$Y 跟用户数基本线性。规模做大，单位成本还会降（共享后端、CDN 摊薄）。

给你画出来就是一张漂亮的一元二次曲线：CAC 跌、LTV 涨、毛利 70%+，投资人看了一拍桌子就签 term sheet。

我做 SaaS 那些年就是这么过的。AWS 月账单从 $800 涨到 $1.2 万，从没慌过——因为 ARPU 也跟着涨，毛利稳在 75%。

## Agent 经济学是博弈题

agent 不一样。算账的变量不是"用户数"，是 **单次任务消耗的 token 数 × 任务频率**。这两个变量 user 自己都不可见，但它们决定了 80% 的账单波动。

我那 $420 是怎么来的：3 个付费用户把我的 agent 接到他们的客服工作流里，每天 200-400 次调用，每次调用 2-5 个 LLM step。0.8M input + 0.4M output per step × 3 step per call × 300 call per day × 30 天 = 21.6 亿 token。GPT-4o 跑这个量就是 $400 上下。

这种增长不是我能用"加机器"解决的。加 EC2 没用——token 钱是 OpenAI 收的，不是我 AWS 上跑的应用服务器钱。

更糟的是 **失败也收钱**。用户问了一个问题，agent 转了 5 个 tool、调了 8 次 LLM、最后用户没拿到答案。账单已经记下了。我卖的是"agent 帮用户干活"，但用户成功不成功，钱我都要付。

## 三件跟 SaaS 反着来的事

**1. 单位成本跟用户行为强相关，不是你控制的。**

用户 A 问"今天北京天气怎么样"，5 个 token 解决。用户 B 把整个项目的代码粘进来让 agent debug，2M token。两个人都是 $9/月订阅，成本差 1000 倍。你做不做 quota？做，用户跑路；不做，你亏。

**2. 规模不摊薄。**

SaaS 加 10 个用户，后端共享那 1 台 EC2。agent 加 10 个用户，每个用户的每次 LLM call 都是新钱。100 个用户每月 1 万次 call，1000 个用户每月 10 万次 call——成本是 10 倍，不是 1 倍（可能还更高，因为大用户会拖更多 token）。

**3. 失败成本不可见。**

传统 SaaS 报错就 500 给用户，没服务器成本。agent 报错前已经烧了 token 试图自愈。reliability 问题在 agent 这儿是双重成本：用户体验差 + token 钱白花。

## unit economics 这个事要早算

我那 $420 的月份里，最大的失误不是没做 cost engineering，是没在产品上线前就算清楚 **cost per success**。

cost per success = (总 token 成本) / (成功完成任务数)

不是 cost per user，不是 cost per call，是 cost per success。我后来算了一下：

- 我那 3 个客服用户：cost per success $0.18
- 他们的 ARPU $29/月
- 假设他们每人每月用 100 次 = $0.29 per success
- 毛利 = $0.11 per success

听起来还行？再算一步：3 个用户 × 100 次 = 300 次/月。ARPU $29 × 3 = $87。但 token 成本 $420。**我在倒贴 $333。**

如果我提前算过 unit economics，我就不会用 GPT-4o 做"格式化答案"这一步。用 gpt-4o-mini 做格式化、用 4o 只做理解和规划，cost 能砍 60% 到 $170，毛利 -$83（还是亏但少亏一半），但至少不是 7 倍账单。

## 写给 side project 的话

这系列不假设你有风投。你也没几个 MAU 大到能谈判 API discount（OpenAI 给 $5M+ ARR 的客户才谈）。

我的视角是 indie 视角：

- 50-500 个用户
- 月账单 $50-$2000
- 一个人写代码 + 一个人处理支持
- 没有专职 SRE、没有 platform team

下面的 9 章都是在这个量级里能落地的做法。不是 AWS 架构师 best practice，是 **怎么在下个月账单出来前别再被吓一跳**。
