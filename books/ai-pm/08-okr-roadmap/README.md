# 08. OKR / 路线图 / 节奏: AI 产品怎么排节奏

去年 Q3 我带 PM 团队复盘, 一个人翻出三个月前的路线图, 上面写的"核心交付物"有 11 项, 标着 P0 的 4 项, 标着 P1 的 5 项, 剩下的 P2。实际交付回去看, 路线图写的 11 项里真正上线且没被回炉的有 6 项, 其余要么半成品要么砍了要么延期到下季度。这意味着我们当时定的优先级有一半是错的, 不是事后聪明那种错, 是当时就没想清楚。

但更让我警觉的是另一件事。同一份路线图是季度初定的, 季度中我们因为一个模型能力突然升级 (那次 Claude 3.5 Sonnet 出来), 把原本排在 Q4 的"多轮对话记忆"功能硬提到当季第二个月做完了。这种事在 2024 年之前几乎不会发生, 一个已经画好的季度计划, 中途因为一项底层技术突进而被改写。我们当时没有应对机制, 全靠产品负责人当天拍桌子喊"这周先把评估搞起来"。

这件事其实就是我后来重写整个团队排节奏方法的起点。我没法再用过去做 SaaS 时那套"季度初定 OKR, 季度末打分"的玩法, 因为 AI 产品的迭代周期塌缩得太厉害。我现在带的 AI native 团队, 一个功能从立项到 AB 上线平均 11 天, 过去 SaaS 是 6 周。11 天, 6 周, 这个倍数变化决定了你如果还在按月度排 OKR, 按季度对齐路线图, 你的产品已经死在别人 Press 出来之前了。

下面这些是我在三个产品里撞出来的东西, 不一定都对, 但每条都有具体故事。

## 为什么 AI 产品的节奏传统模板接不住

先说一个反直觉的事: AI 产品里"按时上线"和"做对"经常是反的。我们之前做传统 SaaS, PRD 写完, 评审, 开发, 测试, 上线, 是线性流程, 一个卡点不会摧毁后续, 至多延期两天。AI 产品不是这样。一个功能上线那一刻, 你其实不知道它"对不对"——因为它的输出是概率的, 不是确定的。一个对话产品你测了 200 条 case, 准确率 92%, 上线后用户拿来玩, 真实 query 分布跟你测试集不一致, 真实准确率可能掉到 78%。这事我去年经历过两次。

所以"按节奏上线"这件事本身被重新定义了。过去我们说"按节奏", 潜台词是"按时间"。现在我团队内部说"按节奏", 指的是"按评测-上线的反馈循环运转"。每个功能上线必须有可度量的 feedback loop, 不是抽象的 metric, 是 24 小时内能跑出来的 metric。

这里有个真实的数字可以分享。我们当时做一个客服场景的 AI 助手, 立项时设的目标是"用户问题一次性解决率 (FCR) 提升 15%"。上线两周后看数据, FCR 实际只提升 6%, 远低于预期。我们花了一周排查, 发现不是模型不行, 是意图分类的边界 case 没覆盖全。这一周排查对应传统研发里可能是两个月——因为传统软件 bug 是确定的, 你能定位代码改; AI 产品的 bug 是分布式的, 你得重训或者改 pipeline。最后我们的修复方式是换了 prompt 模板 + 加了 50 条边界 case 进评测集, 这件事花了三天。

你看到没有, 这个节奏跟传统排期完全不同。传统排期是"先定交付日, 再倒推", AI 产品我现在的做法是"先定最短可验证反馈周期, 再倒推交付日"。一周能跑出真实数据的功能, 优先级远高于一个月才能跑出真实数据的功能, 哪怕后者听起来更"完整"。

这里有个具体例子讲讲我们怎么排错过的。去年 H1 我们做了个智能表单功能, PRD 写了三周, 开发做了五周, 最后上线发现用户根本不用。错在哪? 我们当时按"先做完整再上线"的思路, 等所有场景都覆盖了再发。结果上线时市场已经变了——三家竞品早在两个月前就用"先发最小可用版本, 边发边补"的方式抢了用户心智。这事我复盘了不下十次, 核心教训是, AI 产品里"提前做完整"几乎一定是输的策略。不是因为完整不好, 是因为对手两周就能发一个最小可用版本, 你的"完整"在第六周上线时已经晚了。

## 我现在用什么样的 OKR

绕不开 OKR 这个话题。我不打算讲 OKR 是什么, 想讲我在 AI 产品里看到的 OKR 是怎么失效的, 以及我现在用什么替代。

我们过去的 OKR 长这样: O 是"提升 AI 助手的用户满意度", KR 写的是"用户满意度评分从 4.2 提升到 4.5"。听起来合理对不对? 问题是这样写 KR 的人大概率永远达不成, 因为 AI 产品的"用户满意度"在季度初你是测不准的, 模型一升级体验就变, prompt 一调就变, 用户分布一变又变。我亲眼见过我们组的 OKR 季度末复盘时, 满意度确实从 4.2 涨到 4.5 了, 但活跃用户掉了 20%。KR 是达成了, O 失败了。这种 OKR 写法和打分方法, 在 AI 产品里基本是废纸。

我现在用的不是 OKR 字面意义的"替代", 但思路跟我下面说的接近。我把它叫做 "metric tree + cadence", 翻译过来就是指标树 + 节奏。

指标树的意思是, 产品有一个北极星指标, 下面分两到三层子指标, 每个子指标都要点到一个能跑评测或者能拉数据的具体测量方法。举例, 北极星是"周活用户的 AI 调用深度" (每周至少调 10 次 API 算活跃), 下面是"首调成功率"和"多轮后留存率", 再下面是"意图识别准确率" "对话平均轮数"这些。每层是因果关系, 不是平行罗列。

节奏的意思是, 团队每周有一个固定节拍: 周一上午对上周的评测数据 + 用户反馈做一次 review, 周二列本周要做的实验 (通常是 3-5 个), 周三-周五做实验, 下周一再看效果。这个节奏跟项目管理没关系, 是嵌在产品决策流程里的一个反馈循环。我后来在三个产品都用了这个节奏, 最慢的产品一周一次 review, 最快的 AI agent 产品是两天一次。

讲一个具体细节, 我现在每个实验不超过三天就出结果。三天出不了的数据, 我觉得要么是评测设计错了, 要么是这个实验本身不重要。"三天出结果"听起来激进, 但实际上这逼着我们做更小的实验单元, 比如单独改一个 prompt 模板, 跑 200 条 case 看胜率, 这事一天就能做完。有些 PM 听到"三天出结果"会皱眉, 觉得太草率。但你要考虑一个现实: 你不做小实验, 大版本两个月后上线一次, 失败概率远高于"三天一次小实验"。

这里有个反例, 我第一次推"三天一个实验"时犯过什么错。当时我鼓励团队做小实验, 结果一周下来做了十几个 prompt 微调实验, 每个看起来都有 2-3% 提升, 但具体到真实业务指标上几乎没有一个能稳定复现。我后来把这个错改了: 小实验可以多, 但每周必须有一个"大实验", 这个大实验是改 pipeline 或者改某个核心模块, 不能全是 prompt 调。Prompt 微调是便宜的, 很容易产生"幻觉改进", 让你误以为产品进步了; 但如果你只调 prompt 不动 pipeline, 产品能力的上限是被锁死的。

我把这个写出来是因为, 现在国内很多讲 AI 产品方法论的都在鼓吹"快速迭代 prompt, prompt 是新代码"。这话半对半错。对的那一半是, prompt 确实是高频调整的对象; 错的那一半是, 把 prompt 当唯一可调整对象的产品, 天花板很低。我见过至少两个团队这样做了一年, 后来发现竞争对手一个模型升级把他们打回了原形。

## 路线图怎么画才不会第二天就过时

讲路线图。我现在画路线图只用两类信息: 一是"已锁定的承诺" (通常 4-6 周), 二是"探索中的方向" (通常 8-12 周)。中间这段时间是空的, 不填东西。

为什么这样? AI 产品里 6 周后的事情没人能预测得准。模型会升级, 评测会发现意外的问题, 竞品会出新招数, 用户会变。路线图写满了 12 周, 等于画饼。我看过太多路线图画到季度末的产品, 实际完成率从来不超过 40%, 因为路线图本身就是个谎言。

我现在的具体做法是, 季度初画一个主题方向 (比如 Q3 主题是"提升对话质量"), 不承诺具体功能。然后每两周 (我现在的 cadence 是两周, 不是一个月), PM 团队提交一个"两周 sprint"计划, 这个计划是具体的、有交付物的、能评测的。两周 sprint 计划做完就归入历史, 不影响下个 sprint。

讲一个真实的故事。我们当时做 AI 写作助手, Q3 主题定的是"让 AI 写出能用的营销文案"。第 1 个 sprint 我们做的是"prompt 模板库 + 简单界面", 评估指标是"生成文本的人工评分是否超过 4.0/5.0"。第 2 个 sprint 我们发现 prompt 模板库效果受限于底模, 真正有用的是换更小的微调模型。第 3 个 sprint 我们做了一个完整的微调 pipeline。第 4 个 sprint 我们上线了。整个 Q3 没承诺过什么功能, 但季度末我们上线了一个真实可用的产品, 用户留存也比预期高。

如果你把这事讲给传统 PM 听, 他们会问"那 KOL / 老板怎么知道你在干啥"。我的回答是, 路线图不是给老板汇报的, 是给团队自己用的。给老板的是每周的 demo + 每两周的 review meeting, 不是一份画好的甘特图。

这件事我也走过弯路。前年我刚转 AI 产品时, 还会照搬过去的路线图模板, 哪个月做什么清清楚楚。结果每个 sprint 后路线图都对不齐, 越对不齐越焦虑, 越焦虑越想画得更细。最后陷入"画路线图-路线图对不齐-重画"的循环, 这是个典型的反生产力陷阱。后来我自己做出了一个硬规矩: 路线图超过两页 PPT 我就觉得过细了, 直接撕掉重画。两页是极限, 一页更好。

顺带说一句, 上面这套节奏很依赖一个能力: PM 团队要有"快速判断什么值得做"的能力, 而不是"按计划做完"的能力。我现在招 PM, 第一关是给一个 AI 产品场景, 让他现场给"接下来两周做什么", 理由是什么。这是过去两年我自己刷掉一半 PM 候选人的问题。答得上来的人, 真的能在 AI 团队活下来; 答不上来只是按模板填表的, 进组三个月内会被淘汰。

## 技术债怎么排, 这事 AI 产品比传统软件更急

技术债这事我想单独拿出来讲, 因为这块在 AI 产品里有个特别反常识的点: 传统软件的技术债是"延期还的债", AI 产品的技术债是"过期作废的债"。

传统软件里技术债是这样: 你图快用 hard code 写了一坨, 三个月后再回来重构, 这事没大问题。AI 产品里有些技术债是另一种, 比如评测集过拟合、prompt 模板散落各处没有版本管理、数据 pipeline 跑得太慢无法支持小时级迭代。这些债如果一周不还, 后面再还成本是指数级上涨。

我们去年栽过一个大坑。当时为了赶一个重要 demo, 我们写了一个"评测脚本", 能跑通但没接 CI, 跑一次要手动调参数, 全在 PM 个人机器上。Demo 完了一周没人管它, 后来 Q4 我们要做季度成果总结, 想跑一遍历史数据看产品演进, 发现这个评测脚本跑出来的数据跟当时 demo 用的数据不一样——时隔一个月, 同一个 prompt 在不同模型版本下输出变了, 我们没记录当时的模型版本和参数。

这事让我意识到一个我之前没充分意识到的点: AI 产品的技术债包括"实验的可复现性"这条。这是传统软件里没有的。传统软件你 git log 就能复现, AI 产品你光 git log 不够, 还得记录 prompt 模板版本、模型版本号、评测集快照、随机种子。这四个缺一个, 实验都不可复现。

这件事我后来怎么处理的? 我现在团队里有一个硬规矩, 所有评测脚本必须接入 CI, 所有实验必须登记到一张专门的表里 (我用 Airtable, 但 Notion 也行), 表的字段包括 prompt 版本、模型版本、prompt 模板路径、评测集快照 SHA、跑批时间。漏一个字段算技术债, 必须在下一个 sprint 里补上。

讲一个更细的例子。我们当时做 RAG 应用, RAG 检索的 chunking 策略改了三次, 每次都说"只跑这一个实验, 不写文档没事"。结果一个月后产品出 bug, 想回看 chunking 改之前是什么效果, 找不回来了。后来我们规定 RAG 的关键参数 (chunk size, overlap, embedding model) 任何改动必须记到专门的 changelog 里, 跟代码 changelog 分开。这事不是新发明, 是 MLOps 的基础, 但多数 PM 团队是不管的——因为 PM 不懂, 工程师也懒。我后来把这件事画成 PRD 的必备字段, 谁漏谁打回。

我下面想讲一下, 为什么这件事在 AI 产品里比传统产品"债"的属性更强。一个类比: 传统软件的技术债像房贷, 你慢慢还总有还完的时候; AI 产品的"可复现性"债像过期作废的优惠券, 一旦过期, 你想还都还不了。这个比喻不严谨, 但意思是到位的, 希望你 get 到。

所以我现在团队排节奏时, 技术债这块专门预留 20-25% 的容量。不是那种"季度末尾补一下"的做法, 是每个 sprint 都预留。如果是高优债 (比如评测脚本没接 CI 这种), 这一周就修, 不延后。

## 一句话建议你这一周就能用

讲到这里, 我其实就一句话建议, 我反复跟团队说, 也希望正在看这章的你这一周就能试着做一下:

**把你团队现在的"路线图"这一页文档从 12 周填到 4 周, 中间那段空白不填, 改成"探索中"三个字。** 然后把会议从月度 review 改成双周 review, 每次 review 必须有可评测的实验数据, 没有数据不开。下个 sprint 重复一次。

这事听上去小, 但我过去两年带三个产品反复验证过, 它会立刻把团队从"画路线图的奴隶"变成"做产品的决策者"。区别在哪? 画路线图是被规划的, 做决策是主动的。AI 产品里, 后者比前者值钱得多。

当然, 你可能又会问"那老板问起来我怎么答"。这是另一个话题, 我在下一章会讲怎么用评测数据和用户反馈代替路线图给老板汇报, 这事在 AI 产品里反而比传统软件更好做, 因为 AI 产品的进展是"实验-评测-上线"的循环, 每个节点都有数字, 数字就是汇报材料, 不需要画路线图这种纸面文章。

这是我从去年到现在一直在迭代的方法, 还在变, 不是终稿。如果有更好的做法你正在用, 欢迎线下交流, 我大概率会偷师。
