# 用 agent 替代 PM 一部分工作: 文档 / 分析 / 竞品 / 会议

去年 Q3 我让组里 4 个 PM 跑了一次内部 audit, 统计自己每周到底把时间花在了哪儿. 结果有点打脸: 一个做了三年的 PM, 周工时 50 小时, 真正花在 "判断优先级 / 跟业务方对齐目标 / 拆解模糊问题" 上的时间, 大概 9 小时. 剩下 41 小时呢, 写文档 12 小时, 开评审会 8 小时, 拉数对数 7 小时, 整理竞品 5 小时, 还有一堆杂事 9 小时.

我当时把这份 audit 拿给我们的 CTO 看, 他说了一句挺狠的话: "你这个组, 一半时间在做 agent 已经能做或者马上能做的工作."

这话我一开始听着不太舒服. 但后来想想, 他说的不是 "PM 没用", 他说的是, PM 的时间花错了地方. 我们的 PM 团队人均工作 4 年, 工资不便宜, 每天在做 50% 可以被 LLM 跑掉的事. 这件事本身就是产品问题.

所以这一章我想聊的就是: 哪些 PM 工作现在真能让 agent 接手, 哪些 PM 价值是 agent 接不住的. 我会讲一些我自己在 2024-2025 试过的具体做法, 包括翻车的地方. 不是为了证明 "PM 会被取代" 那种吓人的论调, 是想把这条线画清楚一点.

## 文档: 从 "写" 到 "审", 但 review 不能省

文档是 PM 最容易被 agent 替代的部分, 也是最容易出事的部分. 写文档本身——尤其是那种结构化、模板化、有现成资料可以喂的文档, 比如 PRD、用户故事、API 文档、版本更新日志, 现在 Claude / GPT 跑起来已经比大部分 PM 写得还像样.

我去年在内部推过一次 "PRD 模板改造", 起因是 review 会上我发现一个事儿: 一个 PRD 改到第三版, 核心逻辑 (业务目标 / 核心场景 / 异常分支) 其实第二版就定了. 第三版 80% 的 diff 是措辞、格式、补几个边界 case. 我让组里 PM 试试用 Claude 直接基于第二版生成第三版, 喂进去 "沿用上版逻辑, 把 XX 章节展开, 补充 XX 字段", 结果 90% 的内容可以直接用, 只需要 PM 改 5-10 处关键判断.

那这一周 12 小时写文档, 是不是就剩 2 小时? 没那么乐观. 原因是: 生成容易, 审稿才花时间. PM 真正的工作不是 "把字敲出来", 是 "判断这段话对不对、漏了什么、谁会误解、跟上次逻辑是不是冲突". 而 review 这件事, agent 替不了.

但我踩过坑, 必须讲讲.

第一次我们试 agent 写 PRD, 是 2024 年初用 GPT-4. 当时觉得效果 "惊艳", PM 一周能省 5 小时. 直到一个 PRD 上线后出了大问题: 业务方按照 PRD 实现, 上线才发现一个合规分支没写. 我去翻 GPT 写的原文, 那个分支确实没出现. 我去翻我们当时的 prompt, 写的是 "请基于 XX 文档生成完整 PRD". 看上去挺正常, 但问题在哪呢——业务方给的需求里, 默认假设 "用户都是大陆身份证", 没人提过港澳台和外籍的情况. agent 不知道要问, PM 也没提醒, 结果上线后第一个被卡住的就是港版 iPhone 用户.

后来我跟组里 PM 讲, 凡是涉及 "分支 / 异常 / 合规 / 跨地区 / 跨角色" 的字段, PRD 模板里必须有强制 checklist, 而且 PM 要逐条 review. agent 不知道这些, 因为它没见过你们公司真实的用户结构. 这种东西必须靠 PM 自己的领域知识硬扛.

我现在用的一个轻量做法是: PRD 模板里加一栏 "假设与边界", 写 "这个 PRD 假设的输入是什么、不覆盖什么、需要哪些人去 validate". 这栏 agent 可以写初稿, 但 PM 必须自己改. 不改的 PRD, 我不签字.

另一个翻车是会议纪要. 当时组里 PM 全员上 Otter / 通义听悟之类的工具自动转写, 然后丢给 Claude 总结. 听上去很美, 实际跑下来, 第一周我翻一份纪要, 发现客户说 "这个功能先不做, 后面再说", agent 给总结成 "客户认可方案, 进入开发". 完全是反的. 后来我加了一条规则: 凡是涉及 "决策 / 拒绝 / 延期 / 责任归属" 的句子, 必须原文摘抄, agent 不准改写. 一年过去了, 这个规则还在用.

文档这块我想说的是: agent 真正替代的是 "把信息按格式铺出来" 这件事. 但 "判断什么信息该出现、判断哪些分支不能漏、判断哪些表述会引起误解"——这是 PM 工作的核心, 没替掉. 而且这块如果 PM 自己不上心, agent 会给你制造一种 "看起来很完整" 的幻觉, 实际上埋雷.

## 分析: 拉数和对数的活儿终于可以外包了

我刚入行那会儿 (2014 年前后吧), 一个 PM 必备技能就是写 SQL. 当时组里流传一句话: "不会写 SQL 的 PM 不是好 PM". 现在回头看, 这句话大概是当时产品团队没有数据分析师、又必须自己跑数逼出来的.

到了 2024-2025, 这件事基本可以宣告结束了. 我现在团队里 4 个 PM, 没有一个会写复杂 SQL, 也不需要会. 他们的日常是: 把自然语言问题丢给 Text-to-SQL agent, agent 生成 SQL, 跑出结果, PM 看一眼对不对, 不对就追问.

具体怎么落地的呢. 我们去年接入了 Metabase + 一个自建的 text-to-SQL 中间层 (其实就是用 Claude function calling 把自然语言转 SQL, 然后去 Metabase 查). 真实的 workflow 是: PM 在群里说 "看一下上周新功能上线后, 7 日留存按渠道拆一下", agent 30 秒返回一张表. PM 不满意, 继续追问 "再按城市等级拆一下, 把新一线单独看", agent 再跑一次. 一个本来要 2 小时的活儿, 现在 10 分钟搞定.

但这里有个坑, 我踩过两次.

第一次是 2024 年 6 月, 一次周会上 PM 报了一个数, 说 "新功能上线后 DAU 涨了 12%". 老板挺高兴, 当场决定加大投放. 后来数据组同事私下找我说, 那个 12% 是 text-to-SQL agent 自己算的, SQL 写错了——把 "活跃用户" 写成了 "登录用户", 分子分母定义都跑偏了. 真实涨幅大概是 2%, 还不显著.

我们后来加了一条硬规则: 任何要在 review / 周会 / 决策会上引用的数据, 必须 PM 自己手动 SQL 验证一次, 或者让数据组同事 spot check. agent 给的数只用于探索和初步判断, 不用于拍板. 这个规则听起来挺烦的, 但我们一年就避免了至少 3 次类似的事故.

第二个坑是 "分析的解释" 容易被 agent 简化. 比如留存下降, PM 让 agent 解释原因, agent 可能直接给一个 "因为 XX 渠道质量变差" 的解释. 但真实的留存下降往往是多因素的: 渠道只是表象, 后面是版本节奏 / 季节性 / 竞品动作 / 内部运营策略. agent 不知道这些, 它只能基于你喂给它的维度去解释. 真正能看穿 "数据背后的故事" 的, 还是 PM 自己.

说白了, 拉数对数这件事, agent 替代得很彻底. 但 "数据意味着什么 / 下一步该干嘛 / 这件事是不是值得投入精力去挖", 这些 agent 替不了. 而且我发现一个反直觉的事: 拉数快了之后, PM 反而要花更多时间在 "判断哪些数值得拉". 之前是数据分析师一天跑 5 个问题, 现在 PM 自己一天能问 50 个, 但其中 45 个是噪音. 这要求 PM 自己对业务有更深的理解, 才能问出对的问题.

## 竞品: 替代的是 "信息收集", 替代不了 "判断"

竞品分析这事我纠结过很久. 2024 年初我用 agent 跑了一次大规模竞品扫描, 喂了 30 个 toB SaaS 产品的官网 + 文档 + 公开案例, 让 Claude 总结每个产品的核心定位、收费模式、目标客户. 跑出来一份 80 页的报告, 看上去挺唬人.

但拿给销售和 CEO 看, 反馈是 "没用". 原因是: 信息都对的, 但没有任何 insight. 全是 "XX 产品定位中小企业, 提供 XX 功能, 收费按席位", 这种东西在销售心里早就有了. 销售想看的是 "我们和 XX 比, 真正差异化在哪 / 客户为什么选我们 / 客户为什么选他不选我们".

我后来意识到, 竞品分析这块, agent 替代的是 "事实层" 的工作, 替代不了 "判断层" 的工作. 事实层就是 "竞品有哪些功能、怎么定价、目标用户是谁", 这些靠爬虫 + LLM 总结, 确实能跑得比人快. 但判断层是 "我们应该学他什么 / 我们应该警惕他什么 / 我们的客户为什么买他", 这些只有 PM 跟客户打过交道、看过 deal cycle、输过单子之后才能讲得出来.

我现在做竞品分析, 流程是这样的: 第一步, agent 跑事实层, 输出 5-10 个竞品的功能矩阵 (这块是体力活, agent 跑). 第二步, PM 拉内部销售 / CS 同事 1v1 聊, 问 "上季度输掉的单子, 客户选的是哪家, 为什么". 第三步, 把内部访谈的判断叠加到 agent 跑的事实上, 形成一份 2-3 页的 insight 文档, 不是 80 页的报告.

这么做之后, 我发现一个好处: 竞品文档变得很轻, 大家真看. 之前 80 页的报告, 销售和老板都懒得翻, 真正决策时还是凭感觉. 现在 2-3 页, 每次复盘都会用上.

我还试过一个更激进的玩法: 用 agent 监控竞品的版本更新 + 公开内容变化. 比如订阅竞品的 changelog、博客、招聘信息, 让 agent 每周总结 "这周 XX 竞品发生了什么变化". 这个挺有用, 因为人工盯竞品基本不可能, 但 agent 盯得住. 唯一要注意的是噪音——很多公开信息其实不重要, agent 会一股脑推过来, PM 要学会过滤. 我现在让 agent 每周只推 "3 条最重要的变化 + 我自己的一句话判断", 不是把所有变更都列出来.

## 会议: 协同效率上来了, 但决策密度反而要 PM 自己扛

会议这块我想单拎出来讲, 因为我之前一直没想清楚, 直到去年底经历了一件事.

当时我们组做了一个实验: 周会之前, 让 agent 把上周的会议纪要、决策项、本周待办全部整理成一页 brief, 发到群里. 开会的时候, 大家不用再花 15 分钟过 context, 直接进入讨论. 听上去很合理, 实际效果是: 会议时间从 60 分钟缩到 35 分钟, 但决策质量反而下降了.

为什么? 后来复盘发现, 原因是: 大家跳过 context 这步之后, 很多隐性的共识没了. 比如 "这个需求我们之前决定不做, 因为 XX", 这种 background 在 brief 里 agent 不会主动写, 因为它不知道这个 background. 结果同样的决策被反复重新讨论, 表面上看会议快了, 实际在原地打转.

后来我改了一个做法: brief 让 agent 写, 但 PM 在会前必须自己过一遍, 把 "隐性共识 / 历史决策 / 容易引起分歧的点" 标出来. agent 给的是骨架, PM 给的是血肉. 这个流程走下来, 会议从 60 分钟缩到 40 分钟, 决策质量也保住了.

另一个跟 agent 相关的会议场景是: 跨时区 / 跨语言会议. 我们当时跟欧洲团队合作, 对方母语不是英语, 实时翻译 + 总结 agent 是真的有用. 但也踩过坑——agent 把对方的 "I'm not sure about this" 翻成 "我不同意", 把氛围搞得很僵. 后来我们规定, 翻译 agent 输出的内容必须在会议结束后人工 spot check 一次, 有偏差的当场澄清. 这条规则救了我们好几次关系.

我自己的体感是: agent 在会议场景里, 替代的是 "记录 / 整理 / 翻译" 这种机械活. 替代不了的是 "判断什么时候该推进、什么时候该暂停、什么时候该 escalate"——也就是会议里的政治和时机. PM 在会议上最大的价值, 不是主持流程, 是在合适的时候插一句 "这个我们先别定, 回去再想想", 或者 "这个事老板还没 buy-in, 我们今天先对齐方向". 这些判断, agent 看不到.

## 那 PM 真正不可替代的是什么

写到这儿我想回头讲讲, PM 真正不可替代的价值是什么. 因为前面讲的都是 "哪些活被替了", 但 PM 还在, 那 PM 到底在干嘛.

我自己想了很久, 觉得有三块东西是 agent 短期内替不了的, 也是 PM 应该把时间往这儿挪的.

第一块是 "判断优先级". 一个组同时有 10 个需求, 哪些先做、哪些不做、哪些砍掉. 这件事的输入是模糊的: 老板的焦虑、客户的抱怨、竞争对手的动作、团队的疲劳度、季度的财报压力. 这些信息有 50% 是不在文档里的, 是 PM 通过闲聊、观察、经验积累出来的. agent 看不到这些, 所以它的优先级建议基本是 garbage. 我之前试过让 agent 基于需求列表 + 业务指标数据建议优先级, 跑出来的东西一塌糊涂, 因为它不知道 "老板最近在焦虑什么 / 销售最近在抱怨什么 / 团队最近已经超负荷了". 这些 PM 自己心里得有数.

第二块是 "建立共识". PM 大量时间花在让一群背景不同的人 (老板、销售、工程、运营、客户) 同意同一个方向. 这件事本质上是政治 + 沟通 + 妥协, 没有任何 agent 能干. 我之前让 Claude 试过 "帮我写一封邮件说服老板砍掉一个功能", 写出来的东西技术上挑不出毛病, 但语气完全是错的——要么太刚, 要么太软. 真实的政治敏感度, 是 PM 长期跟这群人打交道练出来的. 而且这件事没法 "自动化", 因为每次背景都不一样.

第三块是 "定义问题". 这个我放在最后, 但可能是最重要的. PM 真正的工作, 不是 "解决问题", 是 "定义出值得解决的问题". 一个 group PM 跟我讲过一句话, 我印象很深: "大部分 PM 的失败, 不是执行失败, 是定义了错的问题." 比如老板让你 "提升 DAU", 真正的 PM 工作不是想三个提升 DAU 的方案, 是回去问 "为什么 DAU 在掉 / DAU 掉到多少会影响业务 / 提升 DAU 是不是当下最重要的事". 问对了问题, 答案会自己冒出来. 问错了, agent 跑得再快, 也是在错的方向上加速.

agent 在 "执行" 层面能替掉很多, 但 "定义问题" 这件事, 是 PM 工作的根. 这块如果 PM 自己不花时间, 整个团队就是在用更快的速度做错的事.

## 给 PM 的两个本周就能动手的建议

讲到这里, 我想给还在做 PM 的朋友两个具体的建议, 本周就能试.

第一个, 找一周时间, 把自己每天做的事按 "信息整理 / 信息判断 / 信息创造" 分个类. 看一下自己一周有多少小时在做 "信息整理" (写文档、拉数、整理纪要). 如果超过 30%, 那就值得严肃考虑把这一块用 agent 跑起来. 不是为了省时间, 是为了把时间挪到 "信息判断" 和 "信息创造" 上去. 我自己 audit 完之后, 把 PRD 模板重写了一遍, 加了 "假设与边界" 栏, 接了 text-to-SQL, 这一项每周省 8 小时. 但更重要的不是省时间, 是 PM 拿这 8 小时去跟客户多聊了几次, 反而抓住了几个之前没注意到的真问题.

第二个, 找自己写过的三份文档 (PRD / 竞品分析 / 会议纪要都行), 喂给 agent, 让它基于你写过的内容生成新版本. 然后对比一下 "agent 写的" 和 "你自己写的" 的差异在哪儿. 这个差异就是你作为 PM 真正的价值. 比如我自己的 audit 里发现, 我和 agent 写的 PRD 差异最大的地方是 "异常分支" 和 "假设说明"——也就是我自己特别会想到 agent 不会想到的边界条件. 这就是我的护城河, 我应该把更多时间花在这块, 而不是花在措辞和格式上.

最后说一句不太好听的话. 我见过一些 PM, 2025 年了还在用 2018 年的方法做事, 觉得 "工具是工具, 思考是思考, 我是手艺人, 工具不影响我". 这种想法在两年前可能还成立, 现在不成立了. 不是工具替代了你, 是用工具的人替代了你. 不是说你要变成 prompt engineer, 也不是说你必须精通 LLM——而是你得承认, 文档 / 拉数 / 整理 / 转写这些事情, 它的 "稀缺性" 在快速归零. 你的稀缺性必须建立在别的东西上, 那个东西是判断力、共识能力、定义问题的能力. 这些能力不是天上掉下来的, 是 PM 通过一次次 review、一次次客户对话、一次次拍板练出来的.
