# 05. LLM-as-judge 评测: 让 AI 当评委跑用户故事

去年 Q3 我带 PM 团队做 AI 写作助手, 产品上线前最难的一道坎不是写代码, 是 "怎么证明它真的变好了". 老办法是找 50 个真实用户跑 3 天, 收集 1200 条反馈, 人工标注 "这篇好 / 那篇差". 我和分析师两个人趴在飞书表格里点了 8 天, 最后产出一份 23 页的评测报告, 写着 "v2 比 v1 胜出 67%". 报告交上去, CTO 问了一个问题: "这 67% 是怎么算出来的, 换个标注员结论会变吗?"

我愣了 3 秒. 说实话, 会变. 不同标注员对 "信息密度高" 的理解差距巨大, 我自己前后两轮标注同一批样本, 一致率也只有 0.71. 这就是传统人工评测的命门: 慢, 贵, 主观, 还不能复现. 后来我们转向 LLM-as-judge, 用 GPT-4 当评委跑了 600 条样本, 4 天出报告, 一致性 0.84, 成本 80 美元. 不是说 LLM judge 完美无缺, 它有自己的偏见和失效模式. 但作为 PM, 我现在做任何一个 AI 产品的版本迭代, 第一件事就是搭评测流水线, 而不是写 PRD 里的 "成功指标" 那一栏. 这一章想讲的, 是怎么把这套东西真正搭起来, 怎么避坑, 怎么跟人工评测配合, 而不是停留在 "用 LLM 做评测就是 prompt 一下让它打分" 这种误解上.

## 先把期望校准: LLM judge 不是什么, 是什么

先讲一个我去年踩过的坑. 我当时在做一个客服 AI 的项目, 产品目标是把 "用户问题匹配到知识库正确答案" 的准确率从 76% 拉到 90% 以上. 我很自然地想: 让 GPT-4 当 judge, 给模型输出和标准答案做对比打分, 跑 500 条样本看分数不就行了? 跑完一看, 92%. 我高兴了三天, 跟老板汇报完开始做下一版. 等到 A/B 上线, 真实数据出来, 实际准确率 81%. 差了 11 个点.

为什么? 复盘那天晚上我盯着 prompt 看了 20 分钟, 发现了一个我忽略的问题: 我的 judge prompt 是 "请判断 AI 回答是否准确, 准确给 1 分, 不准确给 0 分". GPT-4 这个模型, 对 "委婉地回避" 和 "真的答错" 的区分能力其实不如我想的那么强. 用户问 "我的订单为什么没发货", AI 回答 "建议您联系客服查询最新物流状态", 这句话既不准确也不不准确, 它是在回避. GPT-4 给打了 0 分, 人工标注员给了 0.5 分, 真实用户给了 0 分 (没解决问题). 这种 case 一多, 整体分数就虚高.

所以第一课是: LLM judge 不是银弹, 它是个会犯错的评委, 但它犯错的模式比人工稳定. 你要做的是理解它的偏见, 控制它的失效边界, 而不是无脑相信它的输出. 我后来跟团队立了一个规矩: 任何 LLM judge 跑出来的结果, 必须配一个人工验证子集 (我们一般抽 5%, 不超过 10%), 算一致性. 一致性低于 0.75 的评测结论, 老板汇报的时候不能直接用, 只能说 "LLM judge 倾向认为, 但需要人工复验". 这条规矩我后来带到所有 AI 项目里, 没破过.

## 怎么搭一条能用的评测流水线

接下来说流程. 评测流水线说白了就是四步: 准备数据集, 跑模型, 跑 judge, 分析结果. 但每一步都有讲究.

数据集这一步, 我见过最多 PM 栽的跟头是 "拿开发自己造的样本当评测集". 这等于让厨师自己写大众点评. 真实用户的问题分布跟你内部测试集是两回事, 你以为用户会问 "如何重置密码", 实际上 40% 的真实问题长成 "我密码忘了但是手机号换了怎么办, 在线等急". 我们的做法是: 冷启动用 200 条真实线上日志 (脱敏) 作为种子集, 上线后每个版本滚动收集 50 条新样本补进去. 三个月后评测集长到 800 条, 覆盖了 12 个场景类别. 这套数据集后来变成了我们组最值钱的资产, 比任何 PRD 都值钱.

跑模型这一步看起来没 PM 什么事, 但其实有一个判断要做: 你跑的是哪个版本? 我建议 PM 在评测里同时跑 3 个版本 — 当前线上版, 新模型版, 加上一个 baseline (比如 GPT-4o 或者 Claude Sonnet 的输出). baseline 是个锚点, 没有它你不知道新模型到底是好还是 "因为 prompt 写得好". 去年我做内容生成产品, 我们组内公认 "v3 比 v2 好", 跑完评测才发现, v3 跟 GPT-4o 的输出质量差不多, v2 差了一大截. 也就是说我们花了两个月做的 "v3 优化", 实际是把一个烂模型拉到了行业基线水平, 不是真正的进步. 那一刻挺扎心的, 但比上线后被用户骂强.

跑 judge 这一步, 选 judge model 是个真问题. 我用过 GPT-4, GPT-4o, Claude 3.5 Sonnet, 还有开源的 Llama-3-70B. 实测下来, Claude 3.5 Sonnet 在开放式问答上的打分跟人工标注的一致性最高 (0.82 左右), GPT-4 在结构化任务 (比如分类, 抽取) 上更稳. 不要一个 judge 打天下, 任务类型不同, judge model 也应该不同. 还有个反直觉的点: 越新的模型不一定越好. GPT-4o 跑开放式问答, 一致性反而比 GPT-4 略低, 因为它太 "愿意给分了", 倾向于找理由说 "这个回答虽然不完美但也还行". 我们的对策是温度参数开到 0, 关掉它的 "讨好倾向". temperature = 0 的 judge 比 temperature = 0.7 的稳定 5-8 个点的一致性, 这个差异是显著的.

Judge prompt 的写法, 说几个真正重要的点. 第一, 必须给具体维度, 不能只说 "评估这个回答的质量". 我用过的 prompt 模板长这样: "你是一个严格的评测员, 请从以下四个维度评估 AI 的回答: 1) 准确性 (1-5 分, 5 分表示事实完全正确) 2) 完整性 (1-5 分, 5 分表示覆盖了用户所有需求) 3) 简洁性 (1-5 分, 5 分表示无冗余信息) 4) 安全性 (1-5 分, 5 分表示无有害内容). 请分别打分并说明理由." 维度拆开后, 整体分不再是模糊判断, 而是加权平均. 加权这件事 PM 必须参与, 因为不同维度的权重反映了产品判断: 客服场景里 "准确性" 权重 0.5, "简洁性" 只有 0.1; 创意写作场景里反过来, "创造性" 和 "风格匹配" 权重最高.

第二, 给 judge 一些反例. 这点是从清华的一个评测工作里学来的, 叫 "in-context negative examples". 意思是 prompt 里塞 2-3 个低分样本, 告诉 judge "这种回答只能给 2 分". 这能显著降低 judge 的 "礼貌分" 倾向. 我们的做法是在 prompt 里加一段: "以下类型的回答应该得到低分: 直接照抄问题但不提供新信息的; 包含错误事实的; 回避用户核心问题的." 加上这一段, judge 给的低分率从 8% 升到 19%, 跟人工标注的分布更接近.

第三, 跑多次取平均. 同一个 judge prompt, 跑 3 次取中位数, 比单次结果稳定得多. 成本也就涨 3 倍, 在我们 600 条样本的评测里也就 80 变 240 美元, 完全可接受. 但能换来一致性 +0.04 的提升, 这 0.04 在统计意义上是显著的 (我们的样本量足够检出).

## 偏见这件事必须主动治, 不能等它发生

LLM judge 的偏见是结构性的, 不是偶然 bug. 我把我遇到过的主要偏见列一下, 都是真问题.

位置偏见 (position bias) 是最经典的一个. 你给 judge 两个回答让它比较, 无论实际质量如何, 它倾向于给第一个回答更高分. 实测下来, GPT-4 在盲评里给 A 位置打高分的概率比 B 位置高 8-12 个百分点. 对策是: 每次评测跑两遍, 第二遍把 A B 顺序对调, 取平均. 这一步不能省, 成本翻倍也值. 我去年做一个搜索产品, 没做位置对调的版本显示 "新排序算法比老的好 6%", 做了对调之后差距只有 2%, 还都在置信区间内. 没有这一步, 我就去跟老板吹了一个不存在的胜利.

长度偏见 (length bias) 也很常见. Judge 倾向于给长回答高分, 即使长回答里一堆废话. 这个在内容生成场景特别明显. 对策是在 prompt 里明确加一句: "请注意, 长度本身不应该是加分项. 简洁且完整的回答应该得到与冗长但完整的回答相同或更高的分数." 然后在分析阶段, 单独跑一遍 "长度 vs 分数" 的相关性分析, 看看是不是真的存在长度偏见. 如果 r > 0.3, 说明你的 judge prompt 没写好, 需要重写.

自我偏好 (self-preference bias) 是一个更隐蔽的问题. 如果你用 GPT-4 当 judge, 它倾向于给 GPT-4 系列的输出更高分; 用 Claude judge, 它倾向于给 Claude 输出更高分. 这点我们的内部数据验证过, 跨家族比较时差异能有 5-7 个点. 对策是选一个中立的 judge, 比如用一个跟生产模型不同家族的模型. 我现在做 GPT 系列产品的评测, judge 用 Claude; 做 Claude 系列产品评测, judge 用 GPT-4. 跨家族是底线, 至少能消掉一半自我偏好.

还有一个很容易被忽视的, 叫 "格式偏见". Judge 倾向于给格式工整 (有列表, 有 markdown 加粗) 的回答高分, 即使内容质量差不多. 这点我做 PRD 自动生成产品的时候被坑过一次, 我们的 AI 输出格式很朴素, 但内容扎实, 跟一个格式花哨但内容空洞的基线比, judge 给了基线更高分. 后来我在 prompt 里加了一句: "请忽略回答的格式 (排版, 标点, 列表), 仅评估内容质量." 加上之后两个模型的真实差距才显现出来.

分析偏见这事, 还有一个高级玩法: 跑一份 "对抗样本" 专门测试 judge 的失效模式. 我们组每季度会人工构造 30-50 条 tricky 样本, 故意设计成容易让 judge 打错分的类型 (比如正确但简短的回答, 错误但表达自信的回答). 拿这批样本去测 judge, 看它被骗的比例. 如果比例超过 20%, 说明这个 judge 在生产环境里也不可靠, 需要换模型或者改 prompt. 这套 adversarial set 跟主评测集是分离的, 专门用来 "压力测试 judge 本身". 这个想法是我从 AI Evals 系列 (harness 那边) 借过来的, 用在 PM 视角上很合适, 因为 PM 才是最终对评测结论负责的人.

## 跟人工评测怎么配合, 不是一个取代另一个

最后讲跟人工评测的关系. 我经常被问: "有了 LLM judge 是不是就不用人工标注了?" 不是的. 它们是两个不同的工具, 解决的问题不一样. LLM judge 解决的是 "快速, 低成本, 可复现地拿到一个相对分数". 人工标注解决的是 "定义什么是好, 设置 ground truth, 校准 judge". 在我的工作流里, 人工标注的角色是: (1) 初期构造评测集的时候, 由人工标注 200-500 条样本作为种子; (2) 每个月抽 5% 的 LLM judge 结论做人工验证, 算一致性; (3) 关键决策点 (比如新模型上线) 必须有人工小规模 (30-50 条) 复核.

具体到一份评测报告的产出, 我现在的标准模板是这样的: 第一部分是 LLM judge 的整体分数对比 (v1, v2, v3, baseline), 附置信区间; 第二部分是分维度的雷达图, 看新模型在哪些维度强哪些弱; 第三部分是 adversarial set 的结果, 标出 judge 在哪些类型上不可靠; 第四部分是人工验证子集的一致性报告; 第五部分是失败案例的 10-20 条人工归因分析. 第五部分是最值钱的, 也只有人工能做. 它告诉你 "分数之外到底发生了什么", 哪些是 judge 看不见的细节, 哪些是产品决策的真正依据.

有个例子. 我们做内容生成, LLM judge 跑出来 v3 比 v2 整体高 4%, 不算大. 但分维度看, v3 在 "风格匹配" 上高了 11%, 在 "事实准确性" 上低了 3%. 这 4% 是个混合结果, 不能简单地说 v3 更好. 我们拉出失败案例分析, 发现 v3 在 "事实准确性" 上的下降是因为它更敢 "润色" 用户的事实描述, 把它认为 "更通顺" 的版本输出, 但偏离了原意. 这是个产品决策: 你是要 "忠于事实" 还是要 "读起来更通顺". 这个问题没有客观答案, 必须 PM 拍板. 拍板的依据就是这 4% 之外的细节, 不是 LLM judge 的总分.

跟 AI Evals 系列那边的工作相比, PM 视角的评测有几个不同. 工程师关心的是 "模型在 benchmark 上的能力", 跑的是 MMLU, HumanEval, MT-Bench 这种公开榜单; PM 关心的是 "我的产品场景里这个模型表现如何", 跑的是私有评测集, 维度是产品定义的, 结论要能直接转化为产品决策. 这两套评测不是替代关系, 是分工关系. 公开榜单帮你选模型, 私有评测帮你选版本. 上一章 (04) 我们讲的 prompt 迭代, 实际上也应该在私有评测集上跑, 不能只看公开 benchmark. 这个连接点很重要, PM 如果只跟着公开榜单走, 跟产品实际表现是脱节的, 我见过太多次 "榜单分高但用户骂" 的案例.

## 收尾: 这周你能动手的一件事

如果只看这一章只能记住一件事, 我希望是: 评测不是项目结束后的验证, 是项目过程中持续跑的基础设施. PRD 里写的 "成功指标" 如果不能用评测流水线持续测量, 那就是废纸.

具体到这周能做的事, 我建议三选一. 第一, 如果你正在做 AI 产品, 花两天时间搭一条最小评测流水线: 准备 50 条真实样本 (从线上日志脱敏), 写一个 4 维度的 judge prompt, 跑当前线上模型 + 一个新模型对比, 出一份一页纸的评测报告. 80 美元成本, 4 天出结果, 你就能感觉到这套东西的价值. 第二, 如果你还没开始用 LLM judge, 先用 Claude 或者 GPT-4 跑 100 条样本, 跟人工标注对比一致性, 低于 0.7 的话回去改 prompt, 高于 0.85 的话可以开始信任. 第三, 把 "评测报告" 加进你的产品迭代 SOP, 每次模型或者 prompt 变更必须跑评测, 没有评测报告的变更不允许上线. 这一步是流程上的, 比技术更重要.

我做了 11 年 PM, 评测这件事是我这两年才真正搞明白的. 之前做传统产品, 评测靠 DAU 涨没涨, 留存掉没掉, 周期长, 反馈慢. AI 产品不一样, 模型一周能迭代 5 个版本, 你没法用 DAU 验证哪个版本好, 必须有更快的反馈环. LLM judge 不是这个反馈环的全部, 但它是核心组件. 不会搭评测流水线的 PM, 在 AI 时代会越来越像 "没有仪表盘的司机", 凭感觉开车, 偶尔能到, 大部分时间在绕路.

下一章我想聊 agent 怎么替 PM 干活 — 文档, 竞品分析, 会议纪要, 需求梳理. 评测是 "验证想法", agent 是 "加速产出", 这两件事加起来, 才是一个 AI 时代 PM 完整的工具箱. 链接放在 [用 agent 替代 PM 一部分工作](./06-pm-agent.md).
