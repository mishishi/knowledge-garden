# 07. AI native 团队怎么组建：5-10 人 mini team + agent

去年 Q3 我在一家做代码助手的小团队待了四个月，11 个人，没 VP 没收口人，CEO 直接管 PM 和设计。我当时觉得这种结构撑不过半年——结果上个月问了一下，团队还是 11 个人，ARR 从年初的 800 万做到了 2400 万。

这事对我冲击挺大。我之前在大厂带 40 人 PM 团队，年中汇报要写 18 页 PPT，光对齐口径就要两周。这种节奏放在他们那早就死透了。

那段时间我一直在想一个问题：AI native 的小团队，到底跟传统 50 人产品团队差在哪？是工具更狠？流程更野？还是人更猛？后来跟三个不同阶段的 AI 创业团队 CEO 聊过，加上自己这段经历，我大概摸出来一套不那么漂亮、但确实能跑的规律。

这一章我想讲的是：5-10 人的 mini team 怎么搭、为什么能跑、PM 在这种结构里到底干什么活。

## 先承认一个反直觉的事：人少不是劣势，反而是优势

我之前在大厂带团队的时候，特别迷信"组织能力"——人不够就招，招不到就借，借不到就推。所以去年刚接触这个 11 人团队的时候，我第一反应是"这肯定缺人"。代码后端 3 个人、前端 2 个人、算法 1 个半（有一个是研究科学家只能算半个工程）、PM 1 个、设计 1 个、数据 0 个、QA 0 个。我心想，这配置连个像样的 A/B 测试平台都搭不起来吧？

但看了一个月他们的工作方式，我的认知被刷新了。

他们的 PM 一个人管着三条产品线，每天早上 9 点 15 跟 CEO 站会对齐，10 点开始写 PRD，下午 3 点前 PRD 锁稿，后端已经开始写代码了。晚上 11 点代码合进去，第二天早上 PM 起来跑测试脚本——不是人肉点，是他自己写了个 Playwright 脚本 + Claude API 调用的混合流程，30 分钟跑完关键路径。

我问他，你怎么写这么快？他说 PRD 模板就一页，截图 + 用例 + 数据表，没有别的。

我又问，跨部门对齐呢？他说没有跨部门，就 11 个人，对齐就是在工位喊一声。

这就是第一个反直觉的点：**传统产品团队那些"组织摩擦"——跨部门会议、PPT 汇报、向上管理、跨团队撕需求——在小团队里直接消失了。** 消失的原因不是大家更努力了，是物理上就没人可以撕。

我后来给传统大厂的朋友讲这个观察，他第一反应是"那是因为他们公司小所以可以这样"。但我看着 2400 万 ARR 这个数字想，他们也不小啊。问题不是"小"，是 AI 工具让维护一个产品的成本结构变了——以前 50 个人的团队干的事，现在 5-10 个人 + 一堆 agent 就能干完。

那多出来的人力成本（薪酬、管理成本、沟通损耗）就直接被砍掉了。这不是"省钱"，是结构性重组。

## 第二件反直觉的事：PM 在小团队不是"什么都管"，而是"什么都不写"

我以前在大厂，PM 是个文档机器。一份 PRD 5000 字起步，竞品分析 2000 字起，用户调研报告 3000 字起，需求池 200 条，PRD review 三轮，向上汇报要写一晚上。

但在那家 11 人团队，PM 一天能写 3 个 PRD 上线。我一开始以为是 PM 写得潦草，偷工减料。后来仔细看了他写的 PRD，发现不是潦草，是结构完全不同。

我把他最近一周的 5 份 PRD 拿来看，每份大概在 400-800 字之间。结构就三块：问题（3 句话以内，谁、什么场景、卡在哪）、方案（截图 + 关键逻辑，文字描述不超过 200 字）、验收（数据指标 + 测试用例）。没有市场分析、没有竞品对比、没有风险评估、没有 ROI 测算。

我问他，不担心方案被推翻吗？他说 3 天后代码就上线了，推翻就推翻，下一版再调。如果写 5000 字 PRD 把所有边界 case 都列清楚，那是在给自己挖坑——你以为你想清楚了，其实你把"想清楚"的成本转嫁给了代码，等代码出来发现根本不是用户要的。

这句话我到现在还在琢磨。

在大厂，PRD 是一个"证据"——证明你想清楚了、有逻辑、有依据，老板签字了才敢动手。但在 AI native 小团队，PRD 是个"草稿"——写完就扔给代码，代码跑出来不对就改，改到对为止。**写 5000 字 PRD 的人和写 500 字 PRD 的人，区别不是文档能力，是他们对"什么是验证完成"的理解不同。**

传统 PM 认为"想清楚 = 完成"，AI native PM 认为"代码上线跑通 = 完成"。这两套认知在同一个公司里打架，团队越大打得越狠。

我后来在那个团队观察到 PM 一个很有意思的工作模式：他每天花 1-2 小时跟用户聊（真的是 1-1 视频，不是调研报告），剩下的时间全在写代码。不是写产品代码，是写工具——写 eval 脚本、写自动化测试、写 RAG pipeline 的 prompt、写一些内部 dashboard。CEO 跟我说，这个 PM 是组里最会写代码的 PM，但从来不写产品代码。

我问他为啥不写产品代码？他说写产品代码会让 PM 陷入"我做的功能我做主"的陷阱，反而离用户更远。他写工具，是为了让自己判断"用户卡在哪"的速度更快。

这个 PM 的 GitHub 我后来看了，半年提交了 800 多次，主要是 Python 和 TypeScript。他不是工程师，但用 Claude Code 写了大概 60% 的代码——他原话是"我不写代码，我描述代码，Claude 写，我看 review"。

## AI 工具到底怎么用？我见过三种典型姿势

我这一年接触过大概 7-8 个 AI native 团队，加上我自己带的两个 AI 产品项目，发现 PM 用 AI 工具的姿势大概分三种。**没有谁对谁错，但跟团队阶段强相关。**

第一种姿势我管它叫"AI 替代手"。PM 主要用 AI 来加速自己本来就干的活——写 PRD 让 AI 写、写邮件让 AI 写、做调研让 AI 搜、画原型让 AI 生成图。这种姿势常见于产品已经定型、PM 主要在维护阶段。优势是上手快，缺点是天花板低——PM 自己的工作没变化，只是变快了，等于拿了个更好的笔。

第二种姿势是"AI 扩边界"。PM 用 AI 干以前工程师干的活。我前面提到的那个 PM 就是这种——写 eval 脚本、写自动化测试、写 RAG pipeline。我自己最近半年也是这个姿势，PM 阶段我会自己写 agent 跑用户调研（脚本是 Claude Code 写的，跑了 600 条样本，提炼出来 12 个产品机会点），写出来直接交给产品同事去验证。这种姿势的优势是 PM 真的能"看见"产品在用户手里跑的样子，不是看数据报表猜。难点是 PM 得真学点技术——不是变成工程师，但要能看懂代码、能自己 debug 简单问题、能跟工程师对话不被忽悠。

第三种姿势是"AI 重构流程"。整个团队的 workflow 都是 AI 跑——产品 review 用 AI agent 自动评估代码改动有没有破坏体验、用户反馈用 RAG 自动归类到需求池、需求优先级用 LLM 评分、连周报都让 AI 生成。这种姿势对团队成熟度要求很高，常见于 ARR 过亿、团队 20+ 人的阶段。

我现在带的小团队大概在第二到第三种之间。**老实讲，从第二种往第三种走特别容易翻车。** 我们去年 11 月试过"全 agent 周报"，让一个 agent 每天汇总所有用户反馈 + 工程师 commit + 数据变化，自动生成周报。结果跑了两周大家都不看了——agent 生成的周报里全是"显著提升"、"持续优化"、"用户体验改善"这种废话，没有任何一条是"这个事做错了，需要改"。

后来我们改回人工写周报，但让 agent 当"周报质检"——PM 写完周报，agent 跑一遍把所有"显著"、"大幅"、"持续"这类虚词标红，逼着人写具体的数字。

这个反向用法意外好用。

## 招什么样的人？三个我见过的真问题

接下来讲招人。这是 5-10 人小团队最痛的环节。

我去年帮那个 11 人团队招 PM，他们的标准是：3 年以上 AI 产品经验 + 能写代码 + 大厂背景。挂出去两个月，收到 70 多份简历，约了 18 个面试，最后一个都没成。

我后来跟 CEO 复盘为什么招不到，他说了句特别实在的话：**"我要的人不存在。"**

他要的人是"既懂 AI 产品又懂代码又在大厂待过还不被大厂文化污染"——这四个条件互相矛盾。大厂出来的人 PM 流程重、AI 理解浅；AI 产品出身的人技术深、但 PM 基本功弱；技术强 PM 也强的人，早就自己创业去了，不会来你那 11 人公司打工。

最后他们妥协了，招了一个 95 后、AI 公司 2 年经验、PM 流程不熟但能写 Python 的人。CEO 说：我宁愿要"能学"的人，也不要"懂流程"的人。

第二个真问题是工程师和 PM 的边界。我观察到一个规律：小团队里 PM 越懂技术，工程师越自由；PM 越不懂技术，工程师越累。

为什么？因为 PM 不懂技术的时候，会把"我想要什么"翻译成"我想要一个按钮"——工程师拿到的是 UI 描述，不是问题描述，工程师得自己理解问题、自己想方案、自己评估技术债。这个工作量是隐性的，PM 看不到，CEO 也看不到，最后工程师跑路。

PM 懂技术（哪怕只是懂到能读代码、能在 Cursor 里改简单 bug），他写出来的是"用户卡在 step 3，原因是接口延迟，我们想用 X 方案压到 200ms 以内"——工程师拿到的是问题 + 方案边界，剩下的是技术实现。这种沟通效率差 3-5 倍。

第三个真问题是"AI 产品经验"到底值多少钱。我这两年面过几十个声称有"AI 产品经验"的 PM，认真聊下来大概 30% 是真的干过 AI 产品的，50% 是做 AI 加持的传统产品（用 AI 写文案、用 AI 做客服），剩下 20% 是简历包装的。

真正的 AI 产品经验和假的，面试问一个问题就能分出来：**"你的模型效果是怎么评测的？"** 如果对方讲不清楚 eval 怎么设计、数据怎么采样、bad case 怎么归因，那大概率是包装的。这个话题我下一章会展开讲。

## 我们组当时踩过的坑

讲两个我自己去年踩的坑，给后来人提个醒。

第一个坑是"过度追求工具统一"。我去年非要全组用同一个 RAG 框架、同一个 agent 框架、同一个 eval 平台，理由是"统一管理、降低学习成本"。结果半年下来，工程师怨声载道——因为不同任务适合的工具不一样，强制统一之后效率反而下降。

后来我学乖了：**工具是给人用的，不是给架构师用的。** 谁用谁选，选完报备一下团队就行，别一刀切。现在我们组的 RAG 用了三种（LlamaIndex、纯 prompt 工程的、还有自己写的），agent 框架也是工程师各自选，我只关心两件事：能不能跟其他系统打通、成本控不控得住。

第二个坑是"用 AI 替代 PM 的判断"。这件事我想了大半年。年初我试过让 agent 帮忙判断需求优先级——把需求列表喂给 LLM，让它按 ROI 排序。跑了一版，结果 agent 排出来前 5 的需求里，有 3 个是根本不该做的。

我后来意识到，**需求优先级判断不是信息处理，是价值判断。** Agent 擅长的是"这个需求技术上能不能做"、"这个需求历史上有没有用户提过"——但"这个需求该不该做"、"这个需求做了用户会不会真的用"——这是 PM 的活，不能外包。

我当时犯的错是把"判断"和"信息整理"混为一谈。信息整理可以交给 agent，判断必须 PM 来。

## 收尾：给 PM 两个本周能动手的建议

如果你现在在一个传统产品团队，想往 AI native 转型，别想着一步到位。我给两个小建议，都是我这半年验证过的。

第一个：**本周开始用 Claude Code 写一个真实可用的工具。** 不是 demo，是真的能跑、能给你每天省 30 分钟以上的那种。我自己的起步工具是"用户反馈归类脚本"——把散落在各个渠道的用户反馈拉下来，丢给 Claude 做主题归类 + 紧急度打分。脚本我用了大概 4 个小时写完（其中 3 小时在调试 prompt），现在每天早上自动跑一次，省我大概 45 分钟。

写完这个脚本的意义不是省时间，是让 PM 自己经历一次"AI 工具从 0 到 1"的过程。这个经验比读 10 篇 AI 产品文章都有用。

第二个：**本周开始跟一个用户做 30 分钟视频通话，不做任何产品介绍，只听他讲最近一次用你的产品卡在哪。** 不要用问卷，不要用调研报告，就面对面聊。AI 工具再强，也替代不了这个。

我去年跟用户聊了大概 200 多个小时，其中一半是无效闲聊，但另一半给了我至少 5 个"原来用户真的这么用产品"的反常识发现。这些发现没法通过 eval 跑出来，也没法让 agent 替我聊出来。

这是 PM 这个职业最不会被 AI 替代的部分。
