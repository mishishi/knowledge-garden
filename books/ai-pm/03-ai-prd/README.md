# 03. AI 时代 PRD 怎么写：新模板跟传统区别

去年在一家做企业知识库 AI 化的团队，我连续两个月让 PM 团队用 Claude 起草 PRD 初稿，结果出了一个事：有个 PM 把 PRD 丢给 Claude 一句话——"帮我写一个智能问答的 PRD"，Claude 洋洋洒洒写了 6000 字，结构完整、字段齐全，我看了 5 分钟就发现这东西完全没法用。

为什么？Claude 写的是一份"传统 SaaS PRD"——功能列表、用户故事、流程图描述、字段定义。看起来很专业，但里面没有任何一段是在回答"模型在这个场景里到底要做什么"。没有 prompt 段的草稿，没有评测样本的设计意图，没有失败 case 的预期处理路径，甚至连"幻觉"这个词都没出现过。

这份 PRD 给到工程团队之后会发生什么？工程师会问"模型的 system prompt 谁来定？边界 case 怎么办？评测怎么做？"——然后所有这些问题在 PRD 评审会上被重新提一遍，原本 1 小时能开完的会拖到 3 小时，原本"产品定义"的工作被偷偷转嫁给了工程。

这就是 2024 年还在很多团队真实发生的 PRD 失效问题。模板没变，工具升级了，但 PRD 里的"产品决策密度"反而变低了。

这一章我想聊的是：在 AI native 产品里，PRD 到底应该长什么样？哪些字段其实可以删掉，哪些字段是过去没有但现在必须加的？prompt 段怎么写才不变成"给开发看的伪代码"？评测 spec 怎么嵌进 PRD 里，才不会变成"事后补的 PPT"？以及一个很现实的问题：当 Claude 30 秒能给你生成一份像模像样的 PRD 时，PM 的判断力到底体现在哪里？

## 旧模板里哪些字段可以删了

传统 PRD 模板大概有这些大块：项目背景、用户故事、功能列表、流程图、字段定义、异常处理、验收标准、上线计划。十几年来这套结构在 SaaS 时代是有效的，因为它服务的对象是确定性逻辑——按钮点一下跳哪儿、字段怎么校验、状态怎么流转。

但 AI native 产品里，70% 的逻辑不再是确定性的。模型输出是非确定的，prompt 改一个字行为可能就飘了，传统意义上的"功能列表"在 AI 产品里其实是个伪命题——"智能问答"这个功能本身没有列表可言，决定它好不好的是 200 个评测样本上的表现，而不是 20 条用户故事。

那哪些字段真的可以删？我自己的实践里砍掉了这几块：

**业务流程图**。传统 PRD 必画 UML 或者 Visio 流程图，但 AI 产品里很多流程是模型决策驱动的，画出来也是"用户输入 → LLM → 输出"这种没营养的图。我在 PRD 里现在只画两个东西：数据流（用户问题怎么进系统、检索了什么、最后返回什么）和**人工兜底路径**（模型挂了 / 置信度低 / 命中红线时产品怎么降级）。后者才是 AI PRD 真正需要画的流程图。

**字段定义表**。传统 PRD 会详细列每个输入输出字段的类型、长度、枚举值。AI 产品里，模型的输出是自然语言，字段定义是后处理层的事，不是产品决策。产品应该关注的是"输出应该满足什么语义约束"——比如"必须基于检索内容回答、不能编造文档外的实体名、超过 3 个引用源时优先选最近 30 天的"。这些约束应该写在 prompt 段或者评测 spec 里，而不是字段定义表里。

**详尽的异常处理列表**。传统 PM 喜欢列"网络超时怎么办、用户重复点击怎么办、权限不足怎么办"——这些在 SaaS 里是产品边界。AI 产品里，异常处理的核心变成了"模型输出违反约束怎么办、置信度低于阈值怎么办、用户问题触犯红线怎么办"。前者工程团队自己能 cover，后三类才是 PM 必须定义的。

砍掉这些之后，传统 PRD 模板从原本 20+ 字段压缩到 7 个左右。看起来更轻了，但每段的内容密度反而上去了。我后来复盘发现，**PRD 的失败不是字段不够多，是 PM 真正做了决策的那几段被淹没在文档噪音里**。

## 哪些字段是新加的、必填的

接下来聊反过来——AI 时代 PRD 里有哪些字段是过去没有、现在必须有的。我自己用了大概 6 个月，迭代过 4 个版本才稳定下来。

**场景定义段**。这是最重要的一段，写在最前面。这段要回答的不是"我们要做什么功能"，而是"这个 AI 能力在什么场景下被调用、调用频率多少、用户期待什么形态的输出"。听起来虚，但其实是最决定 prompt 怎么写的一段。

举个例子，同样是"智能问答"，如果场景是客服辅助（坐席用，AI 给建议，错了能改），那 prompt 设计的容错空间很大，可以鼓励模型"多说点、给坐席参考"；如果是面向终端用户的法律咨询，prompt 必须收紧到"只引用文档、不能演绎、必须给出来源"。同样是"问答"功能，产品形态完全不同。

我在 PRD 里强制 PM 写一段不超过 200 字的场景描述，包含三个要素：谁在用、什么场景、失败一次的代价是什么。**"失败一次的代价"是过去 PRD 模板里完全没有的字段，但在 AI 产品里是核心**——它直接决定 prompt 该激进还是保守、评测指标该偏召回还是偏精确、上线门槛该多高。

**Prompt 段（草稿 + 决策）**。这一段不是把最终 prompt 贴上去，那是工程交付的产物。这一段要写的是：PM 怎么思考 prompt 的结构、为什么这么设计、哪些是 hard rule 哪些是 soft rule。

我自己的习惯是把 prompt 段拆成三个子段：核心约束（必须遵守的，比如"不引用文档外内容"）、风格约束（语气、长度、结构，比如"先给结论再给依据"）、边界约束（红线 case，比如"医疗诊断、投资建议、政治话题统一拒答"）。每条约束后面我会标一个"为什么"——不是给工程看的，是给 PM 自己看，提醒自己当初为什么这么设计。

举个我踩过的坑：当时做一个企业知识库的"智能总结"功能，我写"输出简洁、保留关键信息"，结果上线后用户反馈"太短、看不到细节"。我回去查 prompt 设计记录，发现"简洁"和"关键信息"是冲突的——PM 自己都没想清楚"简洁到多简洁、关键信息保留哪些"。后来我把 prompt 段改成了"输出不超过 3 段、每段不超过 80 字、必须包含【结论】【原因】【数据来源】三要素"。可执行、可验证、可评测。

**评测 spec 段**。传统 PRD 里没有这个字段，因为传统功能上线前 QA 测一遍就完事了。AI 产品里，上线前必须有评测数据——一批标注好的样本，PM 写明每条样本的预期行为，模型跑一遍看准确率 / 召回率 / 各种自定义指标。

我自己在 PRD 里强制写三个东西：评测集规模（最少 200 条，低于这个数字 PM 要解释为什么）、评测维度（不是单一的"准确率"，是分维度的——事实性、相关性、安全性、风格匹配度，每个维度单独打分）、**红线 case 单独列**（哪些是绝对不能错的，比如医疗 / 法律的硬性边界）。

这一段如果写不出来，PM 应该回去重写场景定义段——不是工程问题，是产品定义不清。**AI PRD 的本质是"把模糊的产品意图翻译成可验证的模型行为"**，评测 spec 是翻译的最后一公里。

**降级与人工介入路径**。AI 产品的 PRD 必须有这一段，写清楚：模型置信度低时怎么兜底、命中红线时怎么回复、用户不满意怎么转人工、模型宕机时产品怎么降级（降级到搜索？降级到 FAQ？还是直接提示暂不可用？）。

过去 PRD 里的"异常处理"是被动响应，AI PRD 里的降级设计是**主动产品决策**——它直接影响用户对产品的整体信任。我做企业知识库的时候设计过一个很简单的降级：模型置信度 < 0.6 时，回复末尾自动加一句"这个问题我不太确定，建议你也可以问下 XX 部门"。上线后用户调研里"信任度"这一项的得分从 3.2 涨到 4.1（5 分制），比单纯优化模型回答准确率的效果还明显。

**模型行为边界（不是参数，是决策）**。传统 PRD 不涉及模型，所以没有这一段。AI PRD 里 PM 必须写明几件事：哪些事情模型必须做、哪些必须不做、哪些可以做但要标注不确定性。这听起来跟 prompt 段重复，但定位不同——prompt 段是"怎么告诉模型"，行为边界段是"产品对模型的最终期望"，是上游决策。

我一般在 PRD 里画一个简单的边界矩阵：三列（必须做 / 可以做 / 不能做），每列下面列 5-10 条具体行为。这东西给到工程，工程能直接转成 prompt 的 hard rule 和 soft rule 优先级；给到评测团队，能直接转成"必须做"列 100% 通过、"可以做"列随机抽查、"不能做"列 0% 触犯的评测标准。

## Prompt 段怎么写才不变成伪代码

我见过很多 PM 把 prompt 段写成"给开发看的伪代码"——把最终 prompt 一字不漏贴进 PRD，然后写"按这个实现"。这种做法有两个问题：第一，PM 把自己降级成了"prompt 操作员"，没体现产品决策；第二，prompt 是迭代中的，最终 prompt 跟上线时的 prompt 几乎一定不同步。

我在 PRD 里写 prompt 段的逻辑是：**写设计意图，不写最终文本**。

具体说，我会写三件事。第一，prompt 包含哪几个模块（角色设定 / 任务描述 / 输入格式 / 约束 / 输出格式 / 示例）。第二，每个模块的核心意图是什么（角色设定是为了让模型进入什么专业语境、约束是为了防什么风险）。第三，哪些是 hard rule（违反直接拒答）、哪些是 soft rule（违反扣分但不拒）。

举个我去年做"合同条款智能解读"时的 prompt 段草稿（PRD 里的版本）：

> **角色设定**：模型是法律助理，负责把合同条款翻译成非专业人士能理解的语言。
> **任务描述**：用户粘贴一段合同文本，模型输出这段条款的核心权利义务、潜在风险、谈判建议。
> **核心约束（hard）**：
> - 不引用合同文本外的法律条文
> - 不给出"这个条款一定有效/无效"的绝对判断
> - 涉及金钱、责任、期限的条款必须原文引用
> **风格约束（soft）**：
> - 先给"这段在说什么"再给"潜在风险"
> - 风险描述用"建议关注"而不是"会出问题"
> - 不确定的地方明确标注"以下为一般性理解，具体以法律专业人士意见为准"
> **边界**：
> - 不替代律师、不输出可执行的合同修改建议
> - 涉及诉讼、仲裁、刑事的内容统一回复"建议咨询专业律师"

工程团队拿到这段，自己能写出 10 版不同的具体 prompt 文本，但产品意图是一致的。这是 prompt 段应该有的样子——**给原则不给文本，给约束不给实现**。

我也犯过错。有一次我直接把 Claude 生成的 prompt 贴进 PRD，还写了一段详细的解释，结果工程团队就照着实现了，上线后用户反馈"回复太机械、像在念条款"。我复盘发现，**PM 的责任不是写最终 prompt，是定义 prompt 应该满足的语义约束**。最终 prompt 是工程和 PM 在实现阶段共同迭代的产物，不是 PRD 阶段 PM 一个人能拍板的。

## 评测 spec 怎么嵌进 PRD 才不变成事后补的 PPT

这是 AI PRD 最难也最重要的一块。我见过 90% 的团队把评测当"上线前的 QA 环节"——产品先做、最后跑一遍评测、过线就上。这种做法的结果是：评测沦为形式，PM 真正关心的产品行为根本没法在评测里被检验。

我的做法是：**PRD 阶段就写评测 spec，跟 prompt 段同步**。

具体说，PRD 里的评测 spec 段必须包含四个东西：评测集来源（谁标注的、什么时候标的）、评测指标（分维度的，不是单一的准确率）、通过门槛（每条指标达到什么数字可以上线）、**评测集本身的小样例（PRD 里放 3-5 条典型样本，让评审者理解评测标准）**。

最后一点是反直觉但极有用的。我让 PM 在 PRD 里直接列 3-5 条典型样本：每条样本包含用户问题、上下文（如果有）、预期模型行为、为什么这是预期。评审者看 5 条样本，比看 5 页指标定义更能理解"产品到底在追求什么"。

举个例子，评测合同解读的 PRD 里我会放这样的样本：

> **样本 1**
> 用户问题：这段合同里我的年假是多少天？
> 上下文：合同中"乙方享有 10 天年假，须提前 7 天申请"
> 预期行为：直接回答"10 天年假、需提前 7 天申请"，引用原文
> 原因：信息明确、原文可引用，模型应直接给答案
>
> **样本 2**
> 用户问题：这段合同里我辞职的话有什么后果？
> 上下文：合同中"乙方提前 30 日书面通知可解除合同"
> 预期行为：回答"提前 30 天书面通知可解除"，并补充"这是常规条款，无额外违约金"
> 原因：信息明确但用户可能不熟悉行业惯例，模型应在合规范围内给扩展信息
>
> **样本 3**
> 用户问题：这段合同有效吗？
> 上下文：合同中"任何一方违约，违约金为合同总额 30%"
> 预期行为：拒绝回答绝对判断，给出"建议咨询专业律师"
> 原因：违反 hard rule "不给绝对判断"

这种样本写 5 条，PM 自己就会发现定义里有没有漏洞——比如"补充信息"到什么程度算合适？这种问题在抽象指标里看不出来，在具体样本里立刻暴露。

还有一个反例：之前某个团队 PRD 里只写了"准确率 ≥ 90%"，没维度、没样本、没边界。结果上线后用户反馈"回答不专业"，评测团队说"准确率达标 92%"。两边对"准确"的定义完全不同——评测团队算的是"包含正确答案"，用户要的是"答案专业、表达得体、风险提示到位"。**评测 spec 写得越具体，越能在 PRD 阶段就把 PM 自己的模糊意图暴露出来**。这不是负担，是工具。

## 我现在用的 PRD 模板（精简版）

最后给一个我目前稳定在用的 PRD 模板，不是表格化展示（前面说过表格不超过 1 个），是用段落形式把字段顺序和定位讲清楚。

PRD 头部 3 段：场景定义（200 字内，谁用、什么场景、失败代价）、产品目标（不超 3 条、可量化）、非目标（明确不做什么，这块在 AI 项目里尤其重要，模型能力边界很多)。

PRD 中部 3 段：Prompt 段设计意图（按模块写设计理由）、评测 spec（指标 + 维度 + 通过门槛 + 3-5 条典型样本）、降级与人工介入路径（什么时候降级、降级到哪、怎么提示用户）。

PRD 尾部 2 段：模型行为边界矩阵（必须做 / 可以做 / 不能做）、上线观察计划（前 7 天重点观察什么指标、什么情况下回滚）。最后这一段是传统 PRD 没有的，但 AI 产品必备——模型上线后的实际表现会偏离评测集，PM 必须提前定义"什么算异常、异常了怎么办"。

整套 PRD 我一般控制在 2000-3000 字。看起来比传统 PRD 短，但每段都是高密度决策，没有废话。工程团队拿到手不需要再问"你到底想要什么"——文档里每一段都在回答这个问题。

## 给 PM 的两个 actionable 建议

第一个，**这周就把现有 PRD 里"字段定义表"和"流程图"删掉**，看你会不会感到不安。如果你删掉之后觉得 PRD 什么都没了，说明你过去的 PRD 严重依赖"看起来专业"撑场，而不是"定义清晰"撑场。这是个很好的自我诊断。

第二个，**下个迭代里在 PRD 里加一个"失败一次的后果"字段**。每写一个 AI 功能就问自己：用户拿到的回答错了，最坏情况是什么？是用户觉得"这 AI 真笨"（可承受），还是用户基于错误回答做了错误决策（不可承受）？把答案写在 PRD 第一段，这个字段会反过来重塑你整个产品的风险偏好和 prompt 设计哲学。

我自己做产品 8 年，AI 时代之前以为 PRD 是"把想法结构化"——AI 时代之后才明白，**PRD 真正的作用是"把 PM 的产品判断力显性化、变成可验证的约束"**。Claude 能 30 秒给你一份"看起来像 PRD"的文档，但写不出 PM 真正做过的那些模糊决策——哪些边界值得守、哪些代价可以承担、哪些事情宁可不做。这是 AI 时代 PM 真正的价值所在。

下一章我会聊怎么跟工程团队协作——传统 PM 提需求、工程实现的模式在 AI 团队里基本失效了，PM 必须更早介入架构、更深度参与 prompt 迭代、更主动承担评测责任。这块我踩过的坑比 PRD 还多。
