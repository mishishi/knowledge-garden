# 09. AI 产品上线后怎么迭代：data + 评测 + user feedback

去年我做的一个对话产品上线第四周，DAU 到了 28 万，NPS 看着还行 41，但用户次日留存只有 19%。这组数据单独拿出来都看不懂——NPS 41 不算差，DAU 28 万对一个新功能来说已经及格，可次日留存 19% 显然塌方了。三组数据打架的时候，传统 PM 流程一般会让你先开会，对齐一下，然后排个优先级，发个问卷。但 AI 产品迭代的节奏不允许这么干，28 万 DAU 的产品一周能产生几百万条对话数据，你靠问卷回收 300 份再开会，这周的数据已经过期了。

我后来是这么处理的：当天下午让 agent 把四百万条对话按"用户下一次操作"做了分类，发现一个反直觉的现象——次日回访的人里，有 38% 在第一次对话里没完成任务，但他们对产品观感不差；而那 19% 次日留存的人，对话完成率反而高。意思是，产品帮用户"太高效"反而伤害留存，因为高效完成了就没动力回来。这个发现不是问卷问出来的，是对话日志里挖出来的。那一周我推翻了原本的"提升任务完成率"路线图，改成"故意留一些探索性 hooks"，六周后次日留存爬到 31%。没有这三方信号融合，这事根本做不出来。

## 三方信号是怎么打架的

AI 产品迭代和传统产品最大的区别，是信号源的颗粒度。传统产品你看 DAU、留存、转化漏斗、看 AB test 结果，主要靠行为数据。AI 产品不一样，你有三套并行信号：

- 行为数据：DAU、留存、功能渗透率、付费转化，跟传统产品基本一样
- 评测数据（eval）：模型输出质量的量化打分，可以离线算，也可以在线算
- 用户反馈：NPS、客诉、客服工单、对话里用户的"吐槽"——这个在 AI 产品里特别重要，因为 AI 回复质量差用户会直接骂出来

问题来了，这三套信号经常打架。我说个真实场景，去年某电商客服助手项目，行为数据显示功能渗透率 67%，看着挺高；评测数据（我们用 LLM-as-judge 打的分）显示回答准确率 0.82；用户反馈 NPS 38。单独看都还行。但当周客服工单激增 240%，工单关键词是"答非所问"。三方信号完全相反。

我当时第一反应是去看评测数据哪里出问题了。拉了 600 条样本人工复核，发现一个尴尬的事：评测集本身有问题。我们当时评测集是用历史对话标注的，但客服场景已经升级到 2.0 版本，旧的"准确率"定义跟新的业务诉求对不上。0.82 这个分数对老场景是高分，对新场景其实是低分。这就是评测集漂移（eval drift）——评测集没跟上产品迭代。

后来我们的处理方式是：评测集每两周重抽 20%，由 PM + 领域专家联合标注，不再一次性标注一个大评测集然后用半年。这事我后面展开讲。

三方信号的优先级，我自己的经验是：先看行为数据的异常点（数据本身不会撒谎），再看评测数据（判断是不是质量问题），最后看用户反馈（理解用户主观感受）。但用户反馈经常被低估，因为它"不量化"，PM 不愿意花时间读。其实客服工单、用户原话这些是最直接的产品真相。我在那个电商项目里最后发现，工单里用户原话指向一个具体 bug——退款政策类问题模型会过度承诺，导致用户实际去退款时被拒。三个信号源都指向同一个问题，只是粒度不同。

## 评测这件事，传统 PM 大多数是低估的

做 AI 产品之前我根本没系统做过 eval。我之前在传统产品时代，迭代靠 A/B test，靠数据埋点，看留存转化漏斗就完事了。AI 产品不行——你同一个 prompt 改两个词，用户行为数据可能看不出明显差异，但模型输出质量可能已经塌了；反过来，模型输出"看起来"变好了，但实际是讨好用户，用户行为数据反而下降。

我去年花了三个月才真正搞明白 eval-in-prod 这件事。不是说离线评测不重要，离线评测是必要不充分条件，你发布前必须有离线评测卡住质量底线。但离线评测不能告诉你产品上线后真实用户场景下表现怎么样，因为评测集永远是你自己标注的，你的标注分布跟真实用户分布有 gap。

eval-in-prod 怎么做？我自己的做法分三档。

第一档是基础款：在线抽样 + 人工标注。1% 的真实流量抽样，人工评估模型输出质量。这是最土但最有效的方法。我去年带 PM 团队做的对话产品，每周五下午组里三个人专门花两小时标注上周抽样的 200 条对话，打分 1-5。这事听着很笨，但 200 条样本看下来，你能直观感受到模型这一周是变好了还是变差了，比看任何指标都准。

第二档是 LLM-as-judge。模型打分模型，这事争议很大。我自己踩过坑——早期用 GPT-4 当 judge 评估我们自己的模型，发现 judge 给分跟人工给分相关性只有 0.61，做决策风险太高。后来改成几个 trick：用更强的模型当 judge（Claude Opus 比 GPT-4 在我们场景里强一档）、用 chain-of-thought 让 judge 给出 reasoning 再打分、多个 judge 投票。这些加上之后相关性爬到 0.81，但还是不够。我现在的做法是：LLM-as-judge 用于大批量筛选 + 发现异常样本，人工复核只针对异常样本，效率提升 10 倍但判断质量不降。

第三档是 implicit feedback。用户行为本身就是反馈。比如用户重写了 AI 给的文案——说明 AI 给的不行；用户复制粘贴了 AI 给的内容——说明能用；用户在 AI 回复后立刻退出——大概率不满意。这套信号在传统产品里就有，但 AI 产品里颗粒度更细，因为你有了对话流。

这三档不是替代关系，是叠加。我现在做的产品基本是三档并行：1% 人工标注 + 全量 LLM-as-judge 监控 + 行为反馈信号回流到评测集。三套并行才能稳定判断"模型这一周到底变没变好"。

## A/B test 在 AI 产品里为什么经常失灵

说个得罪同行的观点：A/B test 在 AI 产品迭代里不是银弹。我去年好几个项目 AB test 得出的结论都是错的，原因有几个。

第一，AI 产品的方差太大。传统产品你改个按钮颜色，DAU 变化可能很稳定；AI 产品你改个 prompt，同一个用户两次请求可能得到完全不同的回复，方差大到 AB test 跑一周都看不出显著性。我们之前做的一个 AI 写作工具，改 prompt 这件事 AB test 跑了 14 天，p 值一直在 0.1 左右徘徊。最后不是 AB test 给的答案，是用户访谈 + 留存数据综合判断的。

第二，AI 产品的"成功指标"很难定义。传统电商 A/B test 转化率就行，AI 产品你要看什么？看留存？留存受太多因素影响；看任务完成率？任务完成率可能跟用户满意度负相关（前面那个例子）。我现在做 AI 产品 AB test 一般同时跑三四个指标，看综合趋势，不依赖单一指标的显著性。

第三，AI 产品的网络效应让 AB test 分组出问题。一个用户用 A 版本生成的内容发给另一个用户，那个用户看到的内容其实是被污染过的。我们做协作类 AI 工具时遇到过，AB test 跑出来 A 版本好，但实际是因为 A 版本的用户群体更活跃。

那 AB test 是不是就没用了？不是。AB test 还是有用的，但定位要降一级——它不是"决策的唯一依据"，而是"排除明显错误的工具"。我现在的做法是：重大改动先 AB test 跑一周，看有没有显著负面影响；如果没有显著负面影响，就推全量，然后在 eval-in-prod 里持续监控。这比传统 PM 那种"AB test 不显著就不上"的思路要激进，但 AI 产品迭代节奏不允许慢慢来。

## 评测集怎么建，才不会三个月后过期

这事我交过学费。去年初我们做了一个客服 AI 助手，吭哧吭哧标了 5000 条评测集，覆盖 12 类业务场景，跑了一个季度。当时觉得挺扎实。结果 Q3 业务调整，新增了三个业务线，老评测集覆盖率掉到 60% 以下，新场景完全没覆盖，0.82 的"高分"其实只反映了 60% 的真实场景。

从那以后我给自己立了个规矩：评测集是流动资产，不是固定资产。具体做法——

评测集来源必须是真实用户对话，不是 PM 脑补。我之前犯过的错是 PM 团队自己造评测样本，造了 1000 条"理想对话"，跑下来分数很好看，但用户真实对话根本不长那样。现在我的做法是每月从线上对话里抽 5% 进评测集候选池，PM + 领域专家每周标 50 条进评测集，每月淘汰 5% 旧的、补充 5% 新的。这样评测集永远跟真实分布接近。

评测维度不要只看"准确率"一个数字。我现在做 AI 产品评测至少分四维：事实准确性、风格一致性、安全合规、用户价值。每个维度单独打分，不要合成一个总分。原因是合成总分会掩盖问题——你安全合规掉 10 分，可能用风格一致性涨 5 分补回来，看着总分没变其实已经出大事了。

评测集要有对抗样本。说白了就是用户故意刁难的场景。"退款怎么这么慢"和"你们是不是诈骗"和"我要投诉到 12315"——这三种问法意图完全不同，模型表现可能差异巨大。这部分样本你不能等真实用户给你造，要 PM 主动造。

我自己的对抗样本库大概每两周补一轮，每次 20-30 条。现在我带 PM 团队做 AI 产品时，评测集维护是 PM 的核心职责之一，不是工程师的事。PM 必须理解评测集的覆盖范围、维度分布、漂移情况，否则就是在闭眼开车。

## 怎么用 agent 加速迭代

这部分展开讲讲，因为这是 2024-2026 这两年 AI PM 真正的变化。我现在每周工作流里 agent 占了相当大比重，但用法不是"让 agent 写 PRD"那么简单。

第一个用法是 eval agent。我有一个专门跑评测的 agent，跑在 Claude Code 上，每天自动抽样当天对话、跑 LLM-as-judge、把异常样本汇总到一个文档。第二天早上我看文档就行。这事以前要 PM 每周花半天干，现在 agent 半小时搞定，效率提升 10 倍不止。

第二个用法是 user feedback 分类 agent。客服工单、NPS 调研原话、应用商店评论，这些非结构化反馈以前要 PM 自己读、自己分类。现在我让 agent 跑分类，跑完之后我自己复核 20% 边界 case。一个 5000 条评论的分析以前要两天，现在两小时。

第三个用法是 hypothesis 生成 agent。这是最微妙的一个。我让 agent 看本周数据异常 + 评测异常 + 用户反馈高频词，生成 5-10 个"为什么发生这个"的假设，我从中挑 3 个去验证。注意——agent 给的是假设不是结论，我从来不直接相信 agent 的因果推断，但它能帮我快速发散思路。我去年用这个方法发现了一个 PM 自己绝对想不到的现象：周四下午用户的对话平均轮次比周二高 38%，原因是用户周四开始规划周末活动。这种数据点不在任何 dashboard 上，是 agent 帮我挖出来的。

第四个用法是 PRD 草稿生成。这事我自己也用，但用法比较克制——我让 agent 生成 PRD 的"骨架"（目标、用户故事、验收标准），我自己填具体细节和取舍逻辑。因为 PRD 里真正值钱的是 PM 的判断，不是结构化文本。这事讲过很多次了不再展开。

但是——我说个反直觉的事——agent 加速迭代有个副作用：迭代节奏太快，PM 容易失去对产品的深层理解。我去年 Q3 有一个月迭代特别快，每周发三个版本，结果 PM 团队对产品的"感觉"反而变弱了——每个人都盯着数据看，没人真的去用产品。这事后来我反思过，强制要求 PM 每周至少亲自用产品 2 小时，不带任何分析目的，就是用。数据是辅助的，PM 对产品的直觉才是核心。

## 怎么把这些事落地到团队流程

最后讲落地。AI 产品迭代流程跟传统产品不一样，我现在的周节奏大概是这样的——

周一上午：看上周数据 dashboard，重点看异常点。agent 提前一天生成报告，我看 30 分钟。

周二下午：评测数据复盘。重点看模型质量有没有漂移、评测集覆盖率有没有掉。PM + 算法工程师 + 领域专家 30 分钟对齐。

周三：user feedback 深度分析。读 30 条真实用户原话，不量化，纯感受。这事我以前觉得浪费时间，现在觉得是 AI PM 最值钱的事之一。用户的原话里有产品真正的样子，dashboard 反映不出来。

周四：迭代决策会。基于上面三套信号，决定本周做什么、不做什么。决策会上我要求每个候选方案必须有三方信号支持，不能只靠单一信号拍脑袋。

周五：发版 + eval-in-prod 启动。新版本上线后第一件事是看 eval 数据，不是看行为数据。行为数据有滞后，eval 数据实时。

这套流程不是教条，是我自己跑了一年迭代出来的。每个团队不一样，但核心逻辑是一样的——三方信号融合、迭代节奏快于传统产品、PM 必须深度介入评测。

如果你这周就要动手，我建议两件事。第一，找你现在的产品里一个 AI 功能，做一次三方信号对齐——把行为数据、评测数据、用户反馈拉到一起看一遍，三方信号对不上的地方就是你产品最大的改进点。第二，开始建评测集，不用大，100 条就够，但必须是真实用户对话、必须分维度标注。建完之后你会发现你看产品的方式不一样了。
