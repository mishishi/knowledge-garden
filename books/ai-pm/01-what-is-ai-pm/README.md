# AI PM 是什么: 2026 角色重塑

去年 12 月底,我做了一次挺尴尬的盘点. 把过去一年 PM 团队 14 个人每周时间花在哪里扒了一遍, 具体到工时, 不是拍脑袋估的, 是从 Linear / Notion / Slack 自动化日志里跑的. 结果很打脸: 一份典型 PRD, 从立项到定稿, 写文档本身只占 13% 的时间. 剩下的 87% 花在了找数据、找对齐、找 stakeholder 签字、改措辞、开会、再改措辞.

我当时就一个念头——这事儿不对.

到了今年 Q1, 我把团队 14 人分成两组做 A/B: 一组继续老流程, 一组强制用 agent 协作 (主要是 Cursor + Claude Code + 我们自己攒的一堆 prompt 和 MCP). 三个月后, 第二组人均每周节省 11 小时. 不是 11%, 是 11 小时. PRD 的初稿时间从平均 3.2 天压到 1.4 天. 用户访谈整理从一份 800 分钟录音要熬两个晚上, 变成 40 分钟出初稿.

我把这组数字给团队 leader 看, 他就回了一句: "那剩下的人干嘛?"

这其实是这一整章我想回答的核心问题: 2026 年, AI PM 到底是个什么角色?

## 不是 PRD 写得快, 是 PRD 这个东西在变形

先说一个残酷的事实: 过去 PM 最核心的"工作产出", 也就是那份 30-80 页的 PRD, 它的功能正在被快速拆解.

我去年带团队做了一年的企业级 SaaS, 我们当时的 PRD 模板是这样的 (跟很多大厂差不多): 一二三四五六七八九十, 背景、目标、用户故事、流程图、原型、埋点、数据指标、风险、依赖、排期. 完整一份 PRD 平均 47 页, 写完平均 4.5 天.

说实话, 这份模板本身就是反人类的. 哪有人真的读完 47 页 PRD? 工程师读到第三页就开始跳着翻图. Designer 直接问"原型在哪". Stakeholder 看个摘要. 真正被仔细读的部分, 不到 20%.

到今年, 我让团队做了一个实验: 弃用 47 页模板, 强制每份 PRD 不超过 6 页, 必须包含 4 块内容——核心场景的 2-3 个真实用户故事 (不是 As a user I want 这种废话, 是真的"周二下午 3 点, 一个外贸业务员在广州咖啡馆, 客户在催单, 系统突然崩了"这种), 关键决策的技术约束 (不是技术方案, 是约束, 比如"必须支持离线操作"而非"用 SQLite"), 验证失败的指标 (就是什么情况下这个方案算死), 还有一份简短的争议清单 (团队内部分歧记录).

这份新模板跑下来, 真实数据是: 平均写一份 1.8 天, 不是省了 1.4 天, 是把"凑模板"的时间砍掉了, 真正花在思考上的时间反而多了. 工程师反馈是 "终于不用跳着读了", Designer 反馈是 "终于不用问原型在哪了", stakeholder 反馈是 "终于能在 5 分钟内判断要不要签".

这里有个我想说的点: AI 时代 PM 最大的变化不是写 PRD 变快了, 是 PRD 这东西从"交付物"变成了"协作语言". 过去的 PRD 是给 stakeholder 签字用的, 是责任划分的法律文书. 现在的 PRD 是给 agent + 人 + 工程师 + Designer 一起看的, 是对齐用的. 后者不需要 47 页, 前者早该死了.

但这里我得承认一个我自己走过错的路径. 我中间试过完全用 agent 生成 PRD——把一个需求丢给 Claude, 让它产出完整 PRD. 产出了, 格式漂亮, 逻辑通顺, 50 页. 但没用. 为什么? 因为 PRD 里有大量隐性知识, 是 PM 跟业务方吃饭时候聊出来的、跟客服团队翻工单翻出来的、跟工程师吵完架才搞清楚的. 这些东西不在 prompt 里, agent 拿不到. AI 生成的 PRD 像一份精致的地图, 但地图上没有"这条路过去是死胡同, 我们 3 个月前走过". 那种坑, 只能 PM 自己写.

所以现在我的做法是: agent 负责生成初稿、查漏、校验一致性、写埋点 schema. PM 负责 4 块东西——核心场景的真实性、关键约束的边界感、争议清单 (这是 agent 完全做不了的)、还有"为什么不做 B 方案" (这是 PRD 里最值钱也最容易被省略的一段). 这种分工下来, PM 真正的核心能力——商业判断和隐性知识挖掘——反而被放大了.

## 跟 agent 协作, 不是用 agent, 是把它当团队成员

这一点我走了很长的弯路才想明白. 2023 年我刚开始用 GPT-4 的时候, 我的用法就是把它当工具——一个更聪明的搜索框, 一个能写的实习生. 帮我写邮件, 帮我整理会议纪要, 帮我查资料. 这种用法其实没改变 PM 工作, 只是提速了 30% 左右.

2024 年下半年, 我开始重新理解这件事. 转折点是有一天我用 Claude Code 跑一个真实的产品分析任务——把过去 6 个月的客服工单全跑一遍, 按主题聚类, 找出 top 20 高频问题, 然后给我每类问题配 3 个最有代表性的原始工单. 我原本以为这就是个简单的 RAG + 聚类任务, 让一个实习生干也行. 但跑出来以后我意识到, 这件事真正的价值不是省了实习生 3 天时间, 是它让我看到了我自己看不到的东西.

具体说, 它跑出来的聚类里有一类我从来不知道存在: 大约 8.7% 的工单来自"用户登录后看到的是空的, 不知道下一步该干嘛". 这个数字不大, 但它暴露了一个我之前完全没意识到的 onboarding 漏洞. 我们的 onboarding 设计是给新用户一个 onboarding 引导, 但引导完成后用户回到主页时, 主页是空的——还没有数据, 没有内容, 没有任何上下文. 这 8.7% 的用户就这么流失了.

这件事不是 agent "做"的, 是 PM (我) 看完 agent 的输出后做的判断. 但没有 agent, 我可能永远不会知道这个数据存在. 因为它的占比不够大到触发任何 alert, 但又足够大到真实影响留存.

从这件事以后, 我开始用一种新的方式跟 agent 协作. 我把它当成一个团队成员——一个没有上下文的、需要我持续 supply context 的、不会累的、可以被并行 5 个的团队成员. 我现在的标准做法是: 每个重要决策前, 我会同时跑 3-5 个不同角度的 agent 分析. 一个从用户视角, 一个从数据视角, 一个从竞品视角, 一个从风险视角, 一个从技术可行性视角. 跑完以后, 我自己来做综合判断. 这种方式下, PM 的角色从"自己写答案"变成了"管理一群 agent 找答案, 自己拍板".

这里有一个反直觉的发现: agent 越强, PM 的 prompt 能力越重要. 我以为 GPT-5 / Claude Opus 4 出来以后, prompt engineering 会过时. 错了. 完全反过来了. 真正决定 agent 输出质量的, 不是模型本身, 是你给它 supply 的 context. 一个 6 行的 prompt 和一个 200 行的 prompt, 输出质量差距巨大, 而且差距会随着模型变强而放大. 因为弱模型你再怎么给 context 也就那样, 强模型你给它 200 行 context 它真的能读进去. 这就是为什么我后来让团队所有人都要学 prompt 写作——不是那种"教你写好 prompt 的 7 个技巧"的废话, 是真的能写出一个 500 行的、有完整业务 context 的、能跑出有价值输出的 prompt. 这是 PM 在 2026 年最硬的技能之一.

我们团队现在内部有一个不成文的规定: 每个 PM 手上必须有 3-5 个"活"的 prompt, 是那种自己长期打磨、反复迭代、在真实业务里跑过的. 不是网上 copy 的, 是自己写的. 这种 prompt 跟你的业务深度绑定, 别人抄不走. 我手上有一个 prompt 跑了快 2 年, 改了 200 多次, 它就是我的"外置大脑"——专门用来做用户反馈分析. 它比我带的任何实习生都更懂我的业务.

## 跟 Engineering 共创架构这件事, 过去 PM 是不参与的

这里我想说一件很多 PM 会忽略的事: 2024-2026 年, AI 产品最大的变化不是 UI 变了, 是架构变了. 我们以前做产品, PM 定义功能, Engineering 实现功能. 这条线很清晰. 但 AI native 产品里, 这条线模糊了.

举一个我自己的真实例子. 我们团队去年做了一个文档问答产品, 让用户上传 PDF 然后提问. 最开始我们 (PM) 的方案是这样的: 用户上传 PDF → 后端 OCR + chunking → embedding → 存向量库 → 用户提问 → 检索 → LLM 生成答案. 这个流程在 PM 看来天经地义, 因为这就是教科书式的 RAG. 然后 Engineering 实现的时候发现, 90% 的用户问的不是"内容是什么", 是"这一段在第几页". 也就是说, 用户的真实需求是定位, 不是问答. 但 PM 的架构方案把整个产品定义成了问答, 所以最后用户体验很差, 工程师也很憋屈——他们知道问题在哪, 但 PM 没给他们参与架构决策的入口.

这个错我承认. 后来我让团队做了一个新规则: 任何涉及 AI 能力的项目, PM 和 Engineering 必须从需求阶段就一起工作, 而不是 PM 出 PRD, Engineering 出技术方案. 这种共创不是开会, 是真的在一起画架构图. PM 出业务约束 (用户会问什么、答错的代价是什么、延迟容忍度多少), Engineering 出技术选项, 两个人一起决定 "我们到底是用 RAG 还是 fine-tune 还是 hybrid", "我们到底要不要 streaming", "我们到底要不要先做一轮 query rewriting".

这件事在传统 PM 培训里是不教的. 传统 PM 教的是"别越界, 让 Engineering 决定技术". 但在 AI 时代, 这是错的. AI 产品的很多关键决策——比如要不要 fine-tune、用什么 embedding 模型、retrieval 召回率要求多少——本质上是产品决策, 不是技术决策. 如果 PM 不参与, 这些决策就会被工程师用技术视角默认掉, 而技术视角常常跟业务视角冲突. 比如工程师会本能地追求更高的召回率, 但更高的召回率意味着更高的延迟和成本, 这对某些场景是不可接受的. 这种 trade-off 必须 PM 一起拍.

我现在的做法是: 每个 AI 项目的 kickoff 会议, 必须是 PM + Tech Lead 联合主持, 而且必须在会上产出至少一份"技术 + 产品共同签字的架构决策清单". 这份清单不是技术方案, 是约束清单: 召回率下限多少、延迟上限多少、成本上限多少、答错率上限多少、出错时降级策略是什么. 这五项约束定下来, Engineering 才有边界去做方案. PM 也才能在边界内做产品决策.

这个变化带来的副产品是: PM 团队必须懂 AI 技术. 不是说 PM 要会写代码, 是 PM 必须能跟工程师在同一个抽象层对话. 比如 PM 必须知道"召回率"和"准确率"的区别, 必须知道"fine-tune"和"RAG"的 trade-off 在哪里, 必须知道"context window"对产品体验的真实影响. 不懂这些的 PM 在 AI 项目里会被 Engineering 自动降级为需求录入员. 我见过太多这样的案例, PM 觉得自己被边缘化了, 其实是因为他在技术对话里接不上话. 这不是工程师的问题, 是 PM 自己没进化.

但这里有个我想 balance 的点: 我也不想把 PM 变成半个工程师. PM 的核心价值不是技术深度, 是商业判断 + 用户理解 + 跨团队协调. 技术理解是 PM 的必要条件, 不是充分条件. 我见过一些 PM 为了显得懂技术, 跑去学 PyTorch, 学 LangChain 源码, 最后变成了一个不写代码的工程师. 这是错的. PM 需要懂技术到什么程度? 够跟工程师吵架就行, 不够自己写代码. 这条线很微妙, 但很重要.

## 哪些 PM 价值在 agent 替不了

到这里我想正面回答一个问题: 哪些 PM 工作是 agent 替不了的?

我自己跑下来的答案, 大致有这么几块, 按重要性排:

第一, 商业判断. 一个功能该不该做, 投入 3 个人做 6 周还是 1 个人做 1 周, 资源怎么分配, 优先级怎么定. 这种决策需要 PM 理解公司战略、当前资源约束、组织能力、各方利益诉求. 这是纯 context-heavy 的决策, agent 没有这个 context, 做了也白做. 而且这种决策往往没有"正确答案", 只有"在当前约束下的次优解". Agent 不擅长这个.

第二, 隐性知识挖掘. 上面那个 onboarding 的例子就是. 用户没说的需求, 业务方没讲清楚的需求, 客服团队默默在处理但没人告诉你的需求. 这些东西 agent 能帮你 surface, 但解读必须是 PM. 一个工单 cluster 暴露出来, "为什么这个 cluster 存在"这个问题, 只有 PM 能回答. 因为 PM 是唯一同时跟业务方、用户、客服、工程都打过交道的人. 这个 cross-stakeholder 的 context, agent 没有.

第三, 跨团队政治. 这个我不想美化它, 但它是真实存在的. 一份需求推到签字阶段, 卡在某个 stakeholder 那里, 这个 stakeholder 为什么卡, 是真的担心还是想要 credit 还是政治站位. 这种判断, PM 要做. Agent 完全做不了, 也不应该做. 跨团队政治不是脏活, 是组织运转的润滑剂. 没有这块能力的 PM, 项目推不动.

第四, 关键时刻的判断. 一个 AI 产品上线后出问题了, 是模型的问题还是数据的问题还是 prompt 的问题? 是紧急回滚还是 hot fix 还是观望? 这种决策需要 PM 在压力下快速整合信息、做不完美的判断. Agent 能给分析, 但最终拍板必须是 PM. 因为这是责任问题, 不是能力问题.

反过来, 哪些 PM 工作正在被 agent 替代? 这个清单更长: PRD 初稿、用户访谈整理、竞品分析、数据报告生成、会议纪要、邮件草拟、需求 backlog 整理、bug triage 草稿、A/B test 结果初步分析. 这些事情 2026 年的 agent 做得已经比 80% 的 PM 好了. 如果一个 PM 2026 年还在花大量时间做这些事, 他要么是在浪费自己时间, 要么是在做错事——因为 agent 做这些事更快、更便宜、更不容易出错.

我自己的真实数据是: 我现在每周的 40 小时里, 大约 8-10 小时花在直接用 agent 输出 (上面那些被替代的事), 12-15 小时花在跟 agent 协作产出 (写 prompt、review agent 输出、综合多个 agent 视角), 剩下 12-18 小时花在 agent 替不了的事 (商业判断、跨团队沟通、关键拍板、深度的用户理解). 这个分布跟 2023 年比完全反过来了. 2023 年我大概 60% 时间在产出, 40% 在判断. 现在反了.

## 给你的两个 actionable 建议

如果你读完上面这些, 我希望你这两周就做两件事. 不是"思考一下", 不是"评估一下", 是真的做.

第一件事, 盘点你自己的时间. 拿一周的时间, 把你做的每一件事按 30 分钟粒度记下来. 然后把每件事分成三类: agent 能做 (但你还在手动做)、agent 能辅助 (你做 + agent 一起做更快)、agent 做不了 (必须你做). 我赌你 40-50% 的时间会落在第一类. 这 40-50% 就是你马上能省出来的时间. 怎么省? 不是明天就开始学 AI, 是明天就打开 Cursor 或者 Claude Code, 把第一类里的某一项 (我建议从"会议纪要"或"用户访谈整理"开始, 因为最容易上手) 跑起来. 跑完以后把时间对比记下来. 这种 first-hand 数据比任何方法论都管用.

第二件事, 找一个你手上正在做的、稍微有点复杂的决策, 跑一遍"5 agent 并行"的方法. 我上面提到的方法——同时从 5 个视角跑 agent 分析, 然后自己做综合判断. 这个方法你跑一次就能感觉到 agent 协作的真正价值, 不是提速, 是扩维. 它让你看到你单视角看不到的东西. 我建议选那种"我以为我已经想清楚了"的决策, 不要选真正模糊的决策. 因为前者的对比感更强, 你能清楚看到 agent 给你补了什么视角.

这两件事做完了, 你会自然进入下一章——用 AI 跑用户调研. 那是我认为 2026 年 PM 工作流里第二大的变化, 仅次于 PRD 重构. 我们下一章见.
