# 04. Select 策略：RAG 与检索

2024 年我做产品的时候，跟一个 RAG 团队的工程师吵了 3 个小时。

他说："我们 RAG 准确率 95%。"他的 RAG 系统召回 10 个 chunk，喂给 LLM 总结，LLM 输出答案。
我说："95% 召回率不代表 95% 答案准确率。模型读 10 个 chunk，里面 7 个无关，它会幻觉。"
他说："但我们测的 benchmark..."

**Benchmark 测的是检索的准确性，不是端到端答案的准确性。** 这是 2026 年大部分 RAG 团队还在犯的错。

这一章讲 Select 策略——把"对的"信息从外面选回来，放进 context。**重点不是召回率，是端到端答案质量。**

## Select 不是 "向量检索 + TopK"

最常见的误解：RAG = "把用户问题变成 embedding，去向量数据库里找最像的 5 个 chunk"。

错。

RAG 的全称是 **Retrieval-Augmented Generation**。Retrieval 是手段，Augmented Generation 是目的。**你的设计目标是让 LLM 生成更好的答案，不是"找最像的 5 个 chunk"。**

2026 年主流的 Select 策略是**多路召回 + 重排序 + 过滤**：

**多路召回**（Multi-path Retrieval）：
- 路径 1：向量相似度（semantic）
- 路径 2：关键词匹配（BM25 / Elasticsearch）
- 路径 3：元数据过滤（时间 / 作者 / 类型）
- 路径 4：图遍历（图数据库）
- 路径 5：结构化查询（SQL / GraphQL）

每路召回 N 个结果（比如各 10 个），合并去重，得 30-50 个候选。

**重排序**（Reranking）：
- 用 cross-encoder 模型（专门训练用于"判断 query-document 相关性"）给每个候选打分
- 比如 bge-reranker、cohere-rerank、jina-reranker
- 选 top K（比如 5-10）进 context

**过滤**（Filtering）：
- 去掉明显无关的（score 低于阈值）
- 去掉重复 / 高度相似的
- 按业务规则过滤（"退款相关查询只能查 2025 年之后的文档"）

**这套"多路 + 重排序 + 过滤"在 2026 年已经是 RAG 的事实标准**。单路 TopK 已经是 2023 年的玩法了。

## 3 个选错的代价

如果 Select 选错了，3 种灾难：

**1. 召回不全。** 答案需要的信息没在 context 里。LLM 只好 hallucination。**用户问"我的订单什么时候到？"，RAG 召回 5 个 chunk 全是产品介绍，没召回物流信息。LLM 编一个"3-5 天"。**

**2. 召回太多 / 太杂。** Context 塞了 30 个 chunk，模型注意力被分散。**"Lost in the Middle" 现象让中间 60% 的内容几乎没用。** LLM 拿到一堆信息但不知道哪些是关键的。

**3. 召回错误的。** 召回的 chunk 跟问题相关但是错的版本（比如旧政策、过时数据、错误的用户）。LLM 基于错误信息给答案，看起来"自信"但完全错。

我的 2026 年经验：**端到端答案质量跟 RAG 召回准确率的相关性只有 0.6-0.7。**剩下 30-40% 取决于**选多少、怎么排序、怎么过滤**。

## Recursive / Multi-hop Retrieval

复杂问题不能一次检索搞定。比如：

> "我们公司 2024 年发布的、面向小客户的、AI 相关的、最受欢迎的产品是什么？"

一次检索召回 5 个 chunk 解决不了这个问题——需要 4 步：
1. 召回"公司 2024 年产品列表" → 10 个
2. 召回"小客户产品" → 5 个
3. 召回"AI 相关产品" → 8 个
4. 求交集 + 召回"最受欢迎" → 3 个

**这就是 Agentic RAG**。不是一次性检索，是**多步推理 + 多次检索**。第 9 章会详细展开。

## 召回的"粒度"问题

2026 年最被低估的 RAG 问题是**chunk 切多大**。

太小（< 200 token）：召回的 chunk 不完整，缺上下文。LLM 拿到一句话不知道怎么用。
太大（> 1000 token）：召回的 chunk 噪声多，浪费 context window。

我自己的 2026 年经验值：**300-500 token 最好**。配合按段落 + 标题切，不要按固定长度切。

**Markdown 结构化文档**特别容易处理——按 ## 切，保留 200 字 overlap（避免切断句子）。**长 PDF 报告**最麻烦——要按页 + 段落 + 表格 3 种 chunk 策略。

## 选多少进 context

这是个工程参数。我自己的 2026 年默认：
- **简单问答**：3-5 个 chunk（1500-2500 token）
- **多文档综合**：8-12 个 chunk（4000-6000 token）
- **复杂研究**：15-25 个 chunk（8000-12000 token）

**超过 25 个 chunk，模型就开始被淹没了**。Lost in the Middle 现象变得明显。

**最贵的不是"选得不够"，是"选得太多"**。选多了的代价是污染 context 有效区。选不够的代价是漏信息——**漏信息的代价可以通过"用户追问第二轮"补回来**。

## Reranker 的真实价值

cross-encoder reranker 比纯向量检索**显著更准**。具体数据（2025 年 LangChain 的 benchmark）：
- 纯向量检索：top-5 召回准确率 ≈ 65%
- 向量 + BM25 混合：top-5 召回准确率 ≈ 75%
- 上面 + cross-encoder rerank：top-5 召回准确率 ≈ 88%

**Reranker 是性价比最高的优化**。成本是每 query 多调用一个模型（bge-reranker-large 之类的，~$0.001/query）。回报是 15-20% 的端到端准确率提升。

但有个坑：**Reranker 模型本身要选对**。bge-reranker-v2-m3 是 2026 年中文场景的事实标准，cohere-rerank-v3 是英文场景，jina-reranker 是多语言通用。

## 实战：客服 RAG 系统的 Select 设计

讲一个具体例子。一个客服 RAG 系统，2026 年线上在跑的：

**用户问题**："我的订单 #12345 还没收到，怎么办？"

**多路召回**（并行）：
- 路径 1：向量检索 "订单没收到" → 召回 SOP 相关 5 个
- 路径 2：BM25 "订单" + "12345" → 召回订单相关文档 3 个
- 路径 3：PostgreSQL 查 #12345 订单状态 → 1 条（订单在途，已发货 3 天）
- 路径 4：向量检索 "用户历史投诉" → 3 个相关历史

**合并去重**：12 个候选。

**Reranker**：cross-encoder 排序，top 5 进 context。

**过滤**：
- 去掉过期 SOP（< 2024 年的）
- 去掉与当前问题无关的
- 保留 5 个 chunk 喂给 LLM

**Context 大小**：约 4K token。占 128K 模型的 3%。

**端到端答案质量**：95%+ 首次解决率。

**对比纯向量 TopK**：首次解决率只有 60-70%。差别就在 4 件事：多路、rerank、过滤、结构化数据库查询。

## 3 个常被忽视的 Select 维度

**1. 时间维度。** "今天 AI 行业有什么新闻？" — 召回 2024 年的文章没意义。**给每条文档加时间戳，召回时按时间过滤**。

**2. 用户上下文。** "我之前问过什么？" — 这是情节记忆召回，不是文档检索。**把"用户历史"和"知识库"分开存，分开召回**。

**3. 任务类型。** 不同任务需要不同的 Select 策略。**问答用 top-3 准确率高，总结用 top-10-15 覆盖率高。**不要一套参数走天下。
