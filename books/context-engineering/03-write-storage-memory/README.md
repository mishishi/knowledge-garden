# 03. Write 策略：外部存储与记忆架构

2025 年初我帮一个出海产品做架构 review。技术 leader 很自豪地展示他们"完整的 context"——产品文档、客户历史、订单记录、客服对话、用户偏好，**全部塞在每次 LLM 调用的 system prompt 里**。他们的 system prompt 大约 18K token。

我问他："你模型用的什么？"
"GPT-4o 128K。"
"所以你每次调用花 18K input token 装载这些？"
"对，但质量很好啊。"

我又问："你客户里有多少是回头客？"
"大概 30%。"
"那 70% 的新客户每次都付 18K token 装载一份跟自己完全无关的历史。你一年在 context 上多花多少钱？"

他算了一下。一年大概多花了 11 万美元。

这就是 Write 策略要解决的问题：**什么信息根本不该在 context 里，而是应该存到外面，按需召回。**

## Write 策略的本质

Write 策略 = **把信息从 LLM 的 context window 移出去，存到外部存储里**。

外部存储可以是：
- **向量数据库**（Qdrant / Weaviate / Chroma / pgvector）
- **关系数据库**（PostgreSQL / SQLite）
- **键值存储**（Redis / DynamoDB）
- **图数据库**（Neo4j / Amazon Neptune）
- **文件系统**（本地 / S3）

不同的存储适合不同的数据类型。我自己 2026 年产品里最常用的组合：
- **PostgreSQL**（结构化事实、用户偏好、订单状态）
- **向量数据库**（长文本、对话历史、文档内容）
- **Redis**（session、缓存、当前状态）

## 4 层记忆架构

从人脑记忆系统借来的概念，2026 年已经是 agent 架构的标准做法。

**语义记忆（Semantic Memory）**：事实性知识、概念、定义。比如"客户的会员等级是金卡"。**存储在 PostgreSQL**。**召回方式**：结构化查询。

**情节记忆（Episodic Memory）**：过去的对话和事件。比如"上周三用户问过退货流程"。**存储在向量数据库**。**召回方式**：语义相似度检索。

**过程记忆（Procedural Memory）**：操作技能和工作流程。比如"处理退款的标准流程是 A→B→C"。**存储在向量数据库或代码常量**。**召回方式**：场景触发的规则匹配。

**工作记忆（Working Memory）**：当前 context window 里的内容。这是模型"现在在想"的事，**不存外部，是 model 自身状态的一部分**。

我自己的项目里，这 4 层记忆的存储成本分布大概是：
- 语义记忆：30%（结构化，查询便宜）
- 情节记忆：50%（向量存储，召回稍贵）
- 过程记忆：5%（小，常量）
- 工作记忆：15%（在 context 里）

## 什么时候该 Write

判断标准：**这个信息下次被用到的概率 × 信息重要性 < 当前在 context 里的成本**？

具体几个我自己的"Write 触发器"：

**1. 任何长度 > 2K token 的内容。** 一旦超过 2K，存到外面按需召回比"塞在 context 里"更便宜。

**2. 任何需要"跨 session 持久"的信息。** 用户偏好、订单状态、项目背景——这些不该每次重新加载。

**3. 任何会被多次引用的"参考材料"。** 用户手册、API 文档、代码库——这些是经典的 RAG 场景。

**4. 任何"如果错了代价高"的信息。** 重要决策的依据、合规要求、用户 PII——这些必须有"持久可查"的来源，不能只存在 context 里。

**5. 任何 context 已经"超载"时的备选。** context 满了之前先 Write 出去，比满了之后压缩好。

## 记忆的"重要性评分"

不是所有信息都该存——存太多召回会变差。**给每条记忆打分**，重要性低的淘汰。

我自己的打分（用 LLM 评估 0-1 分）：
- 0.9-1.0：用户明确偏好、关键决策、身份信息
- 0.7-0.9：项目重要约定、常用配置、工作流程
- 0.5-0.7：有用背景、历史记录、对话片段
- 0.0-0.5：日常闲聊、临时任务、过期信息

低于 0.3 的不存。0.3-0.5 的存但低优先级召回。0.5+ 的存 + 高优先级召回 + 长期保留。

## 记忆的去重与冲突

写多了会有重复、会有冲突。

**去重**：写入时先 search 同主题的旧记忆。相似度 > 0.92 的就**合并**而不是新建。**不要为了"完整"保存 5 个版本的同一信息。**

**冲突**：新信息可能跟旧记忆矛盾。比如用户说"我搬去上海了"，但旧记忆是"住在北京"。这种情况**用 LLM 判断**哪个更新，覆盖旧的。

我自己的产品里这个逻辑花了一个工程师 2 周调，**回报是 30% 的存储成本下降 + 15% 的召回准确率提升**。值得做。

## "遗忘" 机制

无限增长的记忆系统是灾难。3 个具体的遗忘策略：

**时间遗忘**：90 天没访问 + 重要性 < 0.5 的记忆，每周日定时清理。

**冗余遗忘**：5 条相似的记忆只留重要性最高的 1 条，剩下的 4 条软删。

**冲突遗忘**：跟"已确立的事实"矛盾的旧记忆，重要性降到 0.1（不删但不再召回）。

我自己做个人产品，记忆系统大约每 2-3 个月做一次完整审计。**总有 20-30% 的记忆其实没用。** 删掉它们之后召回准确率反而上升。

## 实战：客服 agent 的记忆架构

讲一个具体例子。一个客服 agent 的 4 层记忆：

**语义记忆**（PostgreSQL）：
- 用户表：id, name, email, plan, signup_date
- 订单表：order_id, user_id, status, items
- 偏好表：user_id, preference_key, preference_value

**情节记忆**（向量数据库）：
- 历史工单：每次对话的 summary + 关键问题
- 用户行为日志：浏览过什么、点过什么
- 类似问题的解决方法：其他用户怎么解决的

**过程记忆**（代码常量 + 文档 RAG）：
- 退款流程 SOP
- 升级处理流程
- 常见 bug 的标准回复

**工作记忆**（context window）：
- 当前对话历史
- 当前 ticket 信息
- 当前可用的工具

每次用户发消息：
1. 从 PostgreSQL 召回用户基础信息（1-2K token）
2. 从向量数据库召回相关历史工单（3-5K token）
3. 加载当前 ticket 相关 SOP（1-2K token）
4. 加上当前对话 + 工具定义（5-8K token）
5. **总 context 控制在 15-20K**（128K 模型的高效区）

这套架构让客服 agent 的**首次解决率从 35% 提到 68%**，同时**单次 token 成本下降 60%**（因为没用的历史不再每次加载）。

## 3 个常犯的 Write 错误

**1. 把所有东西都 Write 出去。** 不是所有信息都该存。临时上下文、单次任务、一次性的提示——这些留在 context 里更便宜。

**2. Write 了但不更新。** 用户偏好变了，旧的没删，召回时新旧一起出现，模型混乱。**Write 一定要配更新和淘汰机制。**

**3. 写得太碎。** 把一个完整事件切成 10 个小段存——召回时容易召回不完整的片段。**保持记忆的"语义完整性"**，一段记忆最好是一个完整的事实或事件。
