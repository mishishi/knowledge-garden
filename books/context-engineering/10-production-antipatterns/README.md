# 10. 生产反模式与未来

这一章是收尾。我会讲 5 个最常见的 context 灾难（我自己在 2023-2025 年都踩过），然后是 2026 年下半年到 2027 年的几个明确趋势。

不是"未来学"，是基于已发布的研究和产品的预测。**2026 年的 context engineering 跟 2024 年最大的差别是"上下文失败"成为产品失败的第一原因**。

## 反模式 1：把 context 当垃圾桶

**症状**：每次 LLM 调用塞一堆东西——历史对话、用户信息、文档、few-shot、工具定义——**因为"说不定有用"**。

**真实案例**：2024 年我见过一个产品，每次调用塞 12 个历史对话 + 5 个用户标签 + 30 个工具 + 2 段 SOP + 3 个示例。**单次输入 28K token**。**准确率 58%**。

**为什么是反模式**：
1. Lost in the Middle 让中间 60% 的内容无效
2. 注意力分散让模型抓不住关键
3. 成本高、延迟高、准确率低

**正确做法**：
- **每个 token 进 context 都要有"明确用途"**——如果答不出"这段用来让模型知道什么"，就删
- **每加一段内容都做 A/B 测试**——真的让准确率提高吗？还是只是"觉得有用"？
- **激进剪裁**——砍到 30-50% 长度，看准确率是否下降；如果不下降，就砍对地方了

**我自己 2026 年的规则**：每个新加的 context 段落必须解释"为什么加、不加会怎样"。**回答不出来就不加**。

## 反模式 2：忘记 context 衰减

**症状**：context 设计时考虑"模型能看到什么"，但忽略"模型实际用得上什么"。

**真实案例**：2025 年一个客户，agent 的 system prompt 写了 2000 字的"角色定义 + 行为准则 + 风格指南 + 业务规则"。**模型几乎全部忽略**——准确率跟把这段删了差不多。

**为什么是反模式**：
- 模型对 context 开头和结尾记得最准，中间 60% 衰减严重
- 模型并行处理所有 token，**没有"system prompt 优先级更高"这种概念**
- 长 system prompt 让"重要规则"被淹没在大量"次要规则"里

**正确做法**：
- **System prompt 控制在 1-2K token**——只放"最关键的不变约束"
- **重要信息重复**——同一段约束在 system 和 user message 都出现
- **关键规则放在 context 结尾**——模型的"近期效应"让结尾内容权重更高

**我自己 2026 年的规则**：system prompt 优先做"骨架"，关键决策在 user message 触发时**临时插入**到 context 末尾。

## 反模式 3：单 agent 啥都干

**症状**：一个 agent 试图做所有事——查数据、写内容、审内容、发布内容。**System prompt 8000 token，工具 200 个，knowledge base 30 个**。

**真实案例**：2024 年那个"智能分析师"——上面提到过，单 agent 1 周花 1.2 万美元 token 费，转化率 2%。

**为什么是反模式**：
- 单 agent attention 有限，200 个工具它记不住
- 不同任务需要不同的 context 设计，混在一起互相污染
- 调试难：模型说错一句话是哪个 task 错了？system prompt 错了？工具错了？知识库错了？

**正确做法**：
- **按"职责"拆 agent**——每个 agent 一个明确职责 + 一组相关工具 + 一个相关 knowledge base
- **主 agent 只协调，不执行**——把任务分给子 agent，自己只看结果
- **每个 agent 的 context ≤ 30K**——多了就拆

**我自己 2026 年的规则**：超过 8 个工具或 3 个职责的 agent 必须拆。**拆 agent 的成本远低于"维持超 agent"的复杂度**。

## 反模式 4：context 设计无评估

**症状**：凭直觉设计 context，**没测过"加了这段 / 删了这段准确率变了吗"**。

**真实案例**：我自己 2023 年做产品时，凭"感觉"设计 system prompt。**6 个月后才发现，删了 30% 的 prompt 准确率不变**——浪费了 6 个月的时间和 30% 的 token 钱。

**为什么是反模式**：
- "感觉"不准——你觉得重要的，模型可能觉得不重要
- 没有 baseline 测不出变化——"准确率从 60% 到 70%" 不知道是改 prompt 改的还是模型版本升级的
- **context 设计是工程问题，不是艺术问题**——需要数据驱动

**正确做法**：
- **有 golden set**——50-200 个测试 query，标注"正确答案"
- **每次改 context 都跑 golden set**——看准确率变化
- **A/B test 真实用户**——golden set 不一定代表真实场景
- **看分布**——不只是平均准确率，还有"长尾 query 表现"

**我自己 2026 年的规则**：每个 context 改动必须配 golden set 评估。**没评估的改动不发**。

## 反模式 5：完全信任 LLM 的自我评估

**症状**：让 LLM 自己判断"信息够不够"、"格式对不对"、"内容好不好"——**把 LLM 当 LLM 的裁判**。

**真实案例**：2025 年一个 RAG 系统，agent 在"够不够"判断中出错（明明信息不够，LLM 说"够了"），结果给了用户错误答案。**上线 1 个月客户投诉率 15%**。

**为什么是反模式**：
- LLM 的"自我评估"和"生成"用的是同一套能力——有同样的偏差
- LLM 倾向于"自信"——即使不确定也说"确定"
- 缺乏外部信号校准——LLM 不知道"我不知道"

**正确做法**：
- **关键决策用 hard rule**——"必须包含 order_id 才算答案完整"
- **让 LLM 输出"confidence score"**——用户界面展示"我有 80% 把握"或"我无法确定"
- **设 fallback**——"判断不够" → 抛给人工 or 明确告诉用户"信息不足"
- **A/B 评估 LLM 自我评估的准确率**——是不是真的"够"？

**我自己 2026 年的规则**：LLM 的"判断"只做"软参考"，关键决策必有 hard rule 或 human-in-the-loop。

## 2026 下半年的 5 个明确趋势

**1. Long context 不再是"卖点"**。Claude / Gemini 都有 200K-1M context，但**社区共识是"长 context ≠ 好 context"**。2026 年下半年到 2027 年，"context efficiency"（用更少 context 达到同样效果）会成为新指标。

**2. Context 优化成为独立职业。** 就像 2010 年代的"前端工程师"独立成职业，2026 年的"LLM context 工程师"开始从"prompt 工程师"中独立出来。**专门优化"context 设计"的角色**会越来越多。

**3. 标准化 context 协议。** 类似 MCP（工具协议），2026 年下半年开始出现**"context 协议"**——定义"system prompt / user message / tool result / memory recall"如何组织。**目标是让不同框架的 context 设计可以共享最佳实践**。

**4. Context 可视化调试工具爆发。** LangSmith / LangFuse / Helicone 之外，更多专门做"context 调试"的工具出现——可视化"模型实际看了 context 的哪几段"、"哪些 token 被 attention 高"、"决策依据是什么"。**debug context 比 debug 代码更需要的工具**。

**5. 端到端"context-aware"评测成为标准。** 现在的 benchmark（GSM8K、MMLU、HumanEval）都是"模型能力的评测"，跟"context 设计好坏"无关。**2026 年开始出现"context-aware benchmark"**——专门测"给定模型 + context 设计，端到端答案质量"。

## 写完 10 章你应该有的能力

**1. 看 LLM 产品问题先问"context 装了什么"**——而不是"prompt 怎么改"。

**2. 把任何 LLM 应用拆成 4 层架构**：Write（存哪）/ Select（选什么）/ Compress（怎么压）/ Isolate（怎么隔）。**每个问题都能对应到 4 层之一**。

**3. 用 golden set + A/B test 评估 context 改动**——不是凭感觉。

**4. 判断什么时候该拆 agent**——>8 工具 / >3 职责 / >30K context 必拆。

**5. 用 MCP 标准化工具调用**——3+ 工具且需跨应用复用就用 MCP。

**6. 让 LLM 自主决定"查什么、查几次"**——Agentic RAG 是 RAG 的未来形态。

**7. 避免 5 大反模式**：context 垃圾桶 / context 衰减 / 超 agent / 无评估 / 自我评估。

## 最后一个 take

Context engineering 是 2026 年 LLM 应用的"操作系统层"。**prompt 是 1990 年代的 HTML，context 是 2000 年代的 React**——从"手写 markup"到"系统化架构"的跃迁。

**这 10 章不是"理论介绍"，是"工程师上手指南"**。每一章都能直接落地到你现在的项目里。如果你能把第 3 章的 Write 策略 + 第 4 章的 Select 策略 + 第 6 章的 Isolate 策略组合到现有产品里，**你就能在 2 周内把成本降 50% + 准确率升 20%**。

Context engineering 不是 prompt engineering 的子集。是 LLM 时代的系统设计。

写完这本书的目标达到了——**希望 10 章后你看到 LLM 应用的第一反应是"这 context 怎么设计的"，而不是"这 prompt 写得真巧"**。
