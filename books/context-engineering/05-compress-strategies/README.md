# 05. Compress 策略：摘要与上下文压缩

2025 年我帮一个法律 SaaS 客户做 context 优化。客户在做一个合同审查 agent，每份合同平均 80K token。

他们的做法：直接把合同全文塞 context，然后让模型审查。问题：
1. 合同超过 128K context 后无法整本放进去
2. 80K 的合同放在 128K context 里，模型"读不准"
3. 单次审查 token 成本 $1.5

我的方案：**分章节抽取 + 摘要 + 结构化压缩**。

第一步：把合同按章节切（12 章，每章 6-8K）
第二步：每章生成 200-500 字结构化摘要（条款清单 + 风险标记 + 关键数字）
第三步：把 12 个摘要合起来（3-4K token）作为"context 第一层"
第四步：模型在审查时**按需回看原文**（"摘要提到 X 是关键风险 → 去原文 X 章节看细节"）

结果：
- 首次审查准确率 88% → 95%
- 单次审查成本 $1.5 → $0.25
- 审查时间 18 秒 → 6 秒

**这就是 Compress 策略**。核心思想：**context 里只装"够用"的信息，更详细的内容放外面，必要时回看。**

## 3 种 Compress 方法

**1. 摘要压缩（Summarization）**

把长文本压成短摘要。分两种：

**抽取式摘要**：从原文里抽关键句子。优点是保真度高（不引入新信息），缺点是压缩比低（一般 30-50%）。

**生成式摘要**：让 LLM 重新写一份短的。优点是压缩比高（可达 90%）、可读性好，缺点是可能引入 hallucination。

我自己做产品默认用**生成式摘要 + 关键数字/引用保留**。具体 prompt：

> 用 200 字总结以下内容。保留所有数字、日期、专有名词、关键决策。不要添加原文没有的信息。

**2. 结构化压缩（Structured Compression）**

不是把文本压成短文本，而是**把文本转成结构化数据**。比如：
- 合同 → { 甲方, 乙方, 总金额, 关键日期, 风险条款列表 }
- 会议纪要 → { 参会人, 决议, 行动项, 截止日期 }
- 用户反馈 → { 类别, 情感, 关键问题, 建议 }

**结构化的好处是模型读得懂且用得准**。一个 100 行的 JSON 比一个 3000 字的段落更能让模型回答"总金额是多少"。

**3. 渐进式压缩（Progressive Compression）**

不是一次性压缩，而是**分层压缩**：
- 第 1 层：原文（只在 LLM 需要时调用）
- 第 2 层：摘要（常驻 context）
- 第 3 层：标题 + 标签（永远在 context）

模型根据任务需要"向下钻取"。比如：
- "这篇文档讲什么？" → 答第 3 层（标题）
- "它的主要观点是？" → 答第 2 层（摘要）
- "第三段那句话是什么意思？" → 触发第 1 层（原文）的回看

这套模式叫 **"retrieval-augmented summarization"** 或 **"summarization cascade"**。

## 什么时候 Compress

5 个触发器：

**1. Context 接近 80% 满。** 128K context 用了 100K — 是时候压缩了。

**2. 多轮对话进行了 10+ 轮。** 对话历史是 context 占用大头。每 10 轮做一次摘要压缩，把旧的对话 history 转成 summary。

**3. 工具调用结果很大。** 比如 ReadFile 返回 50K token 的代码 — 进 context 前先摘要。

**4. 检索召回的 chunk 很多。** 召回 20 个 chunk（8K）但只需要 5 个相关 — 压缩前 8K，保留 top 5 进 context。

**5. 任务刚开始需要"概览"。** agent 的第一步是"理解全局" — 用压缩的摘要比读全文更高效。

## 摘要的"双层"原则

我自己做产品时发现一个有用的模式：**双层摘要**。

**短期摘要**（recent 10 轮对话的详细摘要）：每轮更新，保留 2-3K token。

**长期摘要**（整个对话的"事件流"）：每 10 轮更新一次，1-2K token。

短期摘要是"细节"，长期摘要是"剧情"。模型用长期摘要判断"现在在第几幕"，用短期摘要判断"最近 10 轮说了什么"。

双层摘要比单层（一个超长摘要）更准 15-20%。因为单层会"信息丢失"——超长摘要里 mid 部分被压缩了。

## Compress 的失败模式

Compress 也有副作用。3 个最常见的失败：

**1. 摘要引入 hallucination。** LLM 写摘要时"加戏"了 — 原文没说的事被写进摘要。**对策：每条摘要留"原始来源指针"（chunk id / 行号），需要时回查**。

**2. 压缩比太高，细节全丢。** 一段 5000 字的合同被压成 100 字摘要，关键的"例外条款"被丢了。**对策：先结构化压缩（保留所有数字/日期/专有名词），再做摘要压缩**。

**3. 摘要被反复压缩后"失真"。** 第一轮摘要 OK，第二轮把第一轮摘要再压，第三轮再压 —— 信息损失累积。**对策：长期摘要不要压缩短期摘要，而是用"增量更新"——只在原摘要上加新信息，不删旧的**。

## 实战：长对话的 Compress pipeline

一个 50 轮对话的 Compress 设计：

**第 1-10 轮**：原样存。Context 里就是这 10 轮完整对话。约 8K token。

**第 11 轮触发**：把第 1-10 轮生成短期摘要（2-3K token）。Context 变成"摘要 1-10 + 第 11 轮原对话"。

**第 12-20 轮**：原样存。Context ≈ 短期摘要 + 第 11-20 轮完整对话。约 10K token。

**第 21 轮触发**：把短期摘要 + 第 11-20 轮生成新的短期摘要。Context 变成"新摘要 1-20 + 第 21 轮原对话"。

**第 50 轮时**：Context 包含"1-20 摘要 + 21-30 摘要 + 31-40 摘要 + 41-50 原对话"。约 12K token。

**用户的"完整记忆"在向量数据库里**（每轮对话的原始 detail）。**context 里只有摘要**。**如果用户问到第 15 轮的细节，模型可以"召回原始对话"补回 context**。

这套设计让 50 轮对话的 context 始终控制在 12-15K，是 128K 模型的高效区。

## 压缩 vs 淘汰

Compress 策略的"姊妹"是**淘汰**（drop）。Compress 是"保留信息但缩短"，Drop 是"直接删除"。

什么时候该 Drop 而不是 Compress？
- 临时上下文（中间计算结果、临时变量）— Drop
- 重复信息（同一个问题问过 3 次）— Drop 旧的
- 已知答案（用户问过、LLM 答过）— Drop 老的问答，保留新的
- 过期信息（5 轮前的工具错误信息）— Drop

**经验法则**：Compress 给"高价值信息"，Drop 给"低价值或可恢复信息"。
