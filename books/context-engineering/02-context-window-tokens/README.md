# 02. 上下文窗口与 token 基础

我自己在 LLM 应用上犯的最贵的一个错是 2023 年。

当时我在做一个客服对话产品，模型是 GPT-3.5-turbo，context window 4K。我把整段对话历史、所有客户资料、所有产品文档全部塞进 prompt 里。最后一个用户发长消息时，模型直接吐回了"as an AI language model..."（截断后的默认回复）。**整整 3 个月的对话数据、用户资料、prompt 调优全部浪费。**

那时候的解决方案是"删除旧消息"。现在的方案是"管理 context"。这两个东西的差别就是这一章要讲清楚的。

## Token 是什么（30 秒版本）

LLM 不直接读"字"或"词"。它读 **token**——模型在训练时切好的最小语言单位。

粗略规则：
- 1 个英文单词 ≈ 1-1.5 个 token
- 1 个汉字 ≈ 1-1.5 个 token（中文是按字切的，不像英文按子词）
- 1 个 emoji ≈ 2-4 个 token
- 1 行代码 ≈ 10-50 个 token（取决于语言）

为什么这件事重要？因为 **context window 不是按字数算的，是按 token 数算的**。Claude Sonnet 4 的 200K 是 200,000 个 token，不是 200,000 个字。这意味着 200K context 大概能装：

- 一本 600 页的书
- 1000 行的代码
- 30 分钟的对话历史
- 50 个工具的定义

**这听起来很多，但实际有效利用率比这低得多。**

## Lost in the Middle 现象

2023 年斯坦福的一项研究给 LLM context 用法提了一个醒：**模型对 context 开头和结尾的信息记得最准，中间的部分准确率显著下降**。

这个现象被复现了无数次。具体数据：
- 放在 context 前 10% 的信息，召回率 ≈ 95%
- 放在 context 50%（中间）的信息，召回率 ≈ 60-70%
- 放在 context 后 10% 的信息，召回率 ≈ 90%

**这意味着你塞 200K context 是没用的。** 模型对中间 100K 的记忆率大概只有 65%。等价于你花了 200K 的 token 钱，只有 130K 在工作。

更糟的是 **"Lost in the Middle" 跟任务难度成正比**。简单任务（事实问答）受影响小，复杂任务（多步推理、引用整合）受影响大。

## 实际有效 context

我自己的经验法则（2026 年）：

- **可靠区**：context window 的前 10% + 后 10% — 准确率 90%+
- **可用区**：前 30% + 后 30% — 准确率 75-85%
- **风险区**：中间 40% — 准确率 60-70%

所以：
- 8K context（GPT-3.5）→ 实际有效 4-5K
- 128K context（GPT-4o）→ 实际有效 50-70K
- 200K context（Claude 4）→ 实际有效 80-100K
- 1M context（Gemini 1.5）→ 实际有效 200-300K（"1M"这个数字水分很大）

**所以"大 context"不是"随便塞"的理由**。塞得越满，有效利用率越低，因为中间区会膨胀。

## 模型怎么"读" context

理解一件事：**context window 里的所有 token 都经过同一个 transformer forward pass**。没有"先读 system 再读 user"这种顺序差异。模型并行处理所有 token，通过 attention 机制建立它们之间的关系。

这意味着：
- System prompt 和 user message 在模型眼里**地位平等**，没有"system 优先级更高"这种概念
- 重要的事情要**重复**或**放在显眼位置**（开头 / 结尾）
- 不要依赖"模型会读懂我的角色" — 模型读的是 token 序列，不是角色

我自己的一个习惯：**关键信息重复 2 次**。一次在 system prompt 里说"你是 X"，一次在 user message 开头说"作为 X，..."。冗余在 context engineering 里不是浪费，是保险。

## 4 个 context size 决策

具体到一个 LLM 产品，context size 选哪个？

**8K — 别用。** 现在 2026 年，4K 和 8K 模型基本只剩"便宜"这一个优势。如果你的产品对质量有要求，8K 装不下任何有意义的工作流。

**32K — 短对话 + 单文档问答。** 单轮 RAG、长文档摘要（≤ 30K token）、短客服对话。性价比高。

**128K — 长文档 + 多文档 RAG。** 一本书的全文、多篇长报告、复杂 agent 工作流（5-10 步）。**主流产品选择。**

**200K+ — 长 context 推理。** 代码库分析（5K+ 行的 repo）、长对话记忆、复杂研究任务。**贵但必要**。

我个人做产品默认从 **128K 模型**开始。如果需要再升。

## Context 设计 checklist

每个 prompt / 每个 agent 决策时，问这 5 个问题：

1. **context 里最重要的 3 件事是什么？** 它们在 context 的开头还是结尾？
2. **哪些东西其实可以不放进来？** 删掉它们能不能让模型更准？（Often yes。）
3. **中间区有什么？** 那些放中间的内容如果模型读不准，是不是可以"挪到结尾"或"删掉"？
4. **重复了什么？** 关键约束是不是在 system + user 都说了？
5. **context 是几 K？** 接近 80% 的可用区了吗？

如果 5 个问题答不上来，context 设计大概率有问题。

## 工具调用怎么"占" context

工具定义（tool schema）会占 context。**100 个工具的 JSON 定义大概 5-8K token**。所以"all tools available"是个陷阱——你付了 8K token 让模型知道 100 个工具，但 99% 时候它只用 1 个。

更糟的是模型在 100 个工具里"找"该用哪个，attention 会被分散，准确率下降。**每次任务只暴露相关的 3-5 个工具**，剩下的用 MCP / 动态加载。这件事在第 7 章展开。

## 我自己踩过的 3 个坑

**坑 1：以为 context window 是"上限"，不是"实际"。** 200K 装满 ≠ 模型能读 200K。Lost in the Middle 让中间 40% 实际只有 60-70% 召回。

**坑 2：以为 system prompt 是"指令"，不是"内容"。** 实际上 system prompt 跟 user message 在模型眼里等价。重要的信息要**多处重复**才能可靠生效。

**坑 3：以为"信息多一点没坏处"。** 多塞一份资料、多加一个示例、多给一段背景 — 这些"没坏处"是错觉。**每多塞一段，都在污染 context 的有效区。** 实际工程经验是：**少即是多**。
