# 01. 为什么需要 RAG

> LLM 不知道你公司的产品手册、昨晚的客户工单、上周的财报。RAG 是给 LLM「外挂大脑」最实用的方式。

## 三个真实场景

**场景 1：客服问答**
- 用户问：「我的订单 #12345 什么时候发货？」
- 纯 LLM：编一个看起来合理的日期（幻觉）
- RAG：先去订单系统检索 #12345 → 把真实状态塞进 prompt → LLM 基于真实数据回答

**场景 2：法律合同分析**
- 律师问：「这份 50 页合同里有没有提到不可抗力条款？」
- 纯 LLM：超长上下文要么截断、要么算力爆表
- RAG：把合同按章节切块、向量化、检索相关段落 → 只喂相关 2-3 段 → LLM 准确回答

**场景 3：私域知识库问答**
- 员工问：「公司年假是几天？」
- 纯 LLM：瞎说（不知道你公司政策）
- RAG：检索 HR 手册 → 用真实政策回答

## RAG ≠ 「把文档塞给 LLM」

最常见的误解：把所有文档塞进 prompt 让 LLM 看。

**为什么这不行：**
1. **上下文窗口有限** — GPT-4 是 128K，Claude 是 200K。但 50 万字的文档要塞进去？
2. **注意力衰减** — LLM 在长上下文中「中间遗忘」明显（Lost in the Middle 现象）
3. **每次回答全量算** — 贵、慢、不环保

RAG 的解法：**先检索相关的几段，再喂给 LLM**。

## RAG 的基本流程

```
用户问题
  ↓
[1] Embedding：把问题变成向量
  ↓
[2] 检索：去向量数据库找 Top-K 最相似的文档块
  ↓
[3] 拼装 Prompt：把检索到的文档块 + 问题拼起来
  ↓
[4] LLM 生成答案
  ↓
回答（基于真实文档）
```

每个环节都可能翻车——这系列 10 章就是拆解每个环节的选型、踩坑、最佳实践。

## 什么时候**不**用 RAG

RAG 不是银弹：

- **任务能用 LLM 内部知识搞定** — 「写首诗」「总结一段话」「翻译」—— 直接调 LLM，不要套 RAG
- **需要实时联网** — 聊天机器人查 2026 体育比赛比分，RAG 解决不了（数据库没实时数据），需要 Web Search 工具
- **精确数字查询** — 「Q3 营收多少」？SQL/聚合查询比 RAG 准——RAG 检索回来的段落里数字可能被截断或合并
- **超大规模结构化数据** — 几亿条记录的统计，专用 OLAP 数据库比向量检索快得多

## RAG vs Fine-tuning

| | RAG | Fine-tuning |
|---|---|---|
| 知识更新 | 改文档就生效 | 要重训 |
| 成本 | 便宜（向量数据库 + 检索 API）| 贵（GPU + 数据 + 时间）|
| 适合 | 知识频繁变 / 需要来源可追溯 | 学特定风格 / 格式 |
| 幻觉率 | 低（基于真实文档）| 中（模型自己编）|

实战里 **90% 的「让 LLM 懂私域知识」需求，RAG 都比 Fine-tuning 更合适**。Fine-tuning 留到学特定输出格式（比如固定 JSON 结构）再用。
