# 10. RAG 生产化 Checklist

> 把 RAG 从「能跑」到「生产能用」要解决的 30 个问题。这一章是上线前的最后一道关。

## 性能（Speed）

- [ ] **首字延迟 < 2 秒**（P95）
  - 检索 < 200ms（向量数据库索引 OK）
  - Rerank < 500ms（候选 ≤ 50）
  - LLM 流式输出，首字 < 1.5s
- [ ] **批处理**：离线 embedding 用批（`model.encode(texts, batch_size=32)` 比单条快 10x）
- [ ] **并行化**：query 改写、向量检索、BM25 检索并发跑（asyncio）
- [ ] **GPU 选型**：embedding 推理用 L4 / A10（性价比）；rerank 用 A10
- [ ] **缓存层**：相同 query 命中 cache 直接返回（30-60% 命中率，省一大笔）

## 质量（Quality）

- [ ] **评测集 ≥ 50 条**（真实用户问题 + 边界 case）
- [ ] **Recall@5 > 0.85**（85% 正确答案在 Top-5）
- [ ] **Faithfulness > 0.70**（70% 答案被判为 A 级忠实）
- [ ] **A/B 测试框架就绪**（每次改 chunk_size / embedding 模型跑对照）
- [ ] **强过滤**：rerank 分数 < 0.3 直接返回「未找到资料」（不浪费 LLM）
- [ ] **引用**：每个回答附上来源（`[1] xxx.pdf p.5`），用户能点开查证
- [ ] **答案多样性**：MMR，避免 Top-5 全是相似内容

## 安全（Security）

- [ ] **Prompt Injection 防御**：用户输入里如果出现「忽略上面的指令」→ 忽略
  ```python
  # 检测 + 截断
  if "ignore previous" in user_input.lower():
      user_input = "[用户输入包含可疑指令]"
  ```
- [ ] **PII 过滤**：检索返回的文档里如果包含个人敏感信息（身份证、手机号）→ 替换为占位符再喂 LLM
- [ ] **输出过滤**：LLM 输出如果是机密（密码、token）→ 拦截
- [ ] **审计日志**：所有问答记录可追溯（user_id、question、retrieved、answer、latency）
- [ ] **访问控制**：用户只能检索自己有权限的文档（metadata 过滤 + 权限系统）
- [ ] **数据隔离**：不同租户的文档物理隔离（不要共享 collection）或逻辑隔离（强 metadata filter）

## 可靠性（Reliability）

- [ ] **限流**：每个用户 / IP 每分钟最多 60 次问答（防滥用 + 成本控制）
- [ ] **降级**：向量数据库挂了 → 临时切到 BM25；LLM 挂了 → 返回缓存结果
- [ ] **超时**：每个环节有超时（embedding 5s / search 3s / rerank 5s / LLM 30s）
- [ ] **重试**：LLM 临时失败时指数退避重试 2-3 次
- [ ] **健康检查**：每个依赖（embedding 服务、向量数据库、LLM）有 health endpoint
- [ ] **告警**：检索为空比例 > 5%、P95 延迟 > 5s、错误率 > 1% → 通知值班

## 成本（Cost）

- [ ] **Token 预算管理**：Top-K chunk 总 token ≤ 上下文窗口的 50%（留空间给回答）
- [ ] **Embedding 缓存**：相同文本不重复 embed（chunk 重新处理时节省大量）
- [ ] **模型分场景**：简单问答用 `gpt-4o-mini`，复杂推理用 `gpt-4o`
- [ ] **reranker 选择**：量大用本地 bge-reranker-base（省钱），量小用云 API
- [ ] **向量数据库**：量 < 1000 万用 pgvector（已有 PG 的话），量更大用 Qdrant
- [ ] **预留成本 buffer**：每月预留 30% buffer 应对峰值

## 文档（Documentation）

- [ ] **架构图**：数据流、组件关系、依赖
- [ ] **API 文档**：每个 endpoint、参数、返回示例
- [ ] **运维手册**：常见故障排查、扩容步骤、回滚流程
- [ ] **评测方法**：怎么跑评测、怎么解读指标、基线分数
- [ ] **变更日志**：每次模型升级、chunking 调整的日期 + 影响

## 团队协作（Team）

- [ ] **Prompt 模板统一管理**：version control（git），不要散落在代码里
- [ ] **评测集共享**：团队用同一份评测集，避免各跑各的
- [ ] **A/B 切换规范**：灰度发布（5% → 25% → 100%），不直接全量
- [ ] **角色权限**：谁能改 prompt、谁能改 chunking、谁能改模型
- [ ] **复盘机制**：每次出问题的 RCA（root cause analysis）记录

## 用户体验（UX）

- [ ] **首字流式**：不要等 LLM 完整生成再返回
- [ ] **加载状态**：检索阶段显示「正在查找资料...」，LLM 阶段显示「正在组织答案...」
- [ ] **可点击引用**：用户点 `[1]` 直接打开原文档对应页
- [ ] **追问能力**：多轮对话支持（query 改写）
- [ ] **负面反馈**：每个回答有 👍 / 👎 按钮，收集数据
- [ ] **空答案兜底**：检索为空时清晰提示「未找到资料」+ 建议换关键词

## 上线前最后 7 件事

- [ ] **1. 灰度 5% 流量** 跑一周，看错误率
- [ ] **2. 真实用户问题评测集**：从历史用户问题采样 50 条 + 人工标注
- [ ] **3. 跑满 24h 不间断**：看是否有内存泄漏 / 连接泄漏
- [ ] **4. 监控大盘就绪**：Grafana / DataDog 看延迟、错误率、token 用量
- [ ] **5. oncall 排班**：谁负责看告警？响应时间 SLA？
- [ ] **6. 回滚方案**：怎么快速回滚到上一个版本？（blue-green / feature flag）
- [ ] **7. 文档全**：新人能在 1 天内接手

## 常见翻车现场

### 翻车 1：上线当天召回率暴跌

**原因**：embedding 模型版本升级，老向量和新向量混用

**修复**：
- 强制所有向量用同一模型版本
- 模型升级时全量重建向量（不要 incremental）

### 翻车 2：用户问敏感问题，LLM 把内部文档全文吐出来

**原因**：检索返回了带敏感信息的 chunk，LLM 直接抄

**修复**：
- PII 检测（presidio）
- 敏感字段从 chunk 移除
- LLM 输出也检测

### 翻车 3：高峰期 latency 飙到 30s

**原因**：reranker 候选太多（200）+ LLM 拼长 prompt

**修复**：
- 减少 reranker 候选（50 → 20）
- 减少 chunk 数量（10 → 5）
- 启用 LLM 流式 + 首字优化

### 翻车 4：用户吐槽「答非所问」

**原因**：query 改写用了太多轮历史，query 跑偏

**修复**：
- 限制历史长度（最多 4 轮）
- 改写 query 时给 LLM 看完整历史 + 当前 query
- 加 query 改写的评测指标

## 终极心法

**RAG 的上限 = 检索质量 × prompt 质量 × LLM 能力**。

三者都要做好，但**先做好检索**——LLM 再强也救不了垃圾检索。

**不要做的事：**
- 不评测就上线
- 不加 metadata 就入库
- 不设限流就开放
- 不监控就睡觉
- 不写文档就招人

**做完这 30 条，RAG 系统就从 demo 变生产。**

## 系列完结

回顾这 10 章：

1. **为什么需要 RAG** — 找准场景，别滥用
2. **Embedding 模型** — 中文 BGE-M3 起步
3. **向量数据库** — 小项目 Chroma，中型 pgvector，大型 Qdrant

8. **评测** — 评测集 + A/B 实验 + 线上监控
9. **高级模式** — Agentic / Multi-hop / HyDE / GraphRAG
10. **生产化** — 性能 + 质量 + 安全 + 成本 + 文档
