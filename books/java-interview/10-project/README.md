# 项目经验怎么讲: STAR 升级版

面试里最让我头疼的回答, 不是候选人技术答不上来, 而是简历上写着"精通"、聊起来变成"参与过"。

我去年面过一个哥们, 5 年经验, 简历写"独立负责订单中台从 0 到 1 建设, 支撑日均 200w 单"。我让他讲讲, 他说了 5 分钟, 我能记住的只有"我们用了 Kafka"和"上线之后没出过事"。技术深度? 没有。业务结果? 没有。个人贡献? 也没有。

但隔壁组刚招的兄弟, 3 年经验, 简历也就写"参与订单系统开发"。聊起来, 他从 2022 年底那波优惠券被刷的 bug 讲起, 怎么定位到 Redis 预扣减在集群 slot 迁移时丢数据, 怎么改造成本地缓存+异步对账, 上线后资损从 8w/天降到 0, 整个过程 15 分钟讲得清清楚楚。leader 当场发了 offer。

你说技术深度谁更深? 显然是后者。**项目经验这道题, 考的不是你做过什么, 是你能讲出什么。**

我面了 500+ 候选人, 越来越确信一件事: 大部分人的项目经验是被自己浪费的, 不是项目本身没料。今天想把我自己作为面试官看到的一些套路和反套路, 系统地聊一下。

## 面试官问"项目经验"的时候, 到底在问什么

很多人以为, 这个问题是走流程——你简历上写了, 让我确认一下是不是真的。但其实, 问项目经验才是整场面试真正的开胃菜。

我通常会在你讲项目的时候, 听三层东西。

第一层叫**真实性**。STAR 那套老掉牙的东西先放一边, 我听的是细节密度。你说"我们用了 Redis", 这谁都能说; 你说"Redis Cluster 13 个 master, 用的 client 是 Jedis 4.2, 因为 Lettuce 在当时跟我们的连接池配置有兼容问题"——这就比较真了。真实的项目里, 决策都不是真空的, 总有约束、有妥协、有 BUG。**讲不出任何一个具体到版本号、配置项、错误码的细节, 我就会怀疑这段经历是不是你的。**

第二层叫**思考深度**。这个项目里有没有让你睡不着觉的问题? 有没有哪个设计你一开始觉得对, 后来被现实打脸? 我去年聊一个做支付的候选人, 他讲了一个细节我到现在还记得: 一开始他设计幂等的时候用 token + Redis SETNX, 跑了三个月很稳, 直到双十一大促, Redis 集群某个 slot 出现网络抖动, SETNX 在 200ms 内重试, 同一个 token 居然产生了两次扣款。最后排查下来, 是因为 SETNX 成功但业务逻辑超时, 客户端重试时 token 没失效。他后来改成了 token + 数据库唯一索引, 牺牲了一点性能换来了强一致。

这种"我曾经以为……后来发现……所以改成了……"的叙述, 比任何技术名词都更能说明你真的在这个项目里下过功夫。

第三层, 也是最容易被忽略的, 叫**协作和影响力**。技术再牛, 如果你的项目就你一个人知道、文档没人看、别人想复用还得来问你, 这在 P6+ 的面试里是要减分的。我会问"这个方案, 团队里其他人后来怎么看?", "你有没有推动过什么变成团队标准?", "下游接入的时候, 别人问过你哪些问题?". 这些问题的答案, 决定了我给你打 P6 还是 P7。

所以回到开头那个对比, 5 年经验的哥们问题在哪? 他在第一层都没站住, 全是形容词, 没有名词。

## STAR 不是不能用, 但你用错了

STAR——Situation、Task、Action、Result——这玩意儿 2010 年从外企 HR 那儿传过来, 到 2024 年已经成了候选人最爱的"八股"模板。

我每次听到有人开篇就是"当时的情况是这样的, 我的任务是……, 我做了……, 最后的结果是……", 心里基本凉了半截。不是 STAR 本身有问题, 是大家用 STAR 的方式, 把一个有血有肉的项目经历, 讲成了填空题。

STAR 真正的用法, 我理解应该倒过来: **从 Result 切入, 再展开 Action, 最后补充 Situation 和 Task。**

为什么? 面试官的时间有限, 他要的是**先知道你搞出了什么名堂**, 再决定要不要听你讲怎么搞的。你上来花 3 分钟铺垫业务背景, 我听到一半已经走神了。

我自己的经验是, 好的项目讲述结构是这样的:

开头 30 秒, 抛一个**具体的、有数字的结果**。比如"我把订单创建的平均耗时从 850ms 降到了 120ms", 或者"上线后那波羊毛党攻击的资损归零了"。这个数字不用特别大, 关键是真实、可信、有指向性。你说"提升了 50%"——这种话我会本能地怀疑, 因为没有上下文。你说"P99 从 1.2s 降到 380ms, 因为大促时锁竞争降了 80%"——这就有可信度了。

接下来 3-5 分钟, 讲**你怎么做到的**。这一段是技术深度的主战场, 也是 STAR 里 Action 的位置。但我要提醒一句: **不要写成技术方案汇报**。方案汇报是"我们经过调研, 选择了方案 B, 理由是 1、2、3", 这是给领导做决策的。面试讲述是"我试过 A, 不行, 换成 B 才搞定", 这是讲你这个人怎么解决问题的。

中间可以插 30 秒到 1 分钟, 说一下**为什么这个项目存在、当时卡在哪了**。Situation 和 Task 不是不重要, 是不能占比太大。如果你讲了半天对方还听不懂你为什么要做这个事, 那是他阅读能力有问题, 不是你铺垫不够。

最后留 30 秒, 回到那个数字, 强化记忆, 顺便提一句**这个经验后来怎么复用的**。比如"这个异步化的套路后来被我复制到了另外两个链路", 或者"复盘的时候, 我意识到如果当时早点压测就能少踩一周的坑"。这种自我反思的能力, P6+ 一定要有。

讲一个我自己面别人时印象深的回答。候选人做电商, 讲他怎么优化一个"用户取消订单"的接口。原来接口是同步的, 要走 5 个下游服务 (库存、退款、优惠券、积分、消息推送), P99 1.8s, 高峰期经常超时。

他讲的故事是这样的: 第一版他加了个线程池, 把 5 个调用改成并行, P99 降到 600ms, 但线程池经常被打满, 因为取消订单有脉冲, 大促瞬间能翻 10 倍。第二版他换成了 CompletableFuture 配合业务线程池, 但发现有个回调地狱, 排查问题特别麻烦。第三版他引入了一个本地的内存队列, 把非关键路径 (推送) 彻底异步化, 关键路径 (库存+退款) 还是同步但超时降级。最后 P99 稳在 250ms 左右, 线程池被打满的事再也没发生。

你看, 这段讲得好, 不是因为技术多牛, 是因为他讲出了**迭代过程**。从 1.8s 到 600ms 到 250ms, 每一步都有问题、有思考、有取舍。这才是面试官想听的东西。

## 业务价值怎么"算"出来, 是个技术活

很多技术同学最怵的就是"你做的这个有什么业务价值?"。我理解, 毕竟写代码的跟算账的, 脑回路不一样。

但面试的时候, 这个问题你躲不开, 也不该躲。我自己面候选人, 如果他做了两年订单系统, 还说不出这个订单系统的钱从哪来、规模多大、波动规律是什么, 我会觉得他没真的在做事。

**业务价值不是夸出来的, 是"算"出来的。**

举几个我见过的高分回答。

"我做的这个对账系统, 上线前财务每月要花 2 个人/周手工对账, 错误率 0.3% 左右 (一个月大概 100 笔长款/短款, 每年光排查就要花 20w 人力成本), 上线后基本 0 人工干预, 系统对账覆盖率 99.7%。"——这是**人力成本**换算。

"我做的这个推荐服务, 在 A/B 测试期间 CTR 从 4.2% 涨到 5.1%, 按当时的 GMV 大概每天多赚 30w。"——这是**业务指标**换算。注意他没说"显著提升", 说了具体数字。

"我没做的东西更值钱: 我挡掉了一个方案。那个方案要做 2 个月, 上线后预估能省一台机器 (一年 5w), 但会带来一个潜在的资损风险, 按历史上类似的 bug 算, 极端情况一年可能亏 100w。我当时的判断是 ROI 不到, 后来我们换了一个更轻的方案。"——这是**机会成本**和**风险规避**。这种回答, P7+ 面试里我特别愿意给高分。

当然, 很多候选人会说"我做的这个功能, 老板说很重要", 或者"业务方很满意"——这种话不是不能说, 但要谨慎。**当你的项目价值只能用别人的评价来证明的时候, 你就输了。** 一定想办法用数据证明。

我给你一个简单的模板, 遇到不知道怎么讲价值的项目, 套这个:

第一, 这个项目**服务谁** (用户? 商家? 内部运营? 财务?)
第二, 这个项目**规模多大** (QPS? 日单量? 用户数? 资金量?)
第三, 你做的这个改动**影响了什么** (耗时? 错误率? 资损? 人力? GMV?)
第四, 改动**前后对比**是多少 (P99 从 X 到 Y, 错误率从 A% 到 B%)
第五, 这个改动**折算成钱**是多少 (人力成本 / 资损规避 / 收入增长)

五个点里你能讲清楚 3 个, 我就觉得你这个项目是立得住的。

要注意的是, **不是所有项目都有直接的、显性的业务价值**。你做的是中间件、是内部框架、是给公司省钱的基础设施, 怎么算?

诚实点。我面过一个人, 他做的是配置中心, 你说他怎么讲业务价值? 他就说"我们公司以前每个业务线自己写配置加载, 出了问题排查要 2 小时, 上了配置中心之后 5 分钟定位"。这就是**效率价值**。还有一种是"风险价值", "以前每次发布都要重启, 现在配置热更新, 一年避免了大概 30 次发版事故"。这些都是业务价值, 只是不那么性感而已。

## 几个一票否决的"踩雷"姿势

聊了这么多正面套路, 说几个我自己会直接判负的"反模式"。

第一种, 叫**"我们组做的"**。所有的主语都是"我们", 从来没有"我"。项目 6 个人做的, 你挑了哪个是你干的? 是写了核心模块? 是做了架构设计? 是扛了上线? 是解决了某个具体的 bug? 你不讲, 我怎么判断你的能力边界?

注意, 这不是说"我们"不能说——**真实的项目一定是合作的, 该说"我们"就说"我们", 这反而是加分项, 说明你不抢功**。但你要在合适的地方切回"我": "在这个项目里, 我主要负责的是 X 这一块, 当时的具体情况是……" 这样的切换, 是 P6+ 候选人的基本功。

第二种, 叫**"面试型 STAR"**。回答像写作文, 起承转合非常完美, 但你听完不知道他到底干了什么。**当一段叙述里形容词比名词多的时候, 我就开始怀疑了**。我听过最离谱的一段, 大概 7 分钟, 用到的具体名词是: "系统"、"接口"、"数据"、"问题"、"方案"、"结果"——一个都没有。全部是抽象概念。

第三种, 叫**"项目是我的, 思考是别人的"**。你说一个项目, 上来就是"我们 leader 决定用 Kafka, 我负责实现"。我追问一句"为什么是 Kafka 不是 RocketMQ", 你说"leader 调研的, 我不太清楚"。**这种"工具人"式的回答, 在 P6 这道关就会被刷掉。** 技术决策可以不是你的最终拍板, 但你必须有自己的理解和判断。

第四种, 叫**"只讲顺风局"**。整个项目讲下来一帆风顺, 没有遇到任何问题、没有踩过任何坑、没有做过任何 trade-off。**我不信。** 真实项目里, 一定有问题、有妥协、有"当时没想到"。**主动暴露问题, 反而是加分项**——前提是你能从中学到东西, 能在面试里讲出"如果让我重新做, 我会怎么做"的反思。

第五种, **只讲"做了啥", 不讲"为啥这么做"**。你做技术方案, 不讲为什么选 A 不选 B, 不讲 B 的问题在哪, 不讲当时还考虑过什么 trade-off, 这个方案在面试官眼里就是个"按文档做出来的东西"。

## 高分选手的"暗功夫"

讲几个我自己观察到的, 区分 P6 和 P7 的"暗功夫"。

P6 的项目讲述, 重心在"我搞定了 X"。P7 的项目讲述, 重心在"我让团队/业务/系统发生了 Y 级别的变化"。

具体一点, P6 讲的是: "我开发了一个对账系统, 跑了 8 个月没问题。"

P7 讲的是: "我搭了对账系统, 后来这个系统被 4 个业务线复用, 沉淀出了一套通用的对账 SDK, 我写了一份 30 页的接入文档, 还做了一个新手 onboarding 视频, 到现在为止 0 业务方来问过我二次接入的问题。"

后者讲的是什么? **复用性、可维护性、跨团队影响。** 这是 P7 区别于 P6 的核心——你做的东西, 不只是交付了, 是**沉淀了**。

还有一个暗功夫, 叫**"边界感"**。我面过很多人, 越聊越觉得他在吹牛。讲一个项目, 恨不得说 80% 都是他做的。真正厉害的候选人, 边界感非常清楚: 哪里是他的核心贡献, 哪里是同事做的, 哪里是 leader 定的方向。

有一次我面一个阿里 P7 出来的候选人, 聊到分布式事务的方案, 他说"这个方案整体是 leader 定的 Seata 大方向, 我负责的是其中 TCC 模式的性能优化, 当时把 GC 调到了 ZGC, 又把序列化从 Hessian2 换成了 Kryo, 单链路耗时从 80ms 降到了 30ms"。你看, **他很清楚自己的边界, 但在自己的边界内, 又讲出了非常具体的技术深度。** 这种候选人, 我会主动帮他推进流程。

最后, 给两个 actionable 的建议, 你下次面试前可以花 20 分钟做一下。

第一, **挑 3 个项目, 提前写好"30 秒电梯版"和"5 分钟完整版"两套讲法**。电梯版用于面试开场, 完整版用于追问时展开。**写下来, 不要只在脑子里想**。写出来跟想出来, 完全是两码事。

第二, **每个项目准备 2-3 个"挖深点"**。这是面试官大概率会追问的技术细节, 你心里要有数。比如你讲了对账系统, "挖深点"可以是: 对账失败重试的策略、金额精度问题、跨时区结算、长短款容忍度。**这些点你能不能讲清楚 2-3 分钟, 直接决定了你这场面试的深度。**

项目经验这道题, 说难不难, 说简单也真不简单。它考的是你**对自己做过的那些事, 到底有没有真正的理解和反思**。大部分人输, 不是输在项目不够大, 是输在项目讲不出来。
