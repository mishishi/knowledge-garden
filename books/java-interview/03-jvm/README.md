# 03. JVM 调优实战: 不止参数表

> GC 算法/工具链 jstat/arthas, 真实 OOM 案例一步步定位

---

## 从一次生产 OOM 说起

去年双十一之前, 我们一个核心交易系统在大促预热期出了个事故. 现象很简单: 某个订单查询接口 P99 突然从 80ms 飙升到 4 秒, 五分钟后整个服务开始疯狂 Full GC, 最后 OOM 被 K8s 干掉.

我赶到机房的时候, 运维已经把 heap dump 拉下来了, 一个 8G 的 bin 文件. 那会儿我刚开始看 JVM, 第一反应是去翻 `-Xmx` 参数, 想着是不是内存给小了. 后来 leader 拦了我一下: "别急着看参数, 先看 dump."

heap dump 拉进 MAT (Memory Analyzer Tool), 打开 Dominator Tree, 一路点下去, 看到一个 `ConcurrentHashMap` 占了 4.2G. 那个 Map 是干嘛的呢? 是用来缓存订单查询结果的, key 是订单号, value 是订单的完整对象 (包含商品列表, 地址, 优惠明细, 物流信息, 等等).

我当时的反应是: "这 Map 怎么这么大? 谁在往里塞?"

去查代码, 看到一个非常常见的反模式:

```java
public Order queryOrder(String orderId) {
    Order cached = orderCache.get(orderId);
    if (cached != null) {
        return cached;
    }
    Order order = orderDao.findById(orderId);
    // BUG: 没有 size 限制, 没有 TTL
    orderCache.put(orderId, order);
    return order;
}
```

订单数据每天增量大概 50 万, 这个 Map 无限增长, 三个月下来堆到 4G+. 等到老年代被打满, Full GC 也回收不掉 (因为 Map 里的对象全是强引用), 最后直接 OOM.

leader 让我做事故复盘的时候, 我写了一段反思, 后面每次面试 JVM 调优的问题, 我都会拿这个案例开场. 因为它能说明一件事: 调优的第一步不是调参数, 是搞清楚**为什么内存涨**. 参数调得再花, 治不了本.

但今天聊 JVM 调优, 我们不只聊这个案例. 面试的时候, 面试官问 JVM, 真正想听的不是 "ParNew+CMS 还是 G1" 这种选择题答案, 他想看你**怎么排查, 怎么思考, 踩过什么坑**. 我把过去几年面过的候选人和自己带新人的经验揉一揉, 聊聊这块在面试里怎么拿高分.

## 面试官问 JVM, 到底在问什么

我自己面过大概 200 多个候选人, 问 JVM 相关的题目, 大概能归成几类. 候选人最常踩的坑是**背参数表**. 上来就背 "-Xms 是初始堆大小, -Xmx 是最大堆大小, -XX:NewRatio 是新生代比例", 背得滚瓜烂熟, 然后呢? 现场给个 OOM 场景, 一问怎么定位, 答不上来.

面试官问 JVM, 我自己的判断标准是三档:

第一档: 知道概念. 你知道堆分代, 知道 Minor GC 和 Full GC 的区别, 知道 STW 是什么意思. 这个级别的候选人, 我一般给 P5.

第二档: 能用工具. 你会 jstat 看 GC 日志, 会用 jmap 拉 heap dump, 会用 MAT 或者 VisualVM 分析. 大多数 P6 的候选人卡在这里.

第三档: 能在生产环境**独立排查和调优**. 你不仅会用工具, 还能从 GC 日志的 pattern 看出 CMS 的 concurrent mode failure, 知道 G1 的 Mixed GC 触发时机, 知道什么样的场景该用 ZGC, 什么样的场景 ZGC 也救不了你. 这个才是 P7 该有的水平.

我面试的时候, 一般不会直接问 "CMS 和 G1 的区别", 这种题目太八股, 准备过的都会答. 我喜欢问开放问题: "你最近一次生产环境 JVM 调优是什么时候? 什么问题? 怎么解决的?"

这种问题, 答得好的候选人能讲十分钟, 答得不好的两句话就完了. 差距非常明显.

## 先把工具链说透, 别上来就背 GC 算法

我见过很多简历上写 "熟悉 JVM 调优", 然后面试一问, 只会一个 jstack. 说白了, JVM 调优的"调"字, 99% 的场景下指的是**排查**, 真正的参数调优只在最后一步. 排查的工具链我把它分成四层, 你脑子里得有这张图.

第一层是 JVM 自带工具. `jps` 看 Java 进程, `jstat` 看 GC 实时情况, `jstack` 看线程栈, `jmap` 拉 heap dump. 这些工具的好处是**任何 JDK 安装目录里都有**, 线上环境一般不会给你装 arthas, 但 jstat 一定有.

`jstat -gcutil <pid> 1000` 这个命令我建议所有人都背下来. 1000 是采样间隔 (毫秒), 输出大概长这样:

```
S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
0.00  45.32  78.21  89.43  94.12  91.33    127   12.432     8    5.234  17.666
```

FGC 在涨, O 区老年代占用 89%, 这个 pattern 一眼就能看出"老年代快撑爆了". 但是! jstat 的问题在于它**只有当下数据**, 没有历史, 你想看五分钟前的 GC 频率, 不好意思, 没了. 所以线上排查, 我几乎一定会**先打开 GC 日志**, 通过 `-Xloggc:<file>` 配合 `-XX:+PrintGCDetails` 让 JVM 持续输出 GC 事件到文件. 日志分析的工具, 我个人比较喜欢用 GCEasy (在线工具, 把日志拖进去就能看到可视化的分析报告), 或者自己写个脚本 grep 关键 pattern.

第二层是 arthas. 阿里开源的, 我觉得是 Java 程序员的瑞士军刀. 线上排查的时候, arthas 的优势是**不用重启服务就能 attach 上去**, 还能 watch 方法的入参返回值, trace 调用链, dashboard 看实时内存. 我用得最多的几个命令:

- `dashboard` 一眼看到当前 JVM 整体状态
- `thread -n 3` 看最忙的三个线程
- `jad com.example.Service` 在线反编译, 看代码到底跑的是不是你想的那版 (我们以前出过事故, 就是因为发布系统 bug, 跑的是旧代码)
- `watch com.example.Service queryOrder '{params, returnObj}'` 监控某个方法的调用

第三层是 MAT 和 VisualVM. heap dump 分析, 这两个工具能解决 80% 的问题. MAT 的 Leak Suspects 报告能直接告诉你**哪些对象占内存最多, 哪些可能是泄漏源**. 我开头那个订单缓存的案例, 就是 MAT 的 Dominator Tree 一路点下去找到的.

第四层是 async-profiler. 这个工具相对小众, 但是真神器. 它能采样 CPU 和内存分配, 生成火焰图. 当你怀疑某个方法分配了大量对象 (导致 Young GC 频繁), 火焰图能直接告诉你瓶颈在哪. 我们组后来做的一个优化, 用 async-profiler 发现某个 JSON 序列化方法每秒分配 800MB 对象, 是 GC 频繁的元凶. 改成对象池 + 复用之后, Young GC 频率从每秒 12 次降到每秒 2 次.

面试的时候, 如果你能把这四层工具讲清楚, 并且说一两个**实际用它们解决了什么问题的案例**, 基本上 P6 稳了. P7 的话, 还需要再加一层: 你能讲清楚**这些工具背后的原理**, 比如 jstat 是怎么读取 JVM 内部数据的, 为什么 jmap -heap 在 JDK 11 之后要加个 -J-djserver.only 参数, 这种细节.

## GC 算法和收集器, 别背表格, 讲演进

网上那种 "ParNew+CMS vs G1 vs ZGC 对比表" 的文章, 我估计你收藏夹里也有几十篇. 面试背这种表格没意义, 因为面试官稍微追一句 "CMS 有什么问题", "G1 怎么预测停顿时间", 你就卡壳了.

我的建议是: **理解演进, 不要背对比**.

JVM GC 的演进, 有一条非常清晰的主线: **如何在更短的停顿时间内, 回收更多的内存**.

最早期的 Serial GC, 单线程 STW, 适合客户端应用. Parallel GC 把它改成多线程, 吞吐量优先, 适合后台批处理. CMS 是第一个真正追求低停顿的收集器, 它把 GC 过程拆成好几个阶段, 大部分阶段和应用线程并发跑, 只有初始标记和重新标记两个阶段 STW. CMS 的问题是: 1) 会产生内存碎片, 2) concurrent mode failure 时会退化成 Serial Old, 3) CPU 敏感.

G1 的设计思想是"把堆切成很多个大小相等的 Region, 每个 Region 单独管理, 可以是 Eden, 可以是 Survivor, 也可以是 Old". G1 引入了**停顿预测模型**, 你告诉它"我希望每次 GC 停顿不超过 200ms", 它会根据历史数据, 估算回收哪些 Region 能达到这个目标. G1 解决了 CMS 的大部分问题, 是 JDK 9 之后的默认收集器.

ZGC 更激进, 目标是**停顿时间不超过 10ms**, 而且不随堆大小增长而增长 (1TB 堆和 1GB 堆, ZGC 的停顿时间差不多). 它用了染色指针 (Colored Pointers) 和读屏障 (Load Barrier) 这两个技术, 但实现复杂, JDK 11 刚出来的时候 bug 不少, 现在 JDK 17/21 稳定多了.

面试的时候怎么答这部分? 别上来就背 "CMS 用标记清除, G1 用标记整理". 试着这样讲:

"我理解 GC 的演进, 核心是**怎么在吞吐量和停顿时间之间找平衡**. CMS 时代, 我们线上服务用 CMS, 主要诉求是低停顿, 因为对外接口对响应时间敏感. 但 CMS 经常出现 concurrent mode failure, 触发条件是老年代占用增长速度超过 CMS 回收速度, 这种情况一旦发生, 停顿时间可能到几秒. 后来我们升级到 G1, G1 的优势是**停顿时间可预测**, 而且它会自动处理碎片. 但 G1 也不是银弹, 在大堆场景下, 我们做过压测, G1 的吞吐量比 Parallel GC 低 10% 左右. 如果业务是 CPU 密集型后台任务, 反而用 Parallel GC 更合适. ZGC 我们去年在新的一个核心系统上试过, 64G 堆, 停顿时间稳定在 1-2ms, 体感非常好, 但需要 JDK 17+, 老项目升级成本高."

这种回答, 不光答了知识点, 还**有真实的对比数据**, 有**业务场景的考虑**, 面试官听着就不一样. 真正的差距在这里: 不是你知道多少收集器, 是你知道**什么场景用哪个**.

## 真实案例: Young GC 频繁, 是谁在疯狂分配对象

讲一个我们去年优化的案例. 一个 Kafka 消费者服务, 部署在 K8s 上, 每个 Pod 配了 4G 堆. 监控上看到 Young GC 非常频繁, 大概每秒 15-20 次, 每次停顿 20-30ms. 服务响应倒是没受影响 (Kafka 消费对延迟不敏感), 但 GC 占用了大量 CPU, 业务吞吐上不去.

第一步还是 jstat 看一眼, 确认是 Young GC 频繁, 老年代没什么压力. 这说明问题不在泄漏, 在**分配速率太快**.

第二步, async-profiler 采样内存分配, 生成火焰图. 火焰图上能看到, `com.fasterxml.jackson.databind.ObjectMapper.writeValueAsBytes` 这个方法吃掉了 60% 的分配.

第三步去看代码, 发现这个消费者每收到一条 Kafka 消息, 都要做一次 JSON 序列化 (把消息体转成业务对象), 然后做业务处理, 然后做一次 JSON 反序列化 (把处理结果发回 Kafka). 每次 JSON 序列化/反序列化都会分配大量临时对象, 包括 String, char[], byte[].

优化方案有两个层面. 第一个层面是**用对象池**复用的方式, 但 Jackson 本身不太友好, 这个方案我们没采用. 第二个层面是**复用 ObjectMapper 实例**, 因为 ObjectMapper 的创建很重, 复用一个实例能省掉很多配置解析的临时分配. 同时, 业务处理过程中, 一些中间 DTO 我们改成**复用同一个对象, 每次清空字段**而不是 new 一个新的.

改造完之后, Young GC 频率从 15-20 次/秒降到 3-4 次/秒, CPU 使用率从 80% 降到 30%, 业务吞吐直接翻倍.

这个案例, 面试的时候如果被问到 "怎么排查 GC 频繁", 你可以套这个故事: jstat 确认是 Young GC, async-profiler 找分配热点, 看代码, 改实现, 复测验证. 这是一个**完整的排查链路**, 比你说"我会看 GC 日志"强一百倍.

## 另一个反面案例: 调参调出来的生产事故

讲一个我自己犯过的错, 提醒大家**别迷信参数调优**.

2020 年, 我们一个报表系统, 每天凌晨跑批, 处理前一天的全量订单, 生成各种统计指标. 跑批 JVM 配置是 `-Xmx8g -Xms8g`, 用 CMS. 跑批过程大概 4 小时, 期间 Full GC 触发 2-3 次, 每次停顿 5-8 秒, 总停顿时间 20 秒左右. 这个停顿对跑批任务来说可以接受, 业务没抱怨过.

后来有个新同事来, 看了一篇文章说 G1 多牛, 就把收集器换成 G1 了. 结果跑批时间从 4 小时变成 5.5 小时, 还多了一个事故: 跑了 2 小时之后 G1 触发了一次 Mixed GC, 停顿了 12 秒, 业务监控报警说"系统无响应".

事后我们分析, 这个场景其实**不适合 G1**. 原因有两个: 第一, 跑批任务是 CPU 密集型, 吞吐量优先, Parallel GC 才是最合适的, G1 反而拖后腿. 第二, 跑批过程是"分配-回收-再分配"的循环, 内存模式很稳定, 不需要 G1 的停顿预测模型.

后来我们做了个对比测试, 用 Parallel GC 跑同样的任务, 总时间是 3 小时 50 分, 比 CMS 还快 10 分钟.

这个案例, 面试的时候是个**反向加分项**: 它能说明你理解**调优没有银弹**, 不同业务场景适合不同的策略. 面试官最怕听到的答案是"我之前用 G1, 性能提升了 30%", 然后问细节就露馅. 真正的高级工程师会说"我用过 CMS, G1, ZGC, 每个都有适用场景, 我会根据业务特点选".

## 几个高频面试问题的回答套路

我挑几个面试出现频率最高的 JVM 问题, 说说怎么答能拿高分.

**"什么是 OOM? 遇到过吗? 怎么排查?"**

差的回答: "OOM 就是内存溢出, 堆内存不够用了."

好的回答: "OOM 在 JVM 里其实分好几种, 堆内存溢出 (java.lang.OutOfMemoryError: Java heap space), 元空间溢出 (Metaspace), 线程栈溢出 (StackOverflowError, 严格说不是 OOM 但常被一起讨论), 还有 Direct Buffer 内存溢出. 我印象最深的一次是 2022 年我们交易系统的 heap OOM, 现象是 Full GC 之后立刻 OOM. 我的排查步骤是: 1) jstat 确认是 Full GC 频繁触发, 老年代一直 100%; 2) jmap -dump 拉 heap dump, 用 MAT 打开; 3) Leak Suspects 报告显示一个 ConcurrentHashMap 占 4G, Dominator Tree 看到是订单缓存; 4) 查代码, 发现是无限增长没有淘汰策略. 后来加上了 Caffeine 的 LRU 淘汰, 加了 size 上限和 TTL, 问题解决. 事后复盘, 这种问题靠加内存解决不了, 必须从业务代码层面根治."

**"G1 和 CMS 怎么选?"**

差的回答: "G1 是新的, 默认用 G1 就行."

好的回答: "我理解这两个收集器最大的区别是**设计目标**. CMS 追求低停顿, 但有内存碎片和 concurrent mode failure 的问题. G1 追求可预测的停顿, 用 Region 划分和增量回收, 还能处理碎片. 实际选型我会看三个维度: 1) 堆大小, 4G 以内我倾向 Parallel, 4-32G 看业务, 32G 以上 G1 优势明显; 2) 业务对停顿的敏感度, 面向用户的低延迟接口用 G1, 后台跑批用 Parallel; 3) JDK 版本, JDK 9 之后 G1 是默认, CMS 已经 deprecated. 我现在新项目基本都用 G1, 老项目看迁移成本."

**"ZGC 你了解吗? 用过吗?"**

差的回答: "ZGC 是最新的, 停顿时间很短, 1ms 以内."

好的回答: "ZGC 我去年在一个新启动的实时风控系统上用过, 64G 堆, JDK 17, 停顿时间稳定在 1-2ms. ZGC 的核心技术是**染色指针**和**读屏障**, 它把 GC 状态信息存在指针里而不是对象头里, 这样可以做到对象移动之后应用线程无感知. 读屏障会拦截所有引用读取, 在读的时候做指针修正. ZGC 的优势是**停顿时间不随堆大小增长**, 这一点对大堆场景特别重要. 但 ZGC 也有局限: 1) JDK 17 之前不建议生产用, bug 多; 2) 因为有读屏障, 吞吐量比 G1 低 5-10%; 3) 对大对象 (几百 KB 以上的对象) 处理不太友好, 会触发额外的 full GC. 选型上, 如果你的业务是延迟敏感 + 大堆, ZGC 是首选; 如果是吞吐量优先, G1 更合适."

## 几个 actionable 建议, 给准备面试的你

第一, **准备 2-3 个真实案例, 每个能讲 5 分钟**. 案例不用很复杂, 关键是**完整**: 现象是什么, 你怎么排查的, 根因是什么, 怎么解决的, 后续怎么避免. 我开头那个订单缓存的案例, 我讲了不下二十遍, 每次讲都会有新的细节补充. 案例熟到一定程度, 面试不管怎么追问, 你都能接住.

第二, **背参数表是最低效的复习方式**. 网上那些 "-Xms 初始堆大小, -Xmx 最大堆大小" 的速记表, 你背一百遍也不如真的去线上看一次 GC 日志. 如果你现在的工作没机会接触线上 JVM 排查, 自己搭个环境压一压, 用 JMeter 打个流量上去, 故意制造一个内存泄漏, 然后自己用 jstat + MAT 排查一遍. 这个过程走完, 比你背 100 张表都管用.

第三, **面试要主动引导到你的案例上**. 很多候选人被问 "G1 怎么工作的" 就老老实实讲 G1 的 Region 模型, 讲完了面试官就下一个问题. 高级的做法是讲完 Region 模型, 主动加一句 "我之前在一个大堆服务上用过 G1, 当时是 X 场景, 遇到了 Y 问题, 后来 Z 解决了". 这样不光是答了问题, 还展示了实战经验. 面试官记案例比记概念深刻得多, 你在他脑子里留个具体的故事, 印象分立马上去.

第四, **承认不熟的地方, 但要展示学习能力**. 我面过一些候选人, G1 不熟, 被问到 Mixed GC 触发时机, 直接说"这块我不太确定, 但是我能猜一下, 是根据 IHOP 触发的吗? 我下去再查一下". 这种回答比硬编一个答案强十倍. 面试官自己也知道 JVM 知识多到没人能全记住, 他看的是你**面对不会的问题时怎么思考**.

最后, 留一句大实话: JVM 调优这个东西, **真正能讲出花来的人, 大多是在生产环境被坑过很多次的**. 没踩过坑的人, 知识再扎实也是纸上谈兵. 所以如果你工作年限不长, 没机会接触生产排查, 尽量**自己造机会**. 本地跑个高并发场景, 用 jfr (Java Flight Recorder) 录一段, 拉进 JDK Mission Control 看看, 比你读 10 本书都直观.

JVM 这块, 我自己觉得是"门槛低, 天花板高". 入门背几个概念一周就够了, 但要真的能解决生产问题, 至少得有过 2-3 次真实的事故排查经历. 面试的本质是**让面试官相信你有过这些经历, 或者你有能力在类似场景下做到**. 案例准备 + 工具熟练 + 思考方式清晰, 这三块做到了, JVM 这关基本不会卡你.
