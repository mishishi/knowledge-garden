# 消息队列：Kafka vs RocketMQ，我们怎么选

上个月帮一个候选人复盘，面的字节 P6 高级岗，技术终面官问了一个很基础的问题：「你们为什么用 Kafka 不用 RocketMQ？」候选人愣了两秒，然后开始背书——吞吐量高、分布式、分区多副本、LinkedIn 出品、社区活跃。背得挺熟，但面试官眉头越皱越紧，最后追问一句：「那你们线上 Kafka 集群多少节点？单个分区峰值多少？你们订单消息怎么保证不丢？」

候选人一个数字都答不上来。

面完出来他跟我说，这块他准备得挺充分的，八股文背了一个礼拜。我问他：你们公司真的在用 Kafka 吗？他说在用。我问，那你连你们集群几个节点都不知道？

这就是我今天想聊的。消息队列这一章，面试官真正在问的从来不是「Kafka 是什么」，而是**你到底有没有真的用过**。八股文能帮你过笔试，但过不了 P6/P7 这一关。

我自己在阿里待过八年，后面带团队也面试了不少人，这章我尽量用真实经历来讲——有踩坑的、有翻车的、有最后发现「哦原来是这样」的。不保证全对，但保证都是真事。

---

## 先说结论：没有银弹，只有场景

面试里如果你被问「Kafka 和 RocketMQ 怎么选」，千万别上来就说「Kafka 更好」或者「RocketMQ 更适合业务」。这种答案一听就是没自己动过脑子。

我带新人说过一句话：**消息队列不是越强越好，是越合适越好**。

我们组 2019 年做一个电商中台的时候，选型是 RocketMQ。但同一个公司隔壁团队做日志收集和用户行为埋点，毫不犹豫上 Kafka。你说哪个选错了？都对，因为场景不一样。

先说**场景**——面试官不在乎你背了多少特性，他想知道你能不能在 30 秒内把场景拆清楚。

如果你做的偏**业务交易**（订单、支付、库存、积分），这些消息的特点是：体量不算特别大（单日千万级到亿级就算大的了），但对**可靠性、顺序、事务**有强诉求，因为丢一条订单就是真金白银。这种场景 RocketMQ 几乎是国内大厂的事实标准。

如果你做的偏**数据管道**（日志采集、用户行为、监控埋点、流计算），消息量是另一个量级，单日百亿、千亿都算正常，但对单条消息的可靠性没那么苛刻——丢一条行为日志，分析师重新跑一遍埋点就行。这种场景 Kafka 几乎没对手。

到这里你应该能感觉到我说的「合适」是什么意思：**不是技术本身分高下，是技术栈和业务特征对不对得上**。

我面试时如果听到候选人说「我们用 Kafka，因为它是行业标准」，我心里基本就会减分。这话太虚了。等价于没说。

## 顺序消息：90% 的人答错的那道题

顺序消息是面试高频题，但据我观察 80% 的候选人会在这里翻车。翻车点在哪？不是不知道有顺序消息这回事，而是答不清楚**顺序的范围**。

我第一次在生产环境里被顺序消息坑是 2017 年。那会儿做一个交易系统的订单状态流转，发了这么一段需求评审的话：「订单消息必须全局有序，消费者必须按订单创建顺序处理状态变更。」

我们组当时用的是 Kafka，设计方案的时候想当然地觉得——一个分区一个消费者不就行了吗？于是按订单 ID hash 到 6 个分区，每个分区一个消费者单线程消费。看起来没毛病。

上线第三天，线上出了一个问题。用户反馈：明明先支付的订单，后支付的那个反而先到账了。我们查了半天才发现，**支付回调的消息走的是另一个 Topic**，那个 Topic 我们用订单 ID hash，但消费者开了 8 个线程并行消费——这下好了，8 个分区的顺序全乱套了。

说白了，**分区内的顺序 Kafka 是能保证的，但全局顺序 Kafka 根本保证不了**。这是 Kafka 的设计哲学决定的——分区的存在就是为了水平扩展，你把分区扩到 1000 个，全局顺序就是个笑话。

RocketMQ 在这里就友好得多。RocketMQ 的顺序消息是**队列级别的**——同一个业务 key（比如订单 ID）会通过 hash 算法固定到同一个队列，这个队列里只有一个消费者线程在跑，所以业务 key 维度天然有序。你不用操心分区扩了顺序就乱。

我后来做架构评审，候选人上来就说「我用 Kafka 也能实现顺序」，我会反问一句：那你怎么扩分区？你扩到 64 个分区，原来按用户 ID 路由的消息是不是就乱套了？你是按业务 key 路由，那 64 个分区的吞吐上限是不是就锁死在一个分区的能力上？

如果他能答出「我们订单场景分区数固定 8 个，订单量评估下来单分区 5000 TPS 够用，宁可牺牲扩展性也要保顺序」，这就有戏。如果他支支吾吾说「应该能解决吧」，那基本就到这里了。

所以**面试高分答案的套路**是：先承认 Kafka 的限制（分区顺序不是全局顺序），再说你们怎么在这个限制下做业务补偿或者接受这个不完美。

一个反问的思路：如果业务真的要求全局强一致顺序，那是不是该反思一下——你到底需不需要消息队列？是不是用数据库事务 + 状态机更直接？消息队列擅长的是**异步解耦**，不是**强同步**。

## 事务消息：最容易讲玄学的地方

事务消息这块，国内面试基本上 RocketMQ 的主场。为啥？因为 Kafka 早期是没有事务消息的，0.11 版本之后虽然加了，但国内大厂用得不多，生态也少。你简历上写 Kafka 事务消息，我作为面试官第一反应是「你真的用过还是看了几篇博客？」

RocketMQ 的事务消息是个很有意思的设计——它不是传统意义上的分布式事务（2PC、XA），而是**本地消息表 + 反向回查**的思想。

我当时在阿里做交易系统，资金扣减 + 订单创建必须在同一个事务里完成。这事儿用本地事务表能做，但太重。后来切到 RocketMQ 事务消息，代码结构一下子清爽了：

```java
TransactionMQProducer producer = new TransactionMQProducer("tx_group");
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 这里是本地事务逻辑：扣库存、扣资金
        // 成功就 COMMIT，失败就 ROLLBACK
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 反向回查：Broker 过一会儿会问「你到底提交没提交」
        // 这里查本地事务状态
    }
});
```

核心思想是**两阶段**：

第一阶段，发起方发一个「半消息」到 Broker，这个消息对消费者不可见。发起方执行本地事务，根据结果告诉 Broker 是 COMMIT 还是 ROLLBACK。如果网络抖动发起方没告诉 Broker，Broker 会主动回查，回查 N 次（默认 15 次）还查不到，就当成 ROLLBACK 处理。

这个设计的好处是：**业务侧只需要关注本地事务，跨系统的一致性 RocketMQ 帮你兜底**。我们当时用这个方案，5 万 TPS 跑了两年没出过一致性问题。

面试里如果被问到事务消息，**千万别讲「RocketMQ 实现了 XA 规范」「它是基于两阶段提交的分布式事务」**。这是错的，RocketMQ 文档里写得清清楚楚它是「最终一致性」而不是「强一致性」。两阶段提交和它的设计完全是两码事，混着说会让有经验的面试官直接判定你是背书的。

我还被问过一个刁钻的问题：「如果本地事务执行成功了，但返回 COMMIT 的时候网络超时，Broker 一直回查不到怎么办？会不会消息发了但本地事务回滚？」

答案是：**会**。这就是分布式系统的根本难题——网络分区谁都不能完全避免。RocketMQ 的兜底是让你**业务侧能感知到回查超时**，通过日志告警 + 人工/对账补偿来处理。说白了，事务消息不是银弹，它把一致性问题从「业务强一致」降级到「业务最终一致 + 异常可感知 + 可补偿」。

如果候选人能讲出这层，说明他真在生产上跑过。

## Exactly-Once：大部分人误解的一个词

Exactly-Once Semantics（精确一次语义）这个词，我面试的时候基本每三个人就有两个人讲错。

错法一：说「Kafka 自己支持 Exactly-Once」。错。Kafka 0.11 之后支持的是**幂等生产者 + 事务**这两个机制组合，理论上能做到「一次且仅一次」的效果，但有严格前提：生产者不挂、Broker 不挂、消费者处理不挂——**任何环节重启就可能引入重复**。

错法二：把 Exactly-Once 等同于「业务不重复」。错。消息中间件的 Exactly-Once 只能保证**消息投递和消费位移提交是原子的**。业务逻辑执行过程中如果挂了，你该补偿还是得补偿。

我自己在生产上处理过这个事儿。2018 年我们做一个营销活动的发奖系统，奖品有金额和库存，重复发奖就是资损。当时想用 Kafka 事务消息搞定 Exactly-Once，结果上线发现一个 bug：消费者业务逻辑里有个外部 HTTP 调用调用营销接口，调用成功了但本地状态没来得及落库，进程就被 OOM Killed 了。重启之后 Kafka 重平衡，又消费了一遍这条消息，营销接口被调了两次，奖品发了两次。

事后我们做了两件事：

第一，**业务侧加了幂等**。每一笔奖品发出去之前，用一个分布式唯一 ID（雪花算法生成的）写到一张 `prize_record` 表里，INSERT IGNORE 一下，重复的消息会被这张表吃掉。

第二，**消费侧 ack 改成了业务处理完成之后**。原来是消费到就 ack，处理失败会重试；改成处理完再 ack，失败会无限重试直到业务处理成功。

这其实就是业内的标准做法：**消息中间件保证 at-least-once，业务侧保证 idempotent**。Exactly-Once 这事儿，分两层：

- **中间件层**：Kafka 通过幂等 + 事务，RocketMQ 通过事务消息 + 消费位点 + 业务幂等，能做到「不丢不重」（近似）
- **业务层**：无论中间件怎么承诺，你自己的业务必须能处理重复消息

面试里如果能讲清楚这两层的区别，胜过背 10 遍 Exactly-Once 的定义。我有时候甚至会追问一个具体的业务问题：你们订单系统的「支付成功」消息，消费者处理逻辑怎么保证幂等？重复消费一次，订单状态会出问题吗？

能答得清楚「我们用订单号做唯一索引，支付成功处理逻辑里先 SELECT FOR UPDATE 锁订单，状态机从『已支付』才能转到下一态，重复消息进来状态机不会迁移」——这就有 P6/P7 的水准了。

## 一次真实的积压排障：那 4 个小时我们都干了啥

2019 年双十一过后第二天，我们组遇到了一个典型的消息积压问题。我把过程完整复盘一下，面试里我经常拿这个案例问候选人。

背景是这样的：双十一当天天猫交易峰值我们系统扛住了，但零点过后流量回落的时候，我们发现一个 Topic 的消息**积压量从 0 飙升到 1.2 亿**，而且还在持续增长。

第一步，**先看监控**。Grafana 上能看到：消费者实例数还是 12 个，lag（消费滞后）在 1 点 30 分开始陡峭上升，生产者 TPS 倒是正常的——生产没爆，问题在消费侧。

第二步，**看消费者日志**。拉了一个消费者实例的日志，发现大量 `ReentrantLock tryLock failed, waiting`。这是个典型的**热点 key 阻塞**信号。我们用的是 RocketMQ，消息按用户 ID hash 到队列。问题来了：零点过后大促结束，用户开始批量操作退款、查物流、改地址——少数几个大客户（VIP 客户、KA 商户）的消息全 hash 到了同一个队列，而这个队列的消费者单线程处理能力就 1000 TPS。这批大客户的消息量是平时的 30 倍。

第三步，**临时方案**。我们做了两件事——

一是**给消费者扩容**，但因为是顺序消费，一个队列只能一个线程，扩了也没用。所以这条路堵死。

二是**临时改成并行消费**。我们给那个积压的 Topic 单独建了一个临时消费组，关闭顺序消费，用 32 个线程并发处理业务逻辑。业务侧保证幂等（这步当时救了我们）。

三是**限流**。联系上游业务方，让大客户的批量操作分批发，每批之间 sleep 200ms。**这个其实是治本的一步**，因为再多的消费者都顶不住源头洪水。

第四步，**根治方案**。我们后来改了两件事——

- **队列数从 8 个扩到 32 个**，hash 算法加了一致性 hash 而不是普通取模，这样扩队列时只迁移部分数据，**业务 key 的顺序性在大多数情况下能保留**（极端情况我们写了个对账脚本，事后校验）。

- **业务侧加了大客户识别**。如果检测到消息是 KA 商户的，走单独的 Topic + 单独的消费资源池，不和普通用户抢资源。

整个排障从发现到控制住，**4 个小时**。事后复盘 2800 字的报告我写了整整一下午。

我面试的时候偶尔会问：「你有没有遇到过消息积压？怎么排查的？」不是每个候选人都经历过，但有经历的人**讲细节的能力**完全不一样——能讲出「我先看了监控发现 lag 上升，然后查日志发现锁竞争，最后定位到热点 key」的人，基本可以判断是有实战经验的。

如果候选人说「我遇到过，但忘了具体怎么处理的」，我会换个问法：「假设一个 Kafka 集群 lag 突然飙到 1 亿，你怎么排？」听思路也是一样的。

排障思路我自己总结了一个套路：

1. **先看生产者和消费者谁爆了**——监控上生产者 TPS 和消费者 TPS 对比。
2. **看消费者日志**——错误堆栈、超时、锁竞争、GC 频繁、Full GC。
3. **看消息分布**——是不是有热点 key 集中在少数分区。
4. **看上下游**——是不是上游业务突然放量，或者下游数据库挂了。
5. **临时方案先止血**——扩消费者、限流、跳过非关键业务。
6. **根治方案后考虑**——改 hash 策略、加分区、业务拆分。

这个套路是通用的，不管是 Kafka 还是 RocketMQ。

## 消息不丢：最考验工程化能力的一道题

「怎么保证消息不丢」——这是消息队列面试里出现频率前三的问题。基本上所有候选人都知道「生产者重试 + Broker 持久化 + 消费者手动 ack」三件套。但**真正能讲出工程化细节的**不多。

我印象很深的一个 P7 候选人，他说他们支付系统用 Kafka，消息丢了 30 万条订单，最后靠对账才补回来。**不是技术不行，是配置细节没拉满**。他们当时的问题是：Kafka 的 `acks` 参数用的是 `1`（leader 写入成功就返回），leader 写完还没来得及同步到 follower，leader 挂了，消息就丢了。

正确的配置应该是 `acks=all` 或者 `-1`，等所有 ISR 副本都写入成功才返回。代价是延迟上升——我们当时测下来 P99 延迟从 8ms 涨到 25ms，但为了不丢消息这个代价是值的。

还有一个细节是**消费位移提交的时机**。我们组当时踩过一个坑：消费者处理完业务逻辑准备提交位移的时候，进程被 kill 了。位移没提交，重启之后这条消息又被消费了一遍。这就是经典的 at-least-once 场景。

解决方案是**业务处理完先提交本地状态，再提交位移**。但这里有个微妙的竞态：如果本地状态提交了、位移没提交，重启会重复消费——业务幂等解决。如果本地状态没提交、位移提交了，业务就漏处理——这个比重复更可怕。

我们当时为了避免第二种情况，做了个**位点和状态的双重校验**：消费消息时先查数据库里有没有这个业务 ID 的处理记录，没有才处理。这样即使位移提交了，状态没写，重启也会再处理一次（因为业务 ID 没记录）。

面试里如果能讲出**这些工程化的权衡**——比如「我们故意保留 at-least-once 的重复，业务用幂等表兜底」——这就是 P6/P7 应该有的深度。

RocketMQ 在这块稍微省心一点，因为它有**同步刷盘 vs 异步刷盘**的选项，同步刷盘能保证 broker 收到消息就落盘成功，代价是吞吐下降。Kafka 没这个选项，靠的是 `acks` + `min.insync.replicas` 来兜底。

## 给候选人的两个具体建议

聊了这么多，最后给准备面试的同学两个 actionable 的建议。一个是准备方向的，一个是回答技巧的。

**第一，去翻你现在的项目里，消息队列相关的所有监控面板和配置项。** 比如：

- Kafka：`acks` 是什么值？`min.insync.replicas` 是几？`unclean.leader.election.enable` 是 true 还是 false？分区数多少？副本数多少？ISR 缩容过没？
- RocketMQ：刷盘是 SYNC 还是 ASYNC？broker 主从同步是 SYNC_MASTER 还是 ASYNC_MASTER？事务消息用没用？回查次数配的多少？

这些配置项是面试官最喜欢问的，因为背书答不上来，只有真用过的人能答。

**第二，回答「为什么选 X 不用 Y」这类问题时，用「场景 → 诉求 → 决策 → 代价」四步法。**

举个例子：

> 我们当时做订单中台，诉求是顺序消息 + 事务消息 + 可靠性优先，单分区 TPS 评估下来 5000 就够用。所以选了 RocketMQ。
>
> 代价是：放弃了 Kafka 的超高吞吐优势；如果以后单分区需要超过 1 万 TPS，我们要重新评估是不是换 Kafka 或者改造业务。

这种回答模式，**比背 100 个特性都有用**。面试官听了一耳朵就知道——这个候选人真正做过决策，不是背书的。

我们组后来招进来一个 P6 候选，背景是二线公司，简历上没写什么牛逼项目，但终面的时候讲了 20 分钟他们怎么用 Kafka 改造一个老旧消息系统——从 0 积压到日均处理 8 亿消息，把生产者和消费者的几十个配置项一个一个讲清楚，最后还说了「我们当时 `acks=1` 改 `acks=all` 之后 P99 涨了 18ms 做了业务补偿」。

**当场给 offer**。不是因为技术多牛，是因为他讲的东西**真实、有细节、有思考**。这种东西背书背不出来。
