# 02. Java 基础深挖: 那些 90% 候选人翻车的题

去年我们组招 P6, 一面二面加起来聊了 47 个人。最后通过的 8 个, 我拉名单看了一下, 有个挺扎心的规律: 大部分挂在 Java 基础, 不是挂在中间件, 也不是挂在项目深度。

不是说中间件、项目不重要。是基础这块, 大家普遍有个错觉——"我都用了五年 HashMap 了, 能有啥?" 然后面试官一问, "你说说 ConcurrentHashMap 1.7 和 1.8 的区别", 或者 "synchronized 升级过程画一下", 立刻就卡壳。卡得五花八门: 有的能把 1.7 的 Segment 答上来, 一问 1.8 的 CAS+synchronized 就开始混; 有的能把 synchronized 升级过程背下来, 问你 "为什么是这四个阶段, 不能跳过去吗", 直接死机。

这一章我就专门聊这个。我不讲教科书里能找到的内容, 我讲面试官到底想听什么, 以及真实项目里这些东西怎么用、怎么踩坑。

## HashMap 那点事, 真不是 "数组 + 链表" 三个字能概括的

面试官问 HashMap, 表面上是问数据结构, 实际上问的是 "你能不能把一个东西讲清楚它的来龙去脉"。大部分候选人上来第一句就是 "HashMap 底层是数组加链表, 1.8 之后加了红黑树", 背得挺顺。但接下来只要追问一句 "为什么负载因子是 0.75, 不是 0.5 或者 1?", 立马就崩。

我当年面过一个候选人有 8 年经验, 项目讲得很溜, HashMap 这块一追问就露馅。他先说 "0.75 是空间和时间的折中", 这个回答算及格——但还不够。面试官想听的其实是: 这个数怎么来的? 泊松分布。在 HashMap 的源码注释里, 作者明确写了, 负载因子 0.75 是在 "容量" 和 "扩容成本" 之间取的平衡点, 而且通过泊松分布算出来, 在负载因子 0.75 的情况下, 链表长度超过 8 的概率是 0.00000006, 几乎不可能。

你把这个说出来, 面试官眼睛会亮一下。但更重要的是——你能讲清楚 "树化阈值为什么是 8, 不是 7 也不是 9"。这个其实还是泊松分布的延伸。如果你说 "因为 8 的时候树化性价比高, 6 的时候会退化成链表", 这就比单纯背数字强多了, 因为你解释了 "为什么不直接一直用红黑树"——红黑树的节点比链表节点大一倍, 空间成本高, 而且小链表查询其实够用。

那 ConcurrentHashMap 怎么问? 这才是重灾区。

我问过很多人 "1.7 和 1.8 的 ConcurrentHashMap 区别", 答 1.7 用 Segment 分段锁、1.8 用 CAS+synchronized 的占绝大多数。但答到这就停了, 显然不够。面试官会接着问: "为什么 1.8 要废弃 Segment?" 你得说清楚: Segment 本质是个 ReentrantLock, 每个 Segment 独立加锁, 锁粒度还是偏粗; 1.8 直接锁在 Node 头节点上, 配合 CAS 做无锁写入, 锁粒度细到桶级别, 并发度上来了。

还有一个我特别爱问的: "get 操作需不需要加锁? 为什么?" 这个能筛掉一半人。答案是 "不需要, 因为 Node 的 value 和 next 都用 volatile 修饰, 写线程对链表的结构性修改 (比如 put) 会创建新的头节点, 不会被读线程看到中间态"。这个回答如果能讲出来, 说明你真看过源码, 不是背的面经。

讲个我自己的踩坑。19 年我们做支付清结算, 当时有个并发场景, 多个线程往一个 Map 里塞交易流水, 用的是 HashMap。跑了两天没事, 第三天 CPU 飙到 100%。后来排查是 HashMap 在并发 put 下导致链表成环, 查询时死循环。那个 bug 让我记住一件事: 并发场景下, HashMap 是定时炸弹, 哪怕你逻辑上 "应该不会出问题", 概率再低, 量大了就一定出。后来那个模块换成了 ConcurrentHashMap, 顺手把 size 统计从 `map.size()` 改成自己维护 AtomicLong——这个细节值得讲, 因为 ConcurrentHashMap 的 size() 是估算的, 不准, 高并发下会偏, 生产上别信。

## synchronized 升级, 真正难的是 "为什么"

这块我面下来, 80% 的候选人能把 "偏向锁 → 轻量级锁 → 重量级锁" 这条线背下来, 包括每个阶段触发的条件。但只要我换个角度问, 比如 "偏向锁为什么在 JDK 15 被废弃了", 或者 "轻量级锁自旋一定比重量级锁快吗", 立刻卡壳。

我先说一个反常识的事: synchronized 在 JDK 1.6 之前是纯重量级锁, 性能很差, 大家才会去用 ReentrantLock。1.6 之后引入了锁升级机制, 性能才追上来。所以你讲升级过程的时候, 一定要把背景带上——不是 Java 设计者闲得没事做, 是因为早期 synchronized 性能拉胯, 才需要这套优化。

具体怎么升级, 我用一个真实场景讲。

我之前做的一个网关系统, QPS 大概 12 万, 接口里有个 `synchronized (this)` 修饰的热点方法。当时监控看到锁竞争很严重, 线程上下文切换开销巨大。后来我把那段代码拆成: 第一次进来走偏向锁 (只有一个线程), 高峰期切到轻量级锁 (CAS 自旋, 几个线程抢), 极端情况下升级到重量级锁 (锁队列)。你猜怎么着? 改完后 P99 从 180ms 降到 45ms。

这个例子在面试里特别好用——你可以说: "我之前做过一个网关, 监控看到 synchronized 竞争激烈, 通过 -XX:+PrintFlagsFinal 和 jstack 抓到大量 BLOCKED 状态的线程, 后来分析发现是同一个锁被多个线程反复争抢, 不是死锁。然后我们做了两件事: 一是把锁粒度拆细 (锁对象从 this 改成局部变量), 二是对非热点路径用 CAS 替代 synchronized。最后 P99 从 180ms 降到 45ms。"

面试官听这种描述会很有画面感, 比你说 "synchronized 升级是为了性能优化" 强 10 倍。

那回到升级的具体过程。偏向锁的核心是 "CAS 替换线程 ID", 第一次进来记录线程 ID, 后续同一个线程进来直接放行, 不做任何同步。这个机制听起来很美, 但有个坑: 如果线程 A 持有偏向锁, 线程 B 来抢, 必须先撤销偏向锁 (需要 STW 的安全点), 这个撤销成本其实不低。所以 JDK 15 干脆废弃了偏向锁, 因为现代应用多线程场景多, 偏向锁用不上, 撤销开销反而成了负担。这个细节能讲出来, 说明你跟过 JDK 的演进, 不是只看过几年前的博客。

轻量级锁这块, 候选人经常答错的一点是: "自旋一定比阻塞快"。其实不对。自旋是 CPU 空转, 如果锁持有时间长, 自旋就是浪费 CPU, 这时候升级到重量级锁挂起线程反而更划算。HotSpot 的自适应自旋会根据之前自旋的结果动态调整自旋次数, 就是为了避免这种浪费。

我面过一个人, 把这块答得很细, 顺带讲了 "锁消除" 和 "锁粗化"——比如 StringBuffer 的 append 在循环里会被锁粗化, 多个 append 合成一个同步块; 比如局部对象永远不会逃逸出去, JIT 会把锁直接消掉。这种细节加上, 面试官基本就放心了。

## AQS 是必考的, 但 90% 的人讲不清它的精妙

AQS (AbstractQueuedSynchronizer) 这个东西, 说实话, 工作里直接用的机会不多——除非你写中间件。但它太重要了, 因为 ReentrantLock、CountDownLatch、Semaphore、CyclicBarrier 全是基于 AQS。面试官问这块, 是在考察你 "读没读过 JDK 源码", 以及 "能不能把一个抽象框架讲明白"。

我一般这么问: "你说说 AQS 的核心思想是什么?" 10 个候选人里 8 个会说 "维护一个 state 和一个 CLH 队列", 这个算入门。但接下来问 "ReentrantLock 怎么基于 AQS 实现可重入", 又是 8 个答不上来。

AQS 的精妙在于: 它把 "锁" 这种抽象的东西, 拆解成了三部分——state (同步状态)、等待队列 (CLH 变种)、以及两个核心方法 (tryAcquire 和 tryRelease, 由子类实现)。这就是模板方法模式的经典应用。AQS 自己管排队和阻塞, 子类只关心 "怎么判断能不能拿到锁"。

ReentrantLock 怎么实现可重入? 它的 tryAcquire 里会先 compareAndSetState(0, 1) 抢锁; 如果失败, 再判断当前线程是不是持有锁的线程 (exclusiveOwnerThread), 是的话把 state 加 1, 表示重入一次。释放的时候, 把 state 减 1, 减到 0 才真正释放。这套机制你讲清楚了, 再回头看 CountDownLatch (state 是计数, countDown 是 CAS 减 1, await 是判断 state==0) 就一目了然。

我第一次读 AQS 源码是 17 年, 当时看的是 ReentrantLock 的 fair 和 nonfair 实现。读完后有个特别深的感受: Doug Lea 把 "不可重入" 变成 "可重入", 把 "不公平" 变成 "公平", 居然只是在 tryAcquire 里改了几行代码。FairSync 的 tryAcquire 里加了个 `hasQueuedPredecessors()` 判断, 如果队列里有等待的线程, 就乖乖排队。NonfairSync 直接抢, 不管队列。这个对比讲出来, 面试官就知道你是真懂。

那 AQS 在项目里怎么用? 我自己有个真实案例——我们做分布式任务调度的时候, 多个 worker 抢同一批任务, 用 Redis 做分布式锁嫌重 (网络开销), 就在单机 JVM 里用 Semaphore 控制并发度。Semaphore 就是 AQS 的子类, state 表示剩余许可, acquire 是 CAS 减 state, release 是加回去。这种场景其实很多, 比如限流、比如连接池控制并发数, 都是 AQS 的应用。

还有个面试官爱问的: "AQS 的 CLH 队列是单向还是双向? 为什么?" 答案是双向。每个节点有 prev 和 next 两个指针, 主要为了取消——当一个线程等待超时或者被中断, 它要从队列里移除, 双向链表方便前驱和后继的衔接。这个细节你讲出来, 至少说明你画过 AQS 的图。

## 真实项目里, 这些东西怎么串联起来

光讲单个知识点没用, 面试官想看的是你能不能把它们串成一个系统性的认知。我自己面 P6, 最后一定会问一个开放性问题: "你说说 Java 并发包里, 哪些类是乐观的, 哪些是悲观的, 为什么这么设计?"

这个问题好在哪? 它没有标准答案, 但能看出你对整个并发体系的理解。

我的标准答案大概是: Atomic 系列 (AtomicInteger、AtomicLong) 是乐观的, 用 CAS; synchronized 和 ReentrantLock 是悲观的, 用阻塞。但悲观锁里又分偏向、轻量级、重量级三档, 其实是 "乐观的悲观"——偏向锁和轻量级锁本质也是乐观 (CAS), 只是兜底才是真悲观。

ConcurrentHashMap 是个混合体: 1.8 之后, 写入用 CAS + synchronized (桶头节点), size 统计用 baseCount + CounterCell (类似 LongAdder 的分散计数)。这套设计你讲出来, 面试官会觉得你脑子里有完整的图。

还有个我特别喜欢的串联问题: "你说说 HashMap、ConcurrentHashMap、Hashtable 三者的区别, 以及它们的演进背后反映了什么?" 这个题考的不是知识点本身, 是你能不能从演进史看到 Java 并发模型的成熟。

我的回答是: Hashtable 是 JDK 1.0 的产物, 所有方法 synchronized, 性能差; HashMap 是 1.2 引入的, 不保证线程安全, 但单线程性能好; ConcurrentHashMap 是 1.5 引入的, 1.7 用分段锁, 1.8 用 CAS+synchronized。这背后反映的是: Java 的并发模型从 "一刀切同步" 演进到 "按场景精细化同步", 是对性能和正确性平衡的不断追求。

## 写给候选人的两个建议

第一个, 别背, 读源码。我当年面试前, 把 HashMap、ConcurrentHashMap、AbstractQueuedSynchronizer、ReentrantLock 的源码全跟了一遍, 每个都画了图 (内存结构和状态机)。画图这个动作特别重要, 比你看 10 篇博客管用。当你能在脑子里跑通 "put 一个 key 进去, 经历了什么" 这个流程, 面试官怎么问都不慌。

第二个, 准备 1-2 个真实踩坑案例。我自己准备的是 19 年那次 HashMap 并发死循环, 以及前面说的网关 synchronized 优化。案例要包含: 背景 (什么系统、什么量级) → 问题 (现象是什么, 怎么发现的) → 定位 (用什么工具、看到什么) → 解决 (具体怎么改) → 结果 (数字验证)。这种 "五段式" 的案例, 任何技术问题都能往里套。面试官听完基本就放心了, 因为他知道你是真干过, 不是背的。

最后说一句, Java 基础这块, 真的不是 "会用" 就行。面试官考察的是你 "懂不懂为什么这么设计", 以及 "出问题的时候你能不能定位到这层"。大部分 P6 候选人栽在这, 不是因为他们不知道 HashMap 是数组加链表, 而是他们没办法在追问下保持逻辑自洽。
