# 04. 并发编程：那些你不敢写在简历上的踩坑

说句得罪人的话，并发编程是我见过**简历包装最狠**的一块领域。

我面过太多候选人，简历上写着"深入理解 JUC 源码""解决过复杂并发问题"，结果一问到 `synchronized` 锁升级的过程就开始念八股文，一问"你生产环境真出过并发问题吗"就支支吾吾。要么是把网上的案例原封不动背一遍，要么是讲一个自己根本没排查完、靠 leader 救场的故事。

这章我想聊点不一样的。不是 `ThreadPoolExecutor` 的构造参数有哪些、不是 AQS 源码怎么走——这些你随便翻本《Java 并发编程实战》或者看个小视频都能会。我想聊的是：我自己、我们组、我们隔壁组，**真金白银踩过的坑**，以及面试时怎么把这些坑讲成加分项。

去年一整年，我们组线上 3 个比较严重的并发 bug，全是和这块相关。我拿这 3 个 case 串起来讲，每个 case 对应面试里最高频的一类问题。

## 死锁：教科书都会写，线上还是年年出

死锁这个东西，我第一次见是在刚工作第二年。一个老员工写的转账服务，跑了一周没出事，某天突然所有转账接口超时。监控一查，线程全部 BLOCKED，jstack 一打，经典的 `Found one Java-level deadlock` 弹出来。

那个 case 我没参与排查，但听 leader 讲完记到现在。代码大概长这样（简化版）：

```java
synchronized(fromAccount) {
    synchronized(toAccount) {
        // 转账
    }
}
```

两个账户互相转，A 给 B 转的同时 B 给 A 转，就死锁了。教科书答案：按账户 id 排序，所有线程都先锁小的、再锁大的。这个答案没错，但**只回答了教科书层面**。

面试官真正想听的是什么？我给你列一下从浅到深：

第一层，**你能讲出死锁的四个必要条件**。互斥、持有并等待、不可剥夺、循环等待。这个是基础，背也得背下来。

第二层，**你能讲出排查命令**。`jstack <pid> | grep -A 30 deadlock`，或者用 `fastthread.io` 这种在线工具分析 thread dump。最好能说清楚 thread dump 里你会关注哪些字段——线程状态、等待锁的地址、持有锁的线程 id。我们当时就是用 jstack 看到线程卡在 `TransferService.lockAccount(TransferService.java:47)` 这种位置。

第三层，**你能讲出真正的预防策略**，而不仅仅是"按 id 排序"。我面过太多候选人只会背这个。说几个更实战的方向：

**用 `tryLock` 替代 `synchronized`**。`ReentrantLock.tryLock(timeout, unit)`，拿不到锁就回滚重来。代价是代码复杂度上升，但能避免无限等待。我们组后来所有涉及多账户操作的代码都改了 tryLock 模式。

**缩小锁粒度**。原来锁整个账户对象，改成锁账户下的一个 "balance" 子对象。前提是业务允许，账户的其他字段（比如用户名）可以被并发读。

**业务层面拆开**。这个是最容易忽略的。比如 A 给 B 转账，能不能拆成"先冻结、然后异步清算、最后入账"？异步化之后，锁的持有时间从秒级降到毫秒级，死锁概率直接降一个数量级。

面试时如果只能讲一点，**讲 tryLock**。这是最能让面试官眼前一亮的答案——因为它说明你真的写过并发代码，不只是看过。

我再讲一个更隐蔽的死锁，是我自己写的。

前两年做的一个对账系统，每天凌晨跑批，从 MySQL 拉数据和一个外部系统对账。逻辑大概是：先查 A 表，循环处理，处理完更新 A 表，再查 B 表，循环处理。看起来很简单对吧？

```java
// 伪代码
List<Task> tasksA = dao.queryA();
for (Task t : tasksA) {
    // 一些耗时操作
    process(t);
    dao.updateA(t);
}
List<Task> tasksB = dao.queryB();
// 类似的循环
```

上线后某天报障：对账任务卡住，凌晨 5 点还没跑完。jstack 看了下，线程卡在 `Connection.getConnection()` 上——**连接池耗尽**。

我一开始以为是连接池配小了，调大到 200 没用。后来仔细看，process(t) 那个方法里有个**远程调用**，下游系统抖动的时候单次调用能卡 30 秒。循环里又同时持有数据库连接不放，结果就是：20 个连接、20 个线程全在 process 里卡着，新任务想拿连接拿不到，全部阻塞。**这不是死锁，是活锁**——没有真正互相等待，但系统已经不干活了。

这个 case 教我的事情是：**线程池的参数不是想当然配的**。核心线程数、最大线程数、队列容量、拒绝策略，每一个都得根据下游的依赖情况算。我们后来改成每处理完一批（10 条）就释放连接回池，让其他任务能跑起来。

面试里被问到"线程池怎么配"，**千万别背那个 2N+1 的公式**（N 是 CPU 核数）。那个公式是 CPU 密集型任务的，**IO 密集型任务完全不能用**。正确的是：根据下游 RT 和想要的 QPS 反推。先做压测，看单线程能跑多少 QPS，然后除一下。要是你的下游 RT 是 200ms，单线程 QPS 就是 5，要扛 1000 QPS 就得 200 个线程。

## CAS 和 ABA：面试高频题，生产基本不直接用

`AtomicInteger` 背后的 CAS 原理，三步——比较、交换、自旋。这个八股文谁都会背。我问候选人一个变种问题："CAS 在生产里有什么坑？"

80% 的人答不上来。

**ABA 问题**。CAS 拿当前值 A 去比较，内存里如果还是 A 就替换。但如果中间被改成了 B、又改回了 A，CAS 不知道。这个问题理论上存在，**生产里我一次都没遇到过**。你简历上写"深入理解 ABA 问题"，面试官内心其实在想"你是不是为了面试硬学的"。

那 CAS 真正值得讲的坑是什么？**自旋开销**。高竞争场景下，多个线程同时改一个 AtomicLong，自旋次数会飙升，CPU 飙到 100% 都不稀奇。我之前做过一个实验：一个 8 核机器，16 个线程改同一个 AtomicLong，吞吐量比 `synchronized` 低 30% 左右。原因是自旋的代价在多核下被放大了。

生产里怎么处理？**两种思路**：

一是**减少竞争**。LongAdder 替代 AtomicLong，JDK 8 之后的高频计数器都改用这个。原理是分散热点，每个线程有自己的 cell，最后求和。代价是求和要遍历所有 cell，比 AtomicLong 的 get() 慢一点，但写多读少的场景下收益巨大。

二是**避免用**。如果业务允许，用 `synchronized` 反而更稳。我之前在 code review 里见过一个同事，把所有 `AtomicInteger` 都换成 `synchronized`，因为他的业务是写少读多，synchronized 在低竞争下的开销几乎为零，而且代码更清晰。**简单胜过复杂**，这是工程里很重要的原则。

面试里被问 CAS，**你主动提一句"高竞争下用 LongAdder 更好"**，基本就稳了。这说明你不是背书的人，是真做过 benchmark 的人。

## 线程池：我见过最离谱的配置

线程池我必须单独拎出来讲，因为这是我**踩坑最多**的地方。

第一个坑：**队列容量配成 `Integer.MAX_VALUE`**。这意味着任务堆积到 21 亿个都不会触发拒绝策略。我见过一个生产事故，配的就是这个，业务高峰时队列里堆了 200 多万条任务，OOM 了。**队列容量一定是要算出来的**，根据业务能容忍的最大延迟反推。延迟 1 秒、QPS 1000，那就是 1000，延迟 10 秒就是 10000，再考虑个 2 倍冗余。

第二个坑：**线程池不复用**。每个接口 new 一个 ThreadPoolExecutor，堆了几十个线程池。后果是线程数失控，机器上跑了 2000 多个线程，上下文切换开销大到业务逻辑跑不满 CPU。线程池必须是全局共享的，**一个业务一个池子**，最多再加个"未知任务"兜底池。

第三个坑：**忽略拒绝策略**。默认的 `AbortPolicy` 会抛异常，抛了之后你的业务方怎么办？我之前见过一个代码，submit 之后没 try-catch，线程池满了直接异常打到上游，结果上游以为是业务逻辑错误，重试了 10 次，**事故放大 10 倍**。**`CallerRunsPolicy`** 是更安全的默认选择——池子满了让调用方自己跑，至少能起到限流的作用。

我们组去年做的最划算的一件事情，就是把全公司几百个服务接入了**动态线程池**。基于美团的 `DynamicTp` 或者 Hippo4J 这种开源方案，线程池的核心参数在运行时可调，**不用重启服务**。带来的好处是：再也不用凌晨爬起来改线程池参数了，监控发现队列堆积，在控制台直接调。

面试里被问"你们生产怎么用线程池"，**能讲出动态线程池**基本是 P7 起步的水平。绝大多数候选人还在背"7 个核心参数"。

## 伪共享：性能优化的最后一公里

这个东西很少有人讲，但讲明白了能体现深度。

**CPU 缓存行**一般是 64 字节。一个 `long[]` 数组，理论上 8 字节一个元素，连续 8 个元素占满一个 cache line。如果两个线程分别改 `arr[0]` 和 `arr[7]`，虽然业务上完全不相关，但**它们在同一个 cache line 上**，CPU 必须让 cache line 在两个核之间来来回回——这就是伪共享，性能杀手。

我们当时做的一个行情推送系统，热点数据放在 `AtomicLong[]` 数组里，16 核机器实测 QPS 卡在 50 万上不去。CPU 监控看着挺闲，但 `perf stat` 一看，`cache-misses` 高得离谱。改成 `LongAdder`（内部用 `@Contended` 注解填充，避免伪共享）之后，QPS 涨到 80 万，**60% 的提升**。

**`@Contended`** 是 JDK 8 引入的注解，加在字段上让 JVM 自动填充。但默认是关闭的，要加 JVM 参数 `-XX:-RestrictContended` 才能用。很多团队不知道这个开关，注解加了等于没加。

面试里被问性能优化，**主动提一句伪共享**几乎必杀。但要小心，**只在确实需要的时候用**——`@Contended` 会让对象变大，浪费内存。不要为了"优化"无脑加。我们当时是因为压测数据明确指向 cache miss 才上的，不是拍脑袋。

## 写在最后：给候选人的两个建议

第一个建议：**准备 2-3 个你亲自排查过的并发 case**。不用多，2-3 个足够，每个讲清楚背景、症状、排查过程、根因、修复方案。面试官一天面 5 个人，能讲出"我当时怎么用 jstack 看到那个 waiting on condition 状态"的，10 个里有 1 个就不错了。

第二个建议：**别只背 API 名字**。`CountDownLatch` 和 `CyclicBarrier` 的区别背得再熟，也不如讲一句"我用 CountDownLatch 等三个数据源拉取完成才继续下游计算，每个数据源 3 秒超时，超时后我让主线程先走、记录告警"。后者才是工程思维。

并发这块，**写过的代码量决定你的深度**。如果你们组并发场景不多，主动去翻一翻 `synchronized`、`ConcurrentHashMap`、`ThreadPoolExecutor` 的源码，重点看注释里写的实现动机——为什么这么设计、解决了什么问题。比看任何书都管用。
