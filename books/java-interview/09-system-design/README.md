# 系统设计 4 道经典题: 高分答案的套路

我面试过大概 200 多个 P6/P7 候选人, 系统设计这轮是挂人最多的。算法题大家多少会刷, 但系统设计一开口, 差距就出来了。有些人讲得又空又散, 20 分钟过去, 面试官心里已经在想 "这个给 P5 吧"。有些人能跟面试官一来一回聊起来, 哪怕方案不是最优, 也能拿到 strong hire。

这章我想把这种 "差距" 讲清楚。我会挑 4 道最常考的题——短链、秒杀、feed、IM——拆开聊, 重点不是技术本身(那些书上一搜就有), 而是**面试场景下怎么讲**。包括开场怎么破冰, 中间怎么跟面试官对话, 什么时候该问澄清问题, 什么时候该收住。

## 为什么系统设计这轮这么容易挂人

先说个我自己的观察。我们组的系统设计题, 跟网上那些面经不太一样。网上面经爱问 "设计一个 Twitter", 一听就很大, 候选人上来就懵, 开始背八股: "读写分离、分库分表、消息队列……" 全是关键词, 但没有一句话是真正在回答问题。

我后来反思, 觉得问题出在两件事: 候选人不知道**面试官到底在考什么**, 以及不知道**怎么用对话推进设计**。

先说第一个。系统设计不是让你真的设计一个 Twitter。你一个人坐在那里 45 分钟, 设计出来的方案肯定不能上线。面试官真正想看的是三件事: 你的**技术深度**(真的懂还是背的), 你的**工程判断力**(知道什么是 trade-off, 不是非黑即白), 还有**沟通能力**(你能不能在模糊的需求里, 找到关键问题)。

第二件事更重要。系统设计应该是**对话**, 不是**演讲**。我见过太多候选人, 一上来就 5 分钟不停地说, 也不看面试官反应, 也不停下来问问题。讲到最后, 面试官忍不住打断: "你这个 Redis 集群具体怎么分片的? 数据一致性怎么保证?" 候选人当场卡住。

好的候选人是什么节奏? 他们会**主动留白**。讲一个方案, 然后停下来问: "这个方向您觉得可以吗? 如果可以的话, 我想再深入聊一下 [某个点]。" 这种互动, 才是面试官想看到的。

下面我会用 4 道题具体讲。

## 短链系统: 怎么开场, 怎么收

短链是入门题, 但很多人答砸了。问题一般是这样: "设计一个像 bit.ly 那样的短链服务, 用户输入长 URL, 返回短 URL, 访问短 URL 能跳到原地址。"

候选人最容易犯的错是什么? 一上来就说哈希算法。MD5、SHA、Base62, 背得滚瓜烂熟, 实际上根本没搞清楚这个系统的核心矛盾是什么。

**核心矛盾是读写比严重失衡, 而且写可以接受慢, 读必须快**。为什么? 短链一旦生成, 就再也不会变了。但它会被点击, 可能被点几百万次。所以读 (302 跳转) 是高频, 写 (生成短链) 是低频。这个是设计的起点。

我第一次面别人这道题的时候, 候选人是这么开场的:

> "我先问几个问题。QPS 大概是什么量级? 是 to C 的还是 to B 的? 短链有有效期吗?"

我心里一下就舒服了。这三个问题问得都很到位: QPS 决定缓存策略, 场景决定容错要求, 有效期决定存储设计。后来这个候选人拿了 strong hire。

但更多的候选人是这么开场的: "这个我会, 我之前看过。" 然后开始讲方案。没有问任何澄清问题, 直接按自己想象中的需求往下走。这是大忌。你想象中的需求, 跟面试官心里的需求, 90% 不一样。

**怎么开场, 我建议 30 秒到 1 分钟, 问 2-3 个澄清问题**。可以问的包括: 量级 (日活多少, 短链生成多少, 跳转多少)、功能边界 (要不要统计点击? 要不要做防封? 要不要支持自定义短码?)、一致性要求 (强一致还是最终一致, 短链生成后多久必须可读?)。

澄清完之后, 怎么推进? 我的建议是**先讲存储, 再讲读写路径, 最后讲优化**。这是最自然的逻辑顺序。

存储选什么? MySQL 足够, 早期甚至单机 MySQL 都行。表结构两张: 一张是短链映射表 (短码 → 长 URL, 创建时间, 过期时间, 用户 ID), 一张是访问日志表 (短码, 访问时间, IP, UA)。映射表用短码做主键, 索引天然就好。

写路径, 短码怎么生成? 三种方案: 哈希 (MD5/SHA 截断)、自增 ID + Base62 编码、随机生成 + 查重。**我建议讲自增, 然后顺手提一下另外两个的坑**。自增的好处是简单、不冲突、短码短; 坑是连续 ID 容易被爬, 还有分布式下的 ID 怎么搞 (Snowflake 或者号段)。讲到这个程度, 面试官基本就点头了。

读路径, 重点讲**缓存**。301 还是 302? 这个细节很多人不知道。301 是永久重定向, 浏览器会缓存; 302 是临时重定向, 浏览器不缓存。我们用 302, 因为短链可能会被封、被回收, 我们希望每次都来服务端校验。缓存用什么? Redis, key 是短码, value 是长 URL, TTL 设成跟短链过期时间一致。

讲完这些, 你的时间大概过去 15 分钟。剩下 30 分钟干嘛? **跟面试官聊 trade-off**。比如: 如果短链生成量是现在的 100 倍, 怎么办? 数据库扛不住, 你会怎么分? (按短码 hash 分库)。如果某个短链被刷了, 怎么办? (限流 + 验证码 + 黑名单)。如果数据要做冷热分离, 老的短链怎么归档? (按时间分区, 或者单独存 OSS)。

这种聊法, 才是面试官想看到的深度。

**踩坑案例**: 我们组真的做过一个短链服务, 当时日均生成 2000 万条, QPS 峰值跳转 8 万。一开始用的自增 ID, 跑了一年没问题。后来因为业务调整, 需要做短链回收 (同一个长 URL 可以重新生成不同的短码), 数据库主键压力上来了, 加上早期没分库, 写入开始抖。

我们改了什么? 主键从自增改成了 Snowflake, 数据库按短码 hash 拆成 8 个库, Redis 前面加了一层 LocalCache (Caffeine), 命中率做到 92%。最关键的改动其实是监控: 之前出问题靠用户反馈, 后来我们把短链跳转的 P99 延迟、Redis 命中率、数据库慢查询都接到了 Grafana, 出问题 5 分钟内能定位。

这些东西在面试里怎么讲? 不要罗列, 讲一两个点就够了。**讲你踩过的坑, 讲你当时怎么想的, 讲最后怎么改的**。这就是你跟其他候选人的差距。

## 秒杀系统: 怎么突出"工程"两个字

秒杀是另一道经典题。"双 11 零点, 1 万件商品, 1000 万人来抢, 怎么设计?"

这道题的迷惑性在于, 很多人会答成**纯技术题**, 上来就讲 Redis 预扣、消息队列削峰、分布式锁。然后呢? 然后就没了。讲得很全, 但很平。

**面试官想问的是, 你有没有真正的工程经验**。一个没有做过秒杀的人, 和一个做过的人, 讲出来的东西细节差很多。没做过的人讲方案, 都是教科书上的词; 做过的人讲方案, 会带具体的数字、具体的失败案例、具体的凌晨三点救火经历。

我自己的经验是, 讲秒杀要抓住三个字: **一致性、性能、容错**。这三个字不是分开讲的, 是要揉进同一个故事里。

先说一致性的坑。秒杀最大的坑不是性能, 是**超卖**。1 万件商品卖出去 1.1 万件, 你试试看, 老板会杀人的。超卖怎么防? 数据库唯一索引, 扣减用 `UPDATE stock SET count = count - 1 WHERE item_id = ? AND count > 0`, affected rows 是 1 才算成功。这是底线。

但数据库扛不住 1000 万 QPS。所以前面要加 Redis 预扣, 库存先扣 Redis, 再异步落库。**Redis 怎么保证原子性? Lua 脚本**。这是一个关键细节, 面试官会问。

```lua
-- 伪代码
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) > 0 then
  redis.call('DECR', KEYS[1])
  return 1
end
return 0
```

讲到这里, 面试官可能会问: "Redis 扣成功了, 但数据库落库失败了, 怎么办?" 这个问题其实是考你**对账和补偿**的理解。答案是: 用一个补偿任务, 定期扫描 Redis 扣减成功但数据库没落库的单子, 重新落库或者回滚 Redis。

这个点, 没做过秒杀的人想不到。

再说性能。1000 万请求进来, 90% 是无效请求 (机器人、刷子、手抖点错的), 怎么挡?**在前面加好几层漏斗**: 入口限流 (Nginx 层), 验证码 (人机识别), 业务校验 (活动是否开始, 用户是否黑名单), 最后才到 Redis 预扣。这个漏斗设计, 是体现你工程经验的地方。

**踩坑案例**: 我们组去年做过一个秒杀, 当时预估 QPS 50 万, 准备了 Redis 集群 32 个分片, 觉得够了。结果活动开始第一秒, Redis CPU 直接打满。后来查原因, 是有个运营在后台改了商品库存, 触发了 Redis key 的迁移, 加上流量一瞬间打过来, 集群直接雪崩。

怎么救的? 第一, 限流先开起来, 把 30% 的流量挡掉; 第二, Redis 集群加了 16 个分片, 临时扩容; 第三, 入口加了一个本地缓存, 把商品基础信息 (库存、活动时间) 缓存 5 秒, 减少 Redis 访问。最后 1 小时内把活动做完, 没有超卖, 但 P99 延迟一度到 8 秒, 体验很差。

复盘的时候我们得出一个结论: **Redis 集群的容量评估, 不能按平均 QPS 算, 要按峰值的 1.5 倍算, 而且要预留 30% 的 buffer 给运维操作**。这个结论, 我后来在面试里讲过好几次, 面试官都挺认的。

最后说容错。秒杀系统最容易挂的是**链路里的弱依赖**。比如短信通知 (发中奖短信) 挂了, 要不要影响下单? 不要, 短信是弱依赖, 失败了就重试, 不能让用户付了钱收不到货。

这个**强弱依赖的隔离**, 是大厂工程实践里很重要的一环。候选人如果能主动提出来, 加分很多。

**收尾怎么收**? 我建议把方案再过一遍, 然后说一句话: "这个方案在 50 万 QPS 以下, 我有信心; 超过 100 万, 单纯靠加机器不现实, 还要在 CDN、客户端协作 (比如答题前置)、业务规则 (比如限购) 上做更多优化。" 这种"我知道我方案的边界"的态度, 比盲目自信好得多。

## Feed 流: 怎么聊"推拉结合"

Feed 流是另一道 P7 必考题, 跟前面两题不太一样。短链和秒杀是**性能密集型**, 重点是技术深度; Feed 是**产品 + 技术混合型**, 重点是你的**抽象能力和权衡能力**。

典型问题: "设计一个像微博那样的信息流, 用户关注了几百人, 打开 App 看到的是这些人最近发的内容, 按时间倒序。"

这道题最容易答错的地方, 是上来就讲"推"还是"拉"。**正确答案是看场景**。

我第一次答这道题, 是在我 P6 晋升答辩的时候。当时我说 "用推, 用户发微博的时候, 把这条微博写入每个粉丝的收件箱"。评委立刻问: "一个大 V 有 5000 万粉丝, 发一条微博要写 5000 万次, 你这个方案能跑吗?"

我说: "能, 用批量写。" 评委又问: "延迟呢? 写完 5000 万个收件箱要多久? 用户关注的大 V 发了微博, 5 分钟才看到, 这产品能用吗?"

我愣住了。后来我意识到, **Feed 不是一个纯技术问题, 是一个产品定义问题**。

- 关注数 < 1000: 拉 (读时聚合)
- 关注数 > 1000 且 < 100 万: 推 (写时扩散, 收件箱)
- 大 V (>100 万粉丝): 推 + 拉的混合, 大 V 不进收件箱, 读的时候单独走一条"热门/关注的人"的通道, 跟普通用户的收件箱合并

这就是**推拉结合**的本质。**不是非此即彼, 是按用户分层**。

面试里怎么讲这个分层? 关键是**先承认问题, 再讲方案**。

> "纯粹的推, 写扩散太重; 纯粹的拉, 读聚合太慢。所以业内一般是混合, 按用户分。大 V 走拉, 普通用户走推, 读取时合并。"

讲完这个, 面试官可能追问: "合并的时候, 去重和排序怎么做?" 这时候再讲技术细节: 用时间戳做分页, 用 bitmap 或者 set 做去重, 用 Redis SortedSet 按时间排序, 落库用 HBase 或者专门的 feed 服务。

**踩坑案例**: 我之前带过一个团队做 Feed, 当时是纯推架构, 收件箱用 Redis SortedSet, 每个用户一个 key。一开始跑得挺好, 后来日活涨到 2000 万, Redis 内存告急, 一个用户 1000 条 feed, 每条 200 字节, 2000 万用户就是 4TB。

怎么优化的? **分层存储**。最近 100 条 feed 放 Redis (SortedSet), 给在线用户读; 100 条之外的放 HBase, 给回溯或者分页用。同时收件箱做了 TTL, 30 天没访问就清理, 释放出大概 30% 的内存。

这个细节, 我在面试里讲出来, 面试官会追问: "Redis 怎么淘汰? LRU 还是按时间?" 这就聊到具体的内存管理了, 深度就出来了。

**还有一点很重要: Feed 系统怎么限流?** 用户刷太快, 读请求打爆后端, 怎么办? 多级缓存 (本地缓存 + Redis), 加上读限流 (每个用户每秒最多 20 次读请求), 加上降级 (高峰期只返回前 50 条, 不做复杂排序)。

这种"上线之后踩过什么坑"的故事, 永远比"我设计了一个完美的系统"更有说服力。

## IM 系统: 怎么聊"长连接"和"消息可靠"

IM 是 4 道题里最有"网络感"的一道。问题通常是: "设计一个微信那样的即时通讯系统, 支持单聊、群聊、消息可靠投递。"

候选人最容易忽略的是**消息可靠性**。很多人讲完长连接、WebSocket、消息存储, 就觉得讲完了。但消息可靠性才是 IM 的核心难点。

**怎么定义可靠**? 三条: 不丢、不重、有序。听起来简单, 实现起来全是坑。

我们当时做 IM 的时候, 踩过最大的坑是**消息丢失**。用户发了消息, 客户端显示发送成功, 但服务端因为一次 GC 停顿, ack 包没发出去, 用户以为对方没收到, 反复发, 结果对方收到 3 条。

怎么修的? 三步: 第一, 客户端本地先存消息, 等收到服务端 ack 才标记"已送达"; 第二, 服务端收到消息后, 先落盘再 ack, 落盘失败不 ack, 客户端会重试; 第三, 消息带唯一 ID (UUID 或者雪花 ID), 客户端和服务端都做去重, 防止网络重传导致重复。

**讲这个的时候, 面试官可能问: "那群聊呢? 群里 500 人, 一个人发消息, 500 个人都要收到, 怎么保证?"**

群聊的难点是**写扩散 vs 读扩散**。小群 (< 100 人) 写扩散, 每个人一个收件箱; 大群 (几百人) 读扩散, 只写一份, 读的时候按群成员列表去拉。我们当时做了一个折中: 100 人以下的群写扩散, 100 人以上的群读扩散, 群列表单独维护。

**长连接怎么讲**? 不要只说"用 WebSocket"。讲讲**心跳机制**。WebSocket 自己有心跳 (ping/pong), 但很多网络中间件 (NAT、负载均衡) 会静默杀掉长连接, 所以客户端要发应用层心跳, 30 秒一次, 60 秒没收到 pong 就重连。

**还有一个细节: 离线消息怎么推?** 用户不在线的时候, 消息存哪里? 我们当时用了一个离线消息表, 用户上线的时候, 服务端主动推下去。但有个坑: 如果用户离线期间攒了 1 万条消息, 一次推下去会把连接打爆。

怎么解? **分页拉取**。客户端上线后, 先调一个 "拉取离线消息数量" 的接口, 如果超过 100 条, 分批拉, 每批 100 条, 拉完一批再拉下一批。

这个细节, 体现的是你**对边界情况的考虑**。面试里主动提一句, 比你讲 10 分钟 WebSocket 协议都好。

## 最后两个建议

讲完 4 道题, 我想给准备面试的候选人两个 actionable 建议。

**第一个, 准备系统设计, 不要背方案, 要练对话**。找一个朋友, 当面试官, 你当候选人, 让他随机出题, 你现场答。每道题讲 30 分钟, 录下来, 自己回看。你会发现, 你讲的跟你以为你讲的不一样。你以为你讲了 5 个点, 实际上听的人只听清了 2 个。**回放, 是最快的提升方式**。

**第二个, 准备 3 个你真正做过的"系统"故事, 反复打磨**。系统设计最后都会聊到 "你实际项目里怎么做的"。如果你没有秒杀、Feed、IM 的经验, 就找你自己最熟的系统 (哪怕是一个普通的订单系统), 讲清楚它的 QPS、存储、缓存、踩过的坑。**真实的项目, 永远比虚构的方案有说服力**。

我面试的时候, 最怕听到的就是 "这个我没做过, 但如果让我设计我会……" 这句话一出来, 基本就减分了。**你可以说"这个我没做过, 但我做过类似的 X, 我讲讲 X 吧"**, 这才是诚实的、有内容的回答。

系统设计这轮, 说白了就是看你**是不是一个能跟人协作的工程师**。技术深度只是基础, 真正的差距在沟通和判断。
