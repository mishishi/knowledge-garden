# 06. Redis 全景: 5 种数据结构 + 8 种场景

我面过一组 P7 候选人, 大部分简历上写"熟练使用 Redis", 但你往下问一层就露馅: "你用 Redis 做过什么? 遇到过什么问题?", 答得最深的那个候选人讲了他们在优惠券系统里被雪崩搞过一次, 凌晨 3 点被叫起来, 从此以后所有缓存 key 都加了随机过期时间。这个细节, 比"了解 Redis 五大数据类型"高到不知道哪里去了。

这一章我打算按面试视角来切, 不按教科书来。面试官问 Redis 的时候, 真正想知道的不是你能背出 SETNX 的时间复杂度, 是你**到底拿它解决过什么业务问题, 踩过什么坑, 选型的时候有没有动脑子**。下面我按场景分, 每一个都给你"面试官问什么 / 高分怎么答 / 真实案例"三段式。

---

## String 和计数器: 别小看它, 用错的人最多

String 是 Redis 最简单的数据结构, 也是候选人最容易答得浅的。面试官问"你用过 String 吗", 高频的两种答案: 一种是"做缓存, 存 JSON", 一种是"做分布式锁"。这两种答法都不差, 但都没踩到点上。

我比较想听到的是这种: String 底层是 SDS (简单动态字符串), 它做 incr/decr 是**原子操作**, 所以天然适合做计数器。比如我们当时做视频播放量系统, 之前用 MySQL 的 `UPDATE video SET view_count = view_count + 1 WHERE id = ?`, 高峰期 MySQL 直接被打爆, 后来把播放量从 MySQL 迁到 Redis, 用 `INCR` 累加, 再起一个定时任务每 5 秒把 Redis 的数据 batch 写回 MySQL。这个改造做完之后, 视频详情页的 P99 从 800ms 降到 60ms, MySQL 的 QPS 降了 70%。

这个回答为什么好? 因为它有三个层次: 用了 String (技术细节), 因为原子性 (原理), 解决了具体问题 (业务价值)。面试官最怕听到的是"我们用 Redis 做缓存, 性能提升了", 这种话谁都会说。

再说一个 String 的实际坑: 大 Key。我们组 2022 年出过事故, 有一个大 JSON 塞进 Redis 的 String, 单个 key 占了 1.2GB, Redis 持久化的时候 fork 进程直接 STW (Stop The World) 了好几秒, 主从切换也卡。教训: **任何 String 的 value 不要超过 10KB, 超过就要考虑分片或者换结构**。这是血泪教训, 比"我了解 Redis 的 SDS"值钱一百倍。

---

## Hash 存对象: 省内存但别乱用

Hash 结构是候选人最容易答错的一个。面试官问"Hash 和 String 存对象有什么区别", 很多人的回答是"Hash 可以存字段, 方便改单个字段"——这没错, 但不是最关键的点。

真正值钱的角度是**内存**。Redis 的 Hash 在 field 数量比较少的时候, 底层用一种叫 listpack 的紧凑结构 (Redis 5.0 之前是 ziplist), 比 String + JSON 序列化要省很多内存。我们做过一个测试: 存 100 万个用户对象 (20 个字段), String + JSON 序列化大约占 3.2GB 内存, Hash 存同结构只占 1.1GB, 差了将近 3 倍。

但是 Hash 不是银弹。有一个真实场景我印象很深: 我们早期做用户画像系统, 把用户的几十个标签全塞进一个 Hash 里, 单个 Hash 的 field 超过 200 个。结果发现, 这种"大 Hash" 在 RDB 持久化的时候性能很差, 而且 `HGETALL` 会一次性返回所有 field, 如果你只是想拿其中一两个字段, 网络开销很大。最后我们拆成了两个结构: 核心信息 (uid / name / age 这种 5-10 个字段) 用 Hash, 标签这种稀疏数据用多个 String 或者单独存到 HBase。

面试中如果被问到 Hash, 我建议这么组织答案: 先说底层紧凑结构 (listpack), 再说省内存的场景 (field < 100, 字段少但对象多), 最后说反例 (大 Hash 不如拆开)。**有正有反, 有数字, 有场景, 这就是高分答案的套路**。

---

## List 和消息队列: 很多人都用错了

List 结构最容易踩的坑是把它当消息队列用。我们组 2021 年做过一个订单异步处理的链路, 用 Redis List + `BRPOP` 做消费, 当时觉得简单, 上线之后发现一个事: **消费者如果崩了, 没消费的消息会丢**。Redis 的 List 没有 ACK 机制, `BRPOP` 拿到消息那一刻, 消息就从 list 里弹出来了, 如果你处理失败, 消息就没了。

后来我们改成了 Stream (Redis 5.0+ 的新结构), 用 `XREADGROUP` + 消费组 + ACK 机制, 至少能保证 at-least-once。但老实讲, 如果你的业务对可靠性要求高 (比如订单 / 支付), **Redis Stream 也不够用, 还是得上 Kafka 或者 RocketMQ**。Redis 适合的是**允许丢、对延迟敏感、量级大但单条价值低**的场景, 比如日志收集、点赞计数、浏览记录这种。

面试官问"Redis 怎么实现消息队列", 答到 BRPOP 算及格, 答到 Stream 算良好, 答到"什么场景下 Redis 不适合做 MQ, 为什么要选 Kafka" 算优秀。**能说出自己选型不选 Redis 的候选人, 比只会吹 Redis 万能的候选人, 高出 3 个档位**。

还有 List 本身的细节: `LPUSH` + `RPOP` 是 FIFO, 但如果你用 `RPUSH` + `RPOP` 那就是栈 (LIFO)。这个我们组真有同事搞反过, 写代码的时候 `LPUSH` 进去想按时间顺序消费, 用了 `RPOP`, 结果取出来是反的, 排查了半天才发现是 BRPOP 的方向问题。面试里如果你能讲出这种"我犯过的错", 比讲教科书细节更打动人。

## Set 和 ZSet: 排行榜与去重

Set 和 Sorted Set (ZSet) 是 Redis 里业务价值最高的两类结构, 面试必问。

Set 最经典的场景是**去重和共同好友**。比如我们当时做内容推荐, 要算两个用户的兴趣标签交集, 用 `SINTER` 一行代码搞定: `SINTER user:1:tags user:2:tags`。如果用 MySQL 做 `JOIN`, 写起来痛苦, 性能也差。面试答 Set 的时候, 你至少能说出: 共同好友 (`SINTER`)、粉丝去重 (`SADD`)、抽奖 (`SRANDMEMBER`)。

ZSet 更值钱, 因为**几乎所有排行榜都是 ZSet**。游戏战力榜、热搜榜、直播间打赏榜, 底层全是 ZSet。我们做过的直播打赏榜, 大概 50 万在线用户, 用 ZSet 存, `ZADD` + `ZREVRANGEBYSCORE` 取 Top 100, 单机 Redis 跑得很轻松, P99 在 5ms 以内。

但 ZSet 有个坑我要专门讲: 跳表的内存占用。ZSet 底层是跳表 + 字典, 跳表每个节点除了存数据还要存多层指针, 内存开销比 Hash 大。我们做过一次压测: 同样 100 万个 user_id, Hash 存 80MB, ZSet 存要 240MB。所以**不要用 ZSet 存你根本不需要排序的数据**。有个候选人当时答 ZSet, 说"我们用 ZSet 存所有在线用户 ID", 我问为什么要用 ZSet, 他说"因为 ZSet 性能好"——这就是典型的不动脑子, Set 就够了, 用 ZSet 纯属浪费内存。

还有 ZSet 的另一个常考点: `ZADD` 的时间复杂度是 O(log N), 千万级数据加一个元素大概是几百微秒。但如果你要做的是"滑动窗口排行榜" (比如最近 24 小时的热搜), ZSet 存历史数据会越积越多, 内存爆炸。**这种情况要加定时清理, 比如 `ZREMRANGEBYSCORE key 0 (now - 86400)`, 把过期的删掉**。这个细节很多 P6 候选人答不上来, 是真正的高频踩坑点。

## 缓存三大问题: 雪崩、穿透、击穿

这一段是 Redis 面试的**主战场**, 我见过 80% 的候选人栽在这里。问题不是他们不知道这三种异常, 是他们的答案是从博客上背下来的, 没有自己的项目。

先说**缓存雪崩**: 大批 key 同时过期, 瞬间所有请求打到 DB。我面过一个候选人, 答"加随机过期时间"——这没错, 但太浅了, 这是最基础的一层。真正能拿高分的回答要包含: 随机过期 (基础)、**缓存预热 + 永不过期 + 后台异步刷新** (进阶)、多级缓存 (Caffeine + Redis, 兜底) (高级)、熔断降级 (Hystrix / Sentinel) (兜底)。我们当时做的是双 11 大促的首页缓存, 不是加随机时间, 是直接**让热点 key 永不过期**, 后台一个定时任务每 30 秒主动刷新一次。这样雪崩就根本不会发生, 因为没有"过期"这个动作。

再说**缓存穿透**: 查一个 DB 里也不存在的数据, 每次都绕过缓存打 DB。最常见的解法是**布隆过滤器**。但布隆过滤器的坑很多人不知道: 它有误判率, 而且**不支持删除**。我们当时做的是商品系统, 10 亿 SKU, 布隆过滤器误判率设的是 0.1%, 用了 1.2GB 内存, 算下来是合理的。但有个细节: **布隆过滤器的数据要异步重建**, 不能直接拿 MySQL 的数据同步灌进去, 否则启动会卡死。我们是每天凌晨从 MySQL 拉全量数据, 在内存里建好布隆过滤器, 然后原子地替换掉旧的那个。

最后是**缓存击穿**: 单个热点 key 过期, 大量请求同时打到 DB。这个和雪崩的区别是: 雪崩是大量 key 一起过期, 击穿是**一个 key**过期但它是热点, 流量特别大。最经典的解法是**分布式锁 + 互斥重建**。我们当时做的明星热点新闻, 经常被微博大 V 一条微博带火, 单个 article_id 的 QPS 能到 5 万。做法是: 请求进来, 先查 Redis, 没有就抢分布式锁 (Redis 的 SETNX 或者 Redisson), 抢到的去查 DB 然后写回缓存, 抢不到的就 sleep 200ms 再重试。**单 key 重建期间, 90% 的请求会快速失败重试, 只有 10% 真正打到 DB**, DB 不会死。

这三种问题, 面试官不只想听你背概念, 他想听**你的系统里具体怎么防的, 防的是哪一种, 用了什么技术栈, 真实数据是多少**。如果你能讲出"我们的 QPS 是 3 万, 单 key 5 万, 用了 Redisson 做分布式锁, 重试间隔 200ms, 误判率 0.1%", 这就是合格答案。如果你只说"用布隆过滤器防穿透, 加随机时间防雪崩, 用锁防击穿"——这是百度百科的水平。

## 分布式锁: Redis 还是 Zookeeper?

分布式锁是 Redis 面试的**必考题**, 也是候选人**最容易翻车**的地方。我面过的人里, 大概 60% 答 Redis 分布式锁的时候会犯致命错误。

最常见的错误答案是: "用 SETNX + EXPIRE"。这个答案在 2016 年之前是对的, 2016 年之后是错的。原因: SETNX 和 EXPIRE 是两个命令, 中间如果进程崩了, EXPIRE 没执行, 这个锁就永远释放不了。**正确答案是: 用 `SET key value NX EX 300` 原子命令**, 一行搞定加锁和过期。

第二个常见错误: 不考虑锁的可重入性。Java 里 synchronized 是可重入的, 但 Redis 原生 SETNX 不是。如果你的业务里同一个线程要多次获取同一把锁, 简单的 SETNX 就会死锁。**正确做法是用 Redisson**, 它内部用 Hash 存了持有锁的线程和重入次数, 完美解决可重入问题。我们组所有项目统一用 Redisson, 不允许业务方自己写 SETNX。

第三个常见错误, 也是最致命的: **不考虑 Redlock 算法, 不考虑主从切换丢锁**。Redis 的主从复制是异步的, 如果主节点上锁了, 还没同步到从节点, 主挂了, 从节点升主, 这个锁就丢了。这个问题 Redis 官方给的方案是 Redlock (在多个独立的 Redis 实例上加锁, 多数派同意才算加锁成功), 但 Redlock 的争议也很大, Martin Kleppmann 写过一篇《How to do distributed locking》专门怼 Redlock。

面试里如果被问"Redis 分布式锁有什么问题", **最高分的答法是承认它有问题, 然后讲你项目里怎么权衡的**。比如我们当时做的对账系统, 对锁的可靠性要求没那么高 (偶尔丢一次锁, 业务上重试就行), 就用 Redis 单实例 + Redisson。如果做的是资金扣减 (丢一次锁就可能造成资损), 就用 Zookeeper 或者直接用数据库的乐观锁。**能讲出"我为什么不选 Redis 分布式锁"的候选人, 比只会吹 Redis 的人, 高出两个 P 等级**。

顺便提一句 Redisson 的看门狗机制: 它默认锁 30 秒过期, 但会每 10 秒自动续期, 业务没执行完锁不会丢。这个机制在 Redisson 的源码 `RedissonLock` 类里, 面试的时候如果你能讲清楚它怎么实现的 (其实是启动了一个 schedule task, 定时执行 Lua 脚本续期), 会非常加分。

## 限流、BitMap、HyperLogLog: 那些刁钻的数据结构

面试官如果想为难你, 会问一些不太常见的数据结构。这些不一定要用得多深, 但**至少要知道它能解决什么问题**, 否则就显得你的 Redis 知识面很窄。

**BitMap**: 用一个 bit 位存布尔值, 适合做用户签到、日活统计。1 亿用户每天的签到状态, BitMap 只占 12MB 内存 (1 亿 bit ≈ 12MB), Hash 存要 1GB 多。我们当时做连续登录奖励, 用 `SETBIT user:sign:20240101 12345 1` 存用户某天是否签到, 用 `BITCOUNT` 算当天 DAU, 用 `BITOP` 算连续 N 天都签到的用户。BitMap 的本质是 String, Redis 把它当成字节数组操作, 性能极高。

**HyperLogLog**: 做 UV (独立访客) 统计的神器, 内存占用固定 12KB, 标准误差 0.81%。我们当时做内容推荐, 要算每篇文章的 UV, 如果用 Set 存, 1 亿 UV 大概要 8GB 内存, 用 HyperLogLog 只要几十 MB。代价是它只能统计基数, 不能取具体数据, 也不能做交集。**它的底层算法是概率算法, 牺牲了精确性换极致内存效率**, 面试的时候能讲清楚这点就赢了。

**GEO**: Redis 3.2 引入, 存经纬度, 做附近的人、附近的店。我们当时做本地生活服务, 用 `GEOADD` 存商家位置, `GEORADIUS` 查用户附近 5 公里的商家, 性能很好。但 GEO 底层其实是 ZSet, 存的是 geohash 编码, 所以它会受 ZSet 的内存问题影响, 不适合海量数据 (亿级以上)。

这些结构, 面试答的逻辑应该是: **说出能解决什么问题 + 用了什么原理 + 真实案例**。不要罗列 API, 面试官不爱听 API。

## 高频踩坑与避坑建议

最后讲几个**真实项目里翻过车**的细节, 这些比任何理论都值钱。

**第一个, 缓存和数据库的一致性**。我们当时做的是电商库存系统, 修改库存的时候先改 DB 再删缓存, 但还是出现了不一致。最后排查下来是因为**删缓存失败**, 网络抖动了一下, 删缓存的请求没到 Redis, 下次读请求又把旧值写回缓存了。解法是**延迟双删**: 改完 DB 之后, 删一次缓存, 然后异步任务 500ms 后再删一次。这个方案不完美 (极端情况下还是会有不一致), 但**能解决 99% 的场景**。如果要 100% 一致, 就要用 Canal 监听 MySQL binlog, 异步同步到 Redis, 但成本很高, 一般业务不值得。

**第二个, 大 Key 和热 Key**。大 Key 我前面提过 (单个 key 超过 10KB 就算大, 超过 1MB 要警惕), 用 `redis-cli --bigkeys` 能扫出来。热 Key 是某个 key 的 QPS 特别高, 单个 Redis 实例扛不住, 解法是**多副本 + 本地缓存**。我们当时做了一个爆款商品的详情页, 突然被某个主播带火, QPS 从 1 万涨到 20 万, Redis 直接打满。临时方案是在所有服务器本地加了一层 Caffeine 缓存, 兜住了 80% 的请求。

**第三个, Pipeline 和 Lua 脚本**。很多候选人不知道 Pipeline, 一次循环里发 1000 个 `SET` 命令, 等于 1000 次 RTT (Round Trip Time), 性能很差。**正确做法是用 Pipeline 一次性发过去**, 网络开销降到 1 次 RTT。Lua 脚本更高级, 它能保证多个命令的原子性, 比如分布式锁的 "get + set + expire" 三个操作, 用 Lua 包成一个原子操作, 比单纯用 SETNX 强很多。**面试的时候如果你能讲清楚 Pipeline 和 Lua 的区别 (Pipeline 减少 RTT 但不保证原子性, Lua 既减少 RTT 又保证原子性), 这就是一个漂亮的加分项**。

**第四个, Redis 的内存淘汰策略**。默认是 `noeviction` (写满了直接报错), 生产环境一般配 `allkeys-lru` 或者 `volatile-lru`。面试官会问"你怎么配的, 为什么这么配", 好的回答是:**先讲业务特点 (有没有冷热数据分离), 再讲数据特点 (全量数据重要还是热点数据重要), 最后讲具体策略**。比如我们做的是商品缓存, 冷数据基本不会再被访问, 所以配 `allkeys-lru` 把最不常用的淘汰掉。

## 收尾: 给你两个建议

第一, **Redis 不是银弹, 面试里能讲出"为什么不用 Redis"的候选人最值钱**。我面了这么多年, 真正让我眼前一亮的回答, 不是"Redis 多好多好", 是"这个场景我们没用 Redis, 用了 XX, 因为 XX"。这背后是选型能力, 是工程素养, 是真正踩过坑的人才会有的判断力。

第二, **别背 API, 讲故事**。面试官已经听过一百遍"Redis 有五大数据类型"了, 他不想再听第一百零一遍。他想听的是: 你在什么业务里用了什么结构, 解决了什么问题, 当时数据量是多少, 你踩了什么坑, 最后怎么解决的。**一个具体的故事, 比十句教科书结论都管用**。我们组的候选人里, 最后拿到 P7 offer 的人, 几乎都有一两个 Redis 的"事故故事"——被雪崩搞过、被大 key 搞过、被锁丢过。这些故事不是耻辱, 是经验, 是比任何证书都硬的背书。

下一章我会讲**分布式系统的选型**, 怎么根据业务选中间件, 怎么在 CAP 三角里做权衡, 怎么避免"为了分布式而分布式"。那一章会比 Redis 更有意思, 因为它考察的是你的架构视野, 不只是某一个中间件。
