# 05. MySQL 深入: 跟面试官聊底层

去年有个候选人我印象特别深。简历上写着"精通 MySQL 优化",聊到索引的时候张口就是"B+Tree 比 B-Tree 矮胖,叶子节点之间有指针"。我问他,你线上真的因为加索引把查询从 30 秒降到 80 毫秒过吗?他说没真遇到过,基本是看书。

我说那你这不叫精通,这叫背书。

后来这哥们没过。不是因为他不会 B+Tree,而是所有答案都停在"是什么",没一句"我怎么用的"。这一章我想聊的就是这个:MySQL 的底层原理怎么从"八股文"变成你嘴里的真东西。

## 面试官问 B+Tree, 到底想听什么

我面过大概 200 多个后端岗的候选人,MySQL 这一块问得最多的问题前三:索引底层、事务隔离级别、锁。九成答 B+Tree 的答案都是这个路子:对比 B-Tree,说叶子节点用双向链表连起来,叶子节点存数据非叶子存主键和指针。

对吗?对。但这种答案在 P5 还行,在 P6/P7 面试里基本属于"没答"。

面试官真正想听的,是你能不能把数据结构和你见过的真实问题对上。比如我自己的习惯是会追问一句:你建索引的时候,为什么尽量用自增主键?能答上"减少页分裂"是合格,能继续说"我们当时有个业务用 UUID,innodb 表删了 1.2 亿行,空洞一大堆,OPTIMIZE TABLE 都救不回来"——这才叫有体感。

再举一个例子。有次我面个 7 年经验的老哥,我问:你说覆盖索引能避免回表,那如果一个表有 20 个字段我都要 select 出来,覆盖索引还有意义吗?他说"那就建一个包含 20 列的联合索引呗"。我心里基本就给他挂了。

正确答案的思路是:覆盖索引不是无脑建的,你得算账。一个二级索引的叶子节点存的是(主键 + 索引列),索引越宽,页能放的条目越少,I/O 反而可能变多。我们之前做过一个 case,广告主列表页要查 12 个字段,一开始硬上联合索引(12 列),结果单表 3 亿数据,查询没快反而慢了一倍。拆成 2-3 个小联合索引 + 主键回表才解决。

你看,这就是差距。原理谁都背得出来,具体场景下"该不该用、用了会怎样"才是真正筛人的地方。

## MVCC 这块, 八股文和真东西差在哪

MVCC 我估计是 MySQL 面试题里被背得最烂的一个。"read view + undo log + 版本链",这套说辞我听得耳朵起茧。但说实话,这套说辞本身没错,问题是你光说概念,不说你踩过的坑。

我第一次被 MVCC 真正教育,是 2019 年做电商订单系统的时候。业务很常规:订单表(status, create_time),有个定时任务每分钟扫一次"待支付超过 30 分钟"的订单去关单。代码大概这样:

```sql
SELECT id FROM orders WHERE status = 'PENDING' AND create_time < ?;
```

跑了一年没事,某天突然 DBA 找过来,说这个查询把主库干到 30 秒了。我一看 EXPLAIN,走的是索引,type 是 range,扫描行数也没炸,为啥慢?

后来查 binlog 和 slow log 才发现,真正的问题不在查询本身,而是 READ COMMITTED 和 REPEATABLE READ 在 MVCC 下的行为差异。MySQL 的默认隔离级别是 RR,在 RR 下,同一个事务里多次快照读,看到的是同一个 read view。我们那个定时任务跑在一个长事务里(批量处理,事务开了 5 分钟),所以每条 SELECT 都看到的是 5 分钟前的视图,那些 5 分钟内新产生的待支付订单它一个都看不到。

更要命的是,因为 read view 是事务开始时定的,即使它命中了索引,InnoDB 还是得在 undo log 里顺着版本链去找符合条件的历史版本。所以你看 EXPLAIN 显示"扫描行数很少",实际上引擎层做了大量版本链回溯。

后来怎么解决?两种思路:一是把隔离级别降到 READ COMMITTED,二是别把所有处理放在一个长事务里,分批提交。我们选了第二种,把每批从 5000 条降到 500 条,事务时间从 5 分钟压到 30 秒以内,问题解决。

这个 case 教我一件事:隔离级别和 MVCC 不是一个抽象概念,它直接决定你线上 SQL 的行为模式。我后来面试就经常问候选人:你们系统用的什么隔离级别?为什么?遇到过 RC 和 RR 行为不一致导致的问题吗?答得出来的,基本都是真踩过坑的。

## Next-Key Lock 这块最容易露馅

Next-Key Lock 是 InnoDB 在 RR 隔离级别下默认的锁算法,八股文里说"等于记录锁 + 间隙锁,防止幻读"。对,但你要是只答到这一层,基本就是 P5 的天花板了。

我之前带过一个 P5 的小伙子,做了个电商秒杀。商品库存表(goods_id, stock),扣库存的 SQL 大概这样:

```sql
UPDATE stock SET stock = stock - 1 WHERE goods_id = 1001 AND stock > 0;
```

上线前没压测,一上秒杀 QPS 刚到 200,数据库 CPU 直接打满。查了半天才发现,这个 UPDATE 在 RR 隔离级别下,会对 goods_id = 1001 的整条索引记录加 Next-Key Lock,而且因为 WHERE 条件里 stock > 0 不是索引列,InnoDB 不得不把这条记录周围的间隙也都锁住,导致同一商品的所有库存操作全部串行化。

更恶心的是,因为库存是热点行,所有请求都挤在同一个 gap lock 上排队,数据库的 lock waits 直接堆积。

这种问题的答案其实有套路:要么把 stock 加上索引(但热点商品一个值,索引意义不大),要么用乐观锁代替悲观锁(`UPDATE ... WHERE stock = ?`,失败重试),要么干脆把库存从 MySQL 挪到 Redis。我们当时因为业务已经上线没时间重构,临时搞了个折中方案:把库存拆成 10 个分片(`stock_0` 到 `stock_9`),扣的时候按用户 ID 哈希到不同分片,把锁竞争分散开,顶过了双 11。

你看,Next-Key Lock 这玩意儿,书里就一句话"防止幻读",但落到业务里,它意味着你得想清楚:你的 WHERE 条件命中索引了吗?命中的是唯一索引还是非唯一索引?是等值查询还是范围查询?这三个问题的答案决定了你线上会不会有锁等待。

面试的时候我经常让候选人现场分析一段 SQL 的加锁行为,答得对不对先不说,你能不能把"InnoDB 会对哪个范围加什么锁"讲清楚,基本能看出你是背的还是真懂。

## 一次慢 SQL 优化的完整复盘

讲了这么多原理,最后给个完整的实战 case,大家感受下真实的优化是什么样的。

场景是这样的:广告系统的报表查询,有个 SQL 是统计每个广告主每天的花费、曝光、点击数。原 SQL 大概长这样(字段脱敏了):

```sql
SELECT 
  advertiser_id, DATE(create_time), 
  SUM(cost), SUM(impression), SUM(click)
FROM ad_events
WHERE create_time BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY advertiser_id, DATE(create_time);
```

数据量大概 6 亿行。首次跑出来 32 秒,DBA 直接找过来了。

我第一反应是建索引,在 (create_time, advertiser_id) 上加了个联合索引。没用,EXPLAIN 看 Extra 里还是 Using filesort,GROUP BY 没用到索引的顺序。

第二反应是改 SQL,把 GROUP BY 拆成子查询 + JOIN,或者干脆用覆盖索引。改了三种写法,最快的也要 18 秒。

后来查了 InnoDB 状态,发现 buffer pool 命中率只有 87%(正常应该 99%+),也就是说大量查询要落到磁盘上读。我们表当时用的 MyISAM(历史遗留),IOPS 早就打满了。

这一步发现之后,问题性质就变了:不是 SQL 写得多烂,而是存储引擎本身扛不住。这种情况改 SQL 收益有限,得从架构层解决。

我们最后做了三件事:

第一,把表从 MyISAM 迁到 InnoDB。MyISAM 不支持行级锁,查询和写入会互相阻塞,而且它的索引缓存和表数据缓存是分开的,缓存命中率天然吃亏。迁完之后单查询从 18 秒降到 11 秒。

第二,按 create_time 做分区表(RANGE 分区,每个月一个分区)。这样 3 个月的数据扫描从 6 亿行降成实际扫描的分区内的数据(大概 1.8 亿行),但 partition pruning 让 InnoDB 只需要访问 3 个分区,IO 大幅下降。从 11 秒到 6 秒。

第三,也是最关键的一步,把这种聚合查询从在线库挪到了离线数仓。我们搭了个 ClickHouse,把 ad_events 通过 Binlog 实时同步过去,同样的查询在 ClickHouse 上 800 毫秒出结果。

最终从 32 秒到 ClickHouse 的 800 毫秒,中间经历了建索引、改 SQL、迁引擎、做分区、上数仓,前后差不多三周。这中间最让我感慨的是:很多时候 SQL 优化的天花板不在 SQL 本身,而在你敢不敢跳出 MySQL 这一个工具去想问题。

面试里我经常拿这个 case 问候选人:一个查询慢,你优化的顺序是什么?上来就建索引的,基本是初级思路;能想到"是不是架构该改了"的,才有点高级感。

## 给候选人两个具体建议

第一,把"原理 + 自己踩过的坑"打包成 3-5 个故事,反复练。

我面过的人里,答得好的都有一个共同点:他们讲原理的时候一定会带一个真实 case。比如讲 MVCC,会先说"我们当时有个定时任务用了 RR 隔离级别,然后...";讲索引覆盖,会先说"我们之前有个列表页查 12 个字段,索引建错了一次..."。

八股文是死的,故事是活的。你只要能讲清楚"为什么这样做 + 当时出了什么问题 + 最后怎么解决的",原理背不背得出来其实没那么重要。

第二,学会反问。

面试是双向的,不是老师考你。真正的高段位候选人,被问倒的时候会说"这块我之前没深入过,但我猜可能是 XXX,对吗?我下去再查一下"。这种回答比硬背答案的"标准八股"得分高得多。

有一次我面个 8 年经验的老哥,问他 Buffer Pool 的 LRU 算法,他直接说"我只知道大概的最近最少使用,但 MySQL 具体怎么实现的我说不上来,我猜是为了防止全表扫描污染缓存所以做了优化?"。我说你下去看看吧,他说好。这个回答比那种磕磕绊绊硬背"midpoint insertion"的答案强太多——因为它说明你知道自己知道什么、不知道什么,这比假装全知更有价值。

写到最后,我反而想对面试官们说几句:别老问 B+Tree 的叶子节点之间是单向链表还是双向链表(答案是双向),这种题筛不出人。问点有质量的——问"你这个项目里最慢的一条 SQL 是怎么优化的,具体走了哪几步"——比问什么都强。
