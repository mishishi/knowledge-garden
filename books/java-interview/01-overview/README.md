# 面试全貌: P6/P7 到底面什么

去年我们组招 P6, 一共收了 217 份简历, 我自己面了 38 个, 最后发 offer 的 4 个里, 有 1 个是技术面全 A 但 HR 面挂掉的, 还有 1 个是我当时差点挂掉、后来加了二轮才捞回来的。

后者是个做了 5 年半的后端, 简历写得很漂亮——分布式事务、高并发、链路追踪全有, 我问他 "你线上 Redis 用什么架构", 他讲了 15 分钟 Sentinel, 我追问 "主从切换你遇到过切错的 case 吗", 沉默了大概 8 秒, 跟我说 "我们没遇到过"。

老实讲, 这 8 秒的沉默, 在我心里基本就 pass 了, 后来是 leader 看了他笔试分很高才加的二轮。我当时心里的判断标准很简单: 一个声称做过高可用的人, 应该有自己的事故回忆, 而不是从博客上搬下来的标准答案。

这个章节我不会讲任何具体技术题怎么答, 那是后面 11 章的事。我要讲的是这场游戏本身——面试官每一轮到底在筛什么, 评分卡长什么样, 候选人最容易在哪个环节翻车。我做了 12 年后端, 前 5 年是被面, 后 7 年是面人, 这篇是我见过的真实样本拼出来的一张地图。

## 评分卡不是白纸, 它有形状

很多候选人以为面试是 "聊技术", 其实它是一张有结构的评分卡, 只是大多数面试官不会把这张卡给你看。我自己用的卡大概是这个样子, 在不同公司叫法不一样, 但骨架都差不多。

技术维度上, 第一层叫 "基础扎实度", 占 25% 左右。这一层筛的是你会不会, 不是你懂不懂原理。说白了, 就是 HashMap 在 1.8 之前为什么头插, 你不需要现场推导, 但你得知道它会导致死循环; 线程池的 ctl 字段高三位存状态, 你不一定要记住 -536870912 这个数, 但你得知道它是把 workerCount 和 runState 揉在一起的 atomic int。

第二层叫 "深度与广度", 占 30%。这一层筛的是你到底做过没有。候选人讲分布式锁, 讲了 Redis SETNX + Lua + Redisson 看门狗, 听起来挺全, 我会追问 "你们业务里锁的粒度多大, 单 key 还是分段", 这一追问基本就能看出是真的做过还是背的——背的人一般会说 "我们用 Redisson, 它解决了这些问题", 做过的人会说 "我们一开始用单 key, 大促的时候 Redis QPS 顶到了 12 万, 后来改成分段锁, 8 段, 顺便加了本地缓存兜底"。

第三层是 "工程能力", 也占 30%。这个最虚, 也最容易拉开差距。它包含线上排查能力、抽象能力、对系统复杂度的判断力。我面过一个人, 讲 JVM 调优讲了 40 分钟, 各种参数如数家珍, 但我问他 "你们线上最近一次 OOM 是怎么排查的", 他说 "我们有监控告警, 自动重启了"。这个回答不是说错, 但它告诉我一件事: 这个人的调优知识是看书来的, 不是从线上长出来的。

最后一层是 "软素质", 占 15%。包括表达清晰度、协作意识、抗压能力。我特别看重一点: 你会不会主动暴露自己不知道的东西。我见过那种什么都答 "基本了解" 的候选人, 全场没出过汗, 最后 P7 是拿不到的——因为你 P6 的位置就是要在不确定里做判断, 你要是说 "这个我得查一下" 都说不出口, 我怎么相信你能在生产事故里扛住压力。

四个维度加起来不是 100 分制, 是 "强项 / 合格 / 弱项 / 灾难" 四档。我自己评下来一票否决的情况很少, 大多数都是 "技术 OK 但工程深度不够" 或者 "工程深度 OK 但表达拉胯"。

说一个真事: 我们去年面过一个 8 年经验的人, 字节出来的, 系统设计那轮画了 30 分钟短链系统, 架构图漂亮得能印教材。我问他 "如果 ID 空间耗尽了你会怎么办", 他愣了一下, 说 "没遇到过"。这一愣, 在我这里是直接从强项降到合格, 因为一个做过短链系统的人, 一定会想过 ID 上限——62^7 = 3.5 万亿, 你服务全人类都用不完, 但这个问题不在于数字本身, 在于你是不是真的想过系统终态。

## 每轮面试在筛什么, 不是每一轮都在筛技术

我自己面人的标准流程, 几乎和阿里、美团、字节 P6/P7 岗一样, 一般是 4-5 轮。每一轮的筛人目标完全不同, 候选人把它们当成同一场考试是大错特错。

**一轮, 简历面 / 技术初面, 30-45 分钟**, 筛的是 "真假"。这一轮我基本不深入问技术, 我只验证简历上写的项目是不是这个人自己做的。最常用的招是问细节里的细节, 比如 "你说你们做了二级缓存, 本地缓存用什么数据结构, 过期策略是什么, 跟 Redis 怎么同步"。能答上来两种可能: 要么是真的做过, 要么是面经背得很熟。我再追一层: "那次缓存不一致出了多久, 怎么发现的, 老板怎么说的"——面经是背不到这种血肉的。

这一轮挂掉的人最多, 我估计 50% 的简历都在这一轮被刷, 但不是因为技术不行, 是因为项目不是自己做的。y'know, 我理解这个事, 有些候选人确实在组里就是个螺丝钉, 真的就只写 CRUD, 简历上为了好看会包装一下。但你要知道, 面试官会闻出来, 因为真做过和包装过的人, 在细节追问下的反应是不一样的——前者会进入具体场景回忆, 后者会进入 "整理语言" 模式, 眼神会飘。

**二轮, 技术深挖面, 60 分钟**, 筛的是 "深浅"。这一轮才是硬仗, 问的全是底层原理 + 线上 case。我会从三个方向问: 第一是语言和中间件底层 (HashMap、ConcurrentHashMap、Netty 内存池、Kafka ISR), 第二是系统设计 (限流、降级、分布式锁、最终一致性), 第三是排查和调优 (JVM 调优、Full GC 定位、慢 SQL 优化)。

P6 和 P7 在这一轮的差别特别明显: P6 候选人能答出原理, 但追问到 "你们怎么用的" 经常卡住; P7 候选人大部分问题直接跳过原理讲 case, 讲完 case 才反推原理。这个差别不是技术差距, 是角色差距——P6 是解题者, P7 是出题者, 出题者必须站在系统的视角看 trade-off。

**三轮, 架构 / 系统设计面, 60 分钟**, 筛的是 "视野"。这一轮我会抛一个开放性题目, 比如 "设计一个支持 10 万 QPS 的红包系统" 或者 "设计一个公司内部的 IM 系统"。这种题没有标准答案, 我看的不是答得多漂亮, 是你思考问题的路径——你怎么拆解需求, 怎么识别瓶颈, 怎么权衡 trade-off, 怎么考虑演进。

我特别烦一种答法: 上来就堆技术名词, "用 Kafka 解耦, Redis 缓存, HBase 存历史, Sentinel 限流"。这种答法在我这里直接扣分, 因为它反映的是这个人脑子里只有技术名词没有业务抽象。你应该先告诉我业务是什么、用户是谁、流量长什么样、哪些是关键路径、哪些是次要路径。right? 架构是业务的形状, 不是技术的堆叠。

**四面, 老板面 / 交叉面, 30-45 分钟**, 筛的是 "匹配度"。这一轮技术含量下降, 但软素质权重上升。leader 一般会聊成长经历、职业规划、过去项目的角色、对团队的期待、为什么跳槽。P7 这一轮一般还加一道压力面: 故意质疑你的设计, 看你怎么回应。

说一个我自己踩过的坑: 我去年面一个 P7, 系统设计答得很好, leader 面的时候 leader 问 "你 8 年都在一家公司, 怎么证明你能在新环境活下来", 这个候选人直接怼回去: "8 年没换过说明我稳定, 这不是减分项吧"。这句话在 leader 面里基本是 fatal——因为他在用道理反驳, 而不是用共情回应。leader 问这话不是真要你证明什么, 是在看你面对挑战性问题的态度。你哪怕说 "我承认这个是我的盲区, 但过去几年我也主导过跨团队项目, 我可以讲一个具体例子", 都比硬怼强一百倍。

**HR 面, 15-30 分钟**, 筛的是 "价值观和稳定性"。我见过的 HR 面挂掉的 case 里, 大部分不是技术问题, 是表达或动机问题。一个常见死亡回答是 "我想找一个大平台", 太虚了; 另一个是 "我离职是因为 leader 太差", 太诚实了。你要把离职原因包装成 "业务方向调整", 把求职动机包装成 "想看看更复杂的技术场景", 把薪资期望说成 "我相信公司的标准, 也希望有一个合理的范围"。

## 候选人最容易翻车的不是不会, 是太想表现

我做了 7 年面试官, 看了几百场面试, 有一个观察很扎心: 大多数 P6/P7 候选人的天花板不在技术, 在表达。

具体怎么说呢? 有三种最常见的翻车姿势, 我一个一个讲。

第一种叫 "知识点撒胡椒面"。候选人 A, 7 年经验, 我问 ConcurrentHashMap, 他讲了 15 分钟, 从 1.7 讲到 1.8, 从 segment 讲到 CAS+synchronized, 各种细节都到位。然后我问他线上怎么用的, 他说 "我们用 ConcurrentHashMap 做本地缓存"。追问 "你们这个本地缓存怎么防雪崩", 他懵了。他知道 ConcurrentHashMap 怎么实现, 不知道它作为缓存组件该怎么用——这就是典型的 "会造零件不会造机器"。

第二种叫 "答非所问的礼貌性表达"。候选人 B, 6 年, 我问 "你们订单系统怎么保证最终一致性", 他讲了 20 分钟 MQ 事务消息, 各种细节都对。但我后来复盘录音, 发现他只回答了 "怎么发消息" 这一段, 没回答 "订单和库存不一致了怎么办" 这一段。为什么会这样? 因为人天然回避自己不会的部分, 会绕到自己熟悉的部分去。

第三种最致命, 叫 "表演式应答"。候选人 C, 5 年半, 简历上写着 "主导过日均 50 亿请求的网关设计"。我问他 "你这个网关遇到过最大的问题是什么", 他讲了 8 分钟, 最后总结 "我们做了 X、Y、Z 优化, 性能提升了 50%"。这个回答听起来挺好, 但我追问 "X 具体是什么, 你当时是怎么定位到这个点的, 监控数据长什么样", 他支支吾吾。我后来给他打了弱项, 不是因为他真的没做过, 而是因为他不会把一个事讲清楚——50% 这个数字从哪儿来的他都没说清楚。

这三种翻车有个共同根源: 候选人把面试当成了 "回答问题", 而不是 "展示思考"。面试不是答题, 是一次工作场景的模拟——你现在是我的同事, 我们要一起做一个项目, 我问你过去的经验, 是想判断你能不能跟我协作。你应该像跟同事讲事情一样讲: 有背景、有过程、有数据、有反思, 而不是像答辩一样念知识点。

## 真实差距不在 "会不会", 在 "怎么用"

我经常被候选人反问一个问题: "P6 和 P7 的能力差距到底在哪?" 这个问题其实很难一句话答清楚, 因为它跟年限、跟公司、跟运气都有关系。但有一件事是确定的: 真正的差距不在你会不会某个技术, 在你把它用在什么场景、解决什么问题、产生了什么结果。

讲两个我自己组的真实案例, 都是近两年的事。

案例一, 2022 年双 11, 我们做秒杀系统优化。当时流量峰值 QPS 大概 8 万, 系统在压测阶段发现一个问题: 下游 Redis 单 key 写入了 12 万 QPS, 触发了大 key 风险。我的同事 P6 小张负责优化这个点。他一开始的思路是分 key, 把 key 按用户 ID hash 到 16 个 Redis key 上, 写入 QPS 从 12 万降到 7500, 问题解决了。我问他 "为什么是 16 不是 32", 他说 "经验值, 分片数一般 16 就够"。这个回答在 P6 是合格的, 在 P7 是不及格的——因为 16 是拍脑袋, 不是算出来的。P7 应该算: 单 Redis 实例写 QPS 上限 5 万, 12 万需要至少 3 个分片, 考虑到未来 3 倍增长需要 9 个, 加上主从切换的安全余量取 16。

这就是 P6 和 P7 的差距, 不是技术深度, 是 "用工程的方式思考而不是用经验的方式思考"。

案例二, 去年我们做链路追踪系统的改造。当时的情况是, 我们的 trace 系统日均处理 200 亿 span, 存储成本非常高, 我们组 P7 老王接了这个需求。他做的第一件事不是想怎么优化存储, 而是回去看业务方到底用 trace 干什么。调研了 3 周, 发现 90% 的查询是看 "某个请求耗时卡在哪一环", 5% 是查错误, 剩下 5% 才是真排查链路。他基于这个数据做了一个决策: 不再全量存储原始 span, 只存储关键节点和耗时分布, 存储成本降了 70%。

这个 P7 的能力体现在哪? 在于他能从业务价值出发反推技术方案, 而不是从技术能力出发选型。right? 这听起来像套话, 但真的能做到的候选人 10 个里不到 2 个。

## 我给候选人的两个 actionable 建议

第一个建议听起来很 boring, 但真的有用: **录音自己的模拟面试**。我面 P6 之前, 会让候选人做一次模拟, 让他们录下来自己回看, 90% 的人会被自己的表达吓到——停顿太多、眼神飘、"呃" 出现的频率极高、"我理解一下你说的" 这种缓冲句太多。你不录下来自己看, 你永远不会知道自己在别人眼里是什么样。

第二个建议更具体一点: **准备 3 个 "自己最失败的项目"**。面试官一定会问你过去失败的事——故障、撕逼、决策错误, 这是筛你是不是能复盘、有没有成长性的关键一题。90% 的候选人准备的是 "自己最成功的项目", 翻车就翻在这里——你讲得太完美, 反而显得不真实。我推荐你准备 3 个真实的失败案例, 每个能讲 5 分钟: 背景是什么、你做错了什么、后果是什么、你怎么复盘的、后来怎么避免的。这 3 个案例准备下来, 你对面试的恐惧会下降一半。

最后说一个数据: 我们组 2023 年发了 23 个 P6/P7 offer, 最终入职 19 个, 一年后还在的 14 个。这个留存率其实不算高, 但比起前几年好多了。前几年留存率只有 50% 左右, 后来我们在面试环节加了一个隐性筛选: 候选人对 "加班"、"故障"、"和 leader 意见冲突" 这几个话题的态度。留下来的 14 个人里, 12 个在这几个话题上的回答是坦诚且具体的——他们承认困难, 但也表达了应对。这种人, 在真实工作里扛得住。

面试这件事, 说白了, 是一场不对等的博弈: 你只有 4-5 个小时, 面试官要判断你未来 3-5 年的产出。但好在, 你不需要完美, 你需要真实——真实的能力、真实的案例、真实的反思。把这些展示出来, 比任何技术八股文都有用。

下一章我会进入 Java 基础深挖, 讲那些 90% 候选人翻车的题——HashMap 为什么线程不安全, 不只是 "会有死循环" 这一句; synchronized 锁升级, 不只是 "偏向锁→轻量级锁→重量级锁" 这一串词; ConcurrentHashMap 1.7 和 1.8 的差异, 不只是 "分段锁改成 CAS+synchronized" 这一行结论。这些东西怎么答才能拿到高分, 怎么把它跟你自己的项目结合起来, 我一个一个讲。
