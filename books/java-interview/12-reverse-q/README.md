# 反问环节: 5 个问题问出团队好坏

我做了 12 年 Java, 面过大概 500 多个候选人, 自己也被面过不少次。说实话, 面试这事儿, 技术面那 60 分钟大家都在演 —— 候选人背八股文, 面试官照着题库问, 最后互相给个台阶下。

真正能看出东西的, 反而是最后那 10 分钟反问环节。

我第一次意识到这事儿, 是 2017 年面某厂 P7, 当时聊了快两小时, 我技术面答得自我感觉良好, 最后面试官问我 "你有什么想问的", 我特别老实地说 "没啥想问的, 流程大概多久"。面试官笑了一下, 没接话。后来 offer 没拿到, HR 反馈原话是 "技术面评价不错, 但是对团队和业务关注度不够, 担心稳定性"。

那次之后我开始琢磨这事儿。后来我自己做面试官, 也专门观察候选人反问环节, 发现这个环节能暴露的东西远比技术面多 —— 候选人的真实动机、对技术的理解深度、对团队的判断力、甚至性格, 10 分钟全出来了。

这一章我不打算写成 "反问环节技巧大全", 那玩意儿网上一搜一堆, 都是 "问成长空间/问技术栈/问业务前景" 这种正确但没用的废话。我聊聊自己真实用过的、确实帮我在 offer 选择上避过坑的 5 个问题, 以及我作为面试官, 怎么通过候选人的反问环节判断这个人值不值得发 offer。

## 为什么反问环节比技术面更能看出人

先说个数据, 我自己组里去年招的 3 个 P6, 技术面评价都是 B+ 到 A-, 拉出来对比反问环节的笔记, 差距非常明显。后来发 offer 的那个, 反问环节问了一个我们组当时刚踩的坑 —— 我们做了一个配置中心, 上线第一周被某个大流量配置变更搞挂了 3 分钟。候选人直接问 "你们配置中心的灰度发布是怎么做的? 如果配置错了能秒级回滚吗?" 这个问题问出来, 我就知道这个人是有真东西的, 而且提前做了功课。

而另一个技术面评价更高的候选人, 反问环节就问了薪资结构、加班情况、晋升周期, 聊了 8 分钟福利, 我后来跟 HR 商量把他放到了备选池, 因为感觉他对我们业务没兴趣。

面试官问反问环节, 表面上是礼貌, 实际上在看三件事:

第一, 候选人对这个岗位是认真准备过还是海投的。海投的人反问环节非常敷衍, 上来就问薪资。认真准备的人会问得很具体, 而且能问到你正在做的事上。

第二, 候选人的思维层级。我在面 P7 的时候会重点看这个, P7 是要带项目的, 你对团队的理解深度直接决定你进来能不能落地。技术再牛, 不理解团队上下文, 进来要么跟人冲突要么干不下去。

第三, 候选人的判断力和品味。这个比较玄, 但面试官能感觉到。同样问技术栈, 有人问 "你们用 Spring Cloud 还是 Dubbo", 有人问 "你们微服务的服务治理是 SDK 化还是 Sidecar 化"。后者明显是有自己的判断框架的, 问出来的不是答案, 是他自己的思考。

我后来面人有个习惯, 如果候选人反问环节问的东西让我觉得 "嗯, 这个人有点意思", 我会主动多聊几句, 把我们组真实的挑战和坑透出去。因为这种人进来大概率能干活。

## 我自己被面的时候常问的 5 个问题

说说我作为候选人, 觉得值得问的几个问题。这些问题不是网上背的那种 "成长空间/团队氛围", 是真的能问出东西的。

第一个问题是关于团队节奏的, 但我从来不直接问 "加班多不多", 太 low 了。我会问:

"咱们组最近半年做的最有挑战的一个需求是什么? 中间卡壳过吗, 最后是怎么解决的?"

这个问题能问出来的东西至少有三层。第一层是这个组到底有没有在做有挑战的事, 如果回答是 "我们就是接需求做需求, 没啥特别难的", 那基本可以判断这是个纯执行团队。第二层是这个团队解决问题的能力和方法, 候选人讲的过程中你能听到他是拍脑袋还是走数据, 是甩锅还是复盘。第三层是面试官本人的状态, 如果他讲这个项目眼睛发亮, 说明他在这个项目里有成就感, 团队大概率还行; 如果他讲得很敷衍, 说 "都是 leader 做的, 我不太清楚", 那这个组的层级文化可能有点问题。

我当年从某厂跳出来的时候, 用这个问题问过 4 个团队。有一个组的面试官讲他们当时做的支付对账系统, 第一版上线后账实差了 200 多万, 整个组 6 个人连续 3 周每天搞到凌晨 2 点, 最后是一个 P6 的哥们写了一个分布式对账核对工具才搞定的。面试官讲的时候语气里都带着那段时间的疲惫, 但是讲到最后解决方案的时候明显有成就感。我当时就决定去这个组了。后来进去才知道, 那哥们就是我要带的 P6, 那 3 周我虽然没赶上, 但是那个对账工具我现在还在用, 写得是真漂亮。

第二个问题是关于技术决策的:

"咱们组最近一次技术选型是怎么做的? 比如为什么用 Kafka 不用 RocketMQ, 这种决策当时是怎么讨论的?"

这个问题直接试探团队的工程文化。如果面试官支支吾吾说 "这是架构组定的", 说明你们组没有技术决策权, 是个纯执行团队。如果面试官能讲出来 "当时我们纠结了 X 和 Y, 最后选了 X 是因为 Z", 不管最后选的对不对, 这个组至少有讨论的氛围。

我个人经历过技术决策氛围最差的一个组, 是 2018 年某金融公司, 他们的技术选型完全由 CTO 一个人定, 而且定了之后不允许质疑。当时 CTO 拍板用了一个非常小众的国产消息队列, 全组没人用过, 上线后各种诡异问题, 但是谁都不敢说 "这个东西不行", 因为 CTO 会觉得你在质疑他的权威。最后那个项目黄了, 整个组 12 个人走了 9 个。

相反, 技术氛围好的组, 选型一定有争论。Yelp 上有人吐槽 Airbnb 的某个组 "monolith 拆微服务拆了 3 年还在拆", 但是从面试官嘴里能听到的是 "我们一开始拆错了, 拆得服务粒度太细, 后来合并了一波"。这种组你进去干活会舒服很多, 因为有讨论的空间, 不会被 leader 一言堂。

第三个问题是我觉得最值得问的, 关于人和人之间的协作:

"咱们组一般怎么做 code review? 谁来 review? 严格吗?"

这个问题能问出团队的技术深度和工程师文化。一个组怎么写代码、怎么 review 代码, 直接决定你进去之后的技术成长速度。

我经历过 code review 最好的组是 Google 那边的朋友待过的组 (是的, 朋友讲的, 我自己没去过), 据说他们 review 非常细, 一个 PR 上来能 review 几十个 comments, 而且 review 的人会真的去读你引用的那篇 paper, 确认你的设计是对的。也经历过最烂的组, 某个创业公司, code review 就是走个过场, leader 看一眼 diff, 说 "LGTM" 就 merge, 进去之后整个 codebase 烂得一塌糊涂。

作为 Java 工程师, 这个问题的隐藏考点是: 这个团队对工程质量的要求高不高。如果他们 review 严格, 说明对代码质量有要求, 你写代码会越写越好。如果 review 走个过场, 那你进去之后大概率是堆屎山, 越写越烂, 后面想走都走不了 (因为简历上这段经历没法写)。

我后来招人的时候, 如果候选人反问环节问到 code review, 我会非常加分。因为这说明他关心工程质量, 关心自己写出来的代码会不会被人骂。这种候选人进来之后大概率不会乱来。

第四个问题是关于 on-call 和故障处理的:

"咱们组的 on-call 怎么轮? 最近一次线上故障是怎么处理的?"

这个问题的真实目的是问团队的应急能力和运维文化。如果你进的组 on-call 一年到头都是同一个人 (往往是那个 leader), 说明要么业务太小没人轮, 要么大家都想躲。如果你进的组故障处理方式是 "先回滚再追责", 这个组待着还行; 如果是 "先开会追责再决定要不要回滚", 这个组一定有问题, 进去之后你会天天背锅。

我之前面过一个创业公司, 面试官讲他们最近一次故障, P0 级别, 数据库被打挂了, 他们花了 40 分钟恢复。40 分钟其实不算长, 但是面试官讲的过程中, 我注意到一个细节: 他一直在强调 "当时 leader 决策果断", 但是对具体的技术原因避而不谈, 还在说 "这个季度的 OKR 不会被影响吧"。我当时就觉得这个组的 leader 关心 KPI 多过技术细节, 婉拒了 offer。后来听说那个组半年内又出了两次大故障, leader 跑了。

第五个问题是关于我的直接 leader:

"如果我有幸进来, 我的直属 leader 是什么风格? 他带人的方式是放养型还是手把手型?"

这个问题很多人不敢问, 但是我建议直接问。因为你进去之后天天跟这个人打交道, 他的风格直接决定你这 1-2 年过得爽不爽。

我面过最舒服的一个 leader, 是某大厂的一个 P9, 当时面试的时候他就讲: "我带人比较放养, 大的方向我会把控, 具体怎么实现你来定, 但是每周五下午我会做一次 1-on-1, 你这周做的决策我会逐个 review, 该拍砖的拍砖。" 这种 leader 我进去之后发现确实如此, 我的 P7 候选人当时基本上是自我驱动, 但是关键节点 leader 会介入。后来我从那个组出来的时候, leader 给我写的推荐语我现在还留着, 是我职业生涯里被评价最高的一次。

相反, 我面过最差的一个 leader, 面试的时候讲 "我会给你充分的空间", 进去之后发现他每周要 review 我所有代码, 而且 review 标准不是质量, 是 "必须按他喜欢的方式写"。有一次我重构了一段代码, 用了一个新的设计模式, 他 review 的时候直接说 "这个写法我看不懂, 你给我改回去"。我当时 28 岁, 没敢顶嘴, 后来那段屎山代码我离职之前还在维护, 估计现在还在。

所以直属 leader 这事儿, 一定要在面试的时候问清楚。问得太直接面试官可能不会全告诉你, 但是他回答的方式和语气会暴露很多东西。

## 面试官视角, 哪些反问环节的回答让我直接 pass

反过来讲, 我作为面试官, 哪些反问环节的回答会让我直接扣分。

最致命的一类, 是上来就问薪资、加班、调薪。这种候选人我会直接在心里降低评级。因为这个信号说明他关心的是短期利益, 对业务和团队没兴趣。即使他技术面表现得很好, 我也会担心他稳定性差, 干半年就走。

我印象最深的一个候选人, 技术面评价 S, 所有题都答得非常好, 分布式锁、JVM 调优、高并发场景题, 全中。但是反问环节上来就问: "咱们 14 薪还是 16 薪? 加班有加班费吗? 调薪周期是一年一次还是两次?" 一口气问了 5 个福利问题, 整个反问环节 10 分钟全在聊钱。我当时心里就在想, 这种人进来之后大概率会很快被挖走, 因为总会有别的厂开更高的价。后来我跟 HR 商量把他放进备选池, 优先发给了另一个技术面 B+ 但是反问环节问业务问团队的候选人。

第二类, 是问得太泛的, 比如 "咱们团队未来三年的规划是什么?" "咱们公司在行业的地位怎么样?" 这种问题不是不能问, 但是太虚了, 面试官要么敷衍回答, 要么讲得很官方。问这种问题说明候选人没有做功课, 没有真正了解过这个公司和团队。

真正问得好的反问环节, 候选人应该是带着问题来的。他应该提前看过公司官网、看团队博客 (如果有的话)、看技术博客上团队发的文章, 然后基于这些信息提出具体的问题。

我之前面过一个候选人, 反问环节上来就问: "我在贵司技术博客上看到您们组去年发了一篇关于分布式配置中心的文章, 里面提到你们用长连接推送配置变更, 我想问一下长连接的保活策略是怎么做的? 为什么不用传统的轮询?" 这个问题直接把我问住了, 因为那篇文章是我写的, 我当时设计那个长连接保活策略的时候踩过几个坑。这个候选人明显做了功课, 而且有自己的思考。后来他拿了 offer, 进来之后我们成了同事, 那段代码后来确实优化了一波。

第三类, 是反问环节完全没有问题的。"没有了" 或者 "刚才聊得挺全面的, 我没啥想问的" —— 这种回答我一般会给中等偏下的评价, 因为这说明候选人要么海投没认真准备, 要么不敢问 (性格可能偏内向, 但是 P6/P7 这个级别, 不敢问是减分项)。

我遇到过最绝的一个候选人, 反问环节说: "我看了您的 LinkedIn, 觉得您做得东西很有意思, 咱们下次能再聊一下吗?" 然后反问环节就结束了。这个回答其实挺机灵的, 但是太油了, 面试官会觉得这个人太会来事儿, 干活未必靠谱。后来我没给他发 offer, 因为我们组需要的是能踏实干活的人, 不是太会来事儿的人。

## 收尾: 给候选人的两个 actionable 建议

第一个建议, 反问环节一定要做功课。面试之前花 30 分钟研究一下这个团队: 看他们的技术博客、看他们 GitHub 上的开源项目 (如果有)、看他们的技术演讲视频、查一下团队成员在技术社区的发帖。这些信息能在反问环节问出非常具体的问题, 直接跟其他候选人拉开差距。

我之前帮一个朋友准备面试, 他面某大厂的 P7, 我让他提前看了这个团队过去半年的技术博客, 总结出他们做的三件大事, 然后每个大事准备一个问题。结果他面试的时候反问环节直接问到了 leader 的痛点上, leader 当场就跟他说 "你下周二能入职吗"。

第二个建议, 反问环节问完之后, 一定要做一个总结性发言。我发现很多候选人反问环节问完就结束了, 其实这是一个很好的机会做一个简短总结, 表达自己对这个岗位的兴趣和理解。比如:

"今天聊下来, 我对咱们团队做的事情有了更清晰的了解。支付对账这块确实是我们组现在的重点, 我之前在 XX 公司也做过类似的系统, 上线后也遇到过账实差的问题, 当时我们花了 X 周用 Y 方法解决的。我感觉咱们团队做的事情和我的经验非常匹配, 我也特别期待能加入。"

这种收尾能让面试官感觉到你是一个会思考、有逻辑、表达清晰的人。即使技术面有不会的问题, 这种收尾也能挽回一些印象分。

面试这事儿说白了是个双向选择。你在考察这个团队, 这个团队也在考察你。反问环节是你考察团队最重要的 10 分钟, 也是你展示自己的最后 10 分钟。认真对待这 10 分钟, 比多背 100 道八股文有用得多。

写到这儿, 「Java 高级开发工程师面试指南」这个系列 12 章就全部结束了。从 JVM 调优到分布式事务, 从项目设计到反问环节, 我尽量把我 12 年里踩过的坑、见过的候选人、用过的套路都写出来了。希望对正在准备面试的你有点用。

最后说一句, 面试这事儿, 技术是一方面, 但说白了, 面试官招的是一起干活的人, 不是招一个答题机器。你技术再牛, 不会表达、不会思考、不会判断团队, 一样拿不到好的 offer。修炼自己, 不只是修炼技术。
