# 08. 市场模式（Market）实战

市场模式是 4 种协作范式里**最前沿、最不成熟、但潜力最大**的模式。它把 agent 当成独立的"经济主体"，通过竞标 / 协商 / 结算机制动态协作。

## 核心思想

传统范式（Pipeline / Debate / Hierarchical）都有"中心化"——某个 agent 决定任务怎么分、结果怎么合。市场模式去中心化：

```
[任务发布]
   ↓
[任务市场 / 黑板]
   ↓ 通知所有订阅的 agent
[Agent A] [Agent B] [Agent C] [Agent D] 都看到任务
   ↓ 各自评估 + 出价
[竞标]
   ↓
[任务分配]
   ↓
[执行 + 交付]
   ↓
[结算 + 评价]
```

每个 agent 是独立的经济主体，追求自己利益（收入、声誉、任务完成率）。任务委托人通过市场协议选择最合适的 agent。

## 真实案例：动态翻译市场

**场景**：用户上传一段文本，要求翻译成英文。文本可能是法律合同 / 技术文档 / 营销文案 / 学术论文。

**市场结构**：

```
[翻译任务]
   ↓
[市场] 通知所有"翻译 agent"
   ↓
[Legal-Translator]   出价 $0.50/千字，置信度 0.92，擅长法律
[Tech-Translator]    出价 $0.30/千字，置信度 0.85，擅长技术
[Marketing-Translator] 出价 $0.20/千字，置信度 0.78，擅长营销
[Academic-Translator]  出价 $0.40/千字，置信度 0.95，擅长学术
[General-Translator]   出价 $0.10/千字，置信度 0.65，通用
   ↓
[委托人根据文本类型选择]
   - 法律合同 → Legal-Translator
   - 技术文档 → Tech-Translator
   - 营销文案 → Marketing-Translator
   ↓
[任务分配]
   ↓
[翻译执行 + 交付]
   ↓
[用户评价 + 结算]
```

## Agent 出价的考量

每个 agent 看到任务后要考虑 5 个因素：

**1. 能力匹配度**——这个任务在我的 skills 范围内吗？法律翻译 agent 接到技术文档任务会主动放弃或出高价。

**2. 当前负载**——我现在手上有几个任务？负载高就出高价或拒绝。

**3. 历史定价**——我同类任务之前定价多少？保持价格稳定有助于积累声誉。

**4. 竞争对手**——其他 agent 出价多少？差异化竞争。

**5. 预期收益**——做完这个任务对我的声誉 / 收入 / 能力成长有什么影响？

```python
async def bid(task):
    if not within_skills(task):
        return None  # 主动放弃
    
    load = current_load()  # 当前在手任务数
    if load > MAX_LOAD:
        return Bid(price=None, reason="overloaded")
    
    base_price = historical_pricing(task.type)  # 历史同类任务定价
    load_multiplier = 1 + (load / MAX_LOAD) * 0.5  # 负载越高越贵
    
    confidence = estimate_confidence(task)  # 0-1
    
    final_price = base_price * load_multiplier / confidence
    # 置信度低 → 加价（高风险）
    # 负载高 → 加价
    # 基础价格不变
    
    return Bid(price=final_price, confidence=confidence, eta=estimate_eta(task))
```

## 任务分配策略

委托人收到多个 Bid 后，怎么选？

**策略 1：价低者得**——简单粗暴，但可能选到能力不够的 agent。

**策略 2：性价比**——按 `confidence / price` 排序，选最优。

**策略 3：能力优先**——只看 confidence，price 在预算内就行。

**策略 4：混合**——能力阈值 + 性价比排序。confidence < 0.7 的直接淘汰，剩下的按性价比选。

实战推荐**策略 4**。纯价低者得容易翻车，纯能力优先太贵。

## 结算机制

市场模式必须解决**信任问题**：用户付了钱 agent 不交付怎么办？agent 交付了用户不付钱怎么办？

**三种结算机制**：

**1. 托管（Escrow）**——用户先把钱打到第三方托管，agent 交付后托管放款。最安全但慢。

**2. 信用体系**——agent 有声誉积分（基于历史交付评价），高分 agent 可以"先交付后结算"，低分 agent 必须托管。

**3. 智能合约**——任务完成度自动验证（如翻译结果 BLEU 评分 > 阈值），合约自动放款。最透明但实现复杂。

实战推荐**信用 + 托管混合**。新 agent 强制托管，老 agent 高信用可后付。

## 评价机制

每个任务结束后用户评价 agent，影响 agent 的声誉积分：

```
5 星：交付质量超预期
4 星：交付质量符合预期
3 星：交付能用但不完美
2 星：交付有问题但能用
1 星：交付完全失败
```

声誉积分影响后续任务的优先级和定价：

```
声誉 >= 4.5：高优先级，可后付
声誉 3.5-4.5：正常，需托管
声誉 < 3.5：低优先级，必须托管 + 高定价
声誉 < 2.0：踢出市场
```

## 真实生产数据

2026 年 Q2 我跑了一个小规模翻译市场测试，6 个翻译 agent 互相竞争 1000 个任务。结果：

- **平均价格下降 22%**（市场竞争压价）
- **平均交付质量提升 8%**（低质 agent 被淘汰）
- **任务分配延迟 80ms**（市场协议开销）
- **纠纷率 3%**（主要是"翻译质量不达预期"，靠托管机制兜底）

市场模式**对成本敏感的场景特别有效**（翻译 / 数据标注 / 内容生成 / 简单客服）。

## 适用 / 不适用

**适合**：任务标准化（可以明确定义输入输出）、agent 能力差异大、需要动态调度、成本敏感。

**不适合**：高度定制化任务（agent 间能力差异不大）、强实时（市场协议开销）、需要强可解释性（黑盒决策）。

## 2026 现状

市场模式 2026 年还在早期。最大的项目是 **MCP.so** 和 **AgentMarket.ai**——这两个是 agent 市场的淘宝 / 拼多多。

**最大障碍**：缺乏标准的市场协议。各家自己定义 Bid 格式、定价模型、结算机制，互不兼容。Google 在 2026 年 5 月开始起草 A2A Market Extension，预计 2026 Q4 发布标准协议。

## 我的判断

市场模式是 4 种范式里**最有可能诞生 agent 时代独角兽**的范式。Cursor / LangChain / CrewAI 这类公司已经做出 agent 框架，但**没有 agent 市场**。谁先把 agent 市场做出来，谁就能吃下未来 5 年 agent 经济的最大蛋糕。
