# 02. Few-shot 实战

> 给 LLM 看几个例子，比讲 100 条规则管用。但 few-shot 写错比不写还糟——这章讲怎么选样、4 大反模式、跑出来的真实数据。

## Few-shot 为什么有效

LLM 通过**模式匹配**做任务——给它看 3-5 个「输入 → 输出」example，它会「照葫芦画瓢」做第 6 个。

```python
# Zero-shot：没例子
prompt = "把下面句子改成正式语气：'Hey 那啥我想问个事'"

# Few-shot：给 3 个例子
prompt = """
把口语改成正式商务语气。

例 1：
口语：'Hey 那啥我想问个事'
正式：'您好，我想咨询一件事'

例 2：
口语：'这玩意儿不好使啊'
正式：'此功能无法正常使用'

例 3：
口语：'帮我看下这个 bug 行不'
正式：'请协助排查此 bug'

句子：'哎妈呀终于搞定了'
"""
```

**实测**：few-shot 准确率比 zero-shot 高 20-40%（按任务复杂度）。但——**不是所有任务都适合 few-shot**。

## 什么时候用 few-shot

**适合**：输出格式复杂（JSON / 特殊排版）、风格 / 语气迁移、边界 case 多（要展示「不能 X」）、分类任务（例 5 个类别各 1-2 个例子）。

**不适合**：简单事实问答、创意写作（会限制发散）、推理 / 数学（要 CoT 而不是 example）、长输出（1000+ 字）。

**核心判断**：如果 zero-shot 跑出来的输出**风格不对 / 格式不对 / 边界不对**——few-shot 能拉一把。如果 zero-shot 跑得**事实错**——few-shot 救不了（要 RAG / 知识库 / 调模型）。

## 几个例子最优

**经验值：3-5 个**。1 个模型没足够模式可能照搬，2 个开始有模式但不够稳，3-5 个是甜区，5-10 个边际收益递减 token 浪费，10+ 个模型开始「记不住」效果反降。

实测数据（Anthropic 内部研究，多任务平均）：

| Few-shot 数 | 准确率 | Token |
|---|---|---|
| 0 (zero-shot) | 62% | 100 |
| 1 | 71% | 200 |
| 3 | 78% | 400 |
| 5 | 81% | 700 |
| 10 | 80% | 1500 |
| 20 | 75% | 3000 |

3-5 个甜区，10+ 反而下降（context 太长稀释注意力）。

## 怎么选例子（4 大标准）

**标准 1：覆盖「典型 case」**——任务：用户评论情感分类。5 个典型 case 覆盖「正面 / 负面 / 中性」三类。

```python
examples = [
    ("这个产品真不错", "正面"),
    ("太差了，物流 3 天还没到", "负面"),
    ("一般般吧", "中性"),
    ("客服回复很快", "正面"),
    ("完全不值这个价", "负面"),
]
```

**标准 2：覆盖「边界 case」**——边界：反讽、隐喻、模糊。

```python
examples += [
    ("真是'好'产品啊，5 天就坏了", "负面"),   # 反讽
    ("这颜色我不太敢穿出门", "中性"),         # 模糊偏好
    ("无敌", "正面"),                          # 极简
]
```

边界 case 是**最容易翻车**的——专门挑出来给模型看。

**标准 3：多样性 > 重复**。5 个例子都是「正面」变体让模型学不到「负面」模式，5 个例子覆盖不同主题让模型学「怎么判断」。

```python
# 错：5 个例子都是「正面」变体
examples_bad = [
    ("产品很好", "正面"),
    ("服务很好", "正面"),
    ("质量很好", "正面"),
    ("物流很好", "正面"),
    ("价格很好", "正面"),
]

# 对：5 个例子覆盖不同主题
examples_good = [
    ("产品很好用", "正面"),
    ("物流太慢", "负面"),
    ("客服态度好", "正面"),
    ("价格贵", "中性"),
    ("说明书写得很差", "负面"),
]
```

**多样性 = 让模型学模式，不是抄答案**。

**标准 4：例子的「正确率」要高**——所有例子都必须是「正确答案」。模型会从错的例子学错模式：

```python
# 错：例子是错的
examples_wrong = [
    ("这个产品好", "负面"),   # 错了：明显是正面
    ("服务差", "正面"),       # 错了：明显是负面
]
# 模型会学到「正面 → 负面」的错模式
```

**写完例子后先肉眼检查一遍**——尤其是标签类任务，标错的例子会污染整个 few-shot。

## 4 大反模式

**反模式 1：例子太长**——例子是教学不是数据，模型学「细节」而不是「模式」。

```python
# 错：200 字例子
example_bad = """
用户输入：'我在 2024 年 3 月 15 日通过 Apple Store 在线购买了一台 MacBook Pro 14 寸...'
提取：{"product": "MacBook Pro 14 inch", ...很多字段...}
"""

# 对：精简到 2-3 行
example_good = """
用户：'3 月买的 MacBook M3 Pro 18GB 512GB，电池只能撑 5 小时。'
提取：{"product": "MacBook Pro 14", "config": "M3 Pro / 18GB / 512GB", "issue": "电池续航短"}
"""
```

**反模式 2：例子风格不一致**——3 个例子 3 种风格让模型学到的模式混乱。**例子要像同一个作者写的**——风格一致 = 模型学到的模式一致。

**反模式 3：例子难度跟任务不匹配**——例子太简单模型简化；例子太复杂模型抓不到重点。**例子的复杂度 = 任务的复杂度**。

```python
# 错：例子太简单
example_simple = "句子：'好' → 情感：'正面'"
# 但实际任务要分析 100 字评论，模型对长文本就懵

# 对：例子跟任务难度匹配
example_realistic = "句子：'这产品吧...怎么说呢...用了一周，感觉还行但又不算惊艳，就是能用。' 情感：'中性'"
```

**反模式 4：在 prompt 里混入零散说明**——few-shot + 长说明混在一起让模型注意力分散。**说明占 20%，例子占 80%**——few-shot 的核心是「看例学例」。

```python
# 对：先说明（少）+ 再例子（多）
prompt = """
分析用户评论情感，输出 0-10 分 + 1 句话解释。

[例子 1]
输入：'产品不错'
输出：8。产品质量好，性价比高。

[例子 2]
输入：'物流太慢'
输出：2。物流时效不达标。

[例子 3]
输入：'一般吧'
输出：5。没有明显优缺点。

[任务]
输入：'{review}'
输出：
"""
```

## Few-shot 写在 prompt 的什么位置

**结论：放在 system 或 user 末尾，紧跟任务**。

```python
# 推荐结构
prompt = """
[Role + Context 写最前面，1-3 段]

[Task 1 句明确]

[Format 1 段明确]

[Few-shot 例子 3-5 个]    ← 放这，紧跟 task

[实际任务 input 在最后]
"""
```

**不要**把 few-shot 放最前面（模型会先学「例子」再学「任务」，注意力分散）。

**不要**在 few-shot 中间插入「提示语」（"现在再看下一个..."），模型会自动按格式继续。

## 实战案例：query 改写 few-shot

任务：把用户口语化搜索改写成结构化搜索 query。

```python
prompt = """
你是电商搜索 query 改写工程师。
把用户口语化搜索改写成结构化 query，便于检索商品。

【改写规则】
1. 提取核心品类词（手机 / 衣服 / 鞋子 / ...）
2. 提取关键属性（颜色 / 尺寸 / 品牌 / 价格 / 用途）
3. 保留口语风格但加结构

【例子 1】
输入：'我想买个能拍照好看的手机'
改写：'拍照手机 推荐 高像素 摄影'

【例子 2】
输入：'夏天穿凉快的鞋'
改写：'凉鞋 夏季 透气 舒适'

【例子 3】
输入：'送女朋友的香水不要太贵'
改写：'女士香水 礼品 中等价位 300-800元'

【例子 4】边界 case：模糊需求
输入：'有便宜的吗'
改写：'平价 商品 不限品类'

【任务】
输入：'{user_query}'
改写：
"""
```

4 个例子：3 个典型 + 1 个边界 case。

## Few-shot 在生产里的 3 个实战技巧

**技巧 1：用真实数据挑例子**——从 1000 条「历史正确 case」里挑 5 条最有代表性的，避免凭空编例子（容易编出「教科书式」case 不真实）。**真实数据的例子比手写例子好**——手写容易「过度标准化」，真实数据保留 case 的多样性和噪声（这正是模型要学的）。

```python
historical_data = load_history_correct_cases(n=1000)
representatives = pick_diverse_representatives(historical_data, k=5)
```

**技巧 2：动态 few-shot**——不同 query 配不同 few-shot。比静态 few-shot 准。

```python
def get_few_shot_examples(user_query):
    if "退货" in user_query:
        return refund_examples   # 退货场景专用例子
    elif "优惠" in user_query:
        return discount_examples  # 优惠场景专用例子
    return generic_examples
```

**技巧 3：few-shot + 验证**——跑完 5 个 example 后跑 1 个「隐藏」example，如果输出对就真的跑任务，输出错就重新选例子。**避免「模型以为懂了但实际没懂」**。

```python
def few_shot_with_validation(prompt, hidden_test):
    response = llm.call(prompt)
    if matches_expected(response, hidden_test):
        return llm.call(prompt + actual_task)
    else:
        return retry_with_better_examples()
```

## 5 个常见坑

**坑 1：例子不够多样**——3 个例子都是「简单正面」让模型学「所有评论都是正面」。修：至少 1 个负面 + 1 个中性。

**坑 2：例子有拼写错误 / 标点错**——模型会学「标点也错」。修：例子必须干净。

**坑 3：例子是「完美 case」，任务有「噪声」**——模型学「完美输入 → 完美输出」对噪声输入无能为力。**例子要带点噪声**（更真实）。

**坑 4：token 预算让 few-shot 装不下**——100 个例子装不进 prompt。**挑最有代表性的 5 个**——不是「越多越好」。

**坑 5：动态 few-shot 例子来自不可信源**——用户输入当 example 可能含 prompt injection。**例子必须来自可信源**（人工标 / 专家写 / 验证过的历史正确 case）。

## 怎么评估 few-shot 效果

**指标 1：准确率**（分类 / 提取任务）：

```python
test_set = load_test_cases(n=200)
correct = sum(
    1 for case in test_set
    if llm_judge(llm_response(case, prompt), case.expected) == "pass"
)
accuracy = correct / len(test_set)
```

**指标 2：一致性**（同样输入跑 5 次，结果一致的比例）：

```python
results = [llm_response(case, prompt) for _ in range(5)]
consistency = len(set(results)) / len(results)   # 越低越好（1.0 = 完全一致）
```

**指标 3：人工 spot check**——跑 30-50 个 case，人眼抽 10 个看是否合理。

**指标 4：用户反馈**——上线后收集用户反馈（「这个回答不对」），看反馈率。

**经验**：few-shot 至少要提升 10% 准确率才值得加。**加了但没提升 = 浪费 token**。
