# 06. 写你的第一个 Skill

2025 年 12 月我开始写第一个 Skill。模仿网上的例子拼出来一个 SKILL.md，**触发不对、输出不对、过度触发**。

**花了 2 周才把它调到 80% 准确**。

这一章把我的 7 步流程完整写出来——**让你写第一个 Skill 不超过 1 小时**。

## 第 0 步：判断要不要写 Skill

不是所有"重复工作"都该写 Skill。**3 个判断标准**：

**1. 我过去 30 天重复 3+ 次的事**？如果不到，**先继续用 prompt 临时**。**别为了 Skill 而 Skill**。

**2. 这事有"标准化输出"**？如果每次都不同，**Skill 价值低**。**有标准模板 / 固定流程 / 必查清单 → 适合**。

**3. 这事能写清楚"怎么做"**？如果你自己都说不出步骤，**别写**——**会写出一个烂 Skill 帮倒忙**。

我自己的 2026 年 4 月：写了 8 个 Skill（在上一章）+ 2 个自己用的私人 Skill。**私人 Skill 写得粗糙，但够用**。

## 第 1 步：定义 Skill 的边界（10 分钟）

用一句话回答这 3 个问题：

**Q1：做什么？**
示例："按团队规范审查 TypeScript 代码"

**Q2：何时用？**
示例："用户提交 PR / 问 review / 看到代码 diff 时"

**Q3：何时不用？**
示例："纯文档、配置文件、外部库代码"

把这 3 句写下来——**这就是你的 description 草稿**。

## 第 2 步：列出执行步骤（15 分钟）

把这个 Skill 的执行流程写出来——**5-10 个步骤**。

示例（code-review Skill）：

```text
1. 读取所有变更文件（git diff）
2. 按 4 个维度检查（安全 / 性能 / 质量 / 错误）
3. 给每个问题打分（严重 / 警告 / 建议）
4. 输出去重 + 按严重度排序
5. 用 markdown 格式输出
6. 给每个问题提供修复建议
```

**步骤数量**：
- 3-5 步：太简单，可能不够
- 5-10 步：合理
- 10+ 步：太复杂，**拆成多个 Skill**

## 第 3 步：写 SKILL.md 草稿（20 分钟）

把前 2 步的产物组合成 SKILL.md。

```yaml
---
name: code-review
description: 按团队规范审查 TypeScript 代码变更（PR review）。检查安全（SQL 注入、XSS、权限校验）、性能（N+1、内存泄漏）、代码质量、错误处理。涉及 src/ 下的代码时自动触发。
paths:
  - "src/**"
---

# Code Review

## 步骤
1. 读取所有变更文件
2. 按 4 个维度检查
3. 打分排序
4. 输出 markdown

## 审查清单

**安全：** SQL 注入 / XSS / 权限校验 / 敏感信息

**性能：** N+1 / 内存泄漏 / 重渲染 / 大循环

**质量：** 命名 / 单一职责 / 可测试性 / 类型

**错误：** try-catch / 错误信息 / log / 用户面错误

## 输出格式
```markdown
## Review
- 🔴 严重：[问题] in [文件:行号]
- 🟡 警告：[问题] in [文件:行号]
- 🟢 建议：[优化] in [文件:行号]
```
```

**关键**：**description 必须清晰**——**这是模型判断"调不调"的核心**。**不清晰的 description = 触发不对**。

## 第 4 步：测试 5 个场景（30 分钟）

Skill 写完，**测 5 个场景**：

**5 个测试场景：**
- 场景 1：用户提交 PR，明确说"review" → 应该自动触发
- 场景 2：用户提交 PR 但没说"review" → 看 description 是否匹配
- 场景 3：用户问代码问题（非 review）→ 不触发
- 场景 4：用户改文档（不在 paths 内）→ 不触发
- 场景 5：用户手动 `/code-review` → 触发

**每个场景测 3 次**：

- **第 1 次**：看是否触发
- **第 2 次**：看输出质量
- **第 3 次**：看是否符合预期边界

**测试用例怎么写**：

```text
场景 1：触发测试
- 用户消息："帮我 review 一下这段代码：[粘贴代码]"
- 预期：模型说"using skill code-review" + 按清单审查
- 实际：[观察]

场景 2：误触发测试
- 用户消息："什么是 TypeScript?"
- 预期：模型不调 code-review
- 实际：[观察]
```

**5 个场景 3 次重复 = 15 次测试**。**30 分钟**。

## 第 5 步：调整 description（15 分钟）

测试发现的问题 90% 都在 description。

**3 个常见调整**：

**问题 1：description 太宽，过度触发**

```yaml
# 错（太宽）
description: 处理代码相关任务

# 对（精准）
description: 按团队规范审查 TypeScript 代码变更。涉及 PR review、code review、git diff 时触发。
```

**问题 2：description 太窄，漏触发**

```yaml
# 错（太窄）
description: 审查 PR diff 里的 TypeScript 函数

# 对（适度宽）
description: 按团队规范审查 TypeScript 代码变更（PR review / code review / diff）。
```

**问题 3：关键词不匹配**

用户说"看看这段代码"但 description 没说"看看"——**模型可能不触发**。

```yaml
# 加上用户可能用的所有同义词
description: 审查代码（review / 看看 / 检查 / 帮我看下这段）
```

## 第 6 步：调整 paths（10 分钟）

paths 太严会漏触发，太宽会过度触发。

**调试步骤**：
1. 临时把 paths 改成 `**/*`（全匹配），看是否触发
2. 如果触发 → paths 写错了
3. 改成更精确的 glob 限制

**常见 paths 模式**：

```yaml
# 代码相关
paths: ["src/**/*.{ts,tsx,js,jsx}"]

# 前端
paths: ["**/*.{tsx,vue,svelte}", "**/*.{css,scss}"]

# 后端 + DB
paths: ["src/api/**", "src/services/**", "migrations/**"]

# 文档
paths: ["**/*.md", "docs/**"]
```

## 第 7 步：发布 + 监控（持续）

发布到 `.claude/skills/` 后，**前 1 周重点观察**：

- **触发频率**：是不是预期内的频率？
- **过度触发**：不该用的时候用了？
- **漏触发**：该用的时候没用？
- **输出质量**：触发了但输出不对？

**3 个改进信号**：

**1 周后**：
- 触发频率符合预期（不过度 + 不漏）→ 继续
- 触发频率过高 → 收紧 description
- 触发频率过低 → 放宽 description

**2 周后**：
- 输出质量稳定 → 考虑分享给团队
- 输出质量不稳 → 修改 body（步骤 / 模板）

**1 个月后**：
- 团队多个项目都用 → 考虑发到 Skills 市场
- 只有 1 个项目用 → 留本地

## 我自己 2026 年的 5 个常错

**错 1：description 写得太"技术"**。

```yaml
# 错
description: AST-based code analysis with security vulnerability detection

# 对
description: 审查代码找安全漏洞（用户说 review / 检查 / 帮我看看）
```

**模型是 LLM 不是工程师**——**description 要"用户能说的话"**，不是"技术规格"。

**错 2：body 写"实现细节"**。

```markdown
# 错
使用 TypeScript 编译器 API 解析 AST，遍历节点，匹配 ESLint 规则
```

**模型不懂"实现细节"，但懂"做什么"**。**写"做什么" + "模板"**，不写"怎么实现"。

**错 3：files 太多导致 paths 冲突**。

5 个 Skills 都配 `paths: ["src/**"]`——**互相冲突，触发混乱**。

**解法**：每个 Skill 的 paths 限定到**不同**文件类型或子目录。

**错 4：测试场景太少**。

只测 1-2 个场景就发布——**实际项目用起来一堆边角问题**。

**我自己的标准**：**5 个场景 + 每个 3 次重复 = 15 次测试**。

**错 5：从不更新**。

项目迭代 6 个月，Skill 还是最初版本——**新加的库、新改的规范都没更新**。

**我自己的规则**：**每 3 个月 review 一次所有 Skills**。

## 第一个 Skill 的推荐路径

如果你第一次写 Skill，**推荐从"code-format"开始**——**最简单、最容易成功**。

```yaml
---
name: code-format
description: 格式化代码（prettier / eslint）。用户说"格式化"、"format"、"修格式"时触发。
paths:
  - "**/*.{ts,tsx,js,jsx,css,json,md}"
---

# Code Format

## 步骤
1. 检测项目用 prettier 还是 eslint
2. 跑对应的格式化命令
3. 输出修改的文件列表

## 命令
```bash
# Prettier 项目
npx prettier --write .

# ESLint 项目  
npx eslint --fix .
```
```

**这个 Skill 触发简单（"format"关键词）、输出简单（跑命令）、错误容易诊断**。**写完能用 → 信心 → 继续写更复杂的**。

## 7 步流程的 1 小时时间表

| 步骤 | 时间 | 产物 |
|---|---|---|
| 0. 判断要不要写 | 5 分钟 | 决策 |
| 1. 定义边界 | 10 分钟 | description 草稿 |
| 2. 列步骤 | 15 分钟 | 5-10 步流程 |
| 3. 写 SKILL.md | 20 分钟 | 完整文件 |
| 4. 测试 5 场景 | 30 分钟 | 测试报告 |
| 5. 调 description | 15 分钟 | 优化版本 |
| 6. 调 paths | 10 分钟 | 精确 paths |
| 7. 发布 + 监控 | 持续 | 迭代 |

**总计 1 小时 45 分钟**。**熟练后可以压到 45 分钟**。

## 我自己用过的"Skill 模板"

我写 Skill 时直接复制这个骨架，填空即可：

```yaml
---
name: <skill-name>
description: <做 X>。当用户说<关键词 1> / <关键词 2> / <关键词 3>时触发。涉及<文件类型>时自动考虑。
paths:
  - "<glob>"
---

# <Skill Title>

## 触发场景
- 用户说"<用户可能说的话 1>"
- 用户说"<用户可能说的话 2>"
- 文件匹配<条件>

## 执行步骤
1. <第一步>
2. <第二步>
3. ...

## 必做
- [ ] <必查项 1>
- [ ] <必查项 2>

## 输出格式
```markdown
<模板>
```

## 不适用
- ❌ <场景 1>
- ❌ <场景 2>
```

**这个骨架 90% 的情况够用**。**剩余 10% 看具体场景加章节**。
