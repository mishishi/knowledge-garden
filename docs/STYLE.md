# 知识花园 — 写作规范 (Series Style Guide)

> 这是**单一 source-of-truth**, 系列文章生成器 (tools/generate_series.py) 和
> `_fix_ai_flavor.py` 都基于这份规范. 修改这里之前请同步两个调用方.

## TL;DR

- 章节 3000-5000 字, 6-12 段正文, 配 2-4 段代码
- 第一人称叙述 ("我"/"我们"), 短长段交替 (3-5 句紧接 1-2 段长的)
- 段落叙述代替表格 (≤ 1 个对比表, 留作"何时用哪个"决策)
- 数字具体: "3.2x 提升" / "省 47 秒" / "失败率从 12% → 1.8%"
- 真实细节: 报错信息 / 具体 API 名称 / 真用过的工具
- 不堆装饰 emoji, 不写教学大纲模板

## 不写 (硬规则)

| 反例 | 原因 |
|---|---|
| ⚠️💡✅❌🎯 装饰 emoji | AI 味, 显得在表演 |
| `## 本章核心` / `## 本章小结` / `## 这一章要回答的问题` | 教学大纲模板 |
| `## 下篇` / `## 下章` (单独成段) | 单行链接就够了: `[下一章](path)` |
| > 3 个 `###` 子标题 | 应该并到段落, 或用 `##` 升级 |
| > 1 个对比表 | 表格堆砌是 AI 标志 |
| 戏剧化 emoji 标签 (🎯 关键 / ⚡ 提速) | 表演性强 |
| 段首总结句 ("核心原则: ..." / "为什么 X 重要: ...") | 老 PPT 教学体 |
| 硬编号 ("误区 1/2/3" / "步骤 1/2/3") | 工具化口吻 |

## 要写 (人味原则)

- 短长段交替: 3-5 句一段 (论述) → 1-2 段长段 (举例 / 推演) → 再 3-5 句收尾
- 口语连接词: "等下" / "其实" / "换句话说" / "我踩过的坑"
- 具体数字: 拒绝"显著提升" / "大幅改善"
- 真实细节: 一个具体的报错信息胜过 10 段抽象总结
- 代码 ≥ 2 段, 每段 ≤ 30 行, 必须可运行 / 接近可运行

## 文件结构 (一本书)

```
books/<slug>/
  _meta.json                 # 书的元信息
  <NN>-<chapter-slug>/       # 章节目录, NN = 2 位数字
    README.md                # 章节内容 (唯一必需)
    code/                    # 代码示例 (可选, 推荐)
      <file>.py
      README.md              # 代码说明
      requirements.txt       # 依赖 (Python 项目)
```

## _meta.json schema

```json
{
  "title": "Multi-Agent in Practice",
  "description": "从零到生产: 手把手教你构建 Multi-Agent 系统",
  "icon": "agents",
  "color": "#5b8c85",
  "priority": 1,
  "level": 3,
  "level_name": "进阶",
  "order": [
    "01-your-first-agent",
    "02-why-multi-agent",
    "..."
  ]
}
```

字段:
- `title` (str): 书的标题
- `description` (str): 1 句话描述, ~30 字内
- `icon` (str): 16 个可选 icon 之一 (见下)
- `color` (str): hex 色值, 选一个跟主题搭
- `priority` (int): 排序, 数字越小越靠前 (1-10)
- `level` (int, optional): 1=入门 / 2=中级 / 3=进阶 / 4=专家
- `level_name` (str, optional): 跟 level 对应的中文标签
- `order` (list[str]): 章节目录的 slug 列表 (必须与目录名完全一致)

## 可用 icon (来自 ICONS dict, build_reader.py L128-183)

agents, sparkles, bot, database, layers, share, brain, stack, terminal,
network, pen-tool, zap, wind, rocket, coin, bookmark, qr, disc, book

## 可用色值 (按书主题倾向)

| 主题 | 候选色 |
|---|---|
| 严肃 / 工程 | `#0d9488` `#0891b2` `#4f46e5` `#059669` |
| 创意 / 探索 | `#ec4899` `#d97706` `#f59e0b` `#dc2626` |
| 入门 / 友好 | `#5b8c85` `#7c3aed` `#0ea5e9` |
| 通用兜底 | `#b08968` |

## 章节 README 结构

```markdown
# 01. 章节标题

> 一句话副标题 / TL;DR (≤ 60 字)

## 第一节 (动机 / 场景 / 痛点)

3-5 段正文, 短长段交替. 第一段必须 hook (具体场景 / 反直觉 /
踩过的坑).

## 第二节 (核心)

2-3 段正文 + 1-2 段代码 (用 ```python 包, ≤ 30 行).
代码必须可运行或接近可运行. 引用本地文件: [`code/foo.py`](./code/foo.py).

## 第三节 (陷阱 / 边界)

列举 1-2 个真实踩过的坑, 每个有具体表现 + 原因 + 解法.
不要写成"误区 1/2/3" 那种硬编号.

[下一章](./02-xxx.md)
```

## 章节字数 / 段数约束 (用于 AI-味 验证)

- 字数: 1500-6000 字 (中文字符)
- `###` 子标题数: ≤ 3
- 表格数: ≤ 1
- 代码块数: ≥ 1
- 装饰 emoji 数: 0 (硬规则)
- 教学模板段: 0 (硬规则)
- 段首总结句: 0 (可选, 自动检测可能误报)

## LLM 生成时的 prompt 模板 (注入到每次生成)

```
你是知识花园的作者, 给技术读者写 3000-5000 字的深度章节. 严格遵守:

[不写]
- 不出现 ⚠️💡✅❌🎯 等装饰 emoji
- 不出现 "## 本章核心" / "## 本章小结" / "## 这一章要回答的问题" / "## 下篇" 单独成段
- ≤ 3 个 ### 子标题, 多用 ## 或直接段落
- ≤ 1 个对比表
- 段首不放"核心原则: ..." / "为什么 X 重要: ..." 这种总结句

[要写]
- 第一人称 ("我"/"我们"), 短长段交替
- 口语连接词 ("等下"/"其实"/"换句话说"/"我踩过的坑")
- 具体数字 ("3.2x 提升" 优于 "显著提升")
- 真实细节 (报错信息 / API 名称 / 用过的工具)
- ≥ 2 段代码, 每段 ≤ 30 行
- 代码用 ```python 包裹, 注释用中文

[输出格式]
- 文件首行: # NN. 标题
- 第二行: > TL;DR
- 不要 _meta.json, 不要 frontmatter, 不要 refs
- 文件末尾一行: [下一章](<相对路径>)
```

## AI-味 验证脚本 (tools/check_ai_flavor.py 用)

自动检查每章是否违反:
- emoji 黑名单 (⚠️💡✅❌🎯) — 任一出现则失败
- 教学模板段 — 段落级匹配 `^##\s+(本章核心|本章小结|下篇|...)$` 则失败
- `###` 子标题数 > 3 则警告
- 表格数 > 1 则警告
- 字数 < 1500 或 > 6000 则警告
- 代码块数 = 0 则警告
